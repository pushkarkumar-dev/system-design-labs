## Why this lab exists

Every distributed storage system you've worked with — Redis Cluster, Cassandra, DynamoDB, Memcached — uses consistent hashing (or a close cousin) to decide which node owns which key. You've explained it in system design interviews: "we hash the key and map it to a ring." You've drawn the ring on a whiteboard. But until you've implemented it, measured the actual distribution variance, and watched the remapping numbers come out exactly at 1/N — you don't really *know* it.

The consistent hashing ring is also the right first distributed-systems lab: small enough to implement completely in a few hours, hard enough to surface the real lessons, and foundational enough that the partitioned KV store and distributed cache labs build directly on it.

## What we're building

A consistent hashing ring library in Go, implemented across four stages. Each stage adds one capability and benchmarks it.

**We build:**
- A sorted-ring data structure mapping keys and nodes to [0, 2^32) via MD5
- Virtual nodes (100 per physical node) to normalize distribution
- Dynamic membership — add/remove nodes with measured remapping stats
- An HTTP server so any language can use the ring for routing decisions

**We explicitly skip (and why):**
- Replication: returning N successors instead of one (that's the distributed KV lab)
- Gossip membership: we use explicit add/remove HTTP calls (that's the SWIM lab)
- Persistence: the ring is rebuilt from membership events on restart

## The problem with modular hashing

Before the ring, there was `key % N`. Simple, fast, deterministic. It works perfectly — until you change N.

```
3 nodes, key "user:42" hashes to 17
  17 % 3 = 2  → node 2

Add a 4th node
  17 % 4 = 1  → node 1
```

Every key remaps. Not some keys — **every key**. For a cache this means a complete cold-start. For a database it means a massive migration. In practice, `key % N` is unusable for any system where node count ever changes.

The ring fixes this by making the mapping independent of N.

## Mental model

Both keys and nodes are mapped to the same continuous space: [0, 2^32). Imagine this as a circle with 4 billion positions. When you want to find which node owns a key:

```
Ring positions (sorted):

    0                    2^32
    |────●──────●────────●──────●────|
         ^      ^        ^      ^
       node-2  node-4  node-1  node-3
                  ↑
             key "user:42" → hash position 1,847,392,847
             clockwise successor = node-1
```

1. Hash the key to a position on the ring
2. Walk clockwise until you hit a node position
3. That node owns the key

**Adding a node** only steals keys from the new node's predecessor range. Everything else is undisturbed. **Removing a node** migrates its keys to the clockwise successor — only those keys move.

The ring is a sorted `[]uint32` of node positions, with a `map[uint32]*Node` for the reverse lookup. MD5 is fine here — we're not doing cryptography, we just need uniform distribution across [0, 2^32).

```go
// hash32 maps any string to [0, 2^32) using the first 4 bytes of its MD5 digest.
// MD5 is fast and distributes well enough for ring positions.
func hash32(key string) uint32 {
    sum := md5.Sum([]byte(key))
    return binary.BigEndian.Uint32(sum[:4])
}

// GetNode finds the clockwise successor of the key's hash position.
func (r *HashRing) GetNode(key string) *Node {
    if len(r.positions) == 0 {
        return nil
    }
    h := hash32(key)
    // Binary search: first position >= h
    idx := sort.Search(len(r.positions), func(i int) bool {
        return r.positions[i] >= h
    })
    // Wrap: if h is past the last position, use position 0 (the ring wraps)
    if idx == len(r.positions) {
        idx = 0
    }
    return r.nodeMap[r.positions[idx]]
}
```

**The core lesson**: `sort.Search` is the entire routing algorithm. Everything else — virtual nodes, remapping stats, the HTTP server — is scaffolding on top of this binary search.

With 5 nodes and 1 position each, one node got 31% of keys and another got 9% — pure accident of where MD5 placed them on the ring. The fix: each physical node occupies many positions. More positions = more evenly distributed load.

```go
// Each physical node gets vnodesN virtual positions, labelled name#0..name#(N-1)
func (r *VNodeRing) AddNode(n Node) {
    node := &Node{Name: n.Name, Addr: n.Addr}
    r.nodes[n.Name] = node

    for i := 0; i < r.vnodesN; i++ {
        pos := hash32(fmt.Sprintf("%s#%d", n.Name, i))
        r.positions = append(r.positions, pos)
        r.nodeMap[pos] = node
    }

    sort.Slice(r.positions, func(i, j int) bool {
        return r.positions[i] < r.positions[j]
    })
}
```

The math behind why this works: with V virtual nodes per physical node, the sizes of adjacent segments on the ring are approximately exponentially distributed. The coefficient of variation (std dev / mean) of segment sizes is ~1/sqrt(V * N) where N is the number of physical nodes. With V=100 and N=5:

```
CV ≈ 1/sqrt(100 * 5) ≈ 1/sqrt(500) ≈ 4.5%
```

That matches our measured 4.2%. With V=1: `CV ≈ 1/sqrt(5) ≈ 45%`, which matches our observed 28–35% range (the exponential distribution has higher variance at low counts).

</LabStage>

The remapping guarantee: adding 1 node to an N-node ring should remap exactly 1/(N+1) of all keys on average. Not approximately. Exactly. Here's why:

Before adding the new node, each key lives in some arc of the ring. The new node's V virtual positions each carve out a slice of their predecessor's arc. On average, the new node claims exactly 1/(N+1) of the total arc length, which is 1/(N+1) of all keys.

```
5 nodes → add 6th → theoretical remap = 1/6 = 16.7%

Worked example:
  Node-6 gets virtual position at ring position 1,200,000,000
  The predecessor arc (node-3's range) ran from 900,000,000 to 1,500,000,000
  Node-6 steals: keys hashing between 900,000,000 and 1,200,000,000
  That's 300M out of 4B positions = 7% of keys from this one vNode

  Sum across all 100 vNodes of node-6: ≈ 16.7% total
```

Measuring it:

```go
func (m *ManagedRing) AddNodeTracked(n Node, sampleKeys []string) RemapStats {
    // 1. Snapshot: who owns each key now?
    oldOwners := m.snapshotOwners(sampleKeys)

    // 2. Add the node (steals predecessor arcs)
    m.AddNode(n)

    // 3. Re-check: how many keys changed hands?
    newOwners := m.snapshotOwners(sampleKeys)
    remapped := countChanged(oldOwners, newOwners)

    return RemapStats{
        RemappedKeys: remapped,
        RemapRate:    float64(remapped) / float64(len(sampleKeys)),
        Theoretical:  1.0 / float64(m.NodeCount()),
    }
}
```

Our test with 10,000 sample keys consistently lands within 2pp of the theoretical 16.7%. With 100,000 keys it's within 0.5pp.

</LabStage>

**Weighted virtual nodes** are the practical answer to heterogeneous hardware. A node with a 32-core machine and 256GB RAM should handle twice the load of an 8-core node. Give it `2 * baseVNodes` positions on the ring.

```go
func (r *WeightedRing) AddNode(n Node, weightFactor float64) {
    vnodesForNode := int(float64(r.baseVNodes) * weightFactor)
    // ... place vnodesForNode positions on the ring
}
```

**Jump consistent hash** (Lamping & Veach, 2014) is a completely different algorithm that achieves the same property with O(log n) time and O(1) space:

```go
// JumpHash maps a key to one of n buckets (nodes) with minimal remapping.
// 6 lines. No data structure. O(log n).
func JumpHash(key uint64, n int) int {
    var b, j int64 = -1, 0
    for j < int64(n) {
        b = j
        key = key*2862933555777941757 + 1
        j = int64(float64(b+1) * (float64(int64(1)<<31) / float64((key>>33)+1)))
    }
    return int(b)
}
```

The catch: jump hash assigns keys to buckets 0..n-1, so it can only remove the *last* bucket. You can't remove an arbitrary node. This makes it ideal for stateless systems (sharding batch jobs across workers) but not for dynamic storage clusters where any node might fail.

The lesson isn't that one is better — it's that "consistent hashing" is a *property* (minimal remapping on resize), not a specific algorithm. The ring and jump hash are both consistent hashes with different trade-offs.

### Part A — HashRingClient: the HTTP bridge

The Go ring server exposes four routes. `HashRingClient.java` wraps them using Spring Framework 6.1's `RestClient`:

```java
public class HashRingClient {
    private final RestClient http;

    public HashRingClient(String baseUrl) {
        this.http = RestClient.builder()
                .baseUrl(baseUrl)
                .defaultHeader("Accept", MediaType.APPLICATION_JSON_VALUE)
                .build();
    }

    // POST /nodes — register a node on the ring
    public void addNode(String name, String addr) {
        http.post()
                .uri("/nodes")
                .contentType(MediaType.APPLICATION_JSON)
                .body(Map.of("name", name, "addr", addr))
                .retrieve()
                .toBodilessEntity();
    }

    // GET /route?key=... — resolve a key to its responsible node
    public NodeInfo route(String key) {
        return http.get()
                .uri("/route?key={key}", key)
                .retrieve()
                .body(NodeInfo.class);
    }

    // Java 16 records — immutable DTOs, no boilerplate
    public record NodeInfo(String node, String addr) {}
    public record RingStats(int nodes, Map<String, Integer> keysPerNode,
                            double stdDevCV, int min, int max) {}
}
```

### Part B — HashRingRouter: routing middleware with cached decisions

The key abstraction: `HashRingRouter` sits between your business logic and your storage tier. Callers ask "where should I send this key?" and get back a host:port. They never need to know the ring topology.

```java
@Service
public class HashRingRouter implements HealthIndicator {
    private final HashRingClient client;

    // Routing decisions are stable until membership changes.
    // Cache them to avoid an HTTP call on every read.
    private final Cache<String, String> routeCache;

    public String routeRequest(String key) {
        // Cache-first: on hit, zero network calls.
        // On miss: query Go ring server, cache the result.
        return routeCache.get(key, k -> client.route(k).addr());
    }

    public void addNode(String name, String addr) {
        client.addNode(name, addr);
        routeCache.invalidateAll(); // ~1/N of routes changed; invalidate all
        log.info("node added: {} — route cache invalidated", name);
    }

    public void removeNode(String name) {
        client.removeNode(name);
        routeCache.invalidateAll(); // successor range changed
        log.info("node removed: {} — route cache invalidated", name);
    }
}
```

**Why `invalidateAll()` instead of selective invalidation?** After adding a node, you'd need to re-query every cached key to know if it moved. With 10,000 cached routes that's 10,000 HTTP calls. `invalidateAll()` is O(1) and forces lazy re-population on the next access. The cold-start cost is amortized across subsequent requests.

**Why Caffeine instead of `@Cacheable`?** `@Cacheable` makes manual invalidation awkward — you need to know the exact key used in the cache annotation. Direct Caffeine gives us `cache.invalidateAll()` with one line. The Caffeine stats also surface as metrics via `/actuator/metrics/cache.gets`.
