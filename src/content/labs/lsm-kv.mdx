## Why this lab exists

Every storage engine you've ever used — RocksDB, LevelDB, Cassandra, TiKV, ScyllaDB — is built on the Log-Structured Merge-Tree. You've read about it in papers and design docs. You've answered "how does a KV store work?" in system design interviews. But until you've implemented the tombstone problem yourself, until you've watched compaction rewrite data you thought you already paid for, until you've measured the false positive rate of your own bloom filter — you don't *know* it.

The LSM-tree is also the natural follow-up to the WAL lab. Our WAL lab builds the append-only log that sits in front of the memtable. This lab builds the memtable and everything above it. Together, they are the complete write path of RocksDB.

## The core read/write asymmetry

This is the central insight of the LSM-tree: **writes are cheap, reads are expensive**.

**Write path**: every write goes to the in-memory memtable. A BTreeMap insert. No disk I/O. Throughput is limited by memory bandwidth, not disk bandwidth.

**Read path**: a key might be in the memtable, or in any of N on-disk SSTables. Without bloom filters, every read must probe all N SSTables. If you have 10 SSTables and the key is in none of them (a common case in practice — "does this user exist?"), you've done 10 disk reads for nothing. This is *read amplification*.

```
Write (cheap):           Read (expensive):
  put(k, v)               get(k)
     │                       │
     ▼                       ▼
  memtable               memtable?
  (BTreeMap)              │  miss
  no disk I/O             ▼
                       SSTable 1 (newest)
                          │  miss
                          ▼
                       SSTable 2
                          │  miss
                          ▼
                       SSTable N (oldest)
                          │  miss = None
                          ▼
                       return None
                       (N disk reads for nothing)
```

Bloom filters (v2) fix the miss case. A negative bloom result means "this key is *definitely not* in this SSTable" — no disk read needed. At 1% false positive rate, 99% of absent-key lookups skip the disk I/O.

## The write amplification problem

Writes are cheap on the hot path. But that cost is deferred, not avoided. Compaction pays it.

When the memtable fills up (4 MB in our implementation), it's flushed to a new SSTable. Over time, many SSTables accumulate. Compaction merges them: it reads all the input SSTables, produces a single sorted output SSTable, and deletes the inputs.

The problem: the *same data gets rewritten many times*. A key written once at the memtable level might be rewritten during L0→L1 compaction, then L1→L2, then L2→L3... At a 10:1 compaction ratio with 5 levels, a single 1 KB write might trigger 50 KB of total disk writes before reaching L5. This is a **write amplification factor (WAF) of 50**.

RocksDB's default WAF in production is typically 10–30×. Cassandra's size-tiered compaction has lower WAF but higher read amplification. This is the fundamental tradeoff — there is no free lunch.

The memtable is a sorted map of `key → Value`, where `Value` is either `Live(bytes)` or `Tombstone`. Using Rust's `BTreeMap` gives sorted order for free.

```rust
pub enum Value {
    Live(Vec<u8>),
    Tombstone,
}

pub struct Lsm {
    memtable: BTreeMap<Vec<u8>, Value>,
}

impl Lsm {
    pub fn put(&mut self, key: impl Into<Vec<u8>>, value: impl Into<Vec<u8>>) {
        self.memtable.insert(key.into(), Value::Live(value.into()));
    }

    pub fn get(&self, key: &[u8]) -> Option<&[u8]> {
        match self.memtable.get(key)? {
            Value::Live(v) => Some(v),
            Value::Tombstone => None,  // tombstone = key is deleted
        }
    }

    pub fn delete(&mut self, key: &[u8]) {
        // NOT: self.memtable.remove(key)
        // Tombstone must survive flush so it can shadow older SSTable entries
        self.memtable.insert(key.to_vec(), Value::Tombstone);
    }
}
```

**Why `delete` inserts instead of removes**: imagine you've written `user:42=alice` to an SSTable last week, then called `delete("user:42")` today. If we just `remove()` from the memtable, the SSTable still has the old entry. After a restart, `user:42` comes back from the dead. The tombstone travels into the SSTable on the next flush and tells compaction "kill everything older than me for this key."

**The sort order invariant**: `BTreeMap` uses lexicographic byte order. This is the same order we'll use on disk in SSTables. When we flush the memtable, we do a sequential scan of `BTreeMap::iter()` — sorted output, zero extra work.

</LabStage>

The SSTable is a sorted, immutable file. Once written, it is never modified — compaction produces new SSTables and deletes old ones. This immutability is what makes SSTables safe to read without locking.

**On-disk format:**

```
┌────────────────────────────────────────────────────┐
│  DATA BLOCK                                        │
│  [key_len(4)][key_bytes][val_tag(1)][val_len+val]  │
│  ... repeated for each entry, sorted by key ...    │
├────────────────────────────────────────────────────┤
│  SPARSE INDEX                                      │
│  [index_count(4)]                                  │
│  [key_len(4)][key_bytes][file_offset(8)]           │
│  ... one entry every INDEX_STRIDE records ...      │
├────────────────────────────────────────────────────┤
│  FOOTER                                            │
│  [index_offset(8)]  ← last 8 bytes in file        │
└────────────────────────────────────────────────────┘
```

- `val_tag = 0x01` → live value, followed by `[val_len(4)][val_bytes]`
- `val_tag = 0xFF` → tombstone, no further bytes

**Sparse index**: one index entry every 16 records. The index maps `key → file_offset` of the data record at that position. To find a key, binary-search the index for the largest key ≤ target, then scan forward from that file offset. Worst case: scan 16 entries. Memory cost: index_entries × avg_key_size (kilobytes for typical key sizes).

```rust
// The read path for a single SSTable
fn sstable_get(path: &Path, index: &[(Vec<u8>, u64)], key: &[u8]) -> io::Result<Option<Value>> {
    // Binary search for largest index key ≤ target
    let pos = index.partition_point(|(k, _)| k.as_slice() <= key);
    let start_offset = if pos == 0 { 0 } else { index[pos - 1].1 };

    // Scan forward from start_offset until key found or passed
    let mut reader = BufReader::new(File::open(path)?);
    reader.seek(SeekFrom::Start(start_offset))?;
    // ... read key_len, key, val_tag, val ...
}
```

**Recovery on open**: scan the directory for `*.sst` files, sort by filename (sequence number = creation order), load each file's sparse index. Newest first in memory so the read path finds recent writes first.

**Bloom filter (no external crates):**

A bloom filter is a bit array of `m` bits. Insert: hash the key with `k` different seeds, set those `k` bits. Query: check all `k` bits — if any is 0, the key is *definitely absent*. If all are 1, the key *might* be present (check the SSTable).

At 10 bits/key and k=7, false positive rate ≈ 1%. This means 99% of "key not found" lookups skip disk I/O entirely.

```rust
pub struct BloomFilter {
    bits: Vec<u64>,
    m: usize,   // total bits
}

impl BloomFilter {
    pub fn insert(&mut self, key: &[u8]) {
        for seed in 0..BLOOM_HASH_COUNT {
            let bit = self.bit_index(key, seed);
            self.bits[bit / 64] |= 1 << (bit % 64);
        }
    }

    pub fn may_contain(&self, key: &[u8]) -> bool {
        (0..BLOOM_HASH_COUNT).all(|seed| {
            let bit = self.bit_index(key, seed);
            (self.bits[bit / 64] >> (bit % 64)) & 1 == 1
        })
    }
}
```

**Size-tiered compaction** merges all L0 SSTables into a single L1 SSTable when L0 count exceeds the threshold. During merge:
1. Read all L0 entries (oldest first, so newer values overwrite older ones in the map)
2. Merge with existing L1 entries (L0 wins on conflict — newer)
3. **Drop tombstones** — at the bottom level, a tombstone has already shadowed all older versions. There are no older versions below L1, so the tombstone can be deleted entirely. This is the only point in the lifetime of a tombstone where it disappears.

**The compaction write amplification**:

```
Initial write: 1 KB → memtable (free)

Flush:         1 KB → L0 SSTable
               (1 KB written)

Compaction (4 L0 files, each 4 MB):
               4 × 4 MB = 16 MB read
               16 MB → 1 L1 SSTable
               (16 MB + 16 MB written = 32 MB I/O for 4 MB of data)

If L1 overflows into L2 (10:1 ratio):
               10 × 16 MB = 160 MB read
               160 MB written
               (total I/O now: 320 MB for the original 4 MB)
```

This is why WAF is the tuning target, not raw write throughput.

</LabStage>

**Level compaction** is RocksDB's default. Each level has a size limit (L1=256MB, L2=2.5GB, ..., each 10× the previous). When a level overflows, pick one file and merge it with the overlapping range in the next level.

Unlike size-tiered (which merges all L0 files into one big file), level compaction selects *partial* files for each compaction. This keeps the number of files per level bounded and reduces read amplification (fewer files to check per read).

**Merge iterator** (the k-way merge heap):

```rust
// Instead of scanning SSTables one by one, we want globally sorted output
// from k SSTables simultaneously.
use std::collections::BinaryHeap;
use std::cmp::Reverse;

struct MergeIterator {
    heap: BinaryHeap<Reverse<(Vec<u8>, Value, usize)>>, // (key, value, sst_id)
    iterators: Vec<SsTableIterator>,
}

// O(k log k) per key — much better than O(k) probes per SSTable
```

The merge iterator is used both during compaction (to produce sorted output) and during range scans (to return globally sorted results from multiple SSTables).

**Manifest file**: tracks which files belong to which level. Without it, recovery doesn't know which files are active vs being compacted. RocksDB's MANIFEST is a WAL of file-level operations — another WAL, sitting above the data WAL.

</LabStage>

## The WAL relationship

This lab and the WAL lab are two halves of the same write path.

In a real LSM store (RocksDB, LevelDB, Cassandra):
1. `put(k, v)` appends to the WAL first (durable)
2. Then inserts into the memtable (fast)
3. When the memtable flushes, the WAL entries it covered are garbage-collected

Our memtable (this lab) is not crash-safe without a WAL. If the process dies between puts, the in-memory state is lost. The previous lab — [Write-Ahead Log](/system-design-labs/wal) — is exactly the WAL you'd pair with this memtable.

```
User write:           Recovery:
  put(k, v)             open WAL
     │                    │
     ├─ WAL append        ▼
     │  (fsync)        replay WAL
     │                    │
     └─ memtable insert   ▼
        (in-memory)    rebuild memtable
                          │
                          ▼
                       open SSTables
                       (already flushed = durable)
```

The WAL covers the memtable. The SSTables cover everything that's been flushed. Together, no data is lost.

### Part A — LsmClient: the HTTP bridge

The Rust LSM server exposes four routes. `LsmClient.java` wraps them using Spring Framework 6.1's `RestClient`.

```java
public class LsmClient {
    private final RestClient http;

    public LsmClient(String baseUrl) {
        this.http = RestClient.builder()
                .baseUrl(baseUrl)
                .defaultHeader("Accept", MediaType.APPLICATION_JSON_VALUE)
                .build();
    }

    // POST /put — body: {"key":"...", "value":"..."}
    public void put(String key, String value) {
        http.post()
                .uri("/put")
                .contentType(MediaType.APPLICATION_JSON)
                .body(new PutRequest(key, value))
                .retrieve()
                .toBodilessEntity();
    }

    // GET /get?key=... — returns Optional.empty() on 404
    public Optional<String> get(String key) {
        try {
            var resp = http.get()
                    .uri("/get?key={key}", key)
                    .retrieve()
                    .body(GetResponse.class);
            return resp != null ? Optional.of(resp.value()) : Optional.empty();
        } catch (RestClientResponseException e) {
            if (e.getStatusCode().value() == 404) return Optional.empty();
            throw e;
        }
    }

    // DELETE /delete?key=... — writes tombstone to LSM
    public void delete(String key) {
        http.delete()
                .uri("/delete?key={key}", key)
                .retrieve()
                .toBodilessEntity();
    }

    // Java 16 records — immutable DTOs, zero boilerplate
    record PutRequest(String key, String value) {}
    public record GetResponse(String key, String value) {}
    public record HealthStatus(String status, int l0, int l1) {}
}
```

**The `Optional<String>` return from `get()`** is the right Java idiom for "key may not exist." It forces callers to handle the absent case at compile time — no NPE risk. This mirrors the Rust `Option<Vec<u8>>` on the server side.

### Part B — LsmService: write-through cache (the block cache analogy)

The key lesson: the Spring `@Service` is a **block cache** equivalent. RocksDB has an in-process block cache (LRU or CLOCK) that holds decompressed SSTable blocks. We implement the same pattern at the service layer using Caffeine.

```java
@Service
public class LsmService implements HealthIndicator {
    private final LsmClient client;
    private final Cache<String, String> kvCache;  // Caffeine = our block cache

    // put() = write-through: cache AND forward to LSM
    public void put(String key, String value) {
        client.put(key, value);
        kvCache.put(key, value);   // cache the write immediately
    }

    // get() = cache-first: no LSM I/O for hot keys
    public Optional<String> get(String key) {
        var cached = kvCache.getIfPresent(key);
        if (cached != null) return Optional.of(cached);

        // Cache miss: full LSM read path (memtable + SSTable scan)
        var result = client.get(key);
        result.ifPresent(v -> kvCache.put(key, v)); // read-through populate
        return result;
    }

    // delete() = invalidate + tombstone
    public void delete(String key) {
        kvCache.invalidate(key);    // remove from cache
        client.delete(key);         // write tombstone to LSM
    }
}
```

**Why this pattern matters for LSM specifically**: LSM read amplification grows with levels. A 5-level LSM may check 5 SSTables for a missing key. A 1% bloom filter false positive rate means 1 in 100 absent-key reads still hits disk. At 100k reads/sec with a hot miss pattern, that's 1,000 unnecessary disk reads per second. The service-layer cache absorbs this for recently-accessed keys entirely.

## RocksDB comparison

RocksDB is this lab, but:

- **Level compaction by default** (7 levels, 10:1 ratio) instead of size-tiered
- **Column families**: separate LSM trees with independent settings
- **Block cache**: shared LRU cache for decompressed SSTable blocks (our Caffeine cache is a service-level analog)
- **Compression**: Snappy/Zstd per block
- **Merge iterator**: k-way heap for range scans
- **WAL**: the same WAL pattern from the previous lab, but with group commit and segment rotation
- **Java JNI wrapper**: exactly the pattern in `LsmService.java` — in-process cache + native calls

The gap between our toy and RocksDB is about 200,000 lines of C++ and 15 years of production tuning. But the *architecture* is the same. Every concept in this lab has a direct counterpart in `db/db_impl.cc`.
