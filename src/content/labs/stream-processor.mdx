## Why this lab exists

Every analytics pipeline, fraud detection system, and IoT platform runs on stream processing. Apache Flink, Kafka Streams, Apache Beam, Google Dataflow — they all solve the same core problem: aggregate a continuous stream of events into time-windowed summaries, handle events that arrive out of order, and guarantee that each event contributes to the output exactly once, even across crashes.

But stream processors are famously difficult to understand. Developers who use Kafka Streams as a black box tend to hit invisible walls: watermarks that cause windows to fire 65 seconds after you expect, out-of-order events silently dropped, at-least-once re-processing that duplicates downstream database writes. The fix is to build the core abstractions yourself.

This lab builds:
- A tumbling window aggregator that groups events into fixed non-overlapping intervals per key
- A sliding window processor with watermarks and out-of-order event handling via an event-time buffer
- An exactly-once processor using two-phase commit with a crash-safe atomic checkpoint store

After this lab, you will understand why Flink's watermarks exist, why "exactly-once" requires coordination between the source and the sink, and what Kafka Streams does internally when you call `windowedBy(TimeWindows.ofSizeWithNoGrace(...))`.

## What we're building

A stream processor in Go, implemented across three stages. Each stage adds one core abstraction.

**We build:**
- `TumblingWindow`: groups events by key into fixed non-overlapping time windows; flushes min/max/sum/avg/count per key when the window boundary is crossed
- `Aggregator`: runs a tumbling window in a goroutine, receiving events from a channel and flushing every `windowSize/10`
- `SlidingWindow`: overlapping windows that fire every `step` duration, each covering the last `size` duration of event time
- `Watermark`: tracks `maxEventTimeSeen - allowedLateness`; events before the watermark are "too late"
- `StreamProcessor`: wires source channel, `SlidingWindow`, watermark, result sink, and late-event side output
- `TxnCoordinator`: two-phase commit between event source (offset tracking) and output sink (result writing)
- `CheckpointStore`: atomic checkpoint via write-to-temp-file then rename — crash-safe by construction
- `AtLeastOnceProcessor`: simpler baseline that shows why exactly-once needs 2PC

**We explicitly skip (and why):**
- Distributed state partitioning — that's the Kafka Streams / Flink worker sharding problem
- Session windows — merging disconnected sessions when new events bridge a gap requires per-key boundary tracking
- RocksDB or any external state backend — our state is in-memory maps
- Full Flink/Kafka Streams API compatibility — we capture the semantics, not the wire protocol

## Mental model

### Event time vs. processing time

This is the single most important distinction in stream processing. It causes more subtle bugs than any other concept.

**Processing time** is when the event arrives at your processor. It's wall-clock time on the machine running your code. It's easy to work with: just call `time.Now()`.

**Event time** is when the event actually happened — the timestamp embedded in the event itself. A sensor reading from 3 seconds ago has an event time 3 seconds in the past, even if it arrives at your processor right now.

Why does the difference matter?

```
Processing time model (wrong for most analytics):
  Event A (happened at T=10s) arrives at T=12s -> assigned to window [10s, 20s) by wall clock
  Event B (happened at T=19s) arrives at T=21s -> assigned to window [20s, 30s) by wall clock
  Result: A and B are in DIFFERENT windows even though they belong together

Event time model (correct):
  Event A (happened at T=10s) arrives at T=12s -> assigned to window [10s, 20s) by event time
  Event B (happened at T=19s) arrives at T=21s -> assigned to window [10s, 20s) by event time
  Result: A and B are in the SAME window
```

The catch: to use event time, you must handle out-of-order events. Event B arrived after window [10s, 20s) nominally ended. You need to know when it is safe to close and emit a window — that's what watermarks solve.

### Tumbling vs. sliding windows

A **tumbling window** is a non-overlapping partition of time. Events at T=0s to T=59s go in window 1. Events at T=60s to T=119s go in window 2. No event belongs to more than one window.

A **sliding window** produces overlapping windows. With size=5m and step=1m, at any moment there are 5 active windows. An event at T=4m belongs to windows [0m,5m), [1m,6m), [2m,7m), [3m,8m), and [4m,9m) — all five. This gives a smoother view of the data at the cost of 5x more computation.

```
Tumbling (size=1m):
  [0:00 --- 1:00)  [1:00 --- 2:00)  [2:00 --- 3:00)
  no overlap       no overlap

Sliding (size=2m, step=1m):
  [0:00 ------- 2:00)
        [1:00 ------- 3:00)
              [2:00 ------- 4:00)
  each window overlaps its neighbors by size-step = 1 minute
```

### Watermarks: when is it safe to close a window?

Without watermarks, you can never close a window — there might always be a late event that belongs to it. With watermarks, you make a bounded lateness assumption: "I will never see an event more than `allowedLateness` behind the latest event time I've observed."

`watermark = maxEventTimeSeen - allowedLateness`

When the watermark advances past a window's end time, that window is safe to emit. Events with `eventTime < watermark` are "too late" — they missed the window they belonged to and are routed to a side output.

The tradeoff: a larger `allowedLateness` catches more late events but delays window output. A smaller value fires windows sooner but drops more events. Flink defaults to 0 allowed lateness (no out-of-order tolerance) for maximum throughput, expecting producers to be nearly in-order.

### Two-phase commit: the source-sink atomicity problem

Imagine your processor reads from Kafka (source) and writes aggregated results to a database (sink). A crash can happen at three points:

1. **Before processing**: safe — restart from committed Kafka offset, nothing was written
2. **After writing to DB, before committing Kafka offset**: dangerous — next restart re-reads the same events and writes to DB again. **Duplicate output.**
3. **After committing Kafka offset, before writing to DB**: dangerous — events are "consumed" but output was never written. **Lost output.**

Two-phase commit makes source and sink atomic:

- **Phase 1 (prepare)**: process the batch, write output to a `PreparedOutput` buffer, record the new source offset. Checkpoint phase 1 (`OutputCommitted=false`). **Do not yet advance the source or write to the real sink.**
- **Phase 2 (commit)**: write `PreparedOutput` to the real sink. Checkpoint phase 2 (`OutputCommitted=true`).

On recovery: if `OutputCommitted=false`, we crashed between phase 1 and phase 2. Re-execute phase 2. If `OutputCommitted=true`, we completed cleanly.

The atomic rename makes the checkpoint itself crash-safe: write to `.tmp`, then rename over the checkpoint file. The OS rename is atomic — you always see either the old checkpoint or the new one, never a partial write.

The simplest useful stream processor: a fixed-size time window per key, emitting statistics on flush.

### The Event and WindowResult types

```go
type Event struct {
    Key       string
    Value     float64
    Timestamp time.Time
}

type WindowResult struct {
    WindowStart time.Time
    WindowEnd   time.Time
    Key         string
    Count       int
    Sum, Min, Max, Avg float64
}
```

`Key` partitions the stream — all events with the same key go into the same bucket within a window. This is equivalent to Kafka Streams' `groupByKey()` — grouping before aggregation.

### The tumbling window state machine

```go
type TumblingWindow struct {
    mu      sync.Mutex
    size    time.Duration
    startAt time.Time
    buckets map[string][]float64
}
```

The state machine has two transitions:

1. **Event in current window** (`event.Timestamp < startAt + size`): append `event.Value` to `buckets[event.Key]`
2. **Event in future window** (`event.Timestamp >= startAt + size`): flush current window, advance `startAt`, reset `buckets`, then process the event in the new window

```go
func (tw *TumblingWindow) Process(e Event) []WindowResult {
    // Initialise on first event — align to window boundary.
    if tw.startAt.IsZero() {
        tw.startAt = e.Timestamp.Truncate(tw.size)
    }
    // Advance window until e fits.
    var flushed []WindowResult
    for e.Timestamp.After(tw.startAt.Add(tw.size).Add(-1)) {
        flushed = append(flushed, tw.flushLocked()...)
        tw.startAt = tw.startAt.Add(tw.size)
        tw.buckets = make(map[string][]float64)
    }
    tw.buckets[e.Key] = append(tw.buckets[e.Key], e.Value)
    return flushed
}
```

**The critical test to write**: call `Process` with events at T=10s and T=70s (crossing a 1-minute boundary). Assert that `Process(T=70s)` returns the flush of the first window before adding T=70s to the second window.

**Why `Truncate` on the first event**: aligning the window to the event's timestamp truncated to `size` means all processors with the same `size` produce windows at the same boundaries — `[0:00, 1:00)`, `[1:00, 2:00)`, etc. — regardless of when they started. This is essential for joining streams in a distributed system.

### The Aggregator goroutine

The `Aggregator` wires a `TumblingWindow` to a `chan Event` input and a `chan []WindowResult` output. A ticker fires every `windowSize/10` to flush the current window even if no boundary-crossing event arrives.

```go
func (a *Aggregator) run() {
    ticker := time.NewTicker(a.windowSize / 10)
    defer ticker.Stop()
    for {
        select {
        case <-a.stopCh:
            // Final flush before shutdown.
            a.Sink <- a.window.Flush()
            close(a.Sink)
            return
        case e := <-a.Source:
            a.window.Process(e)
        case <-ticker.C:
            if results := a.window.Flush(); len(results) > 0 {
                a.Sink <- results
            }
        }
    }
}
```

The ticker is `windowSize/10` not `windowSize` because a flush triggered by Process (on boundary crossing) is more precise — the ticker is only a safety net for slow streams where no events cross the boundary naturally.

### The watermark equation

The watermark is a lower bound on event time. It says: "I will not see any more events with a timestamp earlier than this value."

```
watermark = max(eventTimestamps seen so far) - allowedLateness
```

When the watermark advances past a window's end time, that window is safe to emit. This is the moment Flink calls "triggering" a window.

```go
type Watermark struct {
    mu              sync.Mutex
    maxEventTime    time.Time
    allowedLateness time.Duration
}

func (w *Watermark) Advance(eventTime time.Time) (time.Time, bool) {
    w.mu.Lock()
    defer w.mu.Unlock()
    if eventTime.After(w.maxEventTime) {
        w.maxEventTime = eventTime
        return w.maxEventTime.Add(-w.allowedLateness), true
    }
    return w.current(), false
}
```

**The arithmetic that trips people up**: with `allowedLateness=5s`, the watermark lags reality by exactly 5 seconds. For a 1-minute window `[0s, 60s)`, the window fires when `watermark >= 60s`, i.e., when `maxEventTime >= 65s`. The window fires at event time T=65s — not T=60s. This is the "65-second surprise" mentioned in the WhatSurprisedMe section.

### The sorted event buffer

Unlike tumbling windows (where each event belongs to exactly one window), sliding window events can belong to multiple overlapping windows. The simplest correct approach: buffer all recent events in a sorted slice ordered by timestamp. When the watermark advances, scan the buffer to compute each mature window.

```go
func (sw *SlidingWindow) insertSorted(e Event) {
    idx := sort.Search(len(sw.sortedBuf), func(i int) bool {
        return !sw.sortedBuf[i].Timestamp.Before(e.Timestamp)
    })
    // Insert at idx, shifting right.
    sw.sortedBuf = append(sw.sortedBuf, Event{})
    copy(sw.sortedBuf[idx+1:], sw.sortedBuf[idx:])
    sw.sortedBuf[idx] = e
}
```

`sort.Search` is a binary search that finds the first index where the condition is true — O(log n). Insertion then shifts elements right — O(n). For typical stream processors with short windows and limited buffer sizes (allowedLateness * eventsPerSecond), this is acceptable. Flink uses a more sophisticated skip-list structure.

**When to evict from the buffer**: once the watermark has advanced past `bufferEvent.Timestamp + size`, that event can never contribute to any future window (all windows covering it have already been flushed). The cutoff is: `evict events with Timestamp < watermark - size`.

### The StreamProcessor wiring

`StreamProcessor` ties everything together:

```go
type StreamProcessor struct {
    window   *SlidingWindow
    Source   chan Event
    Sink     chan WindowResult
    LateSink chan Event
    stopCh   chan struct{}
}

func (sp *StreamProcessor) processEvent(e Event) {
    wm, _ := sp.window.watermark.Advance(e.Timestamp)
    if !wm.IsZero() && e.Timestamp.Before(wm) {
        sp.LateSink <- e  // too late — route to side output
        return
    }
    sp.window.insertSorted(e)
}
```

The side output `LateSink` channel lets the caller decide what to do with late events: log them, count them for monitoring, or discard. This mirrors Flink's `OutputTag` side output and Kafka Streams' `allowedLateness` drop-on-late behavior.

### The 3-second lateness window test

The most important test to write for v1: configure `allowedLateness=3s`, send a normal event at T=10s to advance the watermark, then send an event at T=5s (within 3s lateness) and one at T=0s (beyond 3s lateness).

- T=5s event: `watermark = 10s - 3s = 7s`. `5s < 7s` is true... wait. `IsLate(5s)` checks `5s < watermark`. `watermark = 7s`. `5s < 7s = true` — so the 5s event IS late.
- Correct: the 3-second window means events up to 3 seconds behind `maxEventTime` are accepted, not 3 seconds behind the current watermark. An event at T=8s with `maxEventTime=10s` gives `watermark=7s`, and `8s >= 7s` — not late.

This test forces you to get the inequality direction right.

### The atomic rename trick

The checkpoint file is written atomically — you never see a partially-written checkpoint:

```go
func (cs *CheckpointStore) Save(cp Checkpoint) error {
    data, _ := json.Marshal(cp)
    tmp := cs.path + ".tmp"
    os.WriteFile(tmp, data, 0o644) // write to a temp sibling
    return os.Rename(tmp, cs.path) // atomic rename: replaces the old file
}
```

`os.Rename` on POSIX filesystems is atomic at the filesystem level: the directory entry for `cs.path` is updated in a single operation. A reader of `cs.path` always sees either the old file or the new file — never a file that is half-written. This is the same technique SQLite uses for its journal and LevelDB uses for its MANIFEST file.

The key observation: the "atomicity" here is filesystem-level, not storage-level. After `Rename`, the data might still be in the OS page cache, not on durable storage. A power failure could lose it. For true durability, you need `fsync` on both the file and the directory before returning. We omit `fsync` here for simplicity — the WAL lab covers `fsync` in depth.

### Phase 1 and Phase 2 in detail

```go
func (tc *TxnCoordinator) ProcessBatch(events []Event, sink func([]WindowResult) error) error {
    // Phase 1: compute output, checkpoint with OutputCommitted=false.
    // After this checkpoint is durably written, we are "prepared".
    results := computeWindowResults(events)
    newOffset := currentOffset + len(events)
    tc.store.Save(Checkpoint{SourceOffset: newOffset, OutputCommitted: false})

    // Phase 2: write output to real sink.
    // If we crash here, Recover() will re-execute this on restart.
    sink(results)

    // Phase 2 complete: update checkpoint to OutputCommitted=true.
    tc.store.Save(Checkpoint{SourceOffset: newOffset, OutputCommitted: true})
    return nil
}
```

**Crash between phase 1 and phase 2**: `OutputCommitted=false` is on disk. On `Recover()`, we see this and know we must re-execute phase 2. The sink write must be idempotent for this to work correctly (or the results must be stored in the checkpoint so the exact same data is re-written).

**Crash after phase 2**: `OutputCommitted=true` is on disk. On `Recover()`, we see this and know we completed successfully. No re-execution.

**The gap in our implementation**: we don't persist the `PreparedOutput` in the checkpoint file — only the offset and the `OutputCommitted` flag. In a real system, the prepare phase would also write the computed results to a durable buffer (e.g., a Kafka "prepare" topic) so that `Recover()` can read the exact same results and re-write them to the real sink. Our implementation signals to the caller via the checkpoint that re-processing from the previous offset is needed.

### At-least-once vs. exactly-once: the concrete difference

```go
// AtLeastOnceProcessor — wrong order: write output THEN commit offset.
// Crash between these two steps causes duplicate output on restart.
func (p *AtLeastOnceProcessor) ProcessBatch(events []Event, sink func([]WindowResult) error) error {
    results := computeWindowResults(events)
    sink(results)                    // write output first
    p.store.Save(newCheckpoint)      // commit offset second
    // If crash here: restart re-reads same events -> duplicate sink write
    return nil
}

// TxnCoordinator — 2PC: both are coordinated.
// The checkpoint records where we are in the commit protocol.
// Crash in any state leads to correct re-execution on recovery.
```

The test that makes this concrete: process the same batch twice with `AtLeastOnceProcessor` (simulating a crash by resetting the checkpoint before the second run). Assert that the sink is called twice with the same results. Then do the same with `TxnCoordinator` — assert the sink is called only once.

## Stress tests

**Window boundary flood**: produce events that cross a window boundary at 100,000 events/sec. Each boundary-crossing triggers `Flush` inside `Process`, which allocates a new `map[string][]float64`. At high rates, GC pressure from map allocation becomes visible. The mitigation is to pool maps (via `sync.Pool`) and reset them instead of allocating new ones on each flush.

**Watermark stall**: send events at T=0s, T=10s, T=20s — then stop. The `SlidingWindow` has buffered events but the watermark has stalled at T=20s-allowedLateness. No windows are flushed because no new events are advancing the watermark. This is the "idle stream" problem. Real Flink handles this with periodic watermark generators (e.g., `WatermarkStrategy.forBoundedOutOfOrderness()` with `withIdleness()`) that advance the watermark on a timer even when no events arrive.

**High-cardinality keys**: feed 1,000,000 distinct keys into a 1-minute tumbling window. The `buckets` map grows to 1M entries, each holding a `[]float64` slice. Memory usage climbs to ~100MB for the map overhead alone. Flink uses RocksDB for this case — our in-memory map is the honest gap.

**2PC crash simulation**: call `ProcessBatch` on `TxnCoordinator`, then forcibly reset the checkpoint to `OutputCommitted=false`. Call `ProcessBatch` again with the same events. Assert the sink is called a second time (correct — 2PC re-executes phase 2) but that the source offset does not double-advance. This test is the single most important correctness check for the exactly-once implementation.

### The Kafka Streams DSL comparison

The entire v0 tumbling window aggregation, expressed in Kafka Streams DSL:

```java
KStream<String, Double> stream = builder.stream("sensor-events");

KTable<Windowed<String>, Double> windowedSum = stream
    .groupByKey()
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(1)))
    .aggregate(
        () -> 0.0,
        (key, value, agg) -> agg + value,
        Materialized.with(Serdes.String(), Serdes.Double())
    );

windowedSum.toStream().to("sensor-window-sums");
```

Eight lines. This hides:
- Watermark advancement from record timestamps (our `Watermark` struct, v1)
- Exactly-once via Kafka's transactional producer (our `TxnCoordinator`, v2)
- RocksDB state store for the aggregation state (our in-memory `map[string][]float64`)
- Changelog topic for fault tolerance (our `checkpoint.json`)
- Partition assignment and rebalancing across StreamsTasks (not in our scope)

Our 850-line Go implementation makes each of those eight lines concrete.

### Part A: StreamProcessorClient — the HTTP bridge

`StreamProcessorClient` wraps the Go processor's HTTP API with two methods: `sendEvent` and `queryResults`. Under 60 lines, intentionally.

```java
public class StreamProcessorClient {
    private final RestClient http;

    // Analogous to: producer.send(new ProducerRecord<>("sensor-events", key, value))
    // The event enters the stream topology at the source.
    public void sendEvent(String key, double value, Instant timestamp) {
        http.post().uri("/events")
            .contentType(MediaType.APPLICATION_JSON)
            .body(Map.of("key", key, "value", value, "timestamp", timestamp.toString()))
            .retrieve()
            .toBodilessEntity();
    }

    // Analogous to: streams.store(StoreQueryParameters.fromNameAndType(...))
    // Query the materialized window state.
    public List<WindowResultDto> queryResults(String key) {
        return http.get().uri("/results/{key}", key)
            .retrieve()
            .body(WindowResultListDto.class)
            .results();
    }
}
```

| Fault tolerance | `checkpoint.json` (offset only) | Changelog topic (full state) |

### Part C: Why `ofSizeWithNoGrace` is the production footgun

`TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(1))` sets `allowedLateness=0`. Any event that arrives even 1 millisecond after the window has been watermarked-closed is dropped. This is safe only if your producers are reliably in-order — which is common in Kafka but not guaranteed.

The safer alternative: `TimeWindows.ofSizeAndGrace(Duration.ofMinutes(1), Duration.ofSeconds(5))`. This sets `allowedLateness=5s`, equivalent to our v1 `ProcessorConfig{AllowedLateness: 5 * time.Second}`. The window fires 5 seconds later (at T=65s not T=60s), but catches events that arrived up to 5 seconds late.

Our v1 implementation makes this tradeoff visible: with `allowedLateness=5s`, you buffer more events in `sortedBuf` and emit windows 5 seconds later. The buffer size is bounded by `allowedLateness * eventsPerSecond`.
