## Why this lab exists

Every observability stack you have used — Prometheus, InfluxDB, Datadog, VictoriaMetrics — is built on the same fundamental insight: time-ordered float data has radically different access patterns than general-purpose data, and exploiting those patterns produces 10-100x better storage efficiency.

You have dashboarded your services. You have written PromQL queries. You know that `rate()` and `increase()` exist. But until you have personally compressed a stream of CPU-usage floats down from 16 bytes per point to 1.4 bytes per point using nothing but bit arithmetic — until you have felt the difference between a BTreeMap range scan and a chunk decompression loop — you don't know *why* time-series databases are their own category.

This lab also connects directly to the WAL lab. Facebook's Gorilla paper describes a system that keeps an in-memory chunk per metric and flushes it to disk periodically, using a WAL for the current chunk to survive crashes. We deliberately skip the WAL here (see "What the toy misses") to keep the compression algorithm front and center.

## What we're building

A time-series database in Rust across three stages, each building on the last.

**We build:**
- An in-memory TSDB using BTreeMap for O(log n) range scans
- Gorilla-style bit-packed compression: delta-delta timestamps, XOR floats
- A multi-tier downsampling system: raw (1s), minute averages, hour averages
- A background Tokio task for retention enforcement
- An Axum HTTP server exposing insert, query, and metrics endpoints

**We explicitly skip (and why):**
- A WAL for chunk durability (that's lab 5, and the interaction is covered in "What the toy misses")
- A full inverted label index (that's what makes Prometheus a production system vs. a toy)
- Out-of-order ingestion buffer (real but adds significant complexity)
- Quantile approximation (t-digest deserves its own lab)

## Mental model

Time-series workloads have three properties that shape every design decision:

**1. Writes are nearly always append-only and time-ordered.** CPU usage at 10:00:01 is written, then 10:00:02, then 10:00:03. You almost never go back and change a historical value. This means writes are sequential, and sequential writes are fast on any storage medium.

**2. Queries are almost always range scans over recent data.** "What was the p99 latency over the last 5 minutes?" "Show me the last 24 hours of memory usage." This is not random access by key — it is a scan over a contiguous time range. A data structure that keeps data sorted by timestamp (BTreeMap, time-ordered chunks) makes these scans efficient.

**3. Data arrives at regular intervals.** A metrics scraper runs every 15 seconds. An application emits a heartbeat every 1 second. This regularity is the key to Gorilla's compression: if the interval never changes, the delta between consecutive timestamps is always the same, and the delta-of-delta is zero. Zero is cheap to encode.

These three properties together mean: **a time-series database is not a general-purpose database with timestamps bolted on.** It is a specialized structure that trades general-purpose flexibility for dramatic improvements in write throughput, compression ratio, and range scan speed.

```
Write path (time-ordered):            Query path (range scan):

  app emits metric every 1s            "give me cpu from T-5m to now"
          │                                       │
          ▼                                       ▼
  TSDB append (sequential)             find chunks overlapping range
          │                                       │
          ▼                                       ▼
  buffer fills (128 pts)               decompress each chunk
          │                                       │
          ▼                                       ▼
  compress → flush to disk             filter by [start, end]
  (12:1 ratio, ~1.4 bytes/pt)          return sorted points
```

The smallest TSDB that can exist. Two nested BTreeMaps — one keyed by metric name, one keyed by timestamp.

```rust
pub struct Tsdb {
    series: BTreeMap<String, BTreeMap<i64, f64>>,
}

impl Tsdb {
    pub fn insert(&mut self, metric: &str, timestamp: i64, value: f64) {
        self.series
            .entry(metric.to_owned())
            .or_default()
            .insert(timestamp, value);
    }

    pub fn query(&self, metric: &str, start_ts: i64, end_ts: i64) -> Vec<DataPoint> {
        let Some(tree) = self.series.get(metric) else {
            return Vec::new();
        };
        tree.range(start_ts..=end_ts)
            .map(|(&ts, &val)| DataPoint::new(ts, val))
            .collect()
    }
}
```

**The core lesson**: `BTreeMap::range()` is a free range scan. The data is kept in sorted order by the tree itself. You don't pay for sorting at query time — you pay for it at insert time (O(log n) per insert). For append-mostly workloads where queries outnumber writes by 10:1, this is the right tradeoff.

**Why this already has a problem**: 1 million data points × 16 bytes (8-byte timestamp + 8-byte f64) = **16 MB per metric**. A system monitoring 1000 metrics at 1-second resolution generates 16 GB/day of raw data. That is v0's core limit.

</LabStage>

Facebook's Gorilla paper (2015) describes a system that stores 2 billion metrics in memory. The trick: represent each time series as a stream of bits, not bytes. For steady-state production metrics, most of those bits are zero.

### Timestamp compression: delta-of-delta encoding

Every timestamp is stored relative to the previous one. For regular 1-second sampling:

```
Raw timestamps (ms):   1700000000000, 1700000001000, 1700000002000, 1700000003000, ...
Deltas:                              1000,           1000,           1000, ...
Delta-of-deltas:                                     0,              0, ...
```

A delta-of-delta of 0 means "same interval as before." We can encode this as a **single bit**.

The full encoding:

```
DoD == 0:            '0'              → 1 bit
DoD in [-63, 64]:    '10' + 7 bits    → 9 bits
DoD in [-255, 256]:  '110' + 9 bits   → 12 bits
DoD else:            '111' + 12 bits  → 15 bits
```

For real monitoring data sampled every 1 second, DoD is almost always 0. Each timestamp after the first two costs **1 bit** instead of 64 bits. That is a 64x compression on timestamps alone.

### Float compression: XOR encoding

For float values, we exploit that consecutive readings are usually close:

```
CPU readings: 0.452, 0.461, 0.458, 0.463, ...
XOR:                 ^tiny^  ^tiny^  ^tiny^
```

When you XOR two IEEE-754 doubles that are numerically close, the sign bit and exponent bits (which encode the magnitude) are usually identical, leaving only a few changed mantissa bits. The XOR result has many leading zeros and many trailing zeros.

```
If XOR == 0:                    '0'                   → 1 bit
Else, reuse prev lz/tz:         '10' + meaningful bits
Else, new lz/tz:                '11' + 5-bit lz + 6-bit len + bits
```

This is the key insight: we store only the **meaningful bits** — the bits that actually changed between consecutive values.

### On-disk format

Each metric has a file `data/<metric>.chunks`. Each chunk holds up to 128 data points:

```
Chunk {
  first_ts:           i64  (8 bytes, uncompressed)
  first_delta:        i64  (8 bytes, bootstrap for DoD)
  first_value_bits:   u64  (8 bytes, raw f64 bits)
  ts_bits:            Vec<u8>  (compressed timestamp stream)
  val_bits:           Vec<u8>  (compressed value stream)
  ts_bit_len:         usize    (valid bits in ts_bits)
  val_bit_len:        usize    (valid bits in val_bits)
  count:              usize    (number of points)
}
```

A chunk of 128 points sampled at 1s intervals uses roughly:
- Timestamps: 2 full values (16 bytes) + 126 × 1 bit ≈ 32 bytes
- Values: 8 bytes + ~2 meaningful bits each ≈ ~48 bytes total per chunk
- Overhead: 24 bytes

Total: ~104 bytes for 128 points = **0.81 bytes per point** vs. 16 bytes uncompressed.

Even at 12:1 compression, a metric sampled every second still generates:

```
86,400 points/day × 0.81 bytes/point ≈ 70 KB/day per metric (compressed)
```

For 1000 metrics, that is 70 MB/day. After 90 days, 6.3 GB. Manageable — but the pattern is linear growth with no way to ask "show me last year's trend" without decompressing terabytes.

Downsampling breaks the linear growth. After 24 hours, you replace 86,400 raw points with 1,440 minute-averages. The query `"show me the last 7 days of CPU usage"` runs against minute-tier data: 10,080 points instead of 604,800. The precision cost: you can't reconstruct the individual seconds, but for trend analysis, alerting, and capacity planning, minute resolution is almost always sufficient.

Our three tiers:

| Tier   | Resolution | Retention | Points after 90d |
|--------|-----------|-----------|------------------|
| raw    | exact 1s  | 24 hours  | ~86,400          |
| minute | 1-min avg | 7 days    | ~10,080          |
| hour   | 1-hr avg  | 90 days   | ~2,160           |

**Dual-write on insert**: rather than running an expensive background job to downsample old data, we write to all three tiers on every insert. The minute bucket is computed by truncating the timestamp to the minute boundary; the hour bucket by truncating to the hour boundary.

```rust
pub fn insert(&mut self, metric: &str, timestamp: i64, value: f64) {
    let store = self.stores.entry(metric.to_owned()).or_default();
    // Raw tier: exact timestamp
    store.raw.insert(timestamp, value);
    // Minute tier: truncate to minute boundary
    let minute_bucket = (timestamp / MINUTE_MS) * MINUTE_MS;
    store.minute.insert(minute_bucket, value);
    // Hour tier: truncate to hour boundary
    let hour_bucket = (timestamp / HOUR_MS) * HOUR_MS;
    store.hour.insert(hour_bucket, value);
}
```

**Tokio background compactor**: the `spawn_compactor()` function creates a Tokio task that calls `compact()` on a fixed interval. Because the TSDB is wrapped in `Arc<Mutex<Tsdb>>`, the background task and the HTTP handlers share the same database safely.

```rust
pub fn spawn_compactor(db: SharedTsdb, interval: Duration) -> tokio::task::JoinHandle<()> {
    tokio::spawn(async move {
        let mut ticker = tokio::time::interval(interval);
        loop {
            ticker.tick().await;
            let stats = { db.lock().unwrap().compact() };
            tracing::info!("compaction: purged raw={}", stats.raw_points_purged);
        }
    })
}
```

The critical design point: `compact()` holds the lock for the entire purge loop. For a toy with thousands of points this is fine. A production system would use a more granular lock or a copy-on-write approach so writes are not blocked during compaction.

## Stress tests

**What breaks at 10× data rate (10 metrics, 10s samples)?** v0 is unaffected — BTreeMap scales linearly with insertions. v1 flushes chunks more frequently, generating more disk I/O. The JSON serialization of each chunk (used for simplicity here) becomes the bottleneck. At 10× you would want a binary chunk format.

**What breaks at 1000 metrics?** The v2 `compact()` function holds a single mutex while purging all metrics. At 1000 metrics with 86,400 raw points each, the purge loop runs for milliseconds while writes are blocked. Production TSDBs use per-metric locks or epoch-based compaction to avoid this.

**Corruption injection**: truncate a `.chunks` file mid-byte (`truncate -s 50 data/cpu.chunks`), then reopen the database. The JSON deserialization will fail for that metric, and the database will start fresh — no crash, just data loss for that metric. A real TSDB would use CRC checks per chunk (see WAL lab) to detect and skip corrupt chunks rather than failing the entire file.

**Memory pressure**: insert 10 million points into v0 without calling query. Memory grows to ~160 MB (10M × 16 bytes). The BTreeMap itself has node overhead, so real usage is higher. v2 helps here only after compaction runs — until then, all points are still in memory.

### The Micrometer PushMeterRegistry pattern

Micrometer is the metrics facade that Spring Boot Actuator uses. It ships a `MeterRegistry` interface with dozens of implementations: Prometheus (pull), InfluxDB (push), Datadog (push), etc. Any gauge, counter, or timer you register with `MeterRegistry` is automatically reported to all bound registries.

`PushMeterRegistry` is the abstract base for push-based implementations. You override one method: `publish()`. The base class calls it on a fixed interval in a background thread. This is *exactly* the right abstraction for our TSDB.

```java
public class TsdbMicrometerRegistry extends PushMeterRegistry {

    private final TsdbClient client;

    @Override
    protected void publish() {
        long nowMs = config().clock().wallTime();

        // Collect all gauges and push them
        for (Gauge gauge : getMeters().stream()
                .filter(m -> m instanceof Gauge)
                .map(m -> (Gauge) m).toList()) {
            double value = gauge.value();
            if (Double.isFinite(value)) {
                String name = sanitize(gauge.getId().getName());
                client.insertWithTimestamp(name, nowMs, value);
            }
        }
        // ... similarly for counters and timers
    }

    static String sanitize(String name) {
        return name.replace('.', '_').replace('-', '_');
    }
}
```

### Why this matters for any Spring Boot app

The moment `TsdbMicrometerRegistry` is registered as a bean, **every metric in the application** starts flowing to the TSDB — JVM heap, GC pause time, HTTP request rates, database connection pool stats — without any code changes. This is the power of the Micrometer abstraction.

```java
// TsdbAutoConfiguration.java — zero-config wiring
@Bean @ConditionalOnMissingBean
public TsdbMicrometerRegistry tsdbMicrometerRegistry(
        TsdbClient client, TsdbProperties props) {
    return new TsdbMicrometerRegistry(client, props);
}
```

Any Spring Boot app with this class on its classpath and `tsdb.base-url` set in `application.yml` gets automatic metric shipping. Compare this to `micrometer-registry-influx` — our registry is structurally identical, just pointed at a Rust binary instead of InfluxDB.

### TsdbClient: the HTTP bridge

```java
public class TsdbClient {
    private final RestClient http;

    public void insertWithTimestamp(String metric, long timestampMs, double value) {
        var body = Map.of("metric", metric, "timestamp", timestampMs, "value", value);
        http.post()
                .uri("/insert")
                .contentType(MediaType.APPLICATION_JSON)
                .body(body)
                .retrieve()
                .toBodilessEntity();
    }

    public List<DataPoint> query(String metric, long startTs, long endTs) {
        var result = http.get()
                .uri("/query?metric={m}&start={s}&end={e}", metric, startTs, endTs)
                .retrieve()
                .body(new ParameterizedTypeReference<List<DataPoint>>() {});
        return result != null ? result : List.of();
    }
}
```

## What surprised me

<WhatSurprisedMe insight="The Gorilla paper's claim of 1.37 bytes/point average sounds theoretical — but it comes almost entirely from the timestamp compression, not the float compression. CPU metrics sampled every 10 seconds have timestamps that are almost perfectly regular (delta = 10000ms, delta-of-delta = 0 nearly always). That means each timestamp in a steady stream compresses to exactly 1 bit. I expected the float XOR encoding to be the impressive part. It isn't. The float XOR saves space when consecutive values are similar, but CPU readings vary enough that you frequently need to store new leading/trailing zero counts. The timestamp side, by contrast, is almost always just 1 bit: 'same interval as before.' Gorilla's real magic is exploiting the boring regularity of timestamps in a system that was designed to be sampled on fixed intervals." />
