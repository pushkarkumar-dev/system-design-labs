## Why this lab exists

Every fast service you've built has a cache in front of it. Session tokens, product catalog, rate-limit counters, order status — if you're hitting a database for these on every request, something is wrong. Redis is usually the answer: trivial to operate, blindingly fast, and understood by every Java framework in existence.

But "trivially fast" hides interesting engineering. A HashMap lookup takes nanoseconds. The network round-trip to Redis takes microseconds. The RESP protocol adds parsing overhead. The Mutex over the entire keyspace is a bottleneck at high concurrency. And persistence (AOF, RDB, fsync semantics) turns a cache into a system you have to reason about for durability.

This lab builds all of it from scratch. The star of the show: **the v1 RESP server is wire-compatible with unmodified Jedis 5.x**. You implement the five RESP types, handle seven commands, and Jedis connects without knowing it's talking to a toy. That's a meaningful lesson about protocol simplicity and why Redis dominates: the protocol is so easy to implement that any server can be Redis.

## What we're building

A Redis-compatible cache in Rust, in three stages:

**We build:**
- `HashMap<String, CacheEntry>` with optional TTL per key (lazy expiry)
- RESP2 protocol parser and encoder (five types, seven commands)
- TCP server with one Tokio task per connection
- LRU eviction: evict least-recently-used key when at capacity
- AOF persistence: append every write to a log file, replay on startup

**We explicitly skip (and why):**
- Cluster mode (sharding across 16384 hash slots — that's a distributed systems lab)
- Replication (that's the Raft lab)
- Sorted sets, hashes, lists (non-trivial data structures — this lab is about the cache *mechanics*, not the data model breadth)
- Pub/sub (v3 skeleton only — it requires a separate command multiplexer)

## Mental model

```
Client (Jedis)                  Server (our Rust kv-cache)
─────────────────────────────────────────────────────────
*3\r\n                      ──► parse RESP array (3 elements)
$3\r\nSET\r\n               ──► arg[0] = "SET"
$4\r\nname\r\n              ──► arg[1] = "name"
$5\r\nAlice\r\n             ──► arg[2] = "Alice"
                                dispatch → cache.insert("name", "Alice")
+OK\r\n                     ◄── encode SimpleString "OK"

*2\r\n                      ──► parse RESP array (2 elements)
$3\r\nGET\r\n               ──► arg[0] = "GET"
$4\r\nname\r\n              ──► arg[1] = "name"
                                dispatch → cache.get("name") = Some("Alice")
$5\r\nAlice\r\n             ◄── encode BulkString "Alice"
```

Jedis doesn't see our Rust code. It sees RESP bytes. The protocol is the contract.

The simplest possible cache: a `HashMap<String, CacheEntry>` where each entry holds a value and an optional expiry `Instant`.

```rust
pub struct Cache {
    store: HashMap<String, CacheEntry>,
}

pub struct CacheEntry {
    pub value: String,
    pub expires_at: Option<Instant>,  // absolute deadline, not a duration
    pub accessed_at: Instant,         // for LRU in v2
}

impl Cache {
    pub fn set(&mut self, key: String, value: String, ttl_secs: Option<u64>) {
        let expires_at = ttl_secs.map(|s| Instant::now() + Duration::from_secs(s));
        self.store.insert(key, CacheEntry::new(value, expires_at));
    }

    pub fn get(&mut self, key: &str) -> Option<&str> {
        // Lazy expiry: check on access, remove if expired
        if let Some(entry) = self.store.get(key) {
            if entry.is_expired() {
                self.store.remove(key);
                return None;
            }
        }
        self.store.get_mut(key).map(|e| {
            e.accessed_at = Instant::now(); // track for LRU
            e.value.as_str()
        })
    }
}
```

**Why lazy expiry?** It's simpler and has zero overhead on the hot path. The tradeoff: expired entries stay in memory until they're accessed. Redis handles cold keys with a periodic sweeper that runs every 100ms and samples 20 random keys with TTLs — if more than 25% are expired, it samples again. We skip the sweeper here.

**Why `Instant` not `SystemTime`?** `Instant` is monotonic — it can't go backwards due to NTP adjustments. For TTL calculations (`now >= expiry`), monotonic is exactly what you want. `SystemTime` would give wrong results if the clock steps backward.

**TTL semantics**: Redis uses a convention that Jedis expects:
- `-1` means the key exists but has no expiry
- `-2` means the key does not exist (or has expired)

We match this exactly so Jedis's `jedis.ttl("key")` returns the same values.

RESP2 is almost trivially simple. Five types, all line-delimited:

```
+OK\r\n                       Simple String — status responses
-ERR some message\r\n         Error — command errors
:42\r\n                       Integer — counts, TTLs
$6\r\nfoobar\r\n              Bulk String — the main value type
*3\r\n<elem><elem><elem>      Array — how clients send commands
$-1\r\n                       Null Bulk — missing key (GET miss)
```

Clients always send commands as Arrays of Bulk Strings. A `SET name Alice` looks like:

```
*3\r\n
$3\r\nSET\r\n
$4\r\nname\r\n
$5\r\nAlice\r\n
```

The parser reads the first byte to determine type, then reads the appropriate framing. The hardest part in async Rust is not the parsing itself — it's **partial reads**: the TCP socket might deliver `*3\r\n$3\r\nSET` and the rest arrives in the next read call. We handle this with a grow-until-complete buffer:

```rust
async fn handle_connection(mut socket: TcpStream, cache: SharedCache) -> io::Result<()> {
    let mut buf = Vec::with_capacity(4096);
    let mut tmp = [0u8; 1024];

    loop {
        let n = socket.read(&mut tmp).await?;
        if n == 0 { break; }
        buf.extend_from_slice(&tmp[..n]);

        let mut pos = 0;
        loop {
            match parse_command(&buf[pos..]) {
                ParseResult::Complete(cmd, consumed) => {
                    let response = dispatch(&cmd, &cache);
                    socket.write_all(response.as_bytes()).await?;
                    pos += consumed;
                }
                ParseResult::Incomplete => break, // wait for more data
                ParseResult::Error(msg) => { /* send -ERR, clear buffer */ break; }
            }
        }
        if pos > 0 { buf.drain(..pos); }
    }
    Ok(())
}
```

**Why `Arc<Mutex<HashMap>>`?** Each connection gets a Tokio task. They all share the same HashMap, so we need synchronization. A coarse `Mutex` over the whole cache is the right choice here — it matches Redis's original single-threaded model where one logical lock covers all operations. At 200k ops/sec, each lock hold time is ~5μs. The Mutex overhead is under 2μs. Perfectly acceptable.

</LabStage>

**LRU eviction**: when the cache is full, evict the key that was least recently accessed. Each `get()` already updates `entry.accessed_at`. Eviction is a linear scan for the minimum:

```rust
fn evict_lru(&mut self) {
    let lru_key = self.store
        .iter()
        .min_by_key(|(_, entry)| entry.accessed_at)
        .map(|(k, _)| k.clone());

    if let Some(key) = lru_key {
        self.store.remove(&key);
    }
}
```

This is O(n). For caches under ~50k keys, a single CPU cache line walk is faster than maintaining a doubly-linked list (pointer-chasing). Beyond that, the canonical O(1) structure is a doubly-linked list of keys with a HashMap of `key → list node`. Access = splice to head (O(1)). Eviction = remove tail node (O(1)). This is what Caffeine implements. Redis uses an approximation: sample 5 random keys, evict the oldest of those.

**AOF persistence**: every write appends a JSON line to a file:

```json
{"op":"set","key":"session:abc","value":"user_id=42","ttl_secs":300}
{"op":"del","key":"old-key"}
{"op":"expire","key":"session:abc","ttl_secs":600}
```

On startup, replay the log sequentially. Each `set` inserts, each `del` removes, each `expire` updates the TTL. The final state matches what the log describes.

We do not `fsync` after every write — that would cap throughput at ~3k ops/sec (same as the WAL lab's v1). Instead we rely on the OS page cache, which flushes to disk within ~1 second. This matches Redis's `appendfsync everysec` mode: at most 1 second of data loss on a crash.

**AOF vs RDB**: Redis offers both. RDB is a point-in-time snapshot (fast to restore, potential for minutes of data loss). AOF is a command log (slower to restore from large files, minimal data loss). Production Redis often uses both: RDB for fast restart, AOF for durability. Our v2 implements AOF only.

</LabStage>

  ]}
  loc={100}

**Pipelining** is already half-implemented in our v1 buffer loop — we parse until `ParseResult::Incomplete` and process multiple commands in one read. The remaining work is ensuring all responses are buffered and sent together, not one by one. Jedis's `Pipeline` sends N commands in one TCP write; a correct RESP server processes them all and returns N responses.

**Pub/sub** requires a separate command path. When a client sends `SUBSCRIBE channel`, it enters a special mode where it can only receive messages, not issue normal commands. The server keeps a `HashMap<String, Vec<Sender<String>>>` of channel → active subscribers. `PUBLISH channel message` sends to all subscriber channels.

The architectural challenge: subscriber tasks need to receive messages *pushed* from publisher tasks. This is a natural fit for Tokio's `broadcast::channel` or `mpsc::channel` per subscription.

### Part A — Jedis direct connection

Jedis 5.x is the standard Java Redis client. It has no knowledge of our server — it speaks RESP2, our server speaks RESP2, and that's the entire contract.

```java
JedisPool pool = new JedisPool("localhost", 6380); // our Rust server, not Redis
try (Jedis jedis = pool.getResource()) {
    // PING
    String pong = jedis.ping();   // → "PONG"

    // SET / GET
    jedis.set("session:abc", "user_id=42");
    String val = jedis.get("session:abc");   // → "user_id=42"

    // SET with TTL
    jedis.setex("token:xyz", 300, "secret");
    long ttl = jedis.ttl("token:xyz");       // → 299 (or 300)

    // EXISTS / DEL
    boolean exists = jedis.exists("session:abc");  // → true
    long deleted   = jedis.del("session:abc");      // → 1
}
```

The pool settings in `CacheConfig.java` control connection reuse:
- `maxTotal` (maxActive): 20 — maximum concurrent checkouts
- `maxIdle`: 10 — connections kept warm in the pool (avoids reconnect overhead)
- `connectTimeout`: 2000ms — fail fast rather than hanging

### Part B — Spring Cache abstraction

`OrderService.java` uses three annotations to cache order lookups. It imports **zero Jedis classes**. The Spring Cache abstraction makes the backend completely transparent.

```java
@Service
public class OrderService {

    // Cache-aside: check cache first, call DB on miss, populate cache
    @Cacheable(value = "orders", key = "#orderId")
    public Optional<Order> getOrder(Long orderId) {
        log.info("CACHE MISS — loading order {} from database", orderId);
        return Optional.ofNullable(database.get(orderId));
    }

    // Always writes to DB + refreshes cache entry
    @CachePut(value = "orders", key = "#order.id()")
    public Order updateOrder(Order order) { ... }

    // Removes from cache so next read is a cache miss
    @CacheEvict(value = "orders", key = "#orderId")
    public void deleteOrder(Long orderId) { ... }
}
```

`CacheConfig.java` wires Jedis to our Rust server and creates the `RedisCacheManager`:

```java
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public RedisConnectionFactory redisConnectionFactory(JedisPoolConfig poolConfig) {
        var standalone = new RedisStandaloneConfiguration("localhost", 6380);
        var clientConfig = JedisClientConfiguration.builder()
                .connectTimeout(Duration.ofMillis(2000))
                .usePooling().poolConfig(poolConfig)
                .build();
        return new JedisConnectionFactory(standalone, clientConfig);
    }

    @Bean
    public RedisCacheManager cacheManager(RedisConnectionFactory connectionFactory) {
        var defaultConfig = RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(5))  // default TTL for all caches
                .prefixCacheNameWith("lab::")      // namespace: "lab::orders"
                .disableCachingNullValues();
        return RedisCacheManager.builder(connectionFactory)
                .cacheDefaults(defaultConfig)
                .build();
    }
}
```

**The abstraction lesson**: replace `RedisCacheManager` with `CaffeineCacheManager` and `OrderService` is unaffected. Not one line changes in the service. The cache backend is an infrastructure decision, not a business logic decision. That is the right separation.

## Where to go deeper

- **RESP protocol spec**: the full specification is remarkably short — [redis.io/docs/reference/protocol-spec](https://redis.io/docs/reference/protocol-spec/)
- **Jedis 5.x source**: read `Connection.java` and `Protocol.java` — the entire RESP codec is ~200 lines — [github.com/redis/jedis](https://github.com/redis/jedis)
- **Redis internals**: how Redis implements its data structures (skip lists for sorted sets, intsets for small integer sets, ziplist for compact hashes) — [redis.io/docs/manual/internals](https://redis.io/docs/manual/internals/)
- **Caffeine vs Redis**: when to use an in-process cache vs a distributed cache — [github.com/ben-manes/caffeine/wiki](https://github.com/ben-manes/caffeine/wiki)
- **Redis 7 LOLWUT**: Redis's easter egg command renders an ASCII art piece. The source is in `src/lolwut*.c`. Worth a read for fun and for seeing how simple commands plug into the dispatch table.
