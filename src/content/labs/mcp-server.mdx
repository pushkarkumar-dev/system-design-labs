## Why this lab exists

Every time Claude uses a tool — searches the web, reads a file, queries a database — it's talking to an MCP server. The Model Context Protocol is Anthropic's open standard for connecting LLMs to external systems, and it's spreading fast: Claude Desktop, Claude.ai, Cursor, and dozens of third-party clients all speak MCP.

You've seen the diagrams: LLM calls tool, tool returns result, LLM generates answer. But until you've implemented the JSON-RPC handshake yourself, registered a tool with a schema, watched the `tools/list` response flow back over a Unix pipe, and then seen Claude actually call your function — you don't understand what's happening at the wire level.

This lab builds an MCP server from scratch: from the stdio transport loop (150 lines of Python) up through resources, prompts, SSE streaming, authentication middleware, and progress notifications. You will understand exactly what Claude Desktop does when it spawns an MCP server process, and what a remote MCP server must implement to work over HTTP.

## What we're building

An MCP server framework in Python, implemented across three stages. Each stage adds one layer of the MCP spec and tests it against the real protocol.

**We build:**
- JSON-RPC 2.0 message framing over stdin/stdout
- MCP initialize handshake with capability negotiation
- Tool registration via `@mcp_tool` decorator with JSON Schema validation
- Resource registration via `@mcp_resource` decorator for data sources
- Prompt templates via `@mcp_prompt` with argument substitution
- HTTP transport via FastAPI with SSE streaming for notifications
- Progress notifications for long-running tool calls
- Middleware pipeline (auth, logging) running around every request
- Sampling support (`sampling/createMessage` — server asks client's LLM)
- A complete example server with 3 tools and 2 resources

**We explicitly skip (and why):**
- Full OAuth 2.0 authorization (see "What the Toy Misses" — requires a token server)
- URI template matching beyond prefix matching (full RFC 6570 templates are complex)
- Persistent session state across reconnects (requires a session store)
- Binary blob resources (base64 encoding of large files; straightforward but verbose)
- Multi-server client routing (client-side concern, not server-side)

## Mental model

MCP is a protocol, not a library. The spec defines three primitives the server exposes and one transport the server runs on.

**Three server-side primitives:**

```
Tools      — functions the LLM can call (side effects: read/write/compute)
             tools/list   → [{ name, description, inputSchema }]
             tools/call   → { content: [{type, text}], isError }

Resources  — data the LLM can read (no side effects: pure data access)
             resources/list → [{ uri, name, mimeType }]
             resources/read → { contents: [{uri, mimeType, text}] }

Prompts    — reusable prompt templates with named parameters
             prompts/list → [{ name, description, arguments }]
             prompts/get  → { messages: [{role, content}] }
```

**Two transports:**

```
stdio  — client spawns server as subprocess; stdin/stdout are the pipe
         one JSON object per line, framed by newline
         85,000 calls/sec — just readline + json.loads + dict lookup

HTTP   — client connects to server's HTTP endpoint
         POST /messages   → synchronous request/response
         GET  /sse        → server-sent events for notifications
         12,000 calls/sec — FastAPI + loopback overhead
```

**The MCP handshake:**

```
Client → Server:  initialize {protocolVersion, clientInfo}
Server → Client:  {protocolVersion, serverInfo, capabilities}
Client → Server:  notifications/initialized   (notification, no response)
--- server is now ready ---
Client → Server:  tools/list
Server → Client:  {tools: [...]}
Client → Server:  tools/call {name: "read_file", arguments: {path: "/tmp/x"}}
Server → Client:  {content: [{type: "text", text: "..."}], isError: false}
```

The entire protocol is this simple at the transport layer. The complexity is in the semantics: JSON Schema validation, URI template matching, progress notification timing, capability negotiation. This lab builds each of those layers explicitly.

The entire MCP stdio transport is a `while True` loop. Every line in is one JSON-RPC message. Every line out is one JSON-RPC response. The OS pipe handles buffering and flow control.

**Why one message per line?**

MCP uses newline-delimited JSON (NDJSON) for the stdio transport. This is the simplest possible framing: `json.loads(line)` on each line. No length prefix, no delimiter scanning. The tradeoff is that no message can contain a literal newline — but JSON strings encode newlines as `\n`, so this is always true for valid JSON.

```python
while True:
    line = sys.stdin.readline()
    if not line:
        break          # EOF — client closed the pipe
    message = json.loads(line.strip())
    response = self.handle(message)
    if response is not None:   # notifications have no response
        sys.stdout.write(response.to_json() + "\n")
        sys.stdout.flush()
```

The `flush()` call is critical. Without it, Python's stdout buffering holds the response in memory until the buffer fills. Claude Desktop would wait forever for the first response. Always flush after writing.

**The tool registry:**

The `@mcp_tool` decorator stores `(ToolDefinition, callable)` in a module-level dict keyed by tool name. This is a simple registry pattern — no framework magic.

```python
_TOOL_REGISTRY: dict[str, tuple[ToolDefinition, Callable]] = {}

def mcp_tool(name, description, input_schema):
    def decorator(fn):
        _TOOL_REGISTRY[name] = (ToolDefinition(name, description, input_schema), fn)
        return fn
    return decorator
```

When `tools/call` arrives, the handler does a dict lookup by name, validates the arguments against the `inputSchema`, and calls the function with `**arguments`. The return value is serialized to JSON and wrapped in `{content: [{type: "text", text: ...}], isError: false}`.

**JSON Schema validation:**

MCP tools declare their inputs as JSON Schema. The `validate_input()` function does a minimal subset: checks that all `required` fields are present, and that field types match the `type` constraint. A production implementation would use the `jsonschema` library for full Draft 7 validation.

**The execute_python tool:**

The restricted eval pattern is instructive. Python's `eval()` can execute arbitrary code, which is dangerous. We constrain it by passing a custom `__builtins__` dict that only includes safe functions (math operations, type conversions). `ast.literal_eval` handles pure data literals (lists, dicts, numbers) safely before falling back to restricted eval.

This is a toy sandbox — a production implementation would use a subprocess with seccomp or a WebAssembly runtime. The "What the Toy Misses" section covers this.

**v0 limitation:**

v0 only supports tools. If a Claude Desktop session asks for resources (to see what files are available) or prompts (to use a code review template), the server returns `METHOD_NOT_FOUND`. v1 adds these.

**Tools vs Resources — the key distinction:**

Tools are for actions: `read_file` reads a file (but could also write one). Resources are for data access with no side effects. The MCP spec distinguishes them because clients can cache resources (they're idempotent reads) but must not cache tool calls (they may have side effects).

In practice, this means:
- `read_file(path)` as a **tool**: Claude calls it when it decides to read a file. It can also be used to write.
- `file:///path` as a **resource**: The client can enumerate it, pre-fetch it, show it in a UI. It's declared upfront, not called on demand.

The resource URI scheme is flexible. Our two sample resources use `file://` (filesystem convention) and `system://` (custom scheme). A database resource might use `db://customers/42`. The client treats the URI as an opaque string — the server defines what it means.

**String.Template for prompt substitution:**

Python's `string.Template` uses `$variable` substitution, which is safe for user-supplied templates because it doesn't execute code. The alternative, f-strings, would require `eval`. We use `safe_substitute` which leaves unmapped variables as `$variable` rather than raising KeyError for missing optional arguments.

```python
from string import Template
template = "Review this $language code:\n\n$code"
rendered = Template(template).safe_substitute({"language": "Python", "code": "..."})
```

**SSE transport:**

Server-Sent Events (SSE) is the MCP spec's choice for HTTP server-to-client streaming. The client opens a long-lived `GET /sse` connection; the server sends events as `data: {...}\n\n` chunks whenever it has something to say (progress updates, list-changed notifications).

```python
async def sse_generator():
    queue = asyncio.Queue()
    self._sse_queues.append(queue)
    try:
        yield f"data: {json.dumps({'type': 'connected'})}\n\n"
        while True:
            message = await asyncio.wait_for(queue.get(), timeout=30.0)
            yield f"data: {message}\n\n"
    finally:
        self._sse_queues.remove(queue)
```

Each SSE client gets its own `asyncio.Queue`. Broadcasting a notification means putting it on every queue. The generator yields from its queue, streaming events to that client. When the client disconnects, the `finally` block removes the queue.

The 30-second timeout sends a keepalive comment (`: keepalive\n\n`) to prevent the connection from being closed by proxies and load balancers that kill idle connections.

**HTTP vs stdio dispatch:**

The protocol layer is identical for HTTP and stdio. The same `McpHttpServer.handle(message: dict)` method processes the message. The transport layer is what differs:

- stdio: `json.loads(sys.stdin.readline())` → `handle()` → `sys.stdout.write(response.to_json())`
- HTTP: `await request.json()` → `handle()` → `JSONResponse(response.to_dict())`

This separation is deliberate. You could add a Unix socket transport or a WebSocket transport without touching the protocol layer.

**Sampling — the bidirectional trick:**

Every other MCP primitive flows client-to-server: the client calls a tool, reads a resource, gets a prompt. Sampling flows server-to-client: the MCP server asks the client's LLM to generate text.

Why would a tool need to call an LLM? Agentic tools. Imagine an MCP tool that summarizes a large codebase. It might:
1. Read all Python files (tool calls to `read_file`)
2. For each batch of files, ask the LLM to summarize them (sampling)
3. Ask the LLM to synthesize the summaries into a final report (sampling)
4. Return the final report to the client

The LLM is both the orchestrator (calling tools) and a service (being called by tools). Sampling makes MCP truly bidirectional.

Our `MockSamplingHandler` echoes the last user message as the "generated" response. This lets the test suite verify that the sampling code path runs correctly without a real LLM endpoint.

**Progress notifications — the async flow:**

When a tool call starts, the client can include `_meta.progressToken` in the request params. If the server sees this token, it should send `notifications/progress` messages while the tool is running:

```
Server → Client: {"method": "notifications/progress",
                   "params": {"progressToken": "abc", "progress": 1, "total": 3}}
Server → Client: {"method": "notifications/progress",
                   "params": {"progressToken": "abc", "progress": 2, "total": 3}}
...
Server → Client: {"method": "tools/call", "result": {...}}
```

The client matches `progressToken` to the pending tool call and updates a progress bar. The server sends progress via the same SSE channel (for HTTP transport) or on stdout (for stdio transport).

In our implementation, `call_tool_with_progress` is async because it needs to `await` the SSE queue puts between progress steps. The synchronous `handle()` method can't do this — it's for the request/response cycle. Progress requires a separate async code path.

**Middleware chain:**

The middleware pattern wraps the handler in pre/post hooks. The chain runs pre-hooks in order and post-hooks in reverse (like a stack of decorators):

```
pre:  auth → logging → handler → logging → auth
      ─────── (inward) ──────────── (outward) ───
```

`AuthMiddleware.pre()` returns a `JsonRpcResponse` immediately if the token is invalid — this short-circuits the chain without calling the handler. `LoggingMiddleware.pre()` records the start time and returns `None` (continue). `LoggingMiddleware.post()` computes elapsed time and appends to `self.log`.

**Roots — advisory access control:**

`roots/list` returns a list of filesystem paths the server has declared access to. This is advisory, not enforced: the server can still read files outside these roots. But the client uses it to build UI hints ("this server can access `/home/user/projects`") and to avoid asking for files the server obviously can't read.

The correct mental model: roots are like a server's "business card" — it declares its scope, but the client still enforces what it asks for based on user consent.

The 7x throughput gap between stdio and HTTP is the defining architectural fact of MCP deployment. Claude Desktop runs every MCP server as a local subprocess using stdio transport — not because HTTP doesn't work, but because stdio is faster, requires no port management, and the subprocess lifecycle is tied to the client process. Remote MCP servers (shared team tools, third-party integrations) use HTTP because they need to be accessible from multiple clients — and accept the latency cost.

## Stress tests

**Schema validation throughput:** Run `tools/call` with intentionally wrong argument types at high frequency. Each call must validate against the JSON Schema before invoking the function. Our `validate_input()` is O(fields) per call — at 85,000 calls/sec, schema validation adds less than 1 microsecond per call. The bottleneck is `json.loads`, not validation.

**SSE fan-out:** Connect 100 clients to `GET /sse`. Broadcast one notification. Each client's queue receives the notification; each SSE generator yields it. At 100 clients, the fan-out is 100 queue puts and 100 async yields. This is fast (microseconds) but blocks the asyncio event loop for each yield. A production server would use a pub/sub broker (Redis, NATS) for fan-out beyond 1,000 clients.

**Middleware chain depth:** Add 10 logging middleware instances to the chain. Each request runs 10 pre-hooks and 10 post-hooks in addition to the handler. At 85,000 stdio calls/sec, 20 extra function calls add approximately 2 microseconds per request. Middleware overhead is negligible — the bottleneck is always `json.loads` and `json.dumps`.

**Large tool output:** A tool that returns 1MB of text (e.g., reading a large log file). The response JSON is over 1MB. `json.dumps` of 1MB takes about 10ms. `sys.stdout.write` of 1MB blocks until the pipe buffer drains. For large outputs, MCP has no built-in streaming — the entire result arrives as one JSON object. Production servers either truncate large outputs or expose them as resources (which can be streamed separately).

### Part A — McpAutoConfiguration: spawning the Python server

`McpAutoConfiguration` creates a `McpSyncClient` bean configured with Spring AI's `StdioClientTransport`. When the Spring context starts, this bean spawns our Python MCP server as a subprocess and runs the MCP initialize handshake automatically.

```java
@Bean(destroyMethod = "close")
public McpSyncClient mcpSyncClient(McpProperties props) {
    ServerParameters serverParams = ServerParameters.builder(props.getServerCommand())
            .args(props.getServerArgs().split("\\s+"))
            .build();

    StdioClientTransport transport = new StdioClientTransport(serverParams);

    McpSyncClient client = McpSyncClient.builder()
            .transport(transport)
            .build();

    client.initialize();  // runs MCP handshake synchronously
    return client;
}
```

The `destroyMethod = "close"` annotation ensures the subprocess is killed cleanly when the Spring context closes. Without it, the Python process would become a zombie.

### Part B — McpClient: the high-level wrapper

`McpClient` wraps `McpSyncClient` with convenience methods. The raw Spring AI API uses `McpSchema.*` types for everything; our wrapper extracts the text content and returns plain strings — the 80% case for most Spring services.

```java
public String callTool(String toolName, Map<String, Object> arguments) {
    McpSchema.CallToolResult result = delegate.callTool(
            new McpSchema.CallToolRequest(toolName, arguments)
    );
    return result.content().stream()
            .filter(c -> c instanceof McpSchema.TextContent)
            .map(c -> ((McpSchema.TextContent) c).text())
            .findFirst().orElse("");
}
```

```java
// 1. List available tools (Python server responds)

// 2. Call a tool

// 3. Read a resource

// sysInfo: JSON with platform, python_version, cwd, timestamp

// 4. Get a rendered prompt
String prompt = client.getPrompt("code_review", Map.of(
        "language", "Java",
        "code", "public int add(int a, int b) { return a + b; }"
));
// prompt: "Please review the following Java code for bugs..."
```

## What surprised me

<WhatSurprisedMe insight="The stdio vs HTTP performance gap surprised me. stdio dispatch handles 85,000 tool calls/sec — it's just readline + json.loads + a dict lookup. HTTP dispatch over loopback handles only 12,000 calls/sec — a 7x slowdown. For Claude Desktop (which uses stdio), this means tool calls complete in 12µs average. For remote MCP servers over real networks (HTTP), latency is network-dominated: 1ms round-trip at minimum.
