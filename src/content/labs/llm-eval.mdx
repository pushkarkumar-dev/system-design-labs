## Why this lab exists

Every benchmark number you've seen on a leaderboard — GPT-4 at 87% on MMLU, Llama 3 at 79% on HellaSwag — comes from an evaluation harness running the same loop: build a prompt, call the model, compare the output to an answer key, aggregate accuracy. But there is a gap between "call the model and check the answer" and "produce a statistically valid, contamination-clean benchmark number."

The gap matters. A naive harness penalizes a model for answering "The answer is A" when the expected output is just "A". A harness without contamination detection celebrates models that memorized the test set. A harness without confidence intervals publishes leaderboard rankings where half the differences are statistically insignificant.

Until you've implemented the few-shot prompt format and watched exact-match fail on correct paraphrases, until you've scored choices by log-likelihood and seen format sensitivity disappear, until you've run bootstrap resampling on 100 examples and realized the CI is ±10 percentage points — you don't *know* why lm-evaluation-harness is 8,000 lines instead of 50.

This lab builds an evaluation harness in three stages: from naive few-shot exact-match in ~200 lines to contamination-aware evaluation with bootstrap confidence intervals in ~780 lines. Each stage fixes one measured problem in the previous one.

## What we're building

An LLM evaluation harness in Python, across three stages. Each stage adds one capability and addresses one real problem in evaluation methodology.

**We build:**
- Few-shot evaluator (v0): prepend n examples, call model, compare with exact-match or multiple-choice extraction
- Log-likelihood scoring (v1): score each choice as a continuation, eliminate format sensitivity, add contextual calibration
- Contamination detection and bootstrap CI (v2): 13-gram fingerprinting, paired bootstrap significance test
- FastAPI server: `POST /eval`, `GET /tasks`
- Spring Boot Java integration: `EvalClient` calling our eval server

**We explicitly skip (and why):**
- Real LLM calls: the harness is model-agnostic via `model_fn: Callable[[str], str]`; integrating GPT-4 or Llama is configuration, not architecture
- ROUGE/BERTScore: requires heavy dependencies and an embedding model; covered in "What the Toy Misses"
- Human evaluation: requires crowdsourcing infrastructure; covered in "What the Toy Misses"
- Full MMLU/HellaSwag datasets: 10-example lite versions illustrate the mechanics without requiring dataset downloads

## Mental model

An evaluation harness is a three-stage pipeline:

```
Benchmark dataset
  |
  v
[Prompt builder]  ← few-shot examples + test question
  |
  v
[Model]           ← Callable[[str], str]
  |
  v
[Scorer]          ← extract answer, compare to gold label
  |
  v
[Aggregator]      ← accuracy + confidence interval + contamination report
```

The naive version of this is 20 lines. The production version (lm-evaluation-harness) is 8,000 lines. The difference is in the details:

**Format sensitivity** — a model that would correctly pick A if scoring log-likelihoods answers "The answer is: option A" in free generation. Exact-match comparison to "A" scores this as wrong. Log-likelihood scoring fixes this by never generating text: compute `log P(A | context)`, `log P(B | context)`, etc. and pick the highest.

**Length bias** — a choice with more tokens has lower raw log probability (probabilities multiply; more terms means lower product). "All of the above" gets unfairly penalized vs "A". Normalized log-likelihood divides by token count to correct this.

**Prior bias** — "A" appears more frequently in training data than "D". The model assigns higher prior probability to A-continuations. Contextual calibration subtracts `log P(choice | empty context)` from every score, removing this prior.

**Statistical validity** — a single accuracy number from 100 examples has 95% CI of ±10pp. A model scoring 72% is indistinguishable from 62–82%. Bootstrap CI makes this explicit.

**Contamination** — if the model saw `"What is the capital of France? Paris"` during training, it's measuring memorization, not generalization. 13-gram fingerprinting detects this before you publish numbers.

The simplest possible evaluation loop. No log-likelihood, no calibration.

**The few-shot prompt format:**

For multiple-choice tasks (HellaSwag, MMLU):

```
Question: She poured water into the pot and placed it on the stove. She then
A. turned on the burner and waited for the water to boil.
B. placed the pot in the refrigerator.
C. filled the pot with sand.
D. put the pot on the floor.

[n_shot demos above — the test example below]

Question: The teacher wrote the problem on the blackboard. The students
A. left the classroom.
B. started talking loudly.
C. took out their notebooks and began working.
D. fell asleep.
Answer:
```

The model must complete "Answer:" with a single letter. The n_shot demonstrations show the expected format.

**Why n_shot matters:**

Zero-shot (no demonstrations) fails on novel formats. The model doesn't know it's supposed to respond with just a letter. With 5 demonstrations, it sees the pattern: question, choices, "Answer: X". This alone often improves accuracy by 10-15pp without any model changes.

**Exact match normalization:**

For open-ended tasks (TriviaQA), the expected answer is `"Paris"` and the model might generate `"Paris."` or `"The capital is Paris"`. Normalization handles the punctuation case but not the paraphrase case. Lowercase + strip punctuation is the minimum; production harnesses also strip articles ("the", "a"), normalize whitespace, and handle number formatting.

```python
def _normalize_string(s: str) -> str:
    s = s.strip().lower()
    s = s.translate(str.maketrans("", "", string.punctuation))
    s = re.sub(r"\s+", " ", s).strip()
    return s
```

**Multiple-choice extraction:**

The model might generate any of: "A", "A.", "Answer: A", "The answer is A", "Option A is correct". The extractor tries patterns in priority order:

```python
# Pattern 1: starts with letter
re.match(r"^([ABCD])[.):]?\s", completion)

# Pattern 2: "Answer: A"
re.search(r"[Aa]nswer[:\s]+([ABCD])", completion)

# Pattern 3: "is A" / "is (A)"
re.search(r"is\s+\(?([ABCD])\)?", completion)
```

If none match, return `""` — counted as wrong. This is where format sensitivity shows up: a model that would correctly identify the answer often generates a format the extractor misses. Log-likelihood scoring (v1) eliminates this entirely.

The `examples_evaluated` list lets you drill into which examples the model got wrong — essential for debugging prompts and spotting systematic failure patterns.

Log-likelihood scoring eliminates format sensitivity by never generating text.

**The core insight:**

Instead of asking the model to generate "A", "B", "C", or "D", score each choice as a continuation of the context:

```
context  = "Question: What is the capital of France?\nA. London\nB. Paris\nC. Berlin\nD. Rome\nAnswer:"
score(A) = log P("A. London" | context)
score(B) = log P("B. Paris"  | context)
score(C) = log P("C. Berlin" | context)
score(D) = log P("D. Rome"   | context)
pick = argmax(score(A), score(B), score(C), score(D))
```

The model never generates anything. It just assigns probabilities. A model that "knows" Paris is correct will assign higher probability to "B. Paris" as a continuation, regardless of how it would format a free-text response.

**`model_fn_logprob` protocol:**

```python
ModelFnLogprob = Callable[[str, str], float]
# Takes (context, continuation), returns log probability (always <= 0)
```

For a real LLM, this is typically:
1. Tokenize context + continuation together
2. Run a forward pass to get token log probabilities
3. Sum log probs for only the continuation tokens

Our implementation accepts any callable that satisfies this protocol — including mock functions for testing.

**Normalized log-likelihood:**

Raw log-likelihood biases toward shorter choices. "A" (1 token) has higher raw log-prob than "All of the above" (4 tokens) even if the model "prefers" the latter. Normalization divides by token count:

```python
def score_normalized(model_fn_logprob, context, continuation) -> float:
    raw = score_logprob(model_fn_logprob, context, continuation)
    token_count = max(1, len(continuation.split()))
    return raw / token_count
```

We approximate token count by whitespace-split — sufficient for short choices. Real harnesses use the model's tokenizer (byte-pair encoding token counts differ from word counts).

**Contextual calibration:**

The model has prior biases. "A" appears more in training text than "D". Even before reading the question, the model assigns higher probability to "A" as a next token. This biases multiple-choice accuracy toward options labeled "A".

Fix: compute `bias[choice] = log P(choice | "")` — the probability under empty context. Then:

```python
calibrated_score = log P(choice | context) - log P(choice | "")
```

After calibration, all choices start at equal expected score. Zhao et al. (2021) showed this improves accuracy by 5-15pp on some tasks, particularly when the answer distribution is uneven.

**EvalSuite:**

```python
suite = EvalSuite(evaluator, model_fn_logprob)
result = suite.run([hellaswag_lite, triviaqa_lite, mmlu_lite], n_shot=5)
print(result.macro_average)    # 0.72
print(result.per_task)         # {"HellaSwag-lite": 0.80, "TriviaQA-lite": 0.60, ...}
```

Macro-average weights all tasks equally regardless of example count. Real benchmarks (HELM) use both macro-average and size-weighted average.

Two critical correctness mechanisms: know what your model actually learned, and know whether your accuracy estimates are meaningful.

**13-gram contamination detection:**

The GPT-3 paper (Brown et al., 2020) used 13-gram overlap to check whether benchmark examples appeared in Common Crawl training data. 13 is a deliberate choice: long enough to be distinctive (a random 13-gram match is near-impossible), short enough to be practical (13-gram indexing over terabytes of text is feasible with sets).

```python
def _make_ngrams(text: str, n: int = 13) -> set[tuple[str, ...]]:
    words = text.lower().split()
    return {tuple(words[i:i + n]) for i in range(len(words) - n + 1)}

class ContaminationDetector:
    def add_training_shard(self, texts: list[str]) -> None:
        for text in texts:
            self._training_ngrams.update(_make_ngrams(text))

    def check_contamination(self, example_text: str) -> float:
        example_ngrams = _make_ngrams(example_text)
        found = example_ngrams.intersection(self._training_ngrams)
        return len(found) / len(example_ngrams)
```

Threshold: examples with more than 10% of their 13-grams in training data are flagged. This is conservative — 10% contamination for a 200-token example means 20+ overlapping 13-grams, strong evidence the example was seen during training.

**Bootstrap confidence intervals:**

The bootstrap is conceptually simple: resample with replacement from your results, compute accuracy for each resample, use the 2.5th and 97.5th percentiles as CI bounds.

```python
def bootstrap_ci(results: list[bool], n_bootstrap=1000, confidence=0.95) -> BootstrapCI:
    arr = np.array(results, dtype=float)
    boot_means = np.zeros(n_bootstrap)
    for i in range(n_bootstrap):
        sample = rng.choice(arr, size=len(arr), replace=True)
        boot_means[i] = sample.mean()
    alpha = 1.0 - confidence
    lower = np.percentile(boot_means, 100 * (alpha / 2))
    upper = np.percentile(boot_means, 100 * (1 - alpha / 2))
    return BootstrapCI(lower=lower, upper=upper, mean=arr.mean())
```

Width reference (at 95% confidence):
- n=100: ~±10pp (upper - lower ≈ 0.20)
- n=500: ~±4.5pp
- n=1000: ~±3pp
- n=14,000 (MMLU full): ~±0.8pp

**Paired bootstrap test:**

To test whether model A is significantly better than model B on the same examples:

```python
def paired_bootstrap_test(results_A, results_B, n_bootstrap=1000) -> float:
    # Resample pairs (A_i, B_i) together
    for _ in range(n_bootstrap):
        idx = rng.choice(n, size=n, replace=True)
        delta = arr_A[idx].mean() - arr_B[idx].mean()
        if delta <= 0:
            count_leq_zero += 1
    return count_leq_zero / n_bootstrap  # p_value
```

If p_value is less than 0.05, model A is significantly better at the 95% level. With 100 examples, you need at least a 7pp difference to achieve p < 0.05.

## Stress tests

**n_shot sensitivity**: run the same model on HellaSwag-lite with n_shot = 0, 1, 3, 5. Plot accuracy vs n_shot. You should see accuracy improve from 0-shot to 5-shot and then plateau. If accuracy drops with more demonstrations, your prompt builder is ordering examples in a way that confuses the model (most recent example should be closest to the test question).

**CI convergence**: run `bootstrap_ci` with n_bootstrap = 100, 500, 1000, 5000. For each, compute 10 CI estimates on the same results and measure how much the CI bounds vary. At n_bootstrap=100, CI bounds vary by several percentage points across runs. At n_bootstrap=1000, they stabilize within 0.5pp. This is why 1000 resamples is the standard — further resampling has diminishing returns.

**Contamination threshold sensitivity**: add a training shard that overlaps with exactly 5%, 10%, 15%, and 20% of an eval example's 13-grams. Verify that the 10% threshold correctly separates near-matches (expected noise) from true contamination (model has seen the example). Adjust the threshold and observe false positive rate.

**Paired test power**: generate pairs of results where model A is better by 1pp, 3pp, 5pp, 7pp, and 10pp, all at n=100. Run `paired_bootstrap_test` for each. You should see p_value drop below 0.05 at approximately 7pp with n=100. At n=1000, the threshold drops to approximately 2pp. This is the empirical power curve for your harness.

**Format sensitivity experiment**: implement a mock model that returns "The answer is A" instead of just "A". Run with v0 (exact-match extraction) and v1 (log-likelihood). The v0 harness incorrectly scores these as wrong when the correct answer is A; the v1 harness scores them correctly by not generating text at all.

### Part A — EvalClient: calling our eval server

`EvalClient.java` wraps our Python FastAPI server's `POST /eval` and `GET /tasks` endpoints using Spring Framework 6.1's `RestClient`. The entire client fits in under 60 lines.

```java
public class EvalClient {
    private final RestClient http;

    public EvalResponse evaluate(String taskName, int nShot, String model) {
        var body = new EvalRequest(taskName, nShot, model);
        return http.post()
            .uri("/eval")
            .contentType(MediaType.APPLICATION_JSON)
            .body(body)
            .retrieve()
            .body(EvalResponse.class);
    }

    public record EvalResponse(
        String task, int nShot, double accuracy,
        int numCorrect, int numTotal,
        double ci95Lower, double ci95Upper, String model
    ) {}
}
```

The response includes accuracy and 95% CI bounds — the client can directly print "72.0% [62.0%, 81.0%]" without any local statistics computation.

The demo application shows three things:

2. **Task listing** — `GET /tasks` returns all built-in tasks with name, metric, and example count
3. **CI interpretation** — prints a reference table of CI widths vs sample size

```java
var result = evalClient.evaluate("hellaswag-lite", 5, "mock-always-A");
System.out.printf("Accuracy: %.1f%% [%.1f%%, %.1f%%]%n",
    result.accuracy() * 100,
```

<WhatSurprisedMe insight="The confidence interval width surprised me. With 100 evaluation examples and 95% confidence, the bootstrap CI is plus-or-minus 10 percentage points. That means a model scoring 72% accuracy is statistically indistinguishable from one scoring 62-82%. The MMLU benchmark has 14,000 examples specifically to get CI width under 1pp. But most 'leaderboard' comparisons use 100-500 examples, making half the ranking differences statistically insignificant. The paired bootstrap test formalizes this: two models with 100 examples each need at least 7pp accuracy difference to achieve p less than 0.05 significance." />
