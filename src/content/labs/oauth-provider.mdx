## Why this lab exists

OAuth 2.0 is one of the most widely deployed protocols on the internet and one of the most frequently misunderstood. Developers implement clients every day — adding "Login with Google" or generating API tokens — but almost no one has built a provider. The result is a generation of engineers who know how to consume OAuth but cannot reason about what the server is actually enforcing and why.

The confusion shows up in production. Authorization codes replayed because single-use enforcement was skipped. Public clients without PKCE, leaving their flows vulnerable to code interception. OIDC implementations that issue ID tokens but never explain why they exist alongside access tokens. Services that "validate JWTs" by base64-decoding the payload but skipping signature verification.

Building a provider from scratch solves this. When you implement single-use code deletion, you understand why replay attacks work against naive providers. When you implement PKCE yourself — hash the verifier, compare against the stored challenge, reject mismatches — you internalize why PKCE matters for SPAs and mobile apps rather than memorizing a spec paragraph. When you add the OIDC discovery document, the difference between "can app X access resource Y" (OAuth) and "who is user Z" (OIDC) stops being abstract.

This lab follows the same progression that the OAuth 2.0 ecosystem went through historically: authorization code flow first (v0), PKCE to fix public client security (v1), then OIDC to add identity on top of authorization (v2). The Java co-primary implementation uses Spring Authorization Server — the production-grade equivalent — to show how the same concepts map to a hardened real-world library.

## What we're building

An OAuth 2.0 + OIDC provider in Go across three stages, with a Java co-primary implementation using Spring Authorization Server.

**We build:**
- Authorization code flow (v0): client registration, redirect URI validation, single-use codes with 10-minute expiry, HS256 JWT access tokens
- PKCE for public clients (v1): `code_challenge` stored at `/authorize`, `code_verifier` SHA-256 verified at `/token`, `/userinfo` endpoint protected by Bearer token
- OIDC layer (v2): `/.well-known/openid-configuration` discovery document, `/.well-known/jwks.json` with RSA public key, RS256 ID token with `sub`, `iss`, `aud`, `exp`, `iat`, `nonce`, `email` claims
- HTTP server with simulated login page (`cmd/server/main.go`): GET `/authorize`, POST `/token`, GET `/userinfo`, GET `/.well-known/openid-configuration`, GET `/.well-known/jwks.json`

**We explicitly skip (and why):**
- Refresh token rotation: requires persistent storage and token family tracking — see What the Toy Misses
- Dynamic client registration (RFC 7591): client credential storage and approval workflows significantly increase scope
- Consent screen: real scope approval UI is important but orthogonal to understanding the protocol mechanics
- Token revocation (RFC 7009): requires a blocklist store; the single-instance in-memory implementation would create a false sense of security

## Mental model

OAuth 2.0 answers one question: **can application X access resource Y on behalf of user Z?** OIDC answers a different question layered on top: **who is user Z?** Understanding why these are separate questions is the mental model for this entire lab.

### The authorization code flow, step by step

```
Browser                  Client App               OAuth Provider
   │                         │                         │
   │  1. User clicks "Login" │                         │
   │─────────────────────────▶                         │
   │                         │  2. GET /authorize      │
   │                         │     client_id=X         │
   │                         │     redirect_uri=...    │
   │                         │     scope=openid email  │
   │                         │     state=csrf_token    │
   │                         │─────────────────────────▶
   │                         │                         │
   │  3. Redirect to login   │                         │
   │◀────────────────────────────────────────────────── │
   │                         │                         │
   │  4. User enters creds   │                         │
   │────────────────────────────────────────────────────▶
   │                         │                         │
   │  5. Redirect: ?code=ABC&state=csrf_token           │
   │◀────────────────────────────────────────────────── │
   │                         │                         │
   │  6. Deliver code to     │                         │
   │     client via URL      │                         │
   │─────────────────────────▶                         │
   │                         │  7. POST /token         │
   │                         │     code=ABC            │
   │                         │     client_secret=...   │
   │                         │─────────────────────────▶
   │                         │                         │
   │                         │  8. {access_token, ...} │
   │                         │◀─────────────────────── │
   │                         │                         │
   │                         │  9. GET /api/resource   │
   │                         │  Authorization: Bearer  │
   │                         │─────────────────────────▶ (resource server)
```

**Why the code exists at all:** the access token cannot travel through the browser. Browser history, Referer headers, and server logs all capture URL parameters. If the access token appeared in step 5 as a URL parameter (the legacy implicit flow), it would be logged everywhere. The authorization code is a short-lived, single-use intermediary. It can safely travel through the browser because it is useless without the client secret (step 7) — and the client secret is never in the browser.

**Why the state parameter matters:** step 5 delivers the code via URL redirect to the client app. Without the state check, an attacker can craft a redirect URL with their own authorization code and trick the victim's browser into delivering it to the client — the client would then exchange the attacker's code and bind the attacker's account to the victim's session. The state parameter is a CSRF token for the OAuth flow itself.

### What PKCE adds for public clients

A SPA running in a browser cannot keep a client secret. The secret would be in the JavaScript source — visible to anyone who opens DevTools. Without a client secret, step 7 is unauthenticated: anyone who intercepts the code (via a malicious redirect URI registered with a different app, or browser history theft) can exchange it.

PKCE (RFC 7636) solves this with a cryptographic proof:

```
Before /authorize:
  Generate random code_verifier  (43-128 chars, high entropy)
  code_challenge = BASE64URL(SHA256(code_verifier))

At /authorize:
  Send code_challenge, code_challenge_method=S256
  Provider stores code_challenge with the authorization code

At /token:
  Send code_verifier
  Provider recomputes BASE64URL(SHA256(code_verifier))
  Compares against stored code_challenge
  Match → legitimate client. Mismatch → reject.
```

An attacker who intercepts the code at step 5 does not have the `code_verifier` — it was never transmitted. They cannot compute the verifier from the challenge because SHA-256 is a one-way function. The `/token` exchange fails with a PKCE mismatch error. Attack blocked.

### ID token vs access token

After the OIDC layer is added, `/token` returns two JWTs:

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiJ9...",  // HS256, for resource servers
  "id_token":     "eyJhbGciOiJSUzI1NiJ9...",  // RS256, for the client app only
  "token_type":   "Bearer",
  "expires_in":   3600
}
```

The **access token** proves authorization: "this user authorized scope X." Resource servers verify it to decide whether to serve a request. They do not care who the user is — only that the scope is correct.

The **ID token** proves identity: "the authenticated user is alice@example.com with sub=user-1." The client app uses it to display the user's name, pre-fill their profile, or look them up in its own database. Resource servers should never receive the ID token — it is for the client app that initiated the login.

Both are JWTs. Access tokens use HS256 (fast, symmetric — only the issuer needs to verify them internally). ID tokens use RS256 (asymmetric — any service can verify them using the public key from `/.well-known/jwks.json` without sharing a secret).

The first thing to get right is the exact-match redirect URI validation. This is not a pedantic spec requirement — it is the primary defense against authorization code injection. If the provider matches by prefix or ignores the redirect_uri at `/token`, an attacker can register a client with a URI that starts with the same prefix as the legitimate client and redirect authorization codes to themselves.

```go
// hasRedirectURI checks EXACTLY — never by prefix
func (c *Client) hasRedirectURI(redirectURI string) bool {
    for _, u := range c.RedirectURIs {
        if u == redirectURI {
            return true
        }
    }
    return false
}
```

The single-use enforcement is a delete-on-first-access pattern. The code is removed from the map before any validation occurs. If the client_id doesn't match or the secret is wrong, the code is already gone. This prevents replay attacks where an attacker observes a failed exchange attempt and then retries with the correct parameters.

```go
// Lock, look up, and delete atomically before any validation
p.mu.Lock()
authCode, ok := p.codes[params.Code]
if ok {
    delete(p.codes, params.Code)
}
p.mu.Unlock()

if !ok {
    return nil, errors.New("oauth: invalid or already-used authorization_code")
}
```

The HS256 access token uses the same signing approach as the JWT Library lab: base64url(header) + "." + base64url(payload) as the signing input, HMAC-SHA256 as the signature. Verification uses `hmac.Equal` for constant-time comparison — the same reason as in the JWT Library lab. Claims: `sub`, `iss`, `aud`, `iat`, `exp`, `scope`.

The `state` parameter is echoed back in the redirect URL unchanged. The client app compares the returned state against the value it sent. If they differ, the client must abort — something tampered with the flow. This is the OAuth-specific CSRF defense; it is the client's responsibility to enforce it, not the provider's.

The PKCE verifier check is four lines that prevent a category of attack that the original OAuth 2.0 spec did not anticipate:

```go
if authCode.CodeChallenge != "" {
    if params.CodeVerifier == "" {
        return nil, errors.New("oauth: code_verifier required (PKCE was used)")
    }
    digest := sha256.Sum256([]byte(params.CodeVerifier))
    computed := b64url(digest[:])
    if computed != authCode.CodeChallenge {
        return nil, errors.New("oauth: code_verifier does not match code_challenge")
    }
}
```

The `b64url` function uses `base64.RawURLEncoding` — base64 with URL-safe alphabet (`-` and `_` instead of `+` and `/`) and no padding characters. This is the encoding specified by RFC 7636 Appendix A. The difference between `RawURLEncoding` and `StdEncoding` is why PKCE implementations sometimes produce verification failures: `+` vs `-` and `==` padding vs no padding.

**Why `code_challenge_method=plain` is rejected:** plain PKCE sends the verifier itself as the challenge. An attacker who sees the challenge (it's in the `/authorize` URL, which may appear in browser history or Referer headers) already has the verifier. S256 defeats this — SHA-256 is irreversible, so the challenge reveals nothing about the verifier.

The `/userinfo` endpoint is straightforward: extract the Bearer token from the `Authorization` header, verify it with `VerifyAccessToken` (same HMAC check as Exchange), look up the user by the `sub` claim, return the profile claims. The important design decision is that `/userinfo` shares the same token verification path as resource servers would. There is no separate "userinfo token" — the same access token serves all purposes.

Public clients (`Public: true`) must use PKCE at `/authorize`. This is enforced by checking `client.Public` before generating the code:

```go
if client.Public {
    if params.CodeChallenge == "" {
        return "", errors.New("oauth: PKCE required for public clients")
    }
    if params.CodeChallengeMethod != "S256" {
        return "", errors.New("oauth: only code_challenge_method=S256 is supported")
    }
}
```

Public clients omit the `client_secret` at `/token` (the field is simply ignored if `client.Public` is true). PKCE is their authentication mechanism for the token exchange.

The OIDC discovery document is the configuration that makes OAuth providers composable. Instead of hardcoding endpoint URLs, clients fetch `/.well-known/openid-configuration` at startup and cache it. Spring Security's OAuth2 client does this automatically when you configure `issuer-uri`. The document is JSON with required fields defined by OIDC Discovery 1.0 §3:

```go
type OIDCDiscovery struct {
    Issuer                           string   `json:"issuer"`
    AuthorizationEndpoint            string   `json:"authorization_endpoint"`
    TokenEndpoint                    string   `json:"token_endpoint"`
    UserinfoEndpoint                 string   `json:"userinfo_endpoint"`
    JWKsURI                          string   `json:"jwks_uri"`
    ResponseTypesSupported           []string `json:"response_types_supported"`
    IDTokenSigningAlgValuesSupported []string `json:"id_token_signing_alg_values_supported"`
    ScopesSupported                  []string `json:"scopes_supported"`
    CodeChallengeMethodsSupported    []string `json:"code_challenge_methods_supported"`
    // ... additional optional fields
}
```

The JWKS endpoint distributes the RSA public key so any service can verify ID token signatures without a shared secret. The key is represented as a JSON Web Key (RFC 7517) with the modulus `n` and exponent `e` as base64url-encoded big-endian bytes:

```go
func (p *Provider) JWKS() *JWKSet {
    pub := &p.rsaKey.PublicKey
    nBytes := pub.N.Bytes()
    eBig := big.NewInt(int64(pub.E))
    eBytes := eBig.Bytes()

    return &JWKSet{Keys: []JWK{{
        Kty: "RSA",
        Use: "sig",
        Kid: p.rsaKeyID,  // matches the "kid" in ID token headers
        Alg: "RS256",
        N:   b64url(nBytes),
        E:   b64url(eBytes), // always "AQAB" for e=65537
    }}}
}
```

`e = "AQAB"` decodes to `[0x01, 0x00, 0x01]` = 65537. This is the RSA public exponent used by virtually every RSA key generated in the last 20 years. It appears in every JWKS endpoint for every OAuth 2.0 provider in the world.

The ID token is a standard RS256 JWT. The critical difference from access tokens: the `kid` field in the header links the token to the key in the JWKS. When a service receives an ID token and wants to verify it, the lookup is:

1. Parse the token header, extract `kid`.
2. Fetch `/.well-known/jwks.json`, find the key with matching `kid`.
3. Reconstruct the RSA public key from `n` and `e`.
4. Verify the RS256 signature.

The `nonce` claim is the OIDC replay prevention mechanism for the client app. The client generates a random nonce at `/authorize`, stores it in session storage, and after receiving the ID token, verifies that the nonce in the token matches what it sent. An attacker who replays a captured ID token cannot satisfy the nonce check.

At 1,000 logins per second, HS256 token issuance consumes roughly 2 CPU-seconds per wall-clock second — still comfortable on a single core. The RS256 ID token path at 1,000 logins/sec would consume 3.5 CPU-seconds, approaching single-core saturation. Production OAuth providers horizontally scale the token endpoint and pre-generate RSA key material at startup to avoid key generation latency on hot paths.

The 0.1ms for JWKS is important: it should be cached for 24 hours (`Cache-Control: max-age=86400`) in production. Clients fetching it on every token verification would add unnecessary load for a document that changes only during key rotation.

Two specific failure modes worth testing manually:

**Code replay attack:** issue a code, exchange it successfully, then immediately replay the same code. The second attempt must return `invalid or already-used authorization_code`. This is enforced by the delete-before-validate pattern in `Exchange()` — the code is gone from the map before any other check runs.

**PKCE verifier brute force:** submit the correct code with a wrong `code_verifier`. The provider computes `SHA256(wrong_verifier)` and compares against the stored challenge. The comparison is string equality, not constant-time, so there is no timing leak to exploit. Even if there were, SHA-256 output is 32 bytes — an attacker cannot narrow down the verifier from timing information on a 256-bit value.

### Go reference — what we built

The Go implementation is a from-scratch OAuth 2.0 + OIDC provider in ~900 lines. No external dependencies beyond the standard library. Every operation is explicit: single-use code enforcement with a delete-before-validate pattern, PKCE SHA-256 verify, HS256 and RS256 JWT construction, JWKS endpoint with RSA public key serialization, and OIDC discovery document generation.

The key design decisions: redirect URIs are exact-match only. Codes are deleted before validation. Public clients require PKCE. The HMAC secret and RSA key are set at construction time and never change. The provider is a single struct with a mutex — simple to reason about, not production-scale.

### Java co-primary — Spring Authorization Server

Spring Authorization Server (spring-authorization-server) is the Spring ecosystem's production-grade OAuth 2.0 + OIDC provider. It is the successor to the deprecated Spring Security OAuth2 project and was built from the ground up for Spring Boot 3.x and Jakarta EE. Every concept from our Go implementation has a direct counterpart.

#### Registered clients: RegisteredClientRepository

Our Go `Client` struct becomes Spring's `RegisteredClient`. The repository holds all registered clients — equivalent to our `Provider.clients` map, but typically backed by a database:

```java
@Bean
public RegisteredClientRepository registeredClientRepository() {
    // Confidential client — equivalent to Go Client{Public: false}
    RegisteredClient confidentialClient = RegisteredClient
        .withId(UUID.randomUUID().toString())
        .clientId("demo-client")
        .clientSecret("{noop}demo-secret")          // {noop} = plaintext for demo
        .clientAuthenticationMethod(ClientAuthenticationMethod.CLIENT_SECRET_POST)
        .authorizationGrantType(AuthorizationGrantType.AUTHORIZATION_CODE)
        .redirectUri("http://localhost:8080/callback")
        .scope(OidcScopes.OPENID)
        .scope(OidcScopes.EMAIL)
        .clientSettings(ClientSettings.builder()
            .requireProofKey(false)    // PKCE optional for confidential clients
            .requireAuthorizationConsent(true)
            .build())
        .build();

    // Public client (SPA) — equivalent to Go Client{Public: true}
    RegisteredClient publicClient = RegisteredClient
        .withId(UUID.randomUUID().toString())
        .clientId("demo-spa")
        .clientAuthenticationMethod(ClientAuthenticationMethod.NONE) // no secret
        .authorizationGrantType(AuthorizationGrantType.AUTHORIZATION_CODE)
        .redirectUri("http://localhost:3000/callback")
        .scope(OidcScopes.OPENID)
        .clientSettings(ClientSettings.builder()
            .requireProofKey(true)     // PKCE required — same as Go Public:true
            .build())
        .build();

    return new InMemoryRegisteredClientRepository(confidentialClient, publicClient);
}
```

`requireProofKey(true)` is the exact equivalent of our `if client.Public { if params.CodeChallenge == "" { return error } }` check. Spring enforces it in `PkceTokenGrantAuthenticationProvider` — the same three-step logic: require challenge at authorize, store challenge with code, verify verifier at token exchange.

#### Security configuration: applyDefaultSecurity

The most important Spring Authorization Server method is `OAuth2AuthorizationServerConfiguration.applyDefaultSecurity()`. It wires all the OAuth 2.0 and OIDC endpoints in a single call:

```java
@Configuration
@EnableWebSecurity
public class AuthorizationServerConfig {

    @Bean
    @Order(1)
    public SecurityFilterChain authorizationServerSecurityFilterChain(HttpSecurity http)
            throws Exception {
        // applyDefaultSecurity registers:
        //   GET  /authorize                        — authorization endpoint
        //   POST /token                            — token endpoint
        //   GET  /userinfo                         — UserInfo endpoint (OIDC)
        //   GET  /.well-known/openid-configuration — OIDC discovery
        //   GET  /.well-known/jwks.json            — JWKS endpoint
        //   POST /oauth2/revoke                    — token revocation (RFC 7009)
        //   POST /oauth2/introspect                — token introspection (RFC 7662)
        OAuth2AuthorizationServerConfiguration.applyDefaultSecurity(http);

        http
            .getConfigurer(OidcConfigurer.class)
            .enable();  // enable OIDC UserInfo and discovery endpoints

        http
            .exceptionHandling(e -> e
                .defaultAuthenticationEntryPointFor(
                    new LoginUrlAuthenticationEntryPoint("/login"),
                    new MediaTypeRequestMatcher(MediaType.TEXT_HTML)
                )
            )
            .oauth2ResourceServer(r -> r.jwt(Customizer.withDefaults()));

        return http.build();
    }

    @Bean
    @Order(2)
    public SecurityFilterChain defaultSecurityFilterChain(HttpSecurity http)
            throws Exception {
        http
            .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
            .formLogin(Customizer.withDefaults()); // the login page our Go demo has manually
        return http.build();
    }
}
```

The difference from our Go implementation: Spring handles PKCE enforcement, single-use code deletion, redirect URI validation, and state parameter handling automatically. Our Go code made all of these explicit — Spring encapsulates them behind `applyDefaultSecurity`.

#### PKCE in Spring: PkceTokenGrantAuthenticationProvider

Spring's PKCE implementation lives in `PkceTokenGrantAuthenticationProvider`. When you call `requireProofKey(true)` on a client, this provider is activated automatically. Its logic mirrors our Go implementation exactly:

```java
// Spring's internal PkceTokenGrantAuthenticationProvider (simplified for clarity)
// This runs at POST /token when the client has requireProofKey=true

private void validateCodeVerifier(
        OAuth2Authorization authorization,
        OAuth2AuthorizationCodeAuthenticationToken authenticationToken) {

    // Retrieve the stored code_challenge from the authorization
    String codeChallenge = authorization.getAttribute(
        PkceParameterNames.CODE_CHALLENGE);
    String codeChallengeMethod = authorization.getAttribute(
        PkceParameterNames.CODE_CHALLENGE_METHOD);

    // Get the code_verifier from the /token request
    String codeVerifier = authenticationToken.getAdditionalParameters()
        .get(PkceParameterNames.CODE_VERIFIER).toString();

    // Compute BASE64URL(SHA256(code_verifier)) — same as our Go code
    MessageDigest md = MessageDigest.getInstance("SHA-256");
    byte[] digest = md.digest(codeVerifier.getBytes(StandardCharsets.US_ASCII));
    String computed = Base64.getUrlEncoder().withoutPadding()
        .encodeToString(digest);

    // Compare against the stored challenge
    if (!computed.equals(codeChallenge)) {
        throw new OAuth2AuthenticationException(OAuth2ErrorCodes.INVALID_GRANT);
    }
}
```

This is functionally identical to our Go implementation:
```go
digest := sha256.Sum256([]byte(params.CodeVerifier))
computed := b64url(digest[:])
if computed != authCode.CodeChallenge {
    return nil, errors.New("oauth: code_verifier does not match code_challenge")
}
```

The algorithm is the same standard (RFC 7636). The implementation is the same computation. Spring just calls it from a provider that is registered into the security filter chain, where our Go code calls it inline in `Exchange()`.

#### RSA key configuration and JWKS

Spring Authorization Server generates and manages the RSA key pair for ID token signing through `JWKSource`. The JWKS endpoint is served automatically once the key source is configured:

```java
@Bean
public JWKSource jwkSource() {
    // Generate RSA key pair — equivalent to our rsa.GenerateKey(rand.Reader, 2048)
    KeyPairGenerator keyPairGenerator = KeyPairGenerator.getInstance("RSA");
    keyPairGenerator.initialize(2048);
    KeyPair keyPair = keyPairGenerator.generateKeyPair();

    RSAPublicKey publicKey = (RSAPublicKey) keyPair.getPublic();
    RSAPrivateKey privateKey = (RSAPrivateKey) keyPair.getPrivate();

    // Build the JWK with kid — equivalent to our Provider.rsaKeyID
    RSAKey rsaKey = new RSAKey.Builder(publicKey)
        .privateKey(privateKey)
        .keyID(UUID.randomUUID().toString())  // the "kid" field in JWKS and token headers
        .build();

    JWKSet jwkSet = new JWKSet(rsaKey);
    // ImmutableJWKSet serves the public key at /.well-known/jwks.json
    // The private key never leaves the JWKSource
    return new ImmutableJWKSet<>(jwkSet);
}

@Bean
public JwtDecoder jwtDecoder(JWKSource jwkSource) {
    return OAuth2AuthorizationServerConfiguration.jwtDecoder(jwkSource);
}
```

The `/.well-known/jwks.json` response matches our Go JWKS output format exactly — same RFC 7517 structure with `kty`, `use`, `kid`, `alg`, `n`, `e` fields. Any client that can verify our Go provider's ID tokens can verify Spring's, because the JWK format is a standard.

#### nimbus-jose-jwt for resource server verification

On the resource server side (a separate microservice that receives access tokens), Spring uses nimbus-jose-jwt to verify tokens against the JWKS endpoint. This is the same library as the JWT Library lab — the resource server configuration is identical whether the issuer is our Go provider or Spring Authorization Server:

```java
// Resource server configuration — works against any OAuth 2.0 / OIDC provider
@Bean
public SecurityFilterChain resourceServerFilterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
        .oauth2ResourceServer(oauth2 -> oauth2
            .jwt(jwt -> jwt
                // Point to our Go provider's discovery document
                // Spring fetches /.well-known/openid-configuration at startup,
                // discovers the jwks_uri, and caches the public key
                .jwkSetUri("http://localhost:9000/.well-known/jwks.json")
            )
        );
    return http.build();
}
```

Or with auto-discovery from the issuer URL:

```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: http://localhost:9000
```

Spring fetches `http://localhost:9000/.well-known/openid-configuration`, parses the `jwks_uri` field, fetches the JWKS, and uses the public key to verify all incoming Bearer tokens. This works against our Go provider without any modification — the discovery document and JWKS format are standards.

#### The shared lesson

Every concept in our Go provider has a direct counterpart in Spring Authorization Server:

| Go Provider | Spring Authorization Server |
|---|---|
| `Client{Public: true}` | `ClientSettings.requireProofKey(true)` |
| `delete(p.codes, code)` before validate | `OAuth2AuthorizationService.save()` with consumed flag |
| `client.hasRedirectURI()` exact match | `OAuth2AuthorizationCodeRequestAuthenticationValidator` |
| `issueIDToken()` RS256 JWT | `OidcIdTokenCustomizer` via `JWKSource` |
| `JWKS()` endpoint | `NimbusJwkSetEndpointFilter` |
| `DiscoveryDocument()` | `OidcProviderConfigurationEndpointFilter` |

Spring Authorization Server is not magic. It is the same protocol, the same RFCs, implemented with production-grade error handling, persistent storage adapters, and Spring's dependency injection model. Building the Go version first means reading the Spring source code becomes straightforward: you know what each class is doing because you implemented the same logic yourself.

## What surprised me

<WhatSurprisedMe insight="The authorization code expires in 10 minutes but is typically used within 2 seconds. The 10-minute window exists entirely for network retries — 99.9% of codes are exchanged in under 500ms. I expected the 10-minute expiry to be a meaningful security window, but it is almost never reached in practice. The real security property of the authorization code is not the expiry — it is the single-use enforcement. A code that is valid for 10 minutes and can be used unlimited times is dramatically less secure than a code valid for 10 minutes that is deleted on first use. The expiry is a fallback for codes that are never exchanged (abandoned flows, network failures). The single-use deletion is the actual defense. I had assumed the expiry was the primary security mechanism until I traced through what happens when a code is intercepted: the attacker races to exchange it before the legitimate client does, which can happen in under 100ms — the 10-minute window is irrelevant to that attack scenario." />
