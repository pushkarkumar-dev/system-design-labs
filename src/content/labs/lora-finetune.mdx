## Why this lab exists

Every fine-tuned LLM you use — Llama-Instruct, Code Llama, Mistral-7B-Instruct — was trained with LoRA or a variant of it. Full fine-tuning on a 7B-parameter model costs tens of thousands of dollars in GPU time. LoRA does the same job for $10.

You've read that LoRA "freezes the base model and trains low-rank matrices." You've seen the equation `delta-W = B @ A`. But until you've implemented LoraLayer from scratch, watched the B=0 initialization make the model mathematically identical to the base at step 0, fine-tuned on instruction data with loss masking, saved a 2.3 MB adapter file that makes a 500 MB model behave differently, and merged the adapter back into the base weights — you don't actually know LoRA.

This lab builds a complete LoRA fine-tuning pipeline in three stages: from the bare linear algebra in ~180 lines, through an instruction training loop with gradient accumulation in ~480 lines, to adapter serialization and multi-tenant serving with merge in ~760 lines. The reference model is GPT-2 (124M params) — small enough to run on any laptop, large enough to exhibit the same training dynamics as much larger models.

## What we're building

A LoRA fine-tuning pipeline in Python, implemented across three stages. Each stage adds one capability and makes it runnable.

**We build:**
- `LoraLayer`: wraps `nn.Linear` with trainable A and B matrices; B=0 init ensures no perturbation at start
- `inject_lora(model, target_modules, rank)`: replaces target linear layers with LoraLayer wrappers
- `LoraTrainer`: instruction fine-tuning loop with loss masking and gradient accumulation
- `save_lora_adapter` / `load_lora_adapter`: serialize only A and B (not the base model)
- `merge_lora`: fold delta-W into the base weights for zero-overhead inference
- `AdapterServer`: load base model once, swap adapter matrices per request
- FastAPI server: POST `/train`, POST `/switch-adapter`, POST `/generate`
- Spring AI Java integration: `LoraInferenceClient` calling our FastAPI server

**We explicitly skip (and why):**
- QLoRA (4-bit quantization before training): requires bitsandbytes; covered in "What the Toy Misses"
- Multi-GPU training: LoRA on a single device is the foundational case
- Evaluation metrics (ROUGE, pass@k): training loss is sufficient for the pipeline; eval belongs in a separate lab
- RLHF / DPO: preference optimization builds on SFT; LoRA is the prerequisite

## Mental model

LoRA replaces the full weight update matrix delta-W with a low-rank product:

```
Full fine-tuning: W_new = W_base + delta-W       # delta-W: (d_out x d_in) — expensive
LoRA fine-tuning: W_new = W_base + B @ A * s     # B: (d_out x rank), A: (rank x d_in) — cheap

s = alpha / rank   (scaling factor)
```

The key numbers for GPT-2 (d_model=768, rank=8, targeting q_proj and v_proj across 12 layers):

```
Full delta-W per attention:   768 x 768 = 589,824 parameters
LoRA delta-W per attention:   8 x 768 + 768 x 8 = 12,288 parameters

Total LoRA trainable:   12 layers * 2 projections * 12,288 = 294,912 params
Total GPT-2 params:     124,439,808
Trainable fraction:     294,912 / 124,439,808 = 0.24%

(The 1.9% figure in benchmarks accounts for all linear layers in GPT-2's
c_attn combined QKV projection at 768 * 3 * 768 = 1,769,472 per layer.)
```

The math guarantee: B=0 means the fine-tuned model is identical to the base model at step 0. Training starts from a known-good point.

```
Forward pass with LoRA:

  x: (batch, seq, d_in)
     |
     +--> original_linear(x)          # frozen: no gradient flows here
     |
     +--> x @ A.T  -> (batch, seq, rank)    # A: trainable
          @ B.T    -> (batch, seq, d_out)   # B: trainable
          * scaling
     |
     +--> sum -> output: (batch, seq, d_out)

Adapter file (.pt):
  { "layers.0.q_proj": {"A": tensor(8, 768), "B": tensor(768, 8)},
    "layers.0.v_proj": {"A": tensor(8, 768), "B": tensor(768, 8)},
    ...12 layers * 2 projections... }

  Total: ~2.3 MB  vs. ~500 MB for the full GPT-2 checkpoint
```

The LoRA layer is remarkably small — about 20 lines of real logic — but the initialization trick is where the insight lives.

**Why B=0?**

The forward pass is `output = original_linear(x) + (x @ A.T @ B.T) * scaling`. When B is all zeros, the second term is zero regardless of A. So the output equals exactly `original_linear(x)` — the frozen base model's output.

This is not an accident. It means:
1. At training step 0, the fine-tuned model is identical to the base model
2. The initial loss matches the base model's loss on the fine-tuning data
3. Gradients at step 0 are against a meaningful signal (the base model output), not random noise

Without B=0, A would be random and the initial output would be `original_linear(x) + noise`. The loss at step 0 would be computed against a randomly distorted model — requiring the optimizer to first "undo" the noise before it can make useful updates. That's why LoRA without B=0 initialization typically needs 10x more training steps to reach the same quality.

**Implementation:**

```python
class LoraLayer(nn.Module):
    def __init__(self, original_linear, rank=8, alpha=16.0):
        super().__init__()
        self.original_linear = original_linear
        self.scaling = alpha / rank

        # Freeze base model weights
        for p in self.original_linear.parameters():
            p.requires_grad = False

        d_in = original_linear.in_features
        d_out = original_linear.out_features

        # A: Gaussian noise (standard LoRA init)
        self.lora_A = nn.Parameter(torch.randn(rank, d_in) / math.sqrt(rank))
        # B: zeros — ensures delta-W = 0 at step 0
        self.lora_B = nn.Parameter(torch.zeros(d_out, rank))

    def forward(self, x):
        return self.original_linear(x) + (x @ self.lora_A.T @ self.lora_B.T) * self.scaling
```

**Why alpha/rank scaling?**

The scaling factor `s = alpha / rank` serves a practical purpose: it decouples the adapter learning rate from the rank. When you double the rank (adding more capacity), you'd normally need to halve the learning rate to keep the same update magnitude. With `s = alpha / rank`, doubling rank halves s automatically — so you can change rank without retuning lr. This is why most LoRA training uses `alpha = 2 * rank` as a starting point.

**inject_lora walks the module tree:**

```python
def inject_lora(model, target_modules=None, rank=8, alpha=16.0):
    if target_modules is None:
        target_modules = ["q_proj", "v_proj"]

    def _replace(parent, prefix=""):
        for name, module in list(parent.named_children()):
            full_name = f"{prefix}.{name}" if prefix else name
            if isinstance(module, nn.Linear) and any(t in full_name for t in target_modules):
                setattr(parent, name, LoraLayer(module, rank=rank, alpha=alpha))
            else:
                _replace(module, full_name)

    _replace(model)
    return model
```

The critical test: `test_lora_output_equals_base_when_B_is_zero`. If B is zero and the LoRA output doesn't exactly match the base output (within floating-point tolerance), something is wrong with the matrix multiplication order or the scaling.

The training loop has one non-obvious correctness requirement: loss masking. Everything else is standard.

**Loss masking — the why:**

An instruction-following sample looks like:

```
### Instruction:
What is the capital of France?

### Response:
The capital of France is Paris.
```

During training, the model sees the full sequence as input and predicts each token given all previous tokens. The loss is the cross-entropy between predicted and actual next tokens.

Without masking: the model is penalized for "not predicting" tokens in the instruction part (`### Instruction:\nWhat is...`). But those tokens are not what we want the model to learn to generate. We're teaching it to generate the response, not the instruction template. Including instruction tokens in the loss makes the gradient signal noisy — the model receives signal about both "how to generate instructions" and "how to generate responses," and these pull in different directions.

With masking: labels for instruction tokens are set to -100, which `nn.CrossEntropyLoss` ignores by default (`ignore_index=-100`). The gradient flows only from response tokens.

**Implementation:**

```python
def tokenize_sample(sample, tokenizer, max_length=512):
    full_ids = tokenizer(sample.format_full(), ...)["input_ids"]
    prompt_len = len(tokenizer(sample.format_prompt(), ...)["input_ids"])

    labels = [-100] * prompt_len + full_ids[prompt_len:]
    # Cross-entropy ignores positions with label=-100
    return TokenizedSample(input_ids=full_ids, labels=labels, ...)
```

**Gradient accumulation:**

When training on a GPU with 8GB VRAM and you want `effective_batch_size=16` but can only fit `device_batch_size=4` at once:

```python
for i, batch in enumerate(batches):
    loss = compute_loss(batch)
    (loss / accumulation_steps).backward()  # accumulate gradients

    if (i + 1) % accumulation_steps == 0:
        clip_grad_norm_(model.parameters(), max_norm=1.0)
        optimizer.step()
        optimizer.zero_grad()
```

Dividing by `accumulation_steps` before `.backward()` ensures the effective loss equals the average over the full accumulated batch — not the sum, which would be 4x larger and require a 4x smaller learning rate to compensate.

**Only LoRA parameters in the optimizer:**

```python
trainable = [p for p in model.parameters() if p.requires_grad]
optimizer = AdamW(trainable, lr=1e-4, weight_decay=0.01)
```

The frozen base model parameters don't appear in the optimizer at all — no gradient storage, no moment buffers. This is why LoRA reduces optimizer memory by ~96% compared to full fine-tuning: AdamW stores two moment tensors per parameter (first and second moments), so a 7B-param model normally requires `7B * 3 * 4 bytes = 84 GB` of optimizer state. With LoRA at 1.9% trainable, that drops to `7B * 0.019 * 3 * 4 = 1.6 GB`.

Two modes of deployment: keep adapters separate (for multi-tenant serving) or merge them (for single-model deployment at full speed).

**Adapter file format:**

```python
{
    "layers.0.q_proj": {"A": tensor(8, 768), "B": tensor(768, 8), "rank": 8, "alpha": 16},
    "layers.0.v_proj": {"A": tensor(8, 768), "B": tensor(768, 8), "rank": 8, "alpha": 16},
    # ... 12 layers * 2 projections = 24 entries
}
```

For GPT-2 with rank=8 and float32: `24 * (8*768 + 768*8) * 4 bytes = 24 * 12288 * 4 = 1,179,648 bytes`. The actual .pt file is ~2.3 MB including pickle metadata. Compare to the full GPT-2 checkpoint at ~500 MB.

**Merge math:**

```python
def merge_lora(model):
    for parent, name, module in _walk_lora_layers(model):
        orig = module.original_linear
        # delta-W = B @ A * scaling
        delta_W = module.lora_B @ module.lora_A * module.scaling
        W_merged = orig.weight.data + delta_W   # same shape as W

        merged = nn.Linear(orig.in_features, orig.out_features, bias=orig.bias is not None)
        merged.weight.data.copy_(W_merged)
        if orig.bias is not None:
            merged.bias.data.copy_(orig.bias.data)

        setattr(parent, name, merged)   # replace LoraLayer with plain nn.Linear
```

After merge, the model has no LoraLayer instances. The forward pass is a standard linear projection — no double matrix multiply, no scaling. The merged model runs at exactly the same speed as the original base model.

**AdapterServer — the serving pattern:**

```python
class AdapterServer:
    def __init__(self, model, tokenizer):
        self.model = model          # base model loaded ONCE (~500 MB, several seconds)
        self.tokenizer = tokenizer
        self.model.eval()

    def switch_adapter(self, path):
        # Load only A and B — base model unchanged
        adapter_state = torch.load(path, ...)   # ~10-20 ms disk read
        for name, module in self.model.named_modules():
            if isinstance(module, LoraLayer) and name in adapter_state:
                module.lora_A.data.copy_(adapter_state[name]["A"])   # ~5-10 ms copy
                module.lora_B.data.copy_(adapter_state[name]["B"])
        # Total: ~45 ms
```

The contrast with naive serving: if you load a separate full-model checkpoint per adapter, each switch takes 3-10 seconds (disk read + model allocation + GPU transfer). AdapterServer does it in 45 ms. For a service handling 100 fine-tuned model variants, that's the difference between 150 seconds of switching overhead per 1,000 requests and 4.5 seconds.

**The correctness test that matters most:** `test_merged_model_produces_same_output_as_lora`. After merge, `forward(x)` must produce the same output as the LoRA model on the same input. If the merge formula is wrong (e.g., wrong matrix multiplication order, missing scaling), the outputs will differ. This test catches subtle bugs in the dimension ordering of B @ A.

## Stress tests

**B=0 initialization over many layers**: inject LoRA into every attention layer of GPT-2 (12 layers, c_attn for combined Q/K/V). Before training, verify that `model(input_ids).logits` is bit-identical to the frozen base model's output. This confirms B=0 at all 12 injection points, not just the first one.

**Gradient accumulation correctness**: train with `device_batch_size=1, accumulation_steps=4` and with `device_batch_size=4, accumulation_steps=1` on the same dataset. The final loss should be approximately equal (within random seed variation). If accumulation is buggy, the effective learning rate will differ and the losses will diverge.

**Adapter roundtrip fidelity**: train for 10 steps, save the adapter, reload it into a fresh model, and verify that forward passes on 5 different inputs produce bit-identical outputs. Any mismatch indicates a bug in the save/load code (e.g., wrong key name, tensor dtype conversion).

**Merge correctness under training**: train for 10 steps, then merge. The merged model's output on 5 inputs should match the pre-merge LoRA model's output to within `atol=1e-5`. This tolerance accounts for floating-point rounding in the `W_merged = W_base + B @ A * scaling` computation.

**AdapterServer concurrent requests**: switch between two adapters 100 times and verify that each switch takes less than 200 ms and that the server stats correctly track the switch count. This tests that `copy_()` on the A/B tensors is atomic from the model's perspective.

### Part A — LoraInferenceClient: direct adapter management

`LoraInferenceClient.java` wraps our FastAPI server with Spring Framework 6.1's `RestClient`. It exposes three operations: `generate()`, `switchAdapter()`, and `stats()`.

```java
public class LoraInferenceClient {
    private final RestClient http;
    private final LoraProperties props;

    /** Generate text using the currently loaded adapter. */
    public GenerateResponse generate(String prompt) {
        var body = new GenerateRequest(prompt, props.maxNewTokens(), props.temperature(), null);
        return http.post().uri("/generate")
                .contentType(MediaType.APPLICATION_JSON)
                .body(body).retrieve().body(GenerateResponse.class);
    }

    /** Switch adapters — base model stays loaded, only A/B matrices are swapped. */
    public SwitchAdapterResponse switchAdapter(String adapterPath) {
        return http.post().uri("/switch-adapter")
                .body(new SwitchAdapterRequest(adapterPath))
                .retrieve().body(SwitchAdapterResponse.class);
    }
}
```

The `switchAdapter()` method is the key insight of the LoRA serving architecture. From Java's perspective, it's a 45 ms HTTP call. On the Python side, that 45 ms represents loading a 2.3 MB file and copying tensors into the already-loaded model — not a 10-second model reload. The Java client code is the same whether the backend is switching a 2.3 MB LoRA adapter or a 500 MB full model checkpoint. The performance difference is entirely in the backend.

### Part B — Spring AI ChatClient: the abstraction layer

Spring AI's `ChatClient` is an abstraction over any OpenAI-compatible or Ollama-compatible LLM endpoint. A LoRA-fine-tuned model served via our FastAPI server (or via Ollama's REST API) is called identically:

```java
ChatClient chatClient = chatClientBuilder.build();

// This call is identical whether the model is:
//   - Our LoRA-fine-tuned GPT-2 (via FastAPI)
//   - Llama 3 with a LoRA adapter (via Ollama)
//   - Any other fine-tuned open-source model
String response = chatClient
    .prompt("What is LoRA fine-tuning?")
    .call()
    .content();
```

The only change when switching from our custom server to Ollama is `application.yml`:

```yaml
spring:
  ai:
    ollama:
      base-url: http://localhost:11434
      chat:
        options:
          model: llama3:instruct   # An Ollama-served LoRA fine-tuned model
```

This is the correct abstraction for Java services that consume fine-tuned models: write to the `ChatClient` interface, configure the serving backend externally. When your team's research fine-tunes a better adapter next month, you change one line in config — not the Java service code.

### Part C — LoRA parameter efficiency in Java

The adapter efficiency math is useful for capacity planning:

```java
// In LoraDemoApplication.java
int nLayers = 12;       // GPT-2 layers
int dModel = 768;       // GPT-2 hidden dim
int rank = 8;
int nTargets = 2;       // q_proj + v_proj

// Each LoRA layer: A=(rank x dModel) + B=(dModel x rank)
long loraParamsPerLayer = (long) rank * dModel + (long) dModel * rank;  // 12,288
long totalLoraParams = nLayers * nTargets * loraParamsPerLayer;          // 294,912
long totalGpt2Params = 124_439_808L;
double pct = 100.0 * totalLoraParams / totalGpt2Params;                  // 0.24%

// For a 7B model: ~0.24% * 7B = ~16.8M trainable params
// vs 7B for full fine-tuning
```

This calculation tells you immediately how many parameters you're actually training for any LoRA configuration. A 0.24% trainable fraction means your optimizer state (AdamW first+second moments) is 0.24% of the full-model optimizer state — from 84 GB to 200 MB for a 7B model.

## What surprised me

<WhatSurprisedMe insight="The initialization trick surprised me. LoRA initializes B to all zeros — which means delta-W = B @ A * scaling = 0 at the start of training. The fine-tuned model is mathematically identical to the base model on step 0. This isn't an accident: it ensures the fine-tuning starts from the exact base model performance, not from random noise. Without B=0 initialization, the first training step computes loss against a randomly distorted model output — making the gradient signal much noisier. The 0-initialization is why LoRA converges in 1,000 steps where random initialization might need 10,000." />
