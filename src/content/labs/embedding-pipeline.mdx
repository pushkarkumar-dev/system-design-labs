## Why this lab exists

Every RAG system, every semantic search, every recommendation engine begins with the same operation: convert text to a vector. That operation is embedding. Understanding embedding at the implementation level — not just calling `openai.embeddings.create()` — means understanding why batching gives 18x throughput, why L2 normalization matters for ANN indexes, and why swapping embedding models without re-indexing is a silent correctness bug.

You've read that embeddings are dense representations. You've used them through an API. But until you've implemented the batching queue and timed the forward pass at batch=1 vs batch=32, until you've built the LRU cache and measured cache hit rates on real RAG workloads, until you've measured cosine drift between model versions and understood what it means for your vector index — you don't *know* it.

This lab builds an embedding pipeline from the ground up: a single-model HTTP server, then dynamic batching with an LRU cache, then model versioning with drift detection. Each stage addresses a concrete limitation of the previous one, measured in numbers.

## What we're building

A Python embedding pipeline implemented across three stages. Each stage is a targeted fix for a measured limitation.

**We build:**
- `SingleModelServer` (v0): load `all-MiniLM-L6-v2`, L2-normalize embeddings, serve over FastAPI
- `BatchingQueue` + `EmbeddingWorker` (v1): dynamic batching — collect for 5ms or 32 texts, whichever fires first
- `EmbeddingCache` (v1): SHA-256-keyed LRU cache with configurable max size
- `ModelRegistry` (v2): hold multiple model versions simultaneously with alias routing
- `EmbeddingDrift` (v2): measure cosine distance between versions to detect unsafe model swaps
- `PinningPolicy` (v2): pin callers to a version during gradual migrations

**We explicitly skip:**
- ANN indexes (FAISS, Weaviate) — covered in the Vector Index lab; our pipeline produces the vectors that go *into* those indexes
- Document chunking — covered in the RAG lab; this lab focuses on the embedding step itself
- GPU inference — CUDA support falls out of using PyTorch via sentence-transformers; covered in What the Toy Misses
- Persistent model registry — our registry is in-memory; production would use a model registry like MLflow or Hugging Face Hub

## Mental model

An embedding pipeline has three concerns that scale at different rates:

**Throughput** — how many texts can you embed per second? The bottleneck is the transformer forward pass. A 22M-parameter model like `all-MiniLM-L6-v2` takes ~10ms for one forward pass on CPU regardless of batch size (up to the memory limit). Single-item encoding wastes 31/32 of each forward pass's capacity; batch=32 amortizes the cost across 32 texts.

**Freshness** — when do cached embeddings go stale? Embeddings are deterministic: the same text always produces the same vector from the same model. Cache invalidation is by model version, not by time. When you upgrade the model, all cached embeddings from the old model become stale.

**Compatibility** — can you swap models without re-indexing? Two models from the same family (MiniLM-L6 vs MiniLM-L12) embed texts into similar vector spaces — their cosine distance on the same text is ~0.08 on average. Two models from different families might have cosine distance ~0.4. The vector index built against model-v1 is unusable against model-v2 if drift is high. Measuring drift before swapping is how you avoid this silent bug.

```
                v0: Request → embed() → L2-normalize → return
                ─────────────────────────────────────────────
  text(s) ──► SingleModelServer.embed()
                │
                ▼
              model.encode(texts)   ← forward pass (~10ms, any batch 1-32)
                │
                ▼
              L2-normalize          ← cosine sim = dot product after this
                │
                ▼
              float32 (N, 384)

                v1: Batching queue + LRU cache
                ──────────────────────────────
  text ──► EmbeddingCache.get()
             │
             ├─[hit]──► return cached embedding
             │
             └─[miss]─► BatchingQueue.put() → Future

                         EmbeddingWorker (background thread)
                              │

                              │

                              │

                v2: Model registry + drift gate
                ────────────────────────────────
  request ──► PinningRegistry.resolve(caller_id)
                │
                ▼
              ModelRegistry.embed(texts, version="stable")
                              │
                         (or "canary" if drift measured + approved)
```

The key insight of v0 is L2 normalization. After normalization, every embedding has unit length: `np.linalg.norm(embedding) == 1.0`. This single property collapses cosine similarity to a dot product:

```python
cosine_similarity(a, b) = (a · b) / (|a| * |b|) = a · b
```

Because `|a| = |b| = 1` after normalization, the denominator disappears. This matters enormously for ANN indexes: FAISS's `IndexFlatIP` (inner product) gives exact cosine similarity for unit vectors. The HNSW index in Weaviate expects unit vectors and fails silently if they're not normalized (it computes inner products and reports wrong nearest neighbors).

The normalization itself is three lines:

```python
def _l2_normalize(embeddings: np.ndarray) -> np.ndarray:
    norms = np.linalg.norm(embeddings, axis=1, keepdims=True)
    norms = np.where(norms == 0, 1.0, norms)  # guard against zero vectors
    return (embeddings / norms).astype(np.float32)
```

The zero-vector guard is important: if sentence-transformers ever returns an all-zeros embedding (it can for empty strings in some configurations), dividing by zero produces NaN. NaN embeddings silently corrupt ANN indexes.

**Why sentence-transformers?**

`SentenceTransformer("all-MiniLM-L6-v2")` downloads the model on first call and caches it in `~/.cache/huggingface/`. It wraps HuggingFace Transformers and handles tokenization, batching, pooling, and (optionally) normalization. We do normalization explicitly to make it visible.

**FastAPI wiring:**

The server exposes three endpoints:
- `POST /embed` — body: `{"texts": ["...", "..."]}` → returns `{"embeddings": [[...]], "dimension": 384}`
- `GET /health` — returns model name and loaded status
- `GET /model-info` — returns name and dimension

The v0 limitation is throughput: single-item requests call `model.encode()` once per text. The forward pass takes ~10ms on CPU whether you embed 1 text or 32 texts. At single-item: 100 texts/sec. With batch=32: 1,800 texts/sec. V1 fixes this.

The core idea of dynamic batching: don't call `model.encode()` per request. Collect requests until either 32 are waiting or 5ms has elapsed, then encode the whole batch at once.

**Why a thread, not asyncio?**

sentence-transformers releases the Python GIL during the forward pass (PyTorch's C extension does this). A real OS thread running the forward pass is truly parallel with the asyncio event loop handling new HTTP requests. With `asyncio.to_thread()`, you'd lose the batching window — each call would immediately dispatch to a thread pool instead of waiting to be grouped.

```python
class EmbeddingWorker:
    def _run(self):
        while not self._stop_event.is_set():
            batch = self._queue.drain()   # blocks up to 100ms for first item
            if not batch:
                continue
            texts = [req.text for req in batch]
            embeddings = encode(self._server.model, texts)
            for i, req in enumerate(batch):
                req.future.set_result(embeddings[i])
```

**The drain() logic:**

`drain()` blocks until at least one item arrives (up to 100ms), then collects additional items non-blocking until `max_batch_size` or `max_wait_ms` expires:

```python
def drain(self) -> list[_PendingRequest]:
    batch = []
    deadline = None
    try:
        first = self._queue.get(timeout=0.1)  # block for first item
        batch.append(first)
        deadline = time.monotonic() + (self.max_wait_ms / 1000.0)
    except queue.Empty:
        return batch
    while len(batch) < self.max_batch_size:
        remaining = deadline - time.monotonic()
        if remaining <= 0:
            break
        try:
            item = self._queue.get(timeout=remaining)
            batch.append(item)
        except queue.Empty:
            break
    return batch
```

**SHA-256 cache key:**

A production RAG system embeds document chunks of 512 tokens (~2,000 characters). Storing those as raw dict keys would use gigabytes of memory for the cache. SHA-256 produces a 32-byte key regardless of text length. Collision probability for 10M cached texts: approximately `10^-61` (birthday paradox on 2^256 outputs). Not a concern in practice.

```python
@staticmethod
def _key(text: str) -> str:
    return hashlib.sha256(text.encode("utf-8")).hexdigest()
```

**LRU eviction:**

`collections.OrderedDict` maintains insertion order. Moving an accessed key to the end (`.move_to_end(key)`) and always evicting from the front (`popitem(last=False)`) implements LRU in O(1) per operation — no sorted data structure required.

**Throughput comparison:**
- Single-item: each text pays the full ~10ms forward pass → ~100 texts/sec
- Batch=32: 32 texts share one ~10.5ms forward pass → ~1,800 texts/sec (18x speedup)
- Cache hit: no forward pass → ~100,000 texts/sec (limited by hash + dict lookup)

The failure mode v2 prevents: you upgrade the embedding model, embeddings look correct in isolation, but your ANN index returns wrong results because the query embedding and the stored document embeddings are from different vector spaces.

**Why different models produce incompatible vectors:**

Each sentence-transformer model is trained with a specific projection layer that maps the transformer's hidden states to the embedding space. Two models from the same family (MiniLM-L6 vs MiniLM-L12) share the same base architecture, so their projections are similar but not identical — cosine distance between same-text embeddings is ~0.08 on average. Two models from different families (MiniLM vs E5-large) have entirely different projections — cosine distance can be 0.3-0.5.

**Measuring drift:**

```python
def measure_drift(self, texts, version_a, version_b) -> float:
    emb_a = self._registry.embed(texts, version=version_a)
    emb_b = self._registry.embed(texts, version=version_b)
    # Dot product of unit vectors = cosine similarity (N,)
    similarities = np.einsum("ij,ij->i", emb_a, emb_b)
    distances = 1.0 - similarities
    return float(np.mean(distances))
```

`np.einsum("ij,ij->i", a, b)` is the row-wise dot product of two matrices — equivalent to `(a * b).sum(axis=1)` but more readable for this pattern. Since both matrices are L2-normalized, this gives cosine similarity per row.

**The compatibility threshold:**

`is_compatible(v_a, v_b, probe_texts, threshold=0.9)` returns True if average cosine similarity is at least 0.9 (equivalently, avg cosine distance at most 0.10). The threshold=0.9 was chosen empirically: at similarity above 0.9, the top-10 retrieved documents from a hybrid BM25+dense index change by less than 5% — acceptable degradation for a live migration.

**PinningPolicy:**

During a gradual migration from "stable" (MiniLM-L6) to "next" (MiniLM-L12), you want:
- New services: use "next" immediately (you chose them for a reason)
- Existing services: keep "stable" for 7 days, allowing teams to test and re-index if needed

```python
policy = PinningRegistry()
policy.pin("svc-legacy", version="stable", until=now + timedelta(days=7))

# svc-legacy gets "stable" regardless of the global default:
resolved_version = policy.resolve("svc-legacy", default_version="next")
# → "stable" (still within 7-day window)
```

After `pinned_until`, `resolve()` returns the default ("next"), so the pin expires automatically. No manual cleanup required.

**Zero-downtime upgrade path:**

1. Load new model as "canary": `registry.register_model("canary", "all-MiniLM-L12-v2")`
2. Measure drift: `drift.measure_drift(probe_corpus, "stable", "canary")` → 0.08
3. Check compatibility: `drift.is_compatible(...)` → True
4. Pin existing services to "stable" for 7 days
5. Shadow-index: re-embed all documents with "canary" into a shadow index
6. Promote: rename "canary" to "stable", atomically swap index pointer
7. Pins expire after 7 days; all callers now use the new stable

The 18x batching speedup is the headline result. The model's forward pass time is nearly constant from batch=1 to batch=32: ~10ms for 1 text, ~10.5ms for 32 texts. Batching amortizes the fixed cost of loading 22M parameters into the compute units across 32 texts simultaneously.

## Stress tests

**Batch accumulation under load**: send 1,000 concurrent single-text requests using `asyncio.gather()`. The batching queue should naturally form batches of 32 as requests arrive faster than the 5ms window. Expected: all 1,000 requests complete in under 1,000ms (vs ~10,000ms for single-item). The queue depth should never exceed 32 sustained.

**Cache eviction correctness**: fill the cache to max_size (10,000 entries), then add 1,000 new entries. The 1,000 oldest entries should be evicted. Access pattern: check that the 1,000 most recently accessed entries are all cache hits, and that an entry accessed only at the beginning of the fill is a miss.

**Drift with mismatched dimensions**: register a model with a different output dimension (e.g. `all-mpnet-base-v2` has 768-dim vs MiniLM's 384-dim). `measure_drift()` should raise immediately when trying to compute `np.einsum("ij,ij->i", emb_a, emb_b)` on mismatched shapes — a fast failure rather than silently wrong results.

**Pin expiry race condition**: create a pin that expires in 100ms. Send requests at 50ms intervals. The first two requests should get the pinned version; requests after 100ms should get the default. Verify with the `GET /pin` endpoint's `active` field.

**Worker thread crash recovery**: inject an exception in `model.encode()` via monkeypatching. The worker's `_run()` loop catches all exceptions and calls `future.set_exception()` on all pending requests. Callers should receive `Future.exception()` rather than hanging indefinitely on `future.result()`.

### Part A — EmbeddingClient: the HTTP bridge

`EmbeddingClient.java` calls our Python FastAPI server using Spring Framework 6.1's `RestClient`. Two embed operations:

```java
public class EmbeddingClient {
    private final RestClient http;

    public EmbeddingClient(String baseUrl) {
        this.http = RestClient.builder().baseUrl(baseUrl).build();
    }

    public EmbedResponse embed(List<String> texts) {
        return embed(texts, null);   // uses "stable" version
    }

    public EmbedResponse embed(List<String> texts, String version) {
        var resp = http.post()
                .uri("/embed")
                .contentType(MediaType.APPLICATION_JSON)
                .body(new EmbedRequest(texts, version, null))
                .retrieve()
                .body(EmbedResponse.class);
        if (resp == null) throw new EmbeddingException("null from server");
        return resp;
    }
}
```

Usage in any Spring service:

```java

```

### Part B — Spring AI's EmbeddingClient interface

Spring AI defines `org.springframework.ai.embedding.EmbeddingClient` as an interface with one method:

```java
float[] embed(String text);
```

The `spring-ai-openai-spring-boot-starter` auto-configures an `OpenAiEmbeddingClient` that calls OpenAI's `text-embedding-3-small` (1536-dim) by default. The downstream code is identical for both backends:

```java
// Using Spring AI's auto-configured client (OpenAI by default):
@Autowired
EmbeddingClient springAiClient;

float[] embedding = springAiClient.embed("Hello world");
// → float[1536] from text-embedding-3-small

// Using our custom client (Python server):
@Autowired
dev.pushkar.embedding.EmbeddingClient ourClient;

float[] embedding = ourClient.embed(List.of("Hello world")).floatArray(0);
// → float[384] from all-MiniLM-L6-v2
```

Both produce `float[]` embeddings. The cosine similarity calculation downstream is identical.

### Part C — Routing Spring AI to our local server

To use Spring AI's full abstraction stack (VectorStore, DocumentRetriever) with our local embedding model instead of OpenAI:

```yaml
# application.yml
spring:
  ai:
    openai:
      base-url: http://localhost:8000   # our FastAPI server
      api-key: not-needed               # local server ignores this
      embedding:
        options:
          model: all-MiniLM-L6-v2
```

With this configuration, `springAiClient.embed("...")` routes to `POST http://localhost:8000/v1/embeddings`. Our FastAPI server needs to accept OpenAI's request format — add a `/v1/embeddings` endpoint that maps to our `/embed` handler. This is the standard OpenAI-compatible API contract that local inference servers (Ollama, LM Studio) also implement.
