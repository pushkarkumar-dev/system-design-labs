## Why this lab exists

Every HTTP response that travels from a database through an application server to a user could instead be served from a cache 50 ms away. CDNs exploit that opportunity at scale. But understanding *which* responses to cache, for *how long*, and *what to do when the cache is stale* requires internalizing HTTP cache semantics that most developers only encounter when they accidentally break production.

Building an edge node from scratch forces you to make the decisions that Cloudflare and Fastly have already made. Do you serve stale content during a revalidation? Do you let 10,000 concurrent misses all hit the origin at once? What happens when the origin is down?

## What we're building

A CDN edge node in Go across three stages:

- **v0**: LRU cache, `Cache-Control` parsing, `stale-while-revalidate`, `stale-if-error`
- **v1**: Request coalescing (100 concurrent misses → 1 origin call), `Vary` header keyspace splitting
- **v2**: Two-level L1/L2 cache hierarchy, adaptive TTL, `Link: rel=prefetch` cache warming

**Explicitly excluded**: geographic distribution, WAL-backed origin shields, signed URLs, QUIC/HTTP3, Range request caching, streaming responses.

## Mental model

Think of the edge node as a caching reverse proxy with four decisions per request:

1. **Fresh cache entry?** → Serve it. Done.
2. **Stale entry within `stale-while-revalidate` window?** → Serve it immediately, revalidate in the background.
3. **Stale entry and origin returned 5xx?** → Serve the stale entry (`stale-if-error`).
4. **Complete miss?** → Fetch from origin, cache the response, return it.

Request coalescing wraps case 4: instead of N concurrent origin requests, one goroutine does the fetch while N-1 goroutines wait on a channel. The winner stores the result and closes the channel; all waiters receive the same `CacheEntry`.

The LRU cache uses a doubly-linked list (`container/list`) plus a hash map. `Get` calls `list.MoveToFront` — O(1). `Set` calls `list.PushFront`; when over capacity, `list.Back()` is the victim.

The core data structure stores the full HTTP response plus three timestamps:

```go
type CacheEntry struct {
    Body                      []byte
    Headers                   http.Header
    StatusCode                int
    ExpiresAt                 time.Time
    StaleWhileRevalidateUntil time.Time
    StaleIfErrorUntil         time.Time
    AccessCount               int64
    OriginalTTL               time.Duration
}

func (e *CacheEntry) IsExpired() bool           { return time.Now().After(e.ExpiresAt) }
func (e *CacheEntry) IsStaleWhileRevalidate() bool {
    now := time.Now()
    return now.After(e.ExpiresAt) && now.Before(e.StaleWhileRevalidateUntil)
}
```

`ParseCacheControl` splits the `Cache-Control` response header on commas and handles `max-age=N`, `stale-while-revalidate=N`, `stale-if-error=N`, `no-store`, `no-cache`. It also parses the `Vary` header for v1.

The `EdgeProxy` implements `http.Handler`. The dispatch logic maps to the four decisions above:

```go
func (p *EdgeProxy) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    key := cacheKey(r)
    if entry := p.cache.Get(key); entry != nil {
        if !entry.IsExpired()            { serve(w, entry); return }
        if entry.IsStaleWhileRevalidate() {
            serve(w, entry)
            go p.revalidate(key, r)
            return
        }
    }
    resp, err := p.fetchOrigin(r)
    if err != nil {
        if entry := p.cache.Get(key); entry != nil && entry.IsStaleIfError() {
            serve(w, entry); return
        }
        http.Error(w, "bad gateway", 502); return
    }
    p.cache.Set(key, toEntry(resp, directives))
    serve(w, ...)
}
```

**The thundering herd problem**: when a popular cache entry expires, 100 concurrent requests all see a miss and all race to the origin — a 100× traffic spike. Request coalescing prevents this:

```go
type inflightReq struct {
    done  chan struct{}
    entry *CacheEntry
    err   error
}

func (cp *CoalescingProxy) fetch(key string, r *http.Request) (*CacheEntry, error) {
    cp.mu.Lock()
    if inf, ok := cp.inflight[key]; ok {
        cp.mu.Unlock()
        <-inf.done         // block until winner finishes
        return inf.entry, inf.err
    }
    inf := &inflightReq{done: make(chan struct{})}
    cp.inflight[key] = inf
    cp.mu.Unlock()

    entry, err := cp.base.fetchOrigin(r)
    inf.entry, inf.err = entry, err
    close(inf.done)        // broadcast to all waiters

    cp.mu.Lock()
    delete(cp.inflight, key)
    cp.mu.Unlock()
    return entry, err
}
```

**Vary header support**: some origins return `Vary: Accept-Encoding` or `Vary: Accept-Language`. Without Vary-aware keying, a compressed response would be served to a client that didn't accept gzip. The cache key in v1 appends the normalized value of each Vary field:

```
key = method + ":" + path + "?" + query + "|vary:" + accept_encoding_value
```

A single `/api/data` URL with `Vary: Accept-Language` stores separate entries for `en`, `fr`, and `zh`.

Real CDNs maintain multiple cache tiers. A tiny hot L1 serves the most popular content at the lowest latency. A larger warm L2 stores the long-tail:

```go
func (tp *TieredProxy) lookup(key string) *CacheEntry {
    if e := tp.l1.Get(key); e != nil {
        tp.stats.L1Hits.Add(1)
        return e
    }
    if e := tp.l2.Get(key); e != nil {
        tp.stats.L2Hits.Add(1)
        tp.l1.Set(key, e) // promote to L1
        return e
    }
    tp.stats.Misses.Add(1)
    return nil
}
```

When L1 evicts an entry (capacity overflow), it is *demoted* to L2 rather than discarded. An item does not fully leave the cache until it is also evicted from L2.

**Adaptive TTL**: popular content should stay fresh longer. Each `CacheEntry` tracks `AccessCount`. On background revalidation, if the count exceeds 10, the TTL is doubled (up to 4× the original `Cache-Control: max-age`).

**Link prefetch**: the `Prefetcher` parses `Link: </next-page>; rel=prefetch` response headers and warms those URLs in background goroutines — before any client requests them.

</LabStage>

## Stress tests

**10,000 concurrent requests**: without coalescing, a popular entry expiring at peak traffic sends 10,000 simultaneous origin requests. With coalescing, only 1 fires. The other 9,999 wait on the channel and add roughly 100 µs of latency — negligible compared to the origin RTT saved.

**Vary explosion**: if an origin returns `Vary: User-Agent`, the cache stores one entry per user-agent string. At 1,000 distinct user-agents and LRU capacity 100, the cache is effectively bypassed — 99% of requests miss. Real CDNs normalize user-agents into buckets (mobile/desktop/bot) before using them in Vary keys.

**Adaptive TTL saturation**: after four doublings, the TTL cap is reached. Content cached with `max-age=60` and heavy access will be held for 240 seconds. If the origin changes the content more frequently, the edge serves stale data for up to 240 seconds — the multiplier must be tuned to the content's rate of change.

## Wiring it into a Java/Spring service

<JavaIntegration
  type="service-client"
  mavenCoords={["org.springframework.boot:spring-boot-starter-web:3.3.0"]}
  springModule="spring-webmvc"
>

The Java side uses Spring `RestClient` to route requests through the CDN edge node. The CDN is transparent to the application — it sees `Cache-Control` headers and caches accordingly:

```java
@Component
public class CdnClient {
    private final RestClient http;

    public CdnClient(RestClient.Builder builder,
                     @Value("${cdn.edge-url:http://localhost:8080}") String base) {
        this.http = builder.baseUrl(base).build();
    }

    public byte[] fetch(String path) {
        return http.get().uri(path).retrieve().body(byte[].class);
    }

    public void purge(String path) {
        http.delete().uri("/cache/purge?path={p}", path)
            .retrieve().toBodilessEntity();
    }
}
```

**Spring Cache comparison**: Spring's `@Cacheable` and `CacheManager` (Caffeine, EhCache) provide in-process caching. The key difference is scope: Spring Cache is per-JVM; a CDN edge is shared across all JVMs and all clients. Use `@Cacheable` to avoid recomputing expensive in-JVM results; use the CDN edge to avoid hitting the application layer at all.
