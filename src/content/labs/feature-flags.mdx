## Why this lab exists

Every time you ship a large feature in one atomic deployment, you are gambling. If the feature breaks production, the only remediation you have is a rollback — a high-stress, all-or-nothing operation that touches every user at once. Feature flags change the calculus: deploy the code, but keep the feature switched off. Turn it on for 1% of users. Watch the error rate. Ramp to 10%, 50%, 100%. If something goes wrong, flip a bit and the feature is gone in milliseconds — no deployment, no rollback, no 2am pager.

This separation of *deploy* and *release* is one of the most underrated architectural patterns in production engineering. It is why Netflix can run hundreds of concurrent A/B experiments. It is how Google rolls out Chrome features to hundreds of millions of users without a coordinated release cycle. It is why the best teams can do continuous deployment (deploy every commit) without continuous user impact.

Building your own flag service reveals something that trips up every engineer the first time: "it's just a map of strings to booleans." And it is — until the moment you need to roll out to exactly 50% of users consistently across a fleet of stateless servers, or push a flag change to ten thousand connected clients in under 100ms, or prove to your security team that you know who changed which flag and when. Each of those requirements peels back another layer of accidental complexity hiding behind the innocent boolean.

This lab builds the whole stack: the evaluation engine, the percentage rollout algorithm, and the real-time push mechanism. The implementation is in Go. The Java integration shows how Spring services can consume it with a `@FeatureFlag` annotation that evaluates flags on every method call without a network hop.

## What we're building

A feature flag HTTP service in Go, implemented across three stages. Each stage adds one layer and measures its cost.

**We build:**
- A JSON file-backed flag store with `fsnotify` hot-reload and `sync.RWMutex` for concurrent evaluation
- A targeting rule evaluator: `user_id` allowlists, email domain matching, and percentage rollout via stable hashing
- Server-Sent Events push so connected clients learn about flag changes within 50ms instead of waiting for the next poll cycle
- An append-only audit log recording every flag evaluation

**We explicitly skip (and why):**
- Persistent storage: flags live in a JSON file, not PostgreSQL or Redis (that is the persistence lab)
- Multi-environment management: dev/staging/prod namespacing is a data modelling exercise on top of the core
- Client SDK: we expose an HTTP API; a real SDK adds reconnection, offline caching, and streaming — a full lab on its own
- Flag dependency graphs: prerequisite flags and circular dependency detection are a graph problem layered on top

## Mental model

Think of the percentage rollout as a consistent hash ring where each user gets a permanent address.

When you evaluate `hash(user_id + ":" + flag_name) mod 100`, you are projecting the user into a one-dimensional space from 0 to 99. A 50% rollout enables everyone whose address falls below 50. A 10% rollout enables everyone below 10. The address never changes — not between requests, not between server restarts, not when you add more servers. The same user always lands in the same bucket.

Why include `flag_name` in the hash input? If you only hashed `user_id`, then every 50% flag would include exactly the same set of users. Your power users (the ones always in the top 50%) would get every new feature simultaneously, making A/B experiments useless because the treatment group is identical across all experiments. Hashing `user_id + flag_name` gives each flag an independent rollout group.

```
user_id="alice"  flag="new-checkout"  →  bucket 23  →  in 50% rollout? yes
user_id="alice"  flag="dark-mode"     →  bucket 71  →  in 50% rollout? no
user_id="bob"    flag="new-checkout"  →  bucket 67  →  in 50% rollout? no
user_id="bob"    flag="dark-mode"     →  bucket 8   →  in 50% rollout? yes
```

The SSE push mechanism follows the classic pub-sub pattern. Each connected client gets a buffered Go channel registered with the `FlagService`. When a flag changes, `notify()` does a non-blocking send into each channel. The HTTP handler reads from its channel and writes `data: <json>\n\n` to the response, flushing immediately via `http.Flusher`. If a client's buffer fills up (it's too slow to read), the update is dropped rather than blocking the flag writer — safety over completeness.

The core data model is small. A `Flag` has a name, a default enabled state, an optional description, and a rules list (empty at v0). The service wraps a `map[string]Flag` behind a `sync.RWMutex`:

```go
type Flag struct {
    Name           string `json:"name"`
    DefaultEnabled bool   `json:"default_enabled"`
    Description    string `json:"description,omitempty"`
    Rules          []Rule `json:"rules,omitempty"`
}

type FlagService struct {
    mu         sync.RWMutex
    flags      map[string]Flag
    configPath string
    // v2 fields added later
}
```

The evaluation path is deliberately minimal:

```go
func (fs *FlagService) IsEnabled(flagName string) bool {
    fs.mu.RLock()
    f, ok := fs.flags[flagName]
    fs.mu.RUnlock()
    if !ok {
        return false // unknown flag → fail safe
    }
    return f.DefaultEnabled
}
```

**The fail-safe default** — returning `false` for unknown flags — is not a minor detail. It means that if you mistype a flag name or the service is unavailable at startup, the feature stays off rather than turning on unexpectedly. Every production flag service uses fail-safe defaults. The alternative (fail-open) causes incidents.

**fsnotify vs polling**: `fsnotify` uses OS-level file system events (`inotify` on Linux, `kqueue` on macOS, `ReadDirectoryChangesW` on Windows). It is more responsive than polling and avoids the O(N) file stat cost. The service falls back to 5-second polling if fsnotify is unavailable — for example, on some container overlay filesystems where inotify events are unreliable.

The JSON config format is a flat array of flags:

```json
[
  {"name": "new-checkout", "default_enabled": false, "description": "New checkout flow"},
  {"name": "dark-mode",    "default_enabled": true},
  {"name": "beta-users",   "default_enabled": false}
]
```

The `Evaluate` method replaces `IsEnabled` as the main entry point for v1 and v2. It accepts an `EvalContext` carrying the per-request identity:

```go
type EvalContext struct {
    UserID     string            `json:"user_id"`
    Email      string            `json:"email"`
    Attributes map[string]string `json:"attributes,omitempty"`
}
```

Rules are evaluated in the order they appear in the flag definition. The first rule whose condition matches determines the result:

```go
func (fs *FlagService) evaluateFlag(f Flag, ctx EvalContext) bool {
    for _, rule := range f.Rules {
        if matchRule(rule, f.Name, ctx) {
            return rule.Enabled
        }
    }
    return f.DefaultEnabled
}
```

The percentage bucket function is the most interesting part of v1. SHA-256 produces 32 bytes of output. Taking the first 8 bytes as a `uint64` and applying `mod 100` gives a number from 0 to 99 with near-perfect uniformity:

```go
func percentageBucket(userID, flagName string) int {
    h := sha256.New()
    h.Write([]byte(userID))
    h.Write([]byte(":"))
    h.Write([]byte(flagName))
    sum := h.Sum(nil)
    n := binary.BigEndian.Uint64(sum[:8])
    return int(n % 100)
}
```

**Why SHA-256 and not a faster hash like FNV or xxHash?** For percentage rollout, the distribution quality matters more than throughput. SHA-256 is a cryptographic hash with provable uniform distribution. FNV is faster but its distribution has known weaknesses for structured inputs like `"user-1"`, `"user-2"`. At 3,000,000 evaluations per second, SHA-256 is not a bottleneck for any realistic flag service — it would require a single Go service handling 3M flag evaluations per second before the hash becomes a concern.

A realistic flag configuration with three rule types looks like this:

```json
{
  "name": "beta-users",
  "default_enabled": false,
  "rules": [
    {"type": "user_id",      "values": ["alice", "bob"],        "enabled": true},
    {"type": "email_domain", "values": ["@pushkar.dev"],        "enabled": true},
    {"type": "percentage",   "percentage": 10,                  "enabled": true}
  ]
}
```

Evaluation order matters: `alice` and `bob` are always in (rule 1 matches first). Everyone else with a `@pushkar.dev` email is always in (rule 2). The remaining 10% of other users are in via the percentage rule. The `default_enabled: false` means everyone who doesn't match any rule is out.

The SSE handler is the most operationally interesting part of the service. It sets the three required headers, sends a snapshot of current flags as the initial state (so clients don't have to call `GET /flags` separately), then blocks waiting for updates or heartbeat ticks:

```go
func handleStream(svc *flags.FlagService) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        flusher, ok := w.(http.Flusher)
        if !ok {
            http.Error(w, "streaming not supported", http.StatusInternalServerError)
            return
        }

        w.Header().Set("Content-Type", "text/event-stream")
        w.Header().Set("Cache-Control", "no-cache")
        w.Header().Set("Connection", "keep-alive")

        // Send current state as initial snapshot.
        for _, f := range svc.ListFlags() {
            sendSSEEvent(w, flusher, flags.FlagUpdate{FlagName: f.Name, Flag: f})
        }

        ch := svc.Subscribe()
        defer svc.Unsubscribe(ch)

        ticker := time.NewTicker(15 * time.Second)
        defer ticker.Stop()

        for {
            select {
            case update, ok := <-ch:
                if !ok { return }
                sendSSEEvent(w, flusher, update)
            case <-ticker.C:
                fmt.Fprintf(w, ": heartbeat\n\n")
                flusher.Flush()
            case <-r.Context().Done():
                return
            }
        }
    }
}
```

**The 15-second heartbeat** is not optional. HTTP proxies (nginx, AWS ALB, Cloudflare) silently close idle connections after 60–90 seconds. The heartbeat keeps the TCP session alive and lets the client detect a dead connection quickly. The SSE spec uses comment lines (starting with `:`) for heartbeats — browsers ignore them, but the TCP keepalive fires.

**The non-blocking notify pattern** is critical for service safety:

```go
func (fs *FlagService) notify(update FlagUpdate) {
    fs.subMu.RLock()
    defer fs.subMu.RUnlock()
    for _, ch := range fs.subscribers {
        select {
        case ch <- update:
        default:
            // Subscriber is not reading — drop rather than block.
        }
    }
}
```

If a subscriber's buffer is full (client is on a slow connection or has stopped reading), the update is dropped. The subscriber will get the next flag update when it catches up. This is correct for flag changes — missing one intermediate state is acceptable because the next change carries the full flag definition. The alternative (blocking on a slow subscriber) would mean one slow browser tab stalls every flag update across the entire service.

The audit log uses a separate mutex from the main flag mutex, allowing evaluation and audit writes to proceed concurrently:

```go
type AuditEntry struct {
    Timestamp time.Time   `json:"timestamp"`
    FlagName  string      `json:"flag_name"`
    UserID    string      `json:"user_id"`
    Result    bool        `json:"result"`
    Context   EvalContext `json:"context"`
}
```

Each entry is a JSON line appended to the audit file. No fsync per entry — the OS will flush periodically. This gives 200,000 audit writes per second at the cost of potentially losing the last few seconds of entries on a hard crash. For an audit log (as opposed to a WAL), this tradeoff is usually acceptable.

## Stress tests

**10,000 concurrent evaluators**: spawn 10,000 goroutines each calling `Evaluate` in a tight loop for 5 seconds. The `sync.RWMutex` allows unlimited concurrent readers — throughput scales linearly with CPU cores because there is no write contention during evaluation. Memory usage stays flat: no per-goroutine allocation in the evaluation path.

**When the SSE channel backs up**: connect 1,000 SSE clients, then pause half of them by stopping their reads (simulate a slow mobile connection). Trigger a flag change. The non-blocking `notify` path drops the update for the paused clients without affecting the active ones. When the paused clients resume reading, they receive the next flag change correctly. The dropped update is lost — this is the documented behavior and is acceptable because the next event carries the full flag state.

**File watch under rapid edits**: write 1,000 flag changes to the config file in 1 second. The `fsnotify` watcher coalesces rapid events — not every write triggers a reload. The service remains consistent because `load()` always reads the entire file atomically and replaces the entire map under the write lock. No partial state is ever visible.

**Flag service unavailable (for the Java client)**: kill the Go server while the Spring service is running. The `FlagCache` continues serving from its `ConcurrentHashMap` — every flag evaluation is a local read with zero network involvement. The SSE subscriber goroutine logs a reconnect attempt every 2 seconds. When the server returns, the subscriber reconnects, receives the initial snapshot, and the cache is back in sync.

### Part A — FlagClient: the HTTP bridge

`FlagClient.java` wraps the Go service's three core routes using Spring Framework 6.1's `RestClient`. The entire client is under 60 lines of non-comment code:

```java
public class FlagClient {
    private final RestClient http;

    public FlagClient(String baseUrl) {
        this.http = RestClient.builder()
                .baseUrl(baseUrl)
                .build();
    }

    public boolean evaluate(String flagName, String userId, String email) {
        String uri = "/flags/{name}/evaluate?user_id={uid}&email={email}";
        var resp = http.get()
                .uri(uri, flagName,
                     userId != null ? userId : "",
                     email  != null ? email  : "")
                .retrieve()
                .body(EvaluateResponse.class);
        return resp != null && resp.enabled();
    }

    public List<FlagInfo> listFlags() {
        var result = http.get()
                .uri("/flags")
                .retrieve()
                .body(new ParameterizedTypeReference<List<FlagInfo>>() {});
        return result != null ? result : List.of();
    }

    public void updateFlag(String name, boolean enabled) {
        http.put()
                .uri("/flags/{name}", name)
                .body(Map.of("name", name, "default_enabled", enabled))
                .retrieve()
                .toBodilessEntity();
    }

    record EvaluateResponse(String flag, boolean enabled, boolean exists) {}
    public record FlagInfo(String name, boolean default_enabled, String description) {}
}
```

### Part B — FlagCache: local in-process cache with SSE subscription

The hot path for flag evaluation must never make a network call. `FlagCache` keeps a `ConcurrentHashMap` of flag states. It is populated in two ways: a `@Scheduled` full refresh every 30 seconds, and an SSE subscriber that updates individual flags within 50ms of a change:

```java
@Service
public class FlagCache {
    private final ConcurrentHashMap<String, Boolean> cache = new ConcurrentHashMap<>();
    private final FlagClient client;
    private final FlagProperties props;

    @PostConstruct
    void init() {
        refresh();
        Executors.newSingleThreadExecutor(r -> {
            Thread t = new Thread(r, "flag-sse-subscriber");
            t.setDaemon(true);
            return t;
        }).submit(this::subscribeToSSE);
    }

    @Scheduled(fixedRateString = "${feature-flags.refresh-interval-seconds:30}000")
    public void refresh() {
        try {
            client.listFlags().forEach(f -> cache.put(f.name(), f.default_enabled()));
        } catch (RestClientException e) {
            log.warn("flag cache refresh failed (using stale): {}", e.getMessage());
        }
    }

    public boolean isEnabled(String flagName) {
        return cache.getOrDefault(flagName, props.defaultEnabled());
    }

    private void subscribeToSSE() {
        while (running) {
            try {
                HttpURLConnection conn =
                    (HttpURLConnection) new URL(props.serviceUrl() + "/flags/stream")
                        .openConnection();
                conn.setRequestProperty("Accept", "text/event-stream");
                conn.setReadTimeout(0); // never time out an SSE stream
                conn.connect();
                try (var reader = new BufferedReader(
                        new InputStreamReader(conn.getInputStream()))) {
                    String line;
                    while (running && (line = reader.readLine()) != null) {
                        if (line.startsWith("data: ")) {
                            applySseEvent(line.substring(6).trim());
                        }
                    }
                }
            } catch (Exception e) {
                if (running) Thread.sleep(2000); // reconnect backoff
            }
        }
    }
}
```

### Part C — @FeatureFlag annotation and AOP aspect

The `@FeatureFlag` annotation gates any Spring bean method behind a flag check. The AOP aspect intercepts the call, reads from the local `FlagCache` (no network), and either proceeds or throws `FeatureDisabledException`:

```java
@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.METHOD, ElementType.TYPE})
public @interface FeatureFlag {
    String value();                        // flag name
    boolean defaultValue() default false;  // fail-safe default
    boolean returnNullIfDisabled() default false;
}

@Aspect
@Component
public class FeatureFlagAspect {
    private final FlagCache cache;

    @Around("@annotation(dev.pushkar.flags.FeatureFlag)")
    public Object aroundFeatureFlag(ProceedingJoinPoint pjp) throws Throwable {
        FeatureFlag ann = ((MethodSignature) pjp.getSignature())
                .getMethod().getAnnotation(FeatureFlag.class);
        boolean enabled;
        try {
            enabled = cache.isEnabled(ann.value());
        } catch (Exception e) {
            enabled = ann.defaultValue(); // fail safe
        }
        if (enabled) return pjp.proceed();
        if (ann.returnNullIfDisabled()) return null;
        throw new FeatureDisabledException(ann.value());
    }
}
```

```java

With 100,000 users and a 50% target, the actual rollout was 49.8% — not because of rounding, but because SHA-256 output has near-perfect uniform distribution. The 0.2% deviation would affect 200 users, which is often within acceptable error bounds for a gradual rollout. What I did not expect was how wrong a naive implementation would be: if you hash only user_id without including flag_name, every 50% flag enrolls the same 50,000 users. ## Where to go deeper

- **Martin Fowler on feature toggles**: the canonical taxonomy — release toggles vs ops toggles vs experiment toggles vs permission toggles — [martinfowler.com/articles/feature-toggles](https://martinfowler.com/articles/feature-toggles.html)
- **LaunchDarkly architecture blog**: how they handle flag evaluation at hundreds of millions of requests per day, including their streaming SDK design — [launchdarkly.com/blog](https://launchdarkly.com/blog/feature-flag-driven-development/)
- **Flipt**: a Go open-source flag service worth reading for its gRPC API and rule engine design — [github.com/flipt-io/flipt](https://github.com/flipt-io/flipt)
- **Unleash**: a Node.js flag service with a mature SDK ecosystem and a strategy plugin model — [github.com/Unleash/unleash](https://github.com/Unleash/unleash)
- **SSE vs WebSocket for flag delivery**: SSE is unidirectional and HTTP/1.1 compatible; WebSocket is bidirectional and adds connection upgrade complexity. For flag delivery (server to client only), SSE is the right tool.
