## Why this lab exists

Every microservice architecture is secretly an RPC framework. When Service A calls Service B, something has to take a Go function call, serialize the arguments, ship them across a TCP socket, deserialize them on the other side, call the real function, serialize the return value, and ship it back. That something is gRPC, or Thrift, or Twirp — or a home-grown HTTP-JSON layer that has no name but does the same job.

The problem with using gRPC as a black box is that its design decisions look like magic. Why do you need `protoc`? What is a `ManagedChannel` actually managing? Why does a streaming RPC require a `.proto` file? Why does gRPC need HTTP/2 when TCP already exists?

The answer to every one of those questions is visible in the implementation. Build the framing layer yourself and you understand why there has to be a length prefix. Build the dispatch table yourself and you understand what `protoc` generates. Build the middleware chain yourself and you understand why panic recovery is non-negotiable. After this lab, reading the gRPC-Java source is not intimidating — it's recognizable.

This lab builds a complete RPC framework in three stages, starting with 9 bytes of binary header on a TCP socket and ending with an IDL parser and production-grade middleware. Every gRPC design decision you'll encounter in the wild has a direct counterpart here.

## What we're building

An RPC framework in Go across three stages. Each stage adds one architectural layer and is measurably better than the previous one on a stated metric.

**We build:**
- A 9-byte binary frame header: `[4-byte length][1-byte flags][4-byte requestID]`
- A reflection-based method dispatch table: map from `"Service.Method"` string to Go function
- A multiplexed client with a pending-call map: one TCP connection, many concurrent in-flight calls
- Streaming support via flag bits: `IsStream`, `EndOfStream`, `IsError`
- Context deadline propagation across the wire
- An IDL parser for `.rpc` service definition files
- A middleware chain: `LoggingMiddleware`, `RecoveryMiddleware`, `MetricsMiddleware`

**We explicitly skip (and why):**
- Protocol Buffers and `protoc` — our serialization is JSON; Protobuf is a separate compiler project
- HTTP/2 transport — our framing is raw TCP; HTTP/2 multiplexing and HPACK are out of scope
- TLS — no certificate management; production would add `tls.Listen` and `tls.Dial`
- Client-side load balancing — single connection per server address
- Bidirectional streaming (reserved for a future exercise)

## Mental model

### TCP is a byte stream, not a message stream

This is the foundational problem every network protocol has to solve, and the 4-byte length prefix is the simplest correct solution.

When you call `conn.Write([]byte("hello"))` and the other side calls `conn.Read(buf)`, there is no guarantee that `Read` returns exactly `"hello"`. TCP is a byte stream. The OS may batch two small writes into one read, or split one large write across three reads, depending on Nagle's algorithm, MTU, and kernel scheduling. Without a framing protocol, the receiver has no way to know where one message ends and the next begins.

The solution is a length-prefix frame:

```
┌─────────────────────────────────────────────────────────────────┐
│  Length    (uint32, 4 bytes, big-endian) — payload byte count   │
│  Flags     (uint8,  1 byte)              — IsStream/EOS/IsError │
│  RequestID (uint32, 4 bytes, big-endian) — in-flight call ID    │
│  Payload   (Length bytes)                — JSON-encoded body    │
└─────────────────────────────────────────────────────────────────┘
```

The receiver always calls `io.ReadFull(conn, header[:9])` first — blocking until exactly 9 bytes arrive. It then reads `Length` from bytes 0–3 and calls `io.ReadFull(conn, payload[:Length])`. `io.ReadFull` loops internally until it has the exact number of bytes requested, regardless of how many `Read` syscalls that takes. The frame is now complete and self-contained.

This is what HTTP/1.1 does with `Content-Length`, what HTTP/2 does with its binary frame header, and what every binary protocol you'll encounter (Redis, Kafka, MySQL wire protocol, gRPC) does in some form.

### The pending-call map and request ID

A single TCP connection can carry many concurrent RPC calls. The client assigns each call a unique `requestID` (an atomically incrementing `uint32`) and stores a per-call channel in a map keyed by that ID. A background goroutine reads frames from the server and routes each response to the correct channel by looking up `frame.RequestID`.

```
Client goroutine A:  Call("Math.Add", …)  → requestID=1 → write frame → block on ch[1]
Client goroutine B:  Call("Math.Mul", …)  → requestID=2 → write frame → block on ch[2]

Background reader:   ReadFrame → requestID=2 → ch[2] ← frame  (B unblocks)
Background reader:   ReadFrame → requestID=1 → ch[1] ← frame  (A unblocks)
```

This demultiplexing model is identical to HTTP/2 stream IDs and Go's `net/rpc` package. The key invariant: register the pending entry in the map **before** writing the frame to the wire. If you register after writing, there is a race where the server responds before the client has anywhere to route the response.

### The reflection dispatch table

The server holds a `map[string]HandlerFunc` where keys are `"ServiceName.MethodName"` strings. `Register(svc interface{})` uses Go's `reflect` package to iterate over exported methods of `svc`, filter for the signature `func(context.Context, *Request) (*Response, error)`, and populate the map.

This is exactly what `net/rpc.Register` does. The difference with gRPC is that `protoc`-generated code skips the runtime reflection entirely — the dispatch map is built from generated Go code at compile time, with known types. Our approach is more flexible (register any struct) but has a runtime cost: every dispatch requires a `reflect.Value.Call()` instead of a direct function call. For our benchmark load (85,000 calls/sec), this is immaterial.

### The frame header

The 9-byte header is the entire protocol. Everything else is policy built on top of it.

```go
const headerSize = 9 // 4 (Length) + 1 (Flags) + 4 (RequestID)

func WriteFrame(w io.Writer, f Frame) error {
    header := make([]byte, headerSize)
    binary.BigEndian.PutUint32(header[0:4], f.Length)
    header[4] = f.Flags
    binary.BigEndian.PutUint32(header[5:9], f.RequestID)
    if _, err := w.Write(header); err != nil {
        return fmt.Errorf("write frame header: %w", err)
    }
    if len(f.Payload) > 0 {
        if _, err := w.Write(f.Payload); err != nil {
            return fmt.Errorf("write frame payload: %w", err)
        }
    }
    return nil
}

func ReadFrame(r io.Reader) (Frame, error) {
    var header [headerSize]byte
    if _, err := io.ReadFull(r, header[:]); err != nil {
        return Frame{}, fmt.Errorf("read frame header: %w", err)
    }
    f := Frame{
        Length:    binary.BigEndian.Uint32(header[0:4]),
        Flags:     header[4],
        RequestID: binary.BigEndian.Uint32(header[5:9]),
    }
    if f.Length > 0 {
        f.Payload = make([]byte, f.Length)
        if _, err := io.ReadFull(r, f.Payload); err != nil {
            return Frame{}, fmt.Errorf("read frame payload (want %d bytes): %w", f.Length, err)
        }
    }
    return f, nil
}
```

`io.ReadFull` is the critical detail. It loops until it has read exactly `len(buf)` bytes, returning `io.ErrUnexpectedEOF` if the connection closes mid-frame. Without `ReadFull`, a partial header read would silently corrupt the length field, causing the next `ReadFull` to request a nonsensical number of bytes.

### The reflection dispatch table

`Register` iterates over the exported methods of any struct value and filters for the exact handler signature:

```go
func (s *Server) Register(svc interface{}) error {
    typ := reflect.TypeOf(svc)
    val := reflect.ValueOf(svc)
    name := typ.Elem().Name() // strip pointer, get struct name

    ctxType  := reflect.TypeOf((*context.Context)(nil)).Elem()
    reqType  := reflect.TypeOf((*Request)(nil))
    respType := reflect.TypeOf((*Response)(nil))
    errType  := reflect.TypeOf((*error)(nil)).Elem()

    for i := 0; i < typ.NumMethod(); i++ {
        m  := typ.Method(i)
        mt := m.Type
        // func(svc, ctx, *Request) (*Response, error)
        if mt.NumIn() != 3 || mt.NumOut() != 2 { continue }
        if !mt.In(1).Implements(ctxType)         { continue }
        if mt.In(2) != reqType                   { continue }
        if mt.Out(0) != respType                 { continue }
        if !mt.Out(1).Implements(errType)        { continue }

        method := val.Method(i)
        key := name + "." + m.Name
        s.handlers[key] = func(ctx context.Context, req *Request) (*Response, error) {
            out := method.Call([]reflect.Value{
                reflect.ValueOf(ctx),
                reflect.ValueOf(req),
            })
            // extract (*Response, error) from reflect.Value slice
            …
        }
    }
    …
}
```

The key insight: the `key` is always `"TypeName.MethodName"`. When the client calls `client.Call(ctx, "Calculator.Add", args, &reply)`, the method name in the JSON payload is `"Calculator.Add"`. The server does a single map lookup and finds the handler. No routing tree, no pattern matching.

**The test to write first**: two goroutines calling the same method concurrently, both expecting correct results. The server handles each connection in its own goroutine, but the handlers map is shared. Verify there are no data races under `go test -race`.

### The client pending map

The client's pending map must be accessed under a mutex because the background reader goroutine and the `Call` goroutine both touch it:

```go
// Register BEFORE writing the frame — avoid a race where the server
// replies before we have a channel to route the response to.
p := &pending{ch: make(chan Frame, 1)}
c.mu.Lock()
c.pendings[id] = p
c.mu.Unlock()

// Write the frame.
c.wrMu.Lock()
err = WriteFrame(c.writer, f)
if err == nil { err = c.writer.Flush() }
c.wrMu.Unlock()

// Block until response arrives or context expires.
select {
case <-ctx.Done():
    c.mu.Lock()
    delete(c.pendings, id)
    c.mu.Unlock()
    return fmt.Errorf("rpc: %w", ctx.Err())
case respFrame := <-p.ch:
    // decode and return
}
```

Two separate mutexes: `wrMu` protects the write path (multiple goroutines can call `Call` concurrently), `mu` protects the pending map. The `bufio.Writer` is not goroutine-safe, so `wrMu` must wrap both `WriteFrame` and `Flush`.

### Flag bits extend the frame without changing the header size

A single `uint8` flags field can carry 8 independent boolean signals. We use three:

```go
const (
    FlagIsStream    uint8 = 1 << 0  // this frame belongs to a streaming call
    FlagEndOfStream uint8 = 1 << 1  // this is the last frame in the stream
    FlagIsError     uint8 = 1 << 2  // payload is an error string, not a response
)
```

The server's connection read loop checks `frame.Flags & FlagIsStream != 0` and routes streaming frames to a separate handler. Non-streaming frames go through the standard dispatch path. A single requestID unifies both: every frame for a given streaming call shares the same `requestID`, allowing the server to correlate them without additional state.

### Server-side streaming: push N messages, then close

The `ServerStream` type wraps the connection's writer and the requestID. The handler calls `Send` for each message and `Close` when done:

```go
type ServerStream struct {
    requestID uint32
    writer    *bufio.Writer
    wrMu      *sync.Mutex
}

func (ss *ServerStream) Send(msg interface{}) error {
    payload, _ := json.Marshal(msg)
    f := NewStreamFrame(ss.requestID, payload, false /* not EOS */)
    ss.wrMu.Lock()
    defer ss.wrMu.Unlock()
    if err := WriteFrame(ss.writer, f); err != nil { return err }
    return ss.writer.Flush()
}

func (ss *ServerStream) Close() error {
    f := NewStreamFrame(ss.requestID, nil, true /* EOS */)
    ss.wrMu.Lock()
    defer ss.wrMu.Unlock()
    if err := WriteFrame(ss.writer, f); err != nil { return err }
    return ss.writer.Flush()
}
```

The client's read loop detects `FlagEndOfStream` and closes the channel it is draining. The stream is complete.

### Client-side streaming: accumulate N frames, reply once

`ClientSendStream.Send` writes frames with `FlagIsStream` set. `CloseAndRecv` sends the `FlagEndOfStream` sentinel and then blocks on the pending-map channel, waiting for the server's single reply. The server-side `ClientStream.Wait()` blocks until `doneCh` is closed — which happens when the EOS frame arrives.

```go
func (cs *ClientSendStream) CloseAndRecv(reply interface{}) error {
    eos := NewStreamFrame(cs.requestID, nil, true)
    cs.client.wrMu.Lock()
    _ = WriteFrame(cs.client.writer, eos)
    _ = cs.client.writer.Flush()
    cs.client.wrMu.Unlock()

    // The background reader will route the server's reply to this channel.
    respFrame := <-p.ch
    // decode into reply …
    return nil
}
```

### Context deadline propagation

Go's `context.Context` carries a deadline but does not transmit it across network boundaries. For the client's deadline to affect server-side work, the client must encode it into the request payload (or a dedicated flag byte). In v1, the client reads `ctx.Deadline()` and, if set, marshals the remaining duration in milliseconds into a `timeout` field in the request JSON. The server decodes this and wraps its `context.Background()` with `context.WithTimeout` for the duration of that call.

This is exactly how gRPC propagates deadlines: the HTTP/2 headers carry a `grpc-timeout` value (e.g. `"100m"` for 100 milliseconds). The server creates a child context from it. If the client cancels before the timeout fires, the `select` in `Call` deletes the pending entry — the response arrives but nobody is waiting, so it is silently discarded.

**The key test**: register a `Slow.Method` that sleeps for 2 seconds. Call it with a 100ms deadline. Assert the call returns an error containing `context.DeadlineExceeded`. The server should not crash; other concurrent calls should continue to succeed.

### The IDL parser

A `.rpc` file describes the service contract. Our grammar is minimal by design — no imports, no message type definitions, no options:

```
service Calculator {
    method Add(AddRequest) returns AddResponse
    method Sum(stream AddRequest) returns AddResponse
}
```

`ParseIDL` is a line-oriented scanner (no parser generator required):

```go
func ParseIDL(src string) (*ServiceDescriptor, error) {
    scanner := bufio.NewScanner(strings.NewReader(src))
    var desc ServiceDescriptor
    inService := false

    for scanner.Scan() {
        line := strings.TrimSpace(scanner.Text())
        switch {
        case strings.HasPrefix(line, "service "):
            desc.Name = strings.Fields(line)[1]
            inService = true
        case line == "}":
            inService = false
        case inService && strings.HasPrefix(line, "method "):
            m, err := parseMethod(line)
            if err != nil { return nil, err }
            desc.Methods = append(desc.Methods, m)
        }
    }
    …
}
```

The `ServiceDescriptor` exposes `Lookup(methodName)` — which returns an error if the method is not declared in the IDL. This lets the server validate at registration time that every registered handler has a corresponding IDL declaration. In gRPC, `protoc` enforces this at compile time; our IDL enforces it at startup.

What the IDL gives you that pure reflection does not: input/output type names, the `stream` keyword on inputs, and a machine-readable service descriptor that could be used to generate client stubs, validate requests, or power a reflection API. The gRPC server reflection protocol is built on exactly this structure — the `FileDescriptorProto` embedded in every gRPC binary is the protobuf equivalent of our `ServiceDescriptor`.

### The middleware chain

`Use(A, B, C)` appends middleware to a slice. When a call arrives, the server builds the chain by wrapping from the inside out — C wraps the handler, B wraps C, A wraps B:

```go
func (s *Server) dispatch(ctx context.Context, f Frame) Frame {
    // …decode request, find handler…

    h := handler
    for i := len(s.middlewares) - 1; i >= 0; i-- {
        mw := s.middlewares[i]
        next := h
        h = func(ctx context.Context, req *Request) (*Response, error) {
            return mw(ctx, req.Method, next)
        }
    }
    resp, err := h(ctx, &req)
    // …encode response frame…
}
```

This is the standard onion model: the outermost middleware is the first to run and the last to return. `Use(LoggingMiddleware, RecoveryMiddleware, MetricsMiddleware)` means logging sees the call first and last, recovery wraps the handler, and metrics is just inside recovery.

### RecoveryMiddleware is non-negotiable

A panicking handler must not crash the server process. In a production RPC server handling 85,000 calls/sec across thousands of connections, a single panicking handler would kill the process and drop all in-flight calls — a denial-of-service vector.

```go
func RecoveryMiddleware(ctx context.Context, method string, next HandlerFunc) (resp *Response, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("handler panic: %v", r)
            resp = nil
        }
    }()
    return next(ctx, &Request{Method: method})
}
```

The named return values (`resp`, `err`) are essential here. The deferred function assigns to them after the panic is recovered, and the caller receives an error response frame instead of a crashed process.

### MetricsMiddleware: latency buckets without allocations

`MetricsRegistry` uses `sync.Map` so methods can be registered concurrently without a global lock. Each `MethodMetrics` struct holds `atomic.Int64` counters — no mutex required on the hot path:

```go
type MethodMetrics struct {
    Calls          atomic.Int64
    Errors         atomic.Int64
    LatencyBuckets [5]atomic.Int64  // 0-100µs, 100-500µs, 500µs-1ms, 1-10ms, >10ms
}
```

The `latencyBucket(d time.Duration) int` function is a simple switch on `d.Microseconds()`. Incrementing an atomic counter takes ~5ns and zero allocations — this is why the middleware chain benchmark reaches 4,500,000 dispatches/sec even with three middleware layers.

## Stress tests

**1000 concurrent connections**: the server spawns one goroutine per accepted connection. At 1000 simultaneous connections each sending one call, the Go scheduler must manage 1000+ goroutines. On the M2, this is handled comfortably — goroutines are cheap (2KB initial stack). The bottleneck shifts to the pending-map mutex contention on the client side if a single `Client` is shared across 1000 goroutines. The fix is a connection pool (one `Client` per goroutine, or round-robin across N clients).

**Frame interleaving test**: two goroutines sharing one client write frames concurrently. The `wrMu` mutex ensures atomicity of `WriteFrame + Flush` as a unit. Without this mutex, a goroutine could write the 9-byte header from one call and then be preempted before writing the payload — the server would read the header, then block waiting for `Length` bytes that belong to a different goroutine's payload. The result would be a garbled frame and a protocol corruption. The test: 100 goroutines calling concurrently with `go test -race` enabled; the race detector will catch any mutex violation immediately.

**Panicking handler**: register a handler that calls `panic("intentional")`. Without `RecoveryMiddleware`, the server goroutine panics and the entire process exits. With `RecoveryMiddleware`, the deferred `recover()` catches the panic and returns an error frame. Other connections continue serving. The test: call the panicking method, assert the error contains `"handler panic"`, then call a healthy method on the same connection and assert it succeeds.

**Large payload**: send a 1MB JSON payload in a single frame. The 4-byte `Length` field supports up to 4GB (uint32 max). The server allocates `make([]byte, frame.Length)` for every frame — so 1000 concurrent 1MB frames would allocate 1GB. In production, enforce a `maxFrameSize` limit in `ReadFrame` and return a protocol error if `Length > maxFrameSize`. gRPC defaults to a 4MB maximum message size for exactly this reason.

### What protoc generates vs. what we hand-rolled

Our `.rpc` IDL file and gRPC's `.proto` file solve the same problem: describe a service contract in a language-neutral format that can be parsed to generate client stubs and server skeletons.

Our IDL for the Calculator service:

```
service Calculator {
    method Add(AddRequest) returns AddResponse
    method Sum(stream AddRequest) returns AddResponse
}
```

The equivalent `.proto` file for gRPC:

```protobuf
syntax = "proto3";
package calculator;

service Calculator {
  rpc Add(AddRequest) returns (AddResponse);
  rpc Sum(stream AddRequest) returns (AddResponse);
}

message AddRequest {
  int32 a = 1;
  int32 b = 2;
}

message AddResponse {
  int32 sum = 1;
}
```

The structural difference is message type definitions. Our IDL names types (`AddRequest`, `AddResponse`) but does not define their fields — we rely on JSON and Go structs for that. Protobuf embeds field definitions in the IDL and uses those definitions to generate binary serialization code. When `protoc` runs against the `.proto` file above, it emits:

- `CalculatorGrpc.java` — the service interface, base server class, and blocking/async/future stub classes
- `CalculatorOuterClass.java` (or `Calculator.java` with `option java_multiple_files = true`) — the `AddRequest` and `AddResponse` message classes with binary serialization

Our `ParseIDL` produces a `ServiceDescriptor` at runtime; `protoc` produces equivalent Java source code at compile time. The compile-time approach catches method-name typos and type mismatches before the program runs.

### ManagedChannel: gRPC's connection abstraction

In gRPC-Java, you never hold a raw TCP connection. `ManagedChannel` manages the connection lifecycle — including reconnection, subchannel health checking, and load balancing across multiple resolved addresses. The equivalent in our framework is a single `*rpc.Client`.

```java
// gRPC-Java: create a managed channel to the server
ManagedChannel channel = ManagedChannelBuilder
    .forAddress("localhost", 50051)
    .usePlaintext()                    // no TLS — equivalent to our net.Dial
    .build();

// Create a blocking stub — equivalent to our client.Call() synchronous API
CalculatorGrpc.CalculatorBlockingStub stub =
    CalculatorGrpc.newBlockingStub(channel);

// Call Add — one generated method per IDL method
AddResponse resp = stub.add(
    AddRequest.newBuilder().setA(7).setB(35).build()
);
System.out.println("7 + 35 = " + resp.getSum()); // 42
```

The `stub.add()` call does exactly what our `client.Call(ctx, "Calculator.Add", args, &reply)` does:
1. Serializes `AddRequest` (protobuf binary vs. our JSON)
2. Assigns a stream ID (HTTP/2 stream ID vs. our `requestID`)
3. Writes a frame to the connection
4. Blocks until the response frame arrives
5. Deserializes `AddResponse` and returns it

The blocking stub is the closest analogue to our synchronous `Call`. gRPC-Java also provides `CalculatorGrpc.newStub()` (async, callback-based) and `CalculatorGrpc.newFutureStub()` (returns `ListenableFuture` for Guava-style composition).

### Client-streaming: the Sum RPC

Our streaming `ClientSendStream` maps to gRPC's `StreamObserver` model:

```java
// gRPC-Java async stub for client-streaming
CalculatorGrpc.CalculatorStub asyncStub =
    CalculatorGrpc.newStub(channel);

// responseObserver receives the server's single reply when the stream closes
StreamObserver<AddRequest> requestObserver = asyncStub.sum(
    new StreamObserver<AddResponse>() {
        @Override public void onNext(AddResponse resp) {
            System.out.println("Sum = " + resp.getSum());
        }
        @Override public void onError(Throwable t) { /* handle */ }
        @Override public void onCompleted() { /* stream finished */ }
    }
);

// Send 100 numbers — equivalent to ClientSendStream.Send() in a loop
for (int i = 1; i <= 100; i++) {
    requestObserver.onNext(AddRequest.newBuilder().setA(i).setB(0).build());
}
requestObserver.onCompleted(); // sends EOS — equivalent to CloseAndRecv()
```

Our `ClientSendStream.CloseAndRecv(reply)` is synchronous — it blocks until the server replies. gRPC's `StreamObserver.onCompleted()` is asynchronous — the server's reply arrives via `responseObserver.onNext`. The model is different, but the wire protocol is identical: a stream of frames, terminated by an EOS frame, followed by one response.

### Mapping our IDL fields to protobuf wire types

| Our IDL concept | Protobuf equivalent |
|---|---|
| `service Calculator` | `service Calculator` |
| `method Add(AddRequest) returns AddResponse` | `rpc Add(AddRequest) returns (AddResponse)` |
| `method Sum(stream AddRequest) returns AddResponse` | `rpc Sum(stream AddRequest) returns (AddResponse)` |
| `ServiceDescriptor.Name` | `FileDescriptorProto.service[*].name` |
| `MethodDescriptor.ClientStream` | `MethodDescriptorProto.client_streaming` |
| JSON payload | `protobuf binary (field tags + varints)` |
| `io.ReadFull` on 9-byte header | HTTP/2 frame header parse (9 bytes also) |

The HTTP/2 frame header is also 9 bytes: 3 bytes for payload length, 1 byte for type, 1 byte for flags, and 4 bytes for stream identifier. The structural similarity to our frame header is not a coincidence — both solve the same length-prefix framing problem.

## Where to go deeper

- **gRPC core concepts** — the official gRPC docs explain the four streaming modes (unary, server, client, bidirectional) and the proto3 type system that backs them: https://grpc.io/docs/what-is-grpc/core-concepts/
- **gRPC over HTTP/2 protocol spec** — the exact framing protocol gRPC uses on top of HTTP/2, including the `Compressed-Flag` and `Message-Length` prefix format: https://github.com/grpc/grpc/blob/master/doc/PROTOCOL-HTTP2.md
- **Apache Thrift binary protocol** — an alternative RPC framing that predates gRPC; its binary protocol document shows the design choices around field tags and type encoding: https://github.com/apache/thrift/blob/master/doc/specs/thrift-binary-protocol.md
- **Twirp (Twitch)** — a minimal RPC framework that runs over HTTP/1.1 instead of HTTP/2; shows that you can have protobuf serialization without the HTTP/2 transport, which clarifies what gRPC's HTTP/2 layer actually buys you: https://twitchtv.github.io/twirp/
- **Go's `net/rpc` package source** — the standard library's built-in RPC package uses the same pending-map + background reader model we implemented; reading it confirms that our design is idiomatic: https://pkg.go.dev/net/rpc
- **`encoding/gob` vs. `encoding/json`** — `net/rpc` uses `gob` by default; the codec interface lets you swap to JSON; comparing gob wire size to JSON illustrates the tradeoff between debuggability and efficiency that protobuf resolves differently

## Related labs and questions

- **Previous in messaging**: Kafka-lite — the segment log uses the same length-prefix record format (`offset int64 + length int32 + payload`) as our frames; the framing problem appears at every layer of a distributed system
