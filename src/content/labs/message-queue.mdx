## Why this lab exists

SQS is one of the most underestimated services in AWS. Developers who've only used it as a task queue — send a job, a worker picks it up, done — miss the operational complexity that makes it interesting: visibility timeouts that simulate consumer crashes, dead-letter queues that catch poison pills, FIFO deduplication that prevents duplicate processing, and long polling that eliminates the cost of empty receives.

These features are not arbitrary. Each one exists because distributed systems fail in specific ways, and each feature is the minimum machinery needed to handle one failure mode correctly.

This lab builds all of it from scratch in Go: from a 180-line in-memory FIFO queue to a 730-line system with DLQs, delayed delivery, per-group FIFO ordering, deduplication, and `sync.Cond` long polling. By the end, you will be able to read an SQS architecture diagram and know exactly what happens in the 30 seconds between a consumer crash and a message reappearing in the queue — and why that 30 seconds is the single most important number to tune.

## What we're building

A message queue in Go, implemented across three stages. Each stage adds one architectural layer that addresses a real operational failure mode.

**We build:**
- A FIFO in-memory queue with `ReceiptHandle`-based deletion and visibility timeout
- A dead-letter queue that captures poison pills after `maxReceiveCount` failed deliveries
- Delayed message delivery where a message is invisible until a future time
- Batch send and batch delete operations
- `QueueAttributes` for visibility into message, in-flight, and delayed counts
- FIFO deduplication using a 5-minute dedup window keyed by `MessageDeduplicationID`
- Per-group message ordering using `MessageGroupID` and a group in-flight map
- Long polling with `sync.Cond` so consumers park instead of busy-waiting
- An HTTP server so any language can interact without a native client library

**We explicitly skip (and why):**
- Durable persistence (disk, WAL) — adds complexity without teaching queue semantics
- Multi-region replication — the Raft lab covers that
- Native SQS wire protocol (AWS Signature v4, SQS XML format) — our broker speaks HTTP
- Message attribute types (String, Number, Binary) — we accept raw bytes
- Per-message encryption — the security labs cover KMS integration

## Mental model

### The queue vs. the log

Before building anything, understand the fundamental distinction between this lab and the Kafka lab:

```
Queue model (this lab):
  Producer → [msg A] [msg B] [msg C] → Consumer
                              ^ message disappears after successful delete

Log model (Kafka lab):
  Producer → [0:A] [1:B] [2:C] → Consumer-1 (cursor at 2)
                                → Consumer-2 (cursor at 0, re-reading A)
                     (messages stay; only cursors move)
```

A queue has **ownership semantics**: when a consumer successfully processes and deletes a message, it is gone. This is the right model for task distribution — send 1,000 emails, each gets processed exactly once by exactly one worker.

A log has **position semantics**: the message stays; only the consumer's cursor moves. This is the right model for event streaming — any number of consumers can read the same event, independently and at their own pace.

### The visibility timeout is not a lock

This is the most common misconception. The visibility timeout is **not a distributed lock** on the message. It is a timer that controls when a message becomes visible again if the consumer disappears.

Here is what actually happens:

1. Consumer calls `ReceiveMessage`. Server marks the message in-flight and returns it with a `ReceiptHandle`. A timer starts.
2. Consumer processes the message. Calls `DeleteMessage(receiptHandle)`. Server permanently removes the message. Timer cancelled.
3. (If consumer crashes) Timer expires. Server returns the message to visible state. Next `ReceiveMessage` call returns it to a (possibly different) consumer.

The message was never "locked" in step 1 — another consumer could receive the same message if they called `ReceiveMessage` at the same time. In practice, `ReceiveMessage` moves messages to in-flight state so they are not returned to other callers, but this is a soft reservation, not a hard lock. The `ReceiptHandle` is proof of the reservation — but only until the timeout expires.

### Why at-least-once is the guarantee, not exactly-once

The visibility timeout means a message can be received more than once in two scenarios:
- The consumer crashes after processing but before calling `DeleteMessage`
- The consumer takes longer than the visibility timeout to process and delete

In both cases, the message reappears. This is **at-least-once delivery**: every message is delivered at least once, but possibly more. If your consumer is not idempotent (safe to run twice with the same input), at-least-once delivery causes bugs.

The deduplication feature (v2) helps with some cases of duplicate delivery from the producer side — if the producer sends the same message twice with the same `MessageDeduplicationID`, only the first arrives. But it does not help with the consumer-side at-least-once scenario.

SQS's exactly-once guarantee for FIFO queues works at the producer level (deduplication) but not at the consumer level — consumers must handle re-delivery.

The smallest message queue that can exist. A `[]Message` slice is the FIFO backing store. `ReceiveMessage` does not pop from the slice — it marks messages as in-flight and returns them with a `ReceiptHandle`. Only `DeleteMessage` permanently removes a message. The goal is to make the at-least-once delivery contract visible without any other complexity.

```go
type Message struct {
    ID            string
    Body          []byte
    EnqueuedAt    time.Time
    ReceiptHandle string
    ReceiveCount  int
    visibleAt     time.Time  // for delayed delivery (v1)
}

type inflightEntry struct {
    msg       Message
    visibleAt time.Time  // when visibility timeout expires
}

type Queue struct {
    mu       sync.Mutex
    name     string
    messages []Message   // FIFO backing store
    head     int         // index of next message to dequeue
    inflight map[string]*inflightEntry
    cond     *sync.Cond  // for long polling (v2)
}
```

The `head` index is the key performance choice: instead of `messages = messages[1:]` (which copies the slice header and may cause allocations), we advance `head`. We compact periodically when `head` is past half the slice length.

**ReceiveMessage**: iterates from `head` to `len(messages)`, skipping messages that are in-flight or delayed. For each candidate, generates a new `ReceiptHandle` (UUID), increments `ReceiveCount`, adds to `inflight` map.

```go
func (q *Queue) ReceiveMessage(maxMessages int, visibilityTimeout time.Duration) []Message {
    q.mu.Lock()
    defer q.mu.Unlock()
    now := time.Now()
    var result []Message
    for i := q.head; i < len(q.messages) && len(result) < maxMessages; i++ {
        msg := &q.messages[i]
        if msg.visibleAt.After(now) { continue }        // delayed
        if q.isInFlight(msg.ID, now) { continue }      // in-flight
        rh := newUUID()
        msg.ReceiveCount++
        msg.ReceiptHandle = rh
        clone := *msg
        q.inflight[rh] = &inflightEntry{msg: clone, visibleAt: now.Add(visibilityTimeout)}
        result = append(result, clone)
    }
    return result
}
```

**The critical test to write**: after receiving a message, verify the second `ReceiveMessage` call returns empty. Then let the visibility timeout expire (wait for the background scanner). Verify the third `ReceiveMessage` returns the same message with `ReceiveCount == 2`. This is the entire at-least-once delivery contract in one test.

**DeleteMessage**: removes the entry from `inflight`, then finds the message in the backing slice by ID and advances `head` past it. The combination of both removals is what prevents the message from reappearing — removing from `inflight` alone would not be enough if the message is still in the slice.

### Dead-letter queues: the poison pill escape valve

Without a DLQ, a message that your consumer cannot process loops forever:
1. Consumer receives message, crashes during processing
2. Visibility timeout expires, message reappears
3. Consumer receives message again, crashes again
4. Repeat indefinitely

This is the poison pill problem. The fix is to track `ReceiveCount` and, after `MaxReceiveCount` receives without a delete, move the message to a separate queue (the DLQ) where it can be inspected without blocking the main queue.

The `Manager` owns queues and enforces DLQ relationships:

```go
type DLQConfig struct {
    MaxReceiveCount int
    DLQName         string
}

// In requeueExpired (background scanner):
if q.dlqConfig != nil && entry.msg.ReceiveCount >= q.dlqConfig.MaxReceiveCount {
    dlqMsg := entry.msg
    dlqMsg.ReceiptHandle = ""
    q.dlq.mu.Lock()
    q.dlq.messages = append(q.dlq.messages, dlqMsg)
    q.dlq.mu.Unlock()
    q.removeByID(entry.msg.ID)
    continue
}
```

**The DLQ chain restriction**: a queue that has a DLQ attached cannot itself be used as a DLQ. This prevents chains like `main → dlq-1 → dlq-2 → ...`. SQS enforces the same restriction. The reason is operational: if DLQ promotion could trigger further DLQ promotion, a single poison pill could cascade across an unbounded chain of queues. The restriction forces DLQs to be terminal.

### Delayed delivery: scheduling messages

A delayed message is enqueued in the normal backing slice but has a `visibleAt` time in the future. `ReceiveMessage` skips messages where `visibleAt.After(now)`. No separate data structure is needed.

```go
func (q *Queue) sendWithDelay(body []byte, delaySeconds int) string {
    id := newUUID()
    now := time.Now()
    visibleAt := now
    if delaySeconds > 0 {
        visibleAt = now.Add(time.Duration(delaySeconds) * time.Second)
    }
    msg := Message{ID: id, Body: body, EnqueuedAt: now, visibleAt: visibleAt}
    q.mu.Lock()
    q.messages = append(q.messages, msg)
    q.mu.Unlock()
    return id
}
```

**Use cases for delayed delivery**: scheduling retries with exponential backoff (send a retry with delay 2s, 4s, 8s), time-based triggers (send a "payment overdue" message with delay = due date - now), and rate limiting (space out API calls by delaying messages).

### Batch operations: why atomicity matters

`BatchSendMessage` enqueues up to 10 messages in a single mutex lock acquisition. This is not just about throughput — it's about atomicity. If the caller sends 5 messages and 3 of them are processed before the other 2 arrive, the system may see a partial update.

The SQS batch API has the same semantics: all messages in a successful batch send are either all enqueued or the call fails. There is no partial success at the batch level (though individual message failures within the batch can be reported separately).

### FIFO ordering with MessageGroupID

A standard FIFO queue guarantees ordering across all messages. This creates a bottleneck: if message N is in-flight, messages N+1, N+2, ... must wait. For a queue handling 10 message groups with independent consumers, this is a 10x throughput reduction.

`MessageGroupID` solves this by partitioning the FIFO guarantee: messages within the same group are delivered in order, but messages from different groups can be interleaved. If group A has a message in-flight, only group A is blocked — groups B, C, ... continue at full speed.

```go
type FIFOQueue struct {
    *Queue
    fifoMu        sync.Mutex
    dedupMap      map[string]dedupEntry  // dedupID → {messageID, expiresAt}
    groupInflight map[string]int         // groupID → count of in-flight messages
    fifoMessages  []FIFOMessage          // parallel to Queue.messages
}

// In ReceiveFIFOMessage:
if fq.groupInflight[fifoMsg.GroupID] > 0 {
    continue  // this group is blocked; skip to the next group's messages
}
```

This is why SQS FIFO queues have a per-group throughput limit (3,000 msg/sec with batching) rather than a global limit — groups are the unit of isolation, not the queue.

### Deduplication: the 5-minute window

The deduplication window prevents the same message from being enqueued twice within 5 minutes. This is useful for producers that retry on network failures — if the first send succeeded but the acknowledgement was lost, the retry produces the same `MessageDeduplicationID` and is silently discarded.

```go
const dedupWindow = 5 * time.Minute

type dedupEntry struct {
    messageID string
    expiresAt time.Time
}

func (fq *FIFOQueue) SendFIFOMessage(body []byte, groupID, dedupID string) string {
    fq.fifoMu.Lock()
    defer fq.fifoMu.Unlock()
    if dedupID != "" {
        fq.pruneDedup(time.Now())
        if entry, ok := fq.dedupMap[dedupID]; ok {
            return entry.messageID  // original ID, no enqueue
        }
    }
    // ... enqueue and record in dedupMap
}
```

Deduplication is a best-effort guarantee at the producer level only. It does not prevent the same message from being delivered twice to consumers (that's the at-least-once consumer delivery scenario we covered in v0). The two guarantees are independent.

### Long polling with sync.Cond

Without long polling, a consumer on an empty queue must poll repeatedly:

```go
// Busy poll — burns CPU and generates meaningless API calls:
for {
    msgs := q.ReceiveMessage(10, 30*time.Second)
    if len(msgs) > 0 { process(msgs) }
    time.Sleep(100 * time.Millisecond)  // 10 polls/sec on empty queue
}
```

With long polling, the consumer parks on a condition variable and is woken when a message arrives:

```go
func (q *Queue) LongPollReceive(maxMessages int, visibilityTimeout, waitTime time.Duration) []Message {
    deadline := time.Now().Add(waitTime)
    // Timer goroutine: broadcast after deadline to unblock the waiter.
    timer := time.AfterFunc(waitTime, func() { q.cond.Broadcast() })
    defer timer.Stop()

    q.mu.Lock()
    for {
        msgs := q.receiveUnlocked(maxMessages, visibilityTimeout)
        if len(msgs) > 0 { q.mu.Unlock(); return msgs }
        if time.Now().After(deadline) { q.mu.Unlock(); return nil }
        q.cond.Wait()  // releases mu, parks goroutine, re-acquires on wake
    }
}
```

`SendMessage` calls `q.cond.Broadcast()` after every enqueue, which wakes all parked consumers. The first consumer to wake and acquire the lock gets the message; others loop back and check (some find the queue empty again and re-park).

**Why `sync.Cond` instead of channels**: a channel-based approach would require one channel per consumer, complicating fan-out. `sync.Cond.Broadcast()` wakes all parked consumers simultaneously with one call, and the mutex ensures only one receives the message. This matches exactly the semantics of SQS long polling, where all connected pollers are eligible to receive any available message.

## Stress tests

**Visibility timeout storm**: send 10,000 messages and receive all of them with a 100ms visibility timeout. Do not call `DeleteMessage`. After 100ms, all 10,000 messages reappear. Your background scanner now has to process 10,000 re-queues in one tick. Verify that the scanner doesn't hold the mutex long enough to block concurrent senders. The fix is to process re-queues in batches and release the mutex between batches.

**DLQ fill rate under crash simulation**: send 100 messages to a queue with `maxReceiveCount=3` and `visibilityTimeout=50ms`. Receive all 100 simultaneously and never delete any. After 150ms (3 timeout cycles), all 100 should be in the DLQ. Verify `QueueAttributes.MessageCount == 0` on the main queue and the DLQ has exactly 100 messages.

**Long poll fan-out**: start 20 goroutines all calling `LongPollReceive(1, 30s, 10s)` on an empty queue. Then send 20 messages rapidly. Verify exactly 20 messages are received (one per goroutine) with no duplicates. The `sync.Cond.Broadcast` wakes all 20, but only 20 receive messages — the mutex ensures serialized access.

**FIFO group contention**: create 2 message groups (A and B), each with 50 messages. Two concurrent consumers process in parallel — one dedicated to group A, one to group B. Verify total throughput is 2x a single-group setup (groups are independent) and that all group A messages arrive in order and all group B messages arrive in order.

**Dedup window expiry**: send a message with `MessageDeduplicationID = "order-1001"`. Wait 5 minutes (in tests, mock the expiry). Send the same dedup ID again. Verify a new message is enqueued (not deduplicated) because the window has expired.

### The @JmsListener comparison

Spring's `@JmsListener` hides the entire send-receive-delete cycle behind a single annotation:

```java
@JmsListener(destination = "orders")
public void onOrder(String body) {
    processOrder(body);
    // Spring commits the JMS "ack" automatically after this method returns.
    // If the method throws, the message is redelivered (JMS session rollback).
}
```

Behind that annotation, `DefaultMessageListenerContainer` is doing exactly what our `QueueClient` does explicitly:
1. Poll `MessageConsumer.receive()` in a thread pool
2. Open a JMS Session (= our visibility timeout window)
3. Deliver the message to your `@JmsListener` method
4. On success: `session.commit()` (= our `DeleteMessage`)
5. On exception: `session.rollback()` (= message reappears after visibility timeout)
6. After N rollbacks: move to dead-letter queue (configured via `activemq.redeliveryPolicy`)

Making this explicit is not a step backward — it forces you to think about what "at-least-once delivery" means in your application:
- What happens if your consumer crashes after step 3 but before step 4?
- What is the right visibility timeout for a 2-minute database update?
- How do you drain a DLQ without re-triggering the bug that put messages there?

### Part A — QueueClient: the HTTP bridge

`QueueClient` wraps three endpoints behind a typed interface. Under 60 lines of logic:

```java
public class QueueClient {
    private final RestClient http;

    public String sendMessage(String queueName, String body) {
        var resp = http.post()
                .uri("/queues/{name}/messages", queueName)
                .contentType(MediaType.APPLICATION_JSON)
                .body(Map.of("body", body))
                .retrieve()
                .body(SendResponse.class);
        return resp.messageId();
    }

    public List<QueueMessage> receiveMessages(String queueName, int max, int visTimeoutSec) {
        var resp = http.get()
                .uri("/queues/{name}/messages?maxMessages={max}&visibilityTimeout={vis}",
                     queueName, max, (double) visTimeoutSec)
                .retrieve()
                .body(ReceiveResponse.class);
        return resp.messages();
    }

    public void deleteMessage(String queueName, String receiptHandle) {
        http.delete()
                .uri("/queues/{name}/messages/{rh}", queueName, receiptHandle)
                .retrieve()
                .toBodilessEntity();
    }

    public record QueueMessage(String id, String body, String receiptHandle, int receiveCount) {}
}
```

The structural difference from the real AWS SQS SDK: our `sendMessage` is synchronous (blocks until the server acknowledges). The real `SqsAsyncClient.sendMessage()` is async — it returns a `CompletableFuture` and batches messages client-side before flushing. Our HTTP client sends one at a time.

### Part B — The full processing loop

```java
// At-most-once (commit BEFORE processing — dangerous):
List<QueueMessage> msgs = client.receiveMessages(queue, 10, 30);
client.deleteMessage(queue, msg.receiptHandle());  // commit first
processOrder(msg.body());                          // if this crashes, message is LOST

// At-least-once (commit AFTER processing — correct default):
List<QueueMessage> msgs = client.receiveMessages(queue, 10, 30);
processOrder(msg.body());                          // process first
client.deleteMessage(queue, msg.receiptHandle());  // if this crashes, message RE-DELIVERS
```

The real Spring JMS `@JmsListener` defaults to at-least-once (session transaction committed after method returns). The AWS Spring Cloud SQS `@SqsListener` also defaults to at-least-once (delete after successful return). Our demo explicitly shows the delete call so you cannot miss it.
