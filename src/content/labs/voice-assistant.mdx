## Why this lab exists

Voice is the fastest-growing AI interface of the decade. Alexa crossed 100 million devices. ChatGPT's voice mode drove millions of new sessions within weeks of launch. Medical transcription, call-center automation, and accessibility tooling all run the same underlying architecture: speech-to-text to extract meaning, a language model to generate a response, text-to-speech to speak it back.

The pipeline architecture — STT to LLM to TTS — is universal. Whether you are building a smart speaker, a medical documentation assistant, or a customer service bot, the fundamental questions are identical: how do you detect when the user has finished speaking, how do you maintain conversational context across turns, and how do you make the response feel fast when three separate models must run sequentially?

This lab forces you to confront those questions concretely. You will implement energy-based voice activity detection (VAD) from 20 lines of math, wire Whisper for transcription, manage a conversation history that does not overflow, and use sentence-level TTS streaming to cut time-to-first-audio in half. The result runs entirely offline on a laptop in under 600ms from speech end to first audio out.

The commercial voice assistants (Siri, Alexa, GPT-4o voice) hide this complexity behind opaque endpoints. This lab makes the seams visible — and shows you exactly which seam costs how many milliseconds.

## What we are building

A streaming voice assistant in Python, built across three stages. Each stage adds one capability and measures its effect on latency.

**We build:**
- `VoiceActivityDetector`: energy-based RMS VAD — no neural model, zero GPU required
- `AudioBuffer`: end-of-speech detection via 500ms silence threshold
- `WhisperSTT`: OpenAI Whisper-tiny wrapping for 16kHz mono PCM transcription
- `ConversationHistory`: rolling Human/Assistant dialogue with max-turns pruning
- `StubLM`: a 2-layer byte-level transformer for offline response generation
- `TextToSpeech` + `StreamingTTS`: pyttsx3-based TTS with sentence-level streaming
- `PipelineOrchestrator`: chains all three stages, measures per-stage latency
- `FastAPI` server with `POST /transcribe` and `WS /ws/voice` binary WebSocket endpoint
- Spring Boot `VoiceWebSocketClient` that connects and streams audio chunks

**We explicitly skip:**
- Wake-word detection (Porcupine, microWakeWord): covered in "What the Toy Misses"
- End-to-end speech models (Moshi, GPT-4o voice): requires specialized training
- Barge-in / interruption handling: requires mid-stream cancellation infrastructure
- Acoustic echo cancellation: requires DSP integration (WebRTC AEC)
- Real-time microphone capture: use PyAudio or sounddevice in production; this lab uses pre-recorded PCM for reproducibility

## Mental model

The voice pipeline is a latency budget. Every stage consumes time, and the user perceives the sum as "response latency" — the gap between finishing their sentence and hearing the assistant begin to reply.

```
User speaks → [VAD] → [Whisper STT] → [LLM] → [TTS] → Speaker

Latency budget (M1 Pro, Whisper-tiny):

  End-of-speech detection:    500ms  (fixed: 50 × 10ms silence chunks)
  Whisper-tiny transcription: 290ms  (for 2s of audio)
  LLM generation (StubLM):     80ms  (50 tokens; real LLM: 200-500ms)
  TTS first sentence:         150ms  (pyttsx3 / macOS AVSpeechSynthesizer)
  ─────────────────────────────────
  Time-to-first-audio:        520ms  (perceived response start)

Without sentence streaming (full response before any TTS):
  TTS all sentences:          550ms  (3-sentence response)
  Time-to-first-audio:       1100ms  (2.1x slower)
```

The key insight is that the user does not need to hear the entire response before they perceive it as "fast." They need to hear the first word. Sentence-level streaming synthesizes and sends sentence 1 while sentence 2 is being generated — the user hears a response in 520ms even though the full pipeline takes 1100ms.

The VAD state machine has three states: IDLE (waiting for speech to begin), RECORDING (accumulating audio chunks), and COMPLETE (500ms of consecutive silence after speech signals utterance end). The 500ms threshold is a deliberate design choice: 300ms feels abrupt for natural pauses within a sentence; 700ms adds noticeable lag.

Voice activity detection is a gating mechanism. You do not want to run Whisper — which costs 290ms — on every 10ms chunk. You run it only when a complete utterance has been detected. The VAD's job is to identify that moment cheaply.

**RMS as a speech proxy:**

Root-mean-square energy measures the average power of an audio signal. For 16-bit PCM samples, the formula is:

```python
rms_raw = sqrt(sum(s*s for s in samples) / n_samples)
normalized = rms_raw / 32767.0   # normalize to [0.0, 1.0]
```

Silence (all zeros) has RMS 0.0. Background hiss has RMS around 0.005. Normal speech has RMS 0.05-0.3. A threshold of 0.02 correctly classifies both in a typical laptop environment.

**End-of-speech state machine:**

```python
CHUNK_SAMPLES = 160           # 10ms at 16kHz
SILENCE_THRESHOLD_CHUNKS = 50 # 50 × 10ms = 500ms

class AudioBuffer:
    def feed(self, chunk: bytes) -> Optional[bytes]:
        is_speech = VoiceActivityDetector.is_speech(chunk)

        if not self._speech_started:
            if is_speech:
                self._speech_started = True
                self._buffer.extend(chunk)
            return None   # drop leading silence

        self._buffer.extend(chunk)

        if is_speech:
            self._silence_chunks = 0
        else:
            self._silence_chunks += 1

        if self._silence_chunks >= SILENCE_THRESHOLD_CHUNKS:
            utterance = bytes(self._buffer)
            self._reset()
            return utterance   # complete utterance ready

        return None
```

**Whisper's audio format requirement:**

Whisper loads audio from a file path, not from bytes. It expects 16kHz mono WAV. If you feed it stereo audio or 44.1kHz audio, transcription quality degrades significantly. The `wave` module handles the conversion:

```python
def transcribe(self, audio_bytes: bytes) -> str:
    with tempfile.NamedTemporaryFile(suffix=".wav", delete=False) as tmp:
        tmp_path = tmp.name
    with wave.open(tmp_path, "wb") as wf:
        wf.setnchannels(1)       # mono
        wf.setsampwidth(2)       # 16-bit
        wf.setframerate(16000)   # 16kHz
        wf.writeframes(audio_bytes)
    result = self.model.transcribe(tmp_path)
    return result["text"].strip()
```

The temporary file pattern is necessary because Whisper uses librosa under the hood, which requires a file path. In production, you would use `io.BytesIO` with a custom loader, but the temp file approach is simpler and correct.

Every voice assistant is a stateful dialogue system. The user says something, the assistant responds, the user says something else informed by the response — and the LLM needs to see that history to give a coherent next reply.

**The Human/Assistant prompt format:**

The simplest stateful dialogue pattern is a text prompt that alternates roles:

```
Human: What is the capital of France?
Assistant: The capital of France is Paris.
Human: What is its population?
Assistant:
```

The LLM is prompted to continue from the final `"Assistant: "` line. This maps to every production API's message format (OpenAI's `role: "user"/"assistant"` fields, Anthropic's `Human:`/`Assistant:` syntax). The `to_prompt()` method generates this string from the message list.

**Max-turns pruning:**

Without pruning, the prompt grows without bound. At Whisper's word rate (~150 words/minute), a 10-minute conversation accumulates ~1,500 words — exceeding typical context windows and adding significant LLM latency. The pruning strategy is FIFO: when the message count exceeds `max_turns * 2`, discard the oldest messages from the front:

```python
max_messages = self.max_turns * 2
if len(self.messages) > max_messages:
    excess = len(self.messages) - max_messages
    self.messages = self.messages[excess:]
```

**StubLM autoregressive loop:**

```python
def generate_response(history, user_text, max_tokens=50):
    history.add("human", user_text)
    prompt_bytes = history.to_prompt().encode("utf-8")[:256]
    token_ids = torch.tensor(list(prompt_bytes), dtype=torch.long)

    generated = []
    with torch.no_grad():
        for _ in range(max_tokens):
            context = token_ids[-256:]      # truncate to MAX_SEQ
            logits = model(context)         # (seq_len, 256)
            next_token = logits[-1].argmax().item()   # greedy
            generated.append(next_token)
            token_ids = torch.cat([token_ids, torch.tensor([next_token])])
            if next_token == 10:            # newline = stop
                break

    response = bytes(generated).decode("utf-8", errors="replace").strip()
    history.add("assistant", response)
    return response
```

The StubLM produces incoherent byte sequences because it has no training — it is initialized with random weights and never updated. Its purpose is to demonstrate the autoregressive pattern at byte level without requiring a model download or API key. Replace `generate_response` with an OpenAI or Ollama API call for coherent output.

The latency breakthrough in this stage comes from a one-line text-splitting function.

**Why sentence streaming matters:**

A 3-sentence response takes about 550ms total TTS time (150ms per sentence × ~3.7 sentences). Without streaming, the client waits 290ms (STT) + 80ms (LLM) + 550ms (full TTS) = 920ms before hearing anything.

With sentence streaming, the server synthesizes sentence 1 (150ms) and sends it while still generating sentences 2 and 3. The client hears the first word at 290ms + 80ms + 150ms = 520ms — a 1.8x reduction in perceived latency.

**The sentence splitter:**

```python
import re

def split_sentences(text: str) -> list[str]:
    parts = re.split(r"(?&lt;=[.!?])\s+", text)
    return [s.strip() for s in parts if s.strip()]
```

The regex uses a lookbehind assertion to split after sentence-ending punctuation followed by whitespace. This preserves the punctuation in each sentence (unlike `str.split(".")`).

**pyttsx3 save-to-file pattern:**

pyttsx3 does not expose an in-memory audio buffer. The only way to get WAV bytes is to save to a temporary file:

```python
def synthesize(self, text: str) -> bytes:
    with tempfile.NamedTemporaryFile(suffix=".wav", delete=False) as tmp:
        tmp_path = tmp.name
    self._engine.save_to_file(text, tmp_path)
    self._engine.runAndWait()   # blocks until synthesis complete
    with open(tmp_path, "rb") as f:
        return f.read()
```

`engine.runAndWait()` is a blocking call. This means each sentence takes 140-200ms. In production, you would use a non-blocking TTS engine (Kokoro, Coqui) that exposes an audio stream. For this lab, the blocking call is acceptable because we synthesize sentences sequentially and stream each one to the client as it completes.

**Pipeline orchestration with per-stage timing:**

```python
class PipelineOrchestrator:
    def process_utterance(self, audio_bytes: bytes):
        t0 = time.time()
        transcript = self._stt.transcribe(audio_bytes)
        t_stt = time.time() - t0

        t1 = time.time()
        response_text = generate_response(self._history, transcript)
        t_llm = time.time() - t1

        t2 = time.time()
        audio_chunks = list(self._tts.synthesize_stream(response_text))
        t_tts = time.time() - t2

        print(f"stt={t_stt*1000:.0f}ms llm={t_llm*1000:.0f}ms tts={t_tts*1000:.0f}ms")
        return transcript, response_text, audio_chunks
```

These timing measurements are not optional logging — they are the primary instrument for identifying where to invest optimization effort. On a machine where Whisper-large-v3 is installed, `t_stt` will be the dominant term. On a machine with a slow Python runtime, `t_llm` with StubLM might dominate. The numbers tell you where to look.

## Stress tests

**Concurrent sessions**: Start the server and connect 5 WebSocket clients simultaneously, each streaming a 2-second audio file. Measure per-client latency. At 5 sessions with Whisper-tiny on a single CPU core, expect 5x latency degradation (sessions queue behind each other) since Whisper is single-threaded. This reveals why production voice servers use async request queues with dedicated inference workers.

**Long conversation overflow**: Drive a single WebSocket session with 25 consecutive utterances (each generating a 3-turn history addition). After turn 20, verify that `ConversationHistory.max_turns` pruning is occurring — the prompt length should plateau rather than growing linearly. Measure LLM inference latency at turn 5 vs turn 20; without pruning, LLM latency would grow with prompt length.

**TTS sentence count scaling**: Generate responses of 1, 3, 5, and 10 sentences by adjusting `max_tokens`. Measure time-to-first-audio for each. It should remain constant at approximately 520ms regardless of total response length — only the first sentence's TTS is on the critical path. Total response duration grows linearly with sentence count (140ms per sentence), but the user does not wait for it.

**VAD false positive rate**: Feed 60 seconds of background noise at RMS 0.01 through the AudioBuffer with default threshold 0.02. Count false utterance triggers. A well-tuned threshold should produce zero false positives on quiet background noise but correctly trigger on speech at RMS 0.05+. Record the false-positive rate for your specific microphone environment to calibrate the threshold.

**Whisper cold start vs warm start**: Time the first call to `WhisperSTT.transcribe()` (cold, model just loaded) vs the second call (warm, model in memory, mel filters cached). Expect 2-5x higher latency on the cold call because Whisper loads its mel filterbank and STFT windows on first inference. Always warm up the model before serving.

### How it works

The Java integration connects to `ws://localhost:8000/ws/voice` using Spring's `StandardWebSocketClient`. It streams pre-recorded audio (or a synthetic sine wave if no WAV file is found on the classpath) as 320-byte binary WebSocket frames at 10ms intervals — exactly matching the server's expected input format.

The server collects chunks, detects the utterance end via VAD, runs the STT-LLM-TTS pipeline, and sends back binary WAV chunks followed by a JSON text frame. The Java client collects these and writes the combined audio to disk.

### VoiceWebSocketClient.java

```java
@Component
public class VoiceWebSocketClient implements CommandLineRunner {

    private static final int CHUNK_BYTES = 320;         // 160 samples x 2 bytes
    private static final int CHUNK_INTERVAL_MS = 10;   // simulate real-time

    @Override
    public void run(String... args) throws Exception {
        byte[] pcmBytes = loadTestAudio();  // from classpath WAV, strips header
        List&lt;byte[]&gt; audioChunks = new ArrayList&lt;&gt;();
        CountDownLatch latch = new CountDownLatch(1);

        AbstractWebSocketHandler handler = new AbstractWebSocketHandler() {
            @Override
            protected void handleBinaryMessage(WebSocketSession s, BinaryMessage m) {
                byte[] chunk = new byte[m.getPayload().remaining()];
                m.getPayload().get(chunk);
                audioChunks.add(chunk);
            }
            @Override
            protected void handleTextMessage(WebSocketSession s, TextMessage m) {
                System.out.println("[response] " + m.getPayload());
                latch.countDown();
            }
        };

        WebSocketSession session = new StandardWebSocketClient()
            .execute(handler, props.wsUrl())
            .get(10, TimeUnit.SECONDS);

        // Stream audio in 10ms chunks at real-time rate
        int totalChunks = pcmBytes.length / CHUNK_BYTES;
        for (int i = 0; i &lt; totalChunks; i++) {
            byte[] chunk = Arrays.copyOfRange(pcmBytes, i*CHUNK_BYTES, (i+1)*CHUNK_BYTES);
            session.sendMessage(new BinaryMessage(ByteBuffer.wrap(chunk)));
            Thread.sleep(CHUNK_INTERVAL_MS);
        }

        latch.await(30, TimeUnit.SECONDS);
        session.close();
    }
}
```

### Why binary WebSocket over HTTP streaming

HTTP chunked transfer (SSE) works well for text token streaming. But audio chunks are binary and latency-sensitive — each 10ms chunk that arrives late causes a glitch in the audio playback stream. WebSocket binary frames have lower framing overhead than HTTP and maintain a persistent connection, avoiding TCP handshake latency on each chunk. Spring's `StandardWebSocketClient` uses the Tyrus WebSocket client under the hood, which supports binary frames natively.
