## Why this lab exists

If you work on distributed systems long enough, you will eventually need to answer one question under pressure: "How does the cluster decide who is in charge?" Every stateful distributed system — etcd, CockroachDB, TiKV, Zookeeper's successor protocols, Kafka KRaft, Consul — has a consensus layer at its heart. Raft is the algorithm that made that layer understandable to mere mortals.

The Raft paper is titled "In Search of an Understandable Consensus Algorithm." That is not false advertising — Raft really is more approachable than Paxos. But there is a gap between reading the paper and actually building it. The paper is a specification. The implementation is where the edge cases live: what happens when two candidates tie? What happens when the leader appends to its own log but crashes before quorum acknowledges? What happens when a follower is so far behind that replaying individual entries would take minutes?

Until you have fought through those questions in real code, you don't know Raft. You know about Raft. This lab closes that gap in three progressive stages.

The implementation is in Go. Three goroutines simulate three Raft nodes communicating over channels — no network required, but the protocol is identical to the real thing. Everything that matters about Raft — election safety, log matching, commitment rules — is present and testable.

## What we're building

A Raft consensus implementation in Go across three stages.

**We build:**
- Leader election with randomized election timeouts
- Log replication with the AppendEntries consistency check
- A simple key/value state machine applied from committed log entries
- Snapshot compaction with InstallSnapshot RPC
- An HTTP server exposing each node's state for the Java client

**We explicitly skip (and why):**
- Network transport (channels simulate RPCs; the protocol is identical)
- Persistent log storage (entries live in memory; production Raft writes to disk before responding)
- Cluster membership changes (joint consensus is a whole separate algorithm)
- Pre-vote and leader lease extensions

## Mental model

Raft provides two guarantees that everything else is built on.

**Election safety**: at most one leader per term. A term is a monotonically increasing integer — think of it as a Lamport clock restricted to leadership epochs. A candidate can only win if it gets votes from a majority, and each node votes at most once per term. Two candidates cannot both get a majority in the same term because there are not enough votes. If they tie (each gets less than majority), the term ends without a leader and the next term starts a new election.

**Log matching**: if two logs have an entry at the same index with the same term, then all preceding entries are identical. This is the invariant that AppendEntries enforces. The leader sends its previous entry's index and term as a consistency check. If the follower's log does not match at that position, it rejects the entry and the leader backs up.

These two guarantees together mean that if a log entry is committed (acknowledged by a majority), it will be present in the log of every future leader. The state machine is just a function of the log. If two nodes have identical committed logs, they have identical state machines.

A five-node election, step by step:

```
Initial state (all Followers, term=0):
  N0  N1  N2  N3  N4

N2's timeout fires first. N2 becomes Candidate, increments term to 1:

  N2 sends RequestVote(term=1, candidate=N2) to N0, N1, N3, N4

Vote decisions (each node, term=1):
  N0: hasn't voted → grants (N0 votedFor=N2)
  N1: hasn't voted → grants (N1 votedFor=N2)
  N3: hasn't voted → grants (N3 votedFor=N2)
  N4: hasn't voted → grants (N4 votedFor=N2)

N2 has 5 votes (including self). Majority = 3. N2 wins.

  N2 becomes Leader. Sends heartbeat AppendEntries(term=1) to all.
  N0, N1, N3, N4 reset timers, state=Follower.

Term 1 vote matrix:
        N0   N1   N2   N3   N4
  N2  [ Y    Y    -    Y    Y  ]  ← 4 votes + self = 5. Majority wins.
```

In practice with randomized timeouts (150–300ms range), the first node to time out almost always wins before any other node even starts an election. Ties happen when two nodes have nearly identical timeouts, but the next term resolves it.

The v0 node has three fields that matter: `state`, `currentTerm`, `votedFor`.

```go
type Node struct {
    mu          sync.Mutex
    id          int
    peers       []*Node

    currentTerm int
    votedFor    int   // -1 means not voted
    state       State // Follower / Candidate / Leader

    stopCh     chan struct{}
    resetTimer chan struct{} // follower resets election timer on this
}
```

The election logic is three goroutine loops:

```go
func (n *Node) runFollower() {
    timer := time.NewTimer(randomTimeout()) // 150–300ms
    for {
        select {
        case <-n.resetTimer:
            timer.Reset(randomTimeout()) // heartbeat received
        case <-timer.C:
            n.state = Candidate          // timeout: start election
            return
        }
    }
}

func (n *Node) runCandidate() {
    n.currentTerm++
    n.votedFor = n.id
    // Send RequestVote to all peers; collect replies; if majority → become Leader.
}

func (n *Node) runLeader() {
    ticker := time.NewTicker(50 * time.Millisecond) // heartbeat every 50ms
    for range ticker.C {
        n.broadcastAppendEntries() // empty entries = heartbeat
    }
}
```

**The key insight about randomized timeouts**: if all nodes had the same timeout, they would all start elections simultaneously, and every election would be a split vote. With timeouts drawn uniformly from 150–300ms, the expected gap between the first and second timeout is about 75ms — more than enough for the first node to win before others even start. The probability of a tie drops to approximately (heartbeat interval / timeout range)^2, which is roughly 0.1% in our setup.

**Why higher term always wins**: when a node sees a message with a higher term than its own, it immediately reverts to Follower and updates its term. This prevents old leaders from making progress after a partition heals. A leader from term 5 that was partitioned cannot override a leader from term 7 that was elected during the partition — it will see the higher term and step down.

The v1 log fields:

```go
type LogEntry struct {
    Term    int
    Command string
}

// Added to Node:
log         []LogEntry
commitIndex int
lastApplied int

// Leader-only (re-initialised on election win):
nextIndex   []int  // next entry to send to each peer
matchIndex  []int  // highest entry known to be replicated on each peer
```

The AppendEntries consistency check, with a concrete example. Suppose the follower's log looks like this (indices 1-4, term annotations):

```
Follower log:   [1:cmd_a] [2:cmd_b] [3:cmd_c]
                  idx=1     idx=2     idx=3
```

The leader wants to send entry at index 4. It attaches `prevLogIndex=3, prevLogTerm=3`. The follower checks: "does my entry at index 3 have term 3?" If yes, accept. If no (because the follower missed a re-election and has an entry from term 2 at index 3), reject.

When the follower rejects, the leader decrements `nextIndex` for that follower and retries with an earlier prevLogIndex. Eventually they find the point of agreement and the follower's diverging tail is replaced.

```go
func (n *Node) HandleAppendEntries(args AppendEntriesArgs, reply *AppendEntriesReply) {
    // ... term check ...
    
    // Consistency check:
    if args.PrevLogIndex > 0 {
        if args.PrevLogIndex > len(n.log) {
            reply.ConflictIndex = len(n.log) + 1
            return // we don't have this entry
        }
        if n.log[args.PrevLogIndex-1].Term != args.PrevLogTerm {
            reply.ConflictTerm = n.log[args.PrevLogIndex-1].Term
            return // term mismatch
        }
    }
    
    // Append (overwriting any conflicting tail):
    n.log = n.log[:args.PrevLogIndex]
    n.log = append(n.log, args.Entries...)
    
    // Advance commitIndex:
    if args.LeaderCommit > n.commitIndex {
        n.commitIndex = min(args.LeaderCommit, len(n.log))
    }
}
```

**Commitment rule — the critical subtlety**: the leader advances `commitIndex` only when a majority of `matchIndex[]` values reach index N, AND the entry at index N has the leader's current term. This is the "commitment without current-term entries" restriction. A new leader cannot commit old entries by counting their replicas — it can only commit them indirectly by appending and committing a new entry in its own term. This restriction is why data loss bugs in Raft implementations almost always appear around leader changes.

The state machine is applied on the applier goroutine:

```go
// Command format: "SET key value" or "DEL key"
func (n *Node) applyToStateMachine(cmd string) {
    parts := strings.Fields(cmd)
    switch parts[0] {
    case "SET": n.stateMachine[parts[1]] = parts[2]
    case "DEL": delete(n.stateMachine, parts[1])
    }
}
```

Two nodes that have applied the same sequence of committed log entries will always have identical state machines — regardless of timing, network delays, or how many terms have elapsed.

Without snapshots, the log grows forever. A node that was offline for an hour needs to replay every entry since it was last seen. A node joining the cluster fresh needs the entire history. Neither is acceptable in production.

The state before snapshot compaction (leader, after 1050 entries):

```
Log:    [1:SET k0 v0] ... [1000:SET k999 v999] [1001:SET k1000 v1000] ... [1050:SET k1049 v1049]
        snapshotLastIndex=0, snapshotLastTerm=0
        commitIndex=1050
```

After taking a snapshot at commitIndex=1050:

```
Log:    [1001:SET k1000 v1000] ... [1050:SET k1049 v1049]  ← entries after snapshot retained
        snapshotLastIndex=1000, snapshotLastTerm=1
        snapshot.StateMachine = {k0:v0, k1:v1, ..., k1049:v1049}
```

The snapshot contains the complete state machine at that point. The truncated log contains only entries after the snapshot. To find the entry at absolute index N, compute `localIndex = N - snapshotLastIndex - 1` and look up `log[localIndex]`.

A follower that is behind the snapshot boundary receives `InstallSnapshot` instead of `AppendEntries`:

```
Before InstallSnapshot:
  Follower log:   [1:SET k0 v0] ... [400:SET k399 v399]
  commitIndex=400, stateMachine={k0:v0,...,k399:v399}

InstallSnapshot(lastIncludedIndex=1000, data=snapshot):
  Follower log:   [] (entries before 1000 discarded)
  snapshotLastIndex=1000
  stateMachine = snapshot.StateMachine = {k0:v0,...,k999:v999}
  commitIndex=1000
  
After replication catches up:
  Follower log:   [1001:SET k1000 v1000] ... [1050:SET k1049 v1049]
  commitIndex=1050, stateMachine={k0:v0,...,k1049:v1049}
```

The follower went from 400 committed entries to 1050 in one snapshot install plus 50 individual entries — instead of replaying 650 individual entries.

**The key insight**: the snapshot is not a shortcut around the log. The snapshot IS the log, compacted. The combination of (snapshot at index N) + (log entries after N) is fully equivalent to the original complete log. Any state machine that replays this combination will reach the same state as one that replayed the complete log from the beginning.

## Stress tests

**Two simultaneous leaders?** In our in-process cluster this cannot happen — the channel model ensures atomic delivery. On a real network, a partitioned old leader may keep sending AppendEntries to the minority partition while a new leader has been elected in the majority. The new leader's term is higher; any node that sees both will reject the old leader's messages immediately.

**10x write rate?** With 180,000 commands/sec, the command channel fills up faster than the leader can append and replicate. `Submit()` returns false when the channel is full. Production Raft systems use bounded queues with backpressure — callers block until the channel drains. Our implementation drops commands silently at 10x; the fix is a blocking submit with a context deadline.

**Killing a follower then rejoining?** Stop one follower with `node.Stop()`. Submit 200 commands to the leader. Restart the follower. The leader detects the follower is behind via the decreasing `nextIndex` loop and replays the missed entries. If the follower is behind the snapshot boundary, it receives `InstallSnapshot` first. Either way, it catches up within the next heartbeat cycle.

**Killing the leader mid-write?** Submit a command. Kill the leader after `Submit()` returns but before the majority acknowledges. The command is in the leader's log but not committed. The new leader does not commit entries from old terms by direct counting — it only commits them as a side effect of committing a new entry in its own term. The uncommitted entry may or may not appear in the final log, and that is correct: Raft only guarantees committed entries are durable.

### Part A — RaftClient: the per-node HTTP bridge

Every node in the cluster exposes the same four routes. `RaftClient.java` wraps them using Spring Framework 6.1's `RestClient`:

```java
public class RaftClient {
    public record NodeState(int id, String state, int term,
                            int commitIndex, int logLen, boolean isLeader) {}
    public record CommandResult(boolean accepted, int nodeId, String error) {}

    private final RestClient http;

    public RaftClient(String baseUrl) {
        this.http = RestClient.builder().baseUrl(baseUrl).build();
    }

    public NodeState getState() {
        return http.get().uri("/state").retrieve().body(NodeState.class);
    }

    public CommandResult submitCommand(String cmd) {
        record Req(String cmd) {}
        try {
            return http.post().uri("/command")
                    .contentType(MediaType.APPLICATION_JSON)
                    .body(new Req(cmd)).retrieve().body(CommandResult.class);
        } catch (HttpServerErrorException ex) {
            // Parse 503 body — it contains accepted=false, error="not the leader"
            return parseErrorBody(ex.getResponseBodyAsString());
        }
    }
}
```

One client per node. The cluster service decides which one to call.

### Part B — RaftClusterService: the follower-redirect loop

This is the pattern every Raft client in every language must implement:

```java
@Service
public class RaftClusterService {
    private final List<RaftClient> clients;
    private final AtomicReference<RaftClient> cachedLeader = new AtomicReference<>();
    private volatile Instant leaderCachedAt = Instant.EPOCH;

    public CommandResult execute(String command) {
        RaftClient leader = leaderClient();           // cached or freshly discovered
        CommandResult result = leader.submitCommand(command);

        if (!result.accepted() && "not the leader".equals(result.error())) {
            invalidateLeaderCache();                  // 503 = leader changed
            leader = leaderClient();                  // re-discover
            result = leader.submitCommand(command);   // retry once
        }
        if (!result.accepted()) throw new IllegalStateException(result.error());
        return result;
    }

    private RaftClient leaderClient() {
        // Return cached leader if TTL (10s) has not expired.
        RaftClient cached = cachedLeader.get();
        if (cached != null && Instant.now().isBefore(leaderCachedAt.plus(props.leaderCacheTtl()))) {
            return cached;
        }
        return refreshLeaderCache(); // poll all nodes, find the one saying isLeader=true
    }
}
```

The TTL is important: re-polling all N nodes on every write would be expensive. We cache the leader for 10 seconds. If leadership changes within those 10 seconds, the first 503 triggers a fresh poll. This means at most one extra round-trip per leader change — acceptable overhead.

**Comparison to database primary/replica routing**: this pattern is identical to how JDBC connection pools handle PostgreSQL streaming replication failover. The driver caches the primary's address, detects "can't write to a replica" errors, and queries the cluster to find the new primary. The Raft client does the same — the only difference is that the "not the leader" signal is explicit (503 with a JSON body) rather than a database error code.

## What surprised me

<WhatSurprisedMe insight="The Raft paper says 'understandable consensus' but the hardest part is not the algorithm — it is the leader-change edge cases. Specifically: what happens when a leader crashes after appending to its own log but before quorum acknowledges? The new leader must not commit those old entries by themselves — it can only commit entries from its own term. This is the 'commitment without current-term entries' restriction in Section 5.4.2 of the paper, and it is the reason why the advanceCommitIndex function checks entry.Term == currentTerm before counting replicas. Our first implementation omitted this check. The result was that entries from a previous term could appear to be committed on the leader but were silently rolled back on followers — a real data loss bug that only manifested when a leader was killed at exactly the wrong moment. The fix is 3 lines, but understanding why those 3 lines are necessary requires understanding the full sequence of events across 2 leader changes and 3 nodes." />
