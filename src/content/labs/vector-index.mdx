## Why this lab exists

Every LLM application you deploy will eventually face this question: "we have a million embeddings — how do we find the 10 most similar ones in under 10ms?" The answer is approximate nearest neighbor (ANN) search, and the dominant algorithm is HNSW — the graph structure powering Pinecone, Weaviate, Qdrant, and the vector search layer inside pgvector.

You've seen these acronyms — HNSW, IVF, PQ — in vector database marketing copy. But until you've implemented the HNSW graph traversal yourself, until you've watched recall drop from 96% to 78% by changing a single parameter, until you've seen a 128-dimensional f32 vector (512 bytes) collapse to 16 bytes of uint8 codes — you don't understand why Faiss exists, why Pinecone charges for "pods", or why "just use pgvector" is sometimes the wrong answer.

This lab also demystifies the Spring AI `VectorStore` interface. By implementing it against our own Rust server, you see exactly what the interface expects and why it is the right abstraction boundary for RAG pipelines.

## What we're building

A vector similarity index library in Rust, implemented across three stages. Each stage adds one algorithmic layer.

**We build:**
- Cosine similarity and Euclidean distance functions from scratch
- A flat brute-force index (exact k-NN, the baseline)
- An HNSW multi-layer graph index (approximate, O(log n))
- An IVF inverted file index with k-means clustering
- Product Quantization: 32x compression with asymmetric distance computation
- An HTTP server so Spring AI can use our index as a `VectorStore`

**We explicitly skip (and why):**
- SIMD acceleration (AVX-512) — that is Faiss's main contribution, not a teaching aid
- Persistence and crash recovery — covered in the WAL and LSM-tree labs
- Distributed sharding — covered in the Consistent Hashing lab
- GPU acceleration — out of scope for a CPU-focused storage lab

## Mental model

Think of finding similar vectors as finding close points in a high-dimensional space. Exact search (flat scan) visits every point. ANN search uses a smart data structure to visit only a fraction of them.

```
Flat (exact):     HNSW (approximate):
                  
  query           layer 2: ●─────────────●
     │                          ↓
     ▼            layer 1: ●──●──────●──●
  scan all                      ↓
  n vectors        layer 0: ●●●●●●●●●●●●●  ← search ef candidates here
     │                               ↓
     ▼                          top-k results
  top-k exact
```

**HNSW's insight**: build a navigable graph where long-range edges live on upper layers and short-range edges live on layer 0. Descend greedily, narrowing the search beam as you go. Expected complexity: O(log n) instead of O(n).

**IVF's insight**: partition vectors into k Voronoi cells using k-means. At search time, only scan the `nprobe` closest cells. Cost drops from O(n) to O(nprobe * n/k).

**PQ's insight**: split each 128-dim vector into 16 subspaces of 8 dims each. Quantize each subspace to 1 of 256 centroids (1 byte). 128 * 4 = 512 bytes becomes 16 bytes — 32x compression — with bounded reconstruction error.

Start with the dumbest thing that works. Linear scan computes cosine similarity between the query and every stored vector — O(n·d) per query, but always correct.

```rust
pub struct FlatIndex {
    entries: Vec<Entry>,
}

impl FlatIndex {
    pub fn search(&self, query: &[f32], k: usize) -> Vec<SearchResult> {
        let mut scored: Vec<SearchResult> = self.entries.iter()
            .map(|e| SearchResult {
                id: e.id.clone(),
                score: cosine(query, &e.vector),
            })
            .collect();

        // Descending by cosine similarity
        scored.sort_by(|a, b| b.score.partial_cmp(&a.score).unwrap());
        scored.truncate(k);
        scored
    }
}
```

**The cosine formula from scratch**:

```rust
pub fn cosine(a: &[f32], b: &[f32]) -> f32 {
    let mut dot = 0.0_f32;
    let mut norm_a = 0.0_f32;
    let mut norm_b = 0.0_f32;
    for (&ai, &bi) in a.iter().zip(b.iter()) {
        dot   += ai * bi;
        norm_a += ai * ai;
        norm_b += bi * bi;
    }
    if norm_a == 0.0 || norm_b == 0.0 { return 0.0; }
    dot / (norm_a.sqrt() * norm_b.sqrt())
}
```

**What this teaches**: cosine similarity is direction, not magnitude. Two vectors pointing the same direction score 1.0 regardless of their lengths. This is why embedding models normalize their outputs — cosine distance becomes equivalent to L2 distance on the unit sphere, which has nicer properties for HNSW's graph traversal.

At 10,000 vectors of 128 dimensions: ~850 queries/sec on a single core. This is the baseline v1 and v2 must beat.

HNSW (Hierarchical Navigable Small World) is a graph index where each node appears in one or more layers. Layer 0 contains all nodes; higher layers contain a geometrically shrinking subset. Long-range edges in upper layers act like express lanes — they let you skip large distances in one hop.

**The layer assignment formula**: each node is promoted to layer L with probability `(1/M)^L`. With M=16, about 1/16 nodes reach layer 1, about 1/256 reach layer 2. This geometric distribution is what makes the graph navigable — the top layer is sparse enough to act as a highway.

```rust
pub struct HnswIndex {
    vectors: Vec<Vec<f32>>,
    ids: Vec<String>,
    /// layers[l][node_id] = Vec<neighbor_id>
    layers: Vec<HashMap<usize, Vec<usize>>>,
    entry_point: Option<usize>,
    max_layer: usize,
    m: usize,              // max neighbors per layer
    ef_construction: usize, // candidate pool during insert
}
```

**Insert algorithm** (simplified):

```
1. Assign max_layer for new node (geometric random)
2. Starting from the entry point, greedily descend to max_layer+1
   (one nearest neighbor per layer — no branching above the node's top layer)
3. From max_layer down to 0:
   a. Beam search with ef_construction candidates on this layer
   b. Select M nearest of those candidates as neighbors
   c. Add bidirectional edges (and prune neighbors that exceed M)
4. Update global entry point if new node reaches a higher layer
```

**The ef parameter** is the key tuning knob. During construction, `ef_construction=200` fills a candidate pool of 200 nodes before selecting M=16 neighbors. Higher ef_construction means better graph quality (and higher recall at search time) at the cost of slower inserts. During search, `ef_search` controls the candidate pool — the same beam search, but you keep all ef candidates and return the top k.

**Search algorithm**:

```
1. Start at the global entry point (highest layer)
2. For layers max_layer → 1: greedy 1-nearest-neighbor descent
3. At layer 0: run full beam search with ef_search candidates
4. Return top-k from the ef_search candidate pool
```

**Recall vs. speed tradeoff**:

| ef_search | recall@10 | queries/sec |
|-----------|-----------|-------------|
| 10        | ~78%      | ~90,000/sec |
| 50        | ~96%      | ~45,000/sec |
| 100       | ~98%      | ~25,000/sec |
| 200       | ~99%      | ~14,000/sec |

The sweet spot for most production workloads is ef=50: you accept 4% misses in exchange for 50x speedup over flat scan.

IVF and PQ are the core of Faiss. They solve different problems: IVF reduces *which vectors you search*; PQ reduces *how much memory each vector occupies*.

**IVF: Inverted File Index**

Partition the corpus into k Voronoi cells using k-means. Each cell has a centroid and a posting list of vectors assigned to it.

At search time:
1. Compute distance from query to all k centroids (fast — k is small, typically 100–4000)
2. Select the `nprobe` closest centroids
3. Exact search within those posting lists only

```rust
pub struct IvfIndex {
    centroids: Vec<Vec<f32>>,               // k cluster centers
    posting_lists: Vec<Vec<(usize, Vec<f32>)>>, // per-centroid vectors
    nprobe: usize,
}
```

Cost: O(k + nprobe * n/k) vs O(n) flat scan. With k=100 and nprobe=4: only scan 4% of the corpus per query.

**Product Quantization**

PQ compresses each vector by splitting it into m subspaces and encoding each subspace as 1 byte (1 of 256 centroids per subspace).

```
128-dim f32 vector → 16 subspaces of 8 dims each
                   → 16 uint8 codes (one per subspace)
                   → 16 bytes total (vs 512 bytes raw)
                   = 32× compression
```

The key insight of **Asymmetric Distance Computation (ADC)**: the query is *not* quantized. Instead, precompute a distance table: `dist_table[subspace][centroid_id]`. Then for each encoded vector, the approximate distance is just `sum(dist_table[s][code[s]])` — a table lookup per subspace, no floating-point multiply.

```rust
// Precompute once per query: m * 256 floats
let dist_table: Vec<Vec<f32>> = (0..m)
    .map(|sub| {
        let query_sub = &query[sub * sub_dim..(sub + 1) * sub_dim];
        codebooks[sub].iter()
            .map(|centroid| euclidean_sq(query_sub, centroid))
            .collect()
    })
    .collect();

// Per-vector cost: m table lookups
let approx_dist: f32 = code.iter().enumerate()
    .map(|(sub, &c)| dist_table[sub][c as usize])
    .sum();
```

For 10,000 vectors with m=16: the inner loop does 16 array lookups (not 16 floating-point multiplications). This is why PQ-encoded search is fast enough to scan millions of vectors.

## Stress tests

**HNSW recall under distribution shift**: insert 10,000 vectors from a standard normal distribution, then query with vectors from a different cluster. Recall@10 drops from 96% to ~82%. This mirrors production failures where embedding model updates shift the distribution — your existing HNSW graph was built for the old distribution and doesn't route the new queries optimally. Fix: periodic full rebuild or incremental graph patching.

**IVF with skewed clusters**: if your data has highly imbalanced clusters (a common real-world pattern with semantic categories), some posting lists are 10x larger than others. nprobe=4 on balanced data (40% coverage) becomes nprobe=4 on skewed data (4% coverage of the large cluster). Fix: use nprobe=20 or switch to a hierarchical IVF (IVF-HNSW coarse quantizer) like Faiss's `IndexIVFHNSW`.

**PQ reconstruction error at high compression**: with m=8 subspaces (64x compression on 128-dim), recall@10 drops below 60% because the per-subspace approximation error compounds. The rule of thumb: `dim / m >= 4` — each subspace should have at least 4 dimensions. For 128 dims, m=16 gives sub_dim=8, which is the minimum viable configuration.

### Part A — VectorClient: the HTTP bridge

`VectorClient.java` wraps the Rust server's two endpoints using Spring Framework 6.1 `RestClient`.

```java
public class VectorClient {
    private final RestClient http;

    public VectorClient(String baseUrl) {
        this.http = RestClient.builder()
                .baseUrl(baseUrl)
                .defaultHeader("Accept", MediaType.APPLICATION_JSON_VALUE)
                .build();
    }

    public int add(String id, float[] vector) {
        var response = http.post()
                .uri("/add")
                .contentType(MediaType.APPLICATION_JSON)
                .body(new AddRequest(id, vector))
                .retrieve()
                .body(AddResponse.class);
        if (response == null) throw new VectorException("null response");
        return response.indexSize();
    }

    public List<SearchResultEntry> search(float[] query, int k, int ef) {
        var response = http.post()
                .uri("/search")
                .contentType(MediaType.APPLICATION_JSON)
                .body(new SearchRequest(query, k, ef, false))
                .retrieve()
                .body(SearchResponse.class);
        if (response == null) throw new VectorException("null response");
        return response.results();
    }

    record AddRequest(String id, float[] vector) {}
    record AddResponse(String id, int indexSize) {}
    record SearchRequest(float[] query, int k, int ef, boolean useFlat) {}
    record SearchResponse(List<SearchResultEntry> results, String indexUsed) {}
    public record SearchResultEntry(String id, float score) {}
}
```

### Part B — VectorAutoConfiguration: wiring into Spring AI

The key learning is the `VectorStore` interface. Spring AI's RAG pipeline (`QuestionAnswerAdvisor`, `ChatClient`) calls `similaritySearch(SearchRequest)` — it does not care whether the backing store is pgvector, Pinecone, or our Rust HNSW server.

```java
@Configuration
@EnableConfigurationProperties(VectorProperties.class)
public class VectorAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public VectorClient vectorClient(VectorProperties props) {
        return new VectorClient(props.baseUrl());
    }

    @Bean
    @ConditionalOnMissingBean(VectorStore.class)
    public VectorStore ourVectorStore(VectorClient client, VectorProperties props) {
        return new OurVectorStore(client, props);
    }

    static class OurVectorStore implements VectorStore {
        @Override
        public void add(List<Document> documents) {
            // expects float[] at metadata key "embedding"
            for (Document doc : documents) {
                float[] v = (float[]) doc.getMetadata().get("embedding");
                client.add(doc.getId(), v);
            }
        }

        @Override
        public List<Document> similaritySearch(SearchRequest request) {
            // plug query embedding here — works with any ChatModel
            // ...
        }
    }
}
```

**Why this matters**: the `@ConditionalOnMissingBean(VectorStore.class)` annotation means your test suite can swap in a mock `VectorStore` without touching any production code. The Rust server is invisible to the RAG pipeline — only the `VectorStore` interface matters. This is the correct way to integrate native services into Spring AI.
