## Why this lab exists

Every LLM inference API you've called — OpenAI, Anthropic, Groq — runs an inference engine. The gap between "run `model(input_ids)` in a loop" and "serve 10,000 concurrent users at 50 tokens/sec" is where the interesting systems work lives.

You've seen diagrams of KV caches. You've read that vLLM uses paged attention. But until you've *implemented* the naive loop and measured the O(n^2) cost, until you've cached K/V tensors and watched the same model jump from 12 to 45 tokens/sec, until you've allocated a paged KV cache and computed that the fragmentation drops from 50% to 3% — you don't *know* it.

This lab builds an LLM inference engine in three stages: from a naive autoregressive loop in ~200 lines to a continuous batching engine with paged attention in ~850 lines. Each stage fixes one measured problem in the previous one. The reference model is GPT-2 (124M params, 12 layers) — small enough to run on any laptop, large enough to exhibit the same bottlenecks as much bigger models.

## What we're building

An LLM inference engine in Python, implemented across three stages. Each stage adds one optimization and measures its impact on tokens/sec and memory.

**We build:**
- Naive autoregressive generation (v0): the baseline everyone should understand first
- KV cache (v1): cache past key/value tensors so decode steps process one token, not n tokens
- Continuous batching with paged KV allocation (v2): group multiple requests, manage memory in fixed pages
- FastAPI server: POST `/generate` with strategy selection, GET `/stats`
- Spring AI Java integration: `ChatClient` calling our FastAPI server

**We explicitly skip (and why):**
- GPU kernel fusion (FlashAttention): requires CUDA kernels; we're CPU-only to keep the lab accessible
- Tensor parallelism: requires multi-GPU setup; the single-device concepts are covered first
- Quantization (INT4/INT8): an orthogonal optimization; covered in "What the Toy Misses"
- Fine-tuning: we serve pre-trained GPT-2; training is the Transformer lab

## Mental model

LLM generation is fundamentally sequential. Each token depends on all previous tokens — you cannot parallelize across the sequence during decode. The inference engine's job is to make each decode step as fast as possible and serve as many concurrent requests as the hardware allows.

```
Autoregressive decode loop (for one request):

  [PROMPT: "The capital of France is"]
              |
              v
  Step 1: forward([token_1...token_6]) -> logit_7 -> sample -> "Paris"
  Step 2: forward([token_1...token_7]) -> logit_8 -> sample -> "."
  Step 3: forward([token_1...token_8]) -> logit_9 -> sample -> EOS

  Naive (v0): step k runs attention over k tokens  = O(k^2) per step
  KV cache (v1): step k runs attention over 1 new token + cached K/V = O(k) per step

  Paged allocation (v2):
    Instead of one contiguous KV tensor per request:
      |--page-0--|--page-1--|--page-2--| (16 tokens per page)
    Free pages return to the pool when a request completes.
    Fragmentation: contiguous = 50% waste (avg); paged = 3% waste.
```

The three bottlenecks this lab addresses are exactly the ones vLLM solved to achieve 24x throughput over naive serving:

1. **Redundant attention computation** — the KV cache eliminates O(n^2) work per token
2. **Sequential request processing** — continuous batching fills the GPU with multiple concurrent requests
3. **Memory fragmentation** — paged allocation lets the engine serve more concurrent requests within the same memory budget

The naive loop: encode the full sequence, forward pass, sample, append, repeat. No caching, no batching.

**The O(n^2) cost in detail:**

For each token we generate, the model runs self-attention over the full sequence. The attention matrix at step k has shape `(k, k)` — k queries attending to k keys. At GPT-2's 12 layers and 12 heads, generating token 513 materializes 12 * 12 attention matrices of shape (512, 512):

```
attention_matrices = 12 * 12 * 512 * 512 * 4 bytes = 150 MB of activations
```

At k=1024: 12 * 12 * 1024 * 1024 * 4 = 603 MB — and this is just the attention activations. Each decode step also recomputes K and V for all prior tokens from scratch. If KV computation costs C per token, step k pays k*C again, even though steps 1 through k-1 already paid it.

**Why measure it?**

Run `generate_naive_with_timing()` and print per-token latency. You'll see a clear upward trend: token 10 takes ~5ms, token 100 takes ~50ms, token 500 takes ~250ms. This is the O(n) growth per token (not O(n^2), because PyTorch's batched attention is O(n^2) but implemented efficiently — the wall-clock time per token grows linearly with context length, which is what matters for users).

**The full loop:**

```python
with torch.no_grad():
    for _ in range(max_tokens):
        outputs = model(input_ids)                  # full forward pass: O(seq_len^2)
        next_logits = outputs.logits[:, -1, :]      # logits at last position
        next_token = sample(next_logits, temperature)
        input_ids = torch.cat([input_ids, next_token], dim=1)
```

**Temperature sampling:**

Temperature controls randomness. At `temperature=0`, we take `argmax` — the most likely token every time. This is greedy decoding: deterministic and reproducible. At `temperature=1.0`, we sample from the softmax distribution. At `temperature=2.0`, the distribution is flatter — more creative output, more likely to derail.

The test for v0 that matters most: `test_greedy_decoding_is_deterministic`. If the same prompt at `temperature=0` produces different output on repeated calls, something is wrong with the sampling implementation. Greedy decode is pure math with no randomness; different outputs mean a bug.

</LabStage>

The KV cache is the single most impactful optimization in LLM inference. It changes decode from O(n^2) to O(n) total work across all steps.

**The insight:**

In the attention mechanism, Q, K, and V are computed from the input tokens. At decode step k, we need:
- Q from the new token only (position k)
- K from all tokens 1 through k
- V from all tokens 1 through k

K and V for positions 1 through k-1 were already computed at step k-1. They don't change. If we cache them, step k only needs to compute K and V for position k — one row, not k rows.

**Memory layout:**

```
KVCache: Dict[layer_index -> (key_tensor, value_tensor)]

key_tensor shape:   (1, n_heads, seq_len, d_head)
value_tensor shape: (1, n_heads, seq_len, d_head)

For GPT-2 (12 layers, 12 heads, d_head=64):
  After 100 tokens:
    1 key tensor per layer:  (1, 12, 100, 64) @ 4 bytes = 307,200 bytes
    1 val tensor per layer:  (1, 12, 100, 64) @ 4 bytes = 307,200 bytes
    12 layers * 2 * 307,200 = 7,372,800 bytes ~ 7 MB

  After 1024 tokens:
    2 * 12 * 12 * 64 * 1024 * 4 = 75,497,472 bytes ~ 72 MB
```

**The exact formula:**

```
kv_cache_bytes = 2 * n_layers * n_heads * d_head * seq_len * bytes_per_element
```

For GPT-2 at seq_len=1024: `2 * 12 * 12 * 64 * 1024 * 4 = 75,497,472 bytes`.

This formula is worth memorizing. It tells you immediately how much memory any model's KV cache will consume at a given sequence length. For a 70B Llama 3 model (80 layers, 64 heads, d_head=128, bfloat16): `2 * 80 * 64 * 128 * 1024 * 2 = 4 GB per request at seq_len=1024`.

**Protocol with HuggingFace:**

HuggingFace models support `past_key_values` out of the box. The protocol is:

```python
# Prefill: process the full prompt, get the initial KV cache
outputs = model(input_ids, past_key_values=None, use_cache=True)
past_key_values = outputs.past_key_values

# Decode: process only the new token each step
outputs = model(next_token, past_key_values=past_key_values, use_cache=True)
past_key_values = outputs.past_key_values   # updated cache
```

At each decode step, `next_token` has shape `(1, 1)` — a single token. The model runs attention with this one Q vector against all cached K/V rows. The result is one output logit vector. This is O(seq_len) per step, not O(seq_len^2).

**Speedup measurement:**

The test `test_kv_cache_produces_same_output_as_naive` verifies correctness: with greedy decoding (temperature=0), KV cache must produce identical output to naive. Then you run both with timing and observe the speedup — typically 3-5x at seq_len=100 on CPU, larger on GPU where memory bandwidth is the bottleneck.

Two orthogonal optimizations for multi-request serving.

**Continuous batching:**

Naive serving processes one request at a time. Batch serving groups N requests into a single forward pass. The forward pass cost is O(batch_size * seq_len^2) — but the fixed per-pass overhead (weight matrix loads, Python dispatch) is amortized across all requests in the batch. At batch_size=4, you pay the overhead once but generate 4 tokens.

"Continuous" means requests can join a batch mid-generation. When request A finishes at step 50 and request B arrives, B immediately joins the batch at step 51. Traditional static batching (used by early inference servers) required all requests to finish before admitting new ones — GPU utilization dropped to near 0 during the admission wait. Continuous batching keeps utilization near 100%.

**Paged attention:**

The fragmentation problem: if you allocate a contiguous KV tensor of size `max_seq_len` for each request at admission, you're likely to waste memory. A request that generates 100 tokens wastes the memory reserved for tokens 101 through `max_seq_len`.

But you can't allocate dynamically and grow the tensor either — tensor memory must be contiguous, and contiguous blocks of the right size may not be available after some requests complete and others grow.

The solution (from the vLLM paper): allocate in fixed-size pages of PAGE_SIZE tokens (we use 16). Each page is independently allocatable. When a request needs more cache space, allocate another page. When a request completes, return its pages to the free pool immediately — no fragmentation between requests.

**Fragmentation math:**

Contiguous allocation: each request wastes `(block_size - seq_len % block_size)` tokens. For PAGE_SIZE=16 and random sequence lengths, average waste per request is 8 tokens — 50% of the last page. With 512-token sequences: 8/512 = 1.5%, which sounds OK. But if you're planning for `max_seq_len` on admission, the waste is `(max_seq_len - actual_len)` tokens per request — potentially 70-80% as reported in the vLLM paper.

Paged allocation: only the last page is partially used. Average waste = PAGE_SIZE/2 tokens per request. For a 128-token sequence: 2 pages allocated, last page wastes avg 8 tokens. Waste = 8/128 = 6.25%. For a 512-token sequence: 32 pages, last page waste = 8/512 = 1.5%. Waste decreases with sequence length because full pages dominate.

**`scheduler_step()` in detail:**

```python
def scheduler_step(self) -> List[InferenceRequest]:
    # 1. Admit new requests from queue up to max_batch_size
    while len(self.active) < self.max_batch_size and queue.has_items():
        req = queue.pop()
        self.active.append(req)
        self.paged_cache.allocate_page(req.request_id)

    # 2. For each active request: prefill (first step) or decode (subsequent steps)
    for req in self.active:
        if req.past_key_values is None:
            outputs = model(full_prompt_ids, use_cache=True)    # prefill
        else:
            outputs = model(last_token, past_key_values=req.past_key_values, use_cache=True)
        req.past_key_values = outputs.past_key_values
        next_token = sample(outputs.logits[:, -1, :], req.temperature)
        req.generated_token_ids.append(next_token)
        paged_cache.record_token(req.request_id)

    # 3. Complete finished requests
    for req in completed:
        paged_cache.free_request(req.request_id)
        req.callback(decoded_text)
        self.active.remove(req)
```

The critical design: `paged_cache.free_request()` returns pages immediately. Another request can use those pages on the very next scheduler step. In contrast, static allocation holds pages until the server process restarts.

## Stress tests

**Sequence length scaling**: run `generate_naive_with_timing()` with `max_tokens=200` and plot `time_sec` vs `context_length`. You'll observe roughly linear growth: each additional token in the context makes the next decode step slightly slower because the attention matrix grows by one row. This is the empirical proof of the O(n^2) cost.

**KV cache memory growth**: run `generate_with_kv_cache()` and log `result.kv_cache_bytes` after each token. At seq_len=k, the cache should hold exactly `2 * 12 * 12 * 64 * k * 4` bytes. Verify the formula holds incrementally — any deviation indicates the cache is being truncated or double-allocated.

**Batch throughput vs batch size**: submit 1, 2, 4, and 8 requests to `BatchedEngine` simultaneously and measure total tokens/sec. You should see near-linear improvement up to batch_size=4 on CPU (where the bottleneck is the forward pass itself), then diminishing returns as memory bandwidth saturates.

**Paged cache under rapid request cycling**: submit 100 short requests (10 tokens each) in rapid succession to the engine. After all complete, verify `paged_cache.stats()["free_pages"]` equals the initial total — no pages should be leaked. This tests that `free_request()` is called exactly once per completed request.

**Fragmentation measurement**: create 16 requests with sequence lengths 1, 17, 33, 49... (all forcing a new page allocation). Measure `paged_cache.fragmentation_rate()`. Compare against a contiguous allocator's theoretical 50% waste. Paged should be under 40% even in the worst case.

### Part A — InferenceClient: direct HTTP access

`InferenceClient.java` wraps our Python FastAPI server with Spring Framework 6.1's `RestClient`. It exposes three operations: `generate()`, `stats()`, and `isHealthy()`.

```java
public class InferenceClient {
    private final RestClient http;
    private final InferenceProperties props;

    public GenerateResponse generate(String prompt) {
        var body = new GenerateRequest(
            prompt, props.maxTokens(), props.temperature(), props.strategy()
        );
        return http.post()
            .uri("/generate")
            .contentType(MediaType.APPLICATION_JSON)
            .body(body)
            .retrieve()
            .body(GenerateResponse.class);
    }
}
```

The `strategy` field lets you switch between `naive`, `kv_cache`, and `batched` from `application.yml` — no Java code change needed to benchmark strategies.

### Part B — Spring AI ChatClient: the abstraction layer

Spring AI's `ChatClient` is an abstraction over any OpenAI-compatible LLM endpoint. Our FastAPI server exposes `POST /generate`, which we configure Spring AI to call directly.

```java
// In InferenceDemoApplication.java
ChatClient chatClient = chatClientBuilder.build();

// This call is identical whether the endpoint is our FastAPI server,
// OpenAI's API (api.openai.com), Azure OpenAI, or Bedrock.
String response = chatClient
    .prompt("Explain KV cache in one sentence.")
    .call()
    .content();
```

The only thing that changes when you swap endpoints is `application.yml`:

```yaml
spring:
  ai:
    openai:
      base-url: http://localhost:8000   # Our server
      api-key: local-dummy-key          # Not validated by our server
```

### Part C — Ollama as a drop-in replacement

The same `ChatClient` code works with Ollama (local open-source LLMs like Llama 3 or Mistral). Change `application.yml`:

```yaml
spring:
  ai:
    ollama:
      base-url: http://localhost:11434
      chat:
        options:
          model: llama3
```

No Java code changes. This is the correct abstraction for AI in Java services: write to the `ChatClient` interface, configure the model externally. Your business logic survives model migrations.

## What surprised me

<WhatSurprisedMe insight="The KV cache memory formula surprised me with its concreteness. GPT-2's KV cache at seq_len=1024 costs exactly 2 x 12 x 12 x 64 x 1024 x 4 = 75,497,472 bytes — about 72 MB. For a 70B-parameter model (Llama 3) with 80 layers, 64 heads, and d_head=128: 2 x 80 x 64 x 128 x 1024 x 2 (bfloat16) = 4 GB — for one request at seq_len=1024. A single A100 with 80 GB HBM can hold at most 20 such requests in their KV cache. The paged attention paper showed that naive contiguous KV allocation wastes 60-80% of that due to fragmentation — meaning the effective capacity drops to 4 to 8 concurrent requests. Paged attention recovers that wasted memory, which is why vLLM reported 24x throughput improvement over naive serving. The math is entirely deterministic once you know n_layers, n_heads, d_head, and seq_len. No approximation needed." />
