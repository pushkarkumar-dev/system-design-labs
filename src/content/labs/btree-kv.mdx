## Why this lab exists

Every SQL database you have ever used — MySQL, PostgreSQL, SQLite, Oracle — stores its indexes as a B+Tree. When you add `CREATE INDEX` to a billion-row table and then write `WHERE user_id = 42`, that lookup completes in 4–5 page reads because of the B+Tree structure underneath. Yet most engineers who can explain B+Tree theory at a whiteboard have never actually implemented one.

This lab changes that. You will build a complete B+Tree key-value store in three stages: from an in-memory proof-of-concept that fits in your head, to a page-managed on-disk tree, to a WAL-protected tree with a free list. By the end you will have touched every concept that makes databases fast: page-aligned I/O, LRU page caches, node splits, write-ahead logging, and the leaf linked list that makes range scans efficient.

This lab is also the natural companion to the LSM-Tree lab. The two data structures solve the same problem — durable ordered KV storage — with fundamentally different tradeoffs. Building both lets you understand why databases choose one over the other.

## What we are building

A B+Tree KV store where:
- Keys and values are byte strings
- The tree is ordered by lexicographic byte comparison
- All data lives in leaf nodes (internal nodes are router-only)
- Leaf nodes are doubly-linked for efficient range scans
- Nodes live on 4096-byte disk pages (v1+)
- Writes are protected by a WAL (v2)

**Explicit scope exclusions**: no transactions, no MVCC, no concurrent access (single Mutex), no merge-on-underflow rebalancing, no prefix compression, no bulk loading.

## Mental model

### B+Tree vs LSM-Tree: read-optimized vs write-optimized

The central architectural choice in storage engines is B+Tree vs LSM-Tree. They solve the same problem with opposite tradeoffs:

```
B+Tree                          LSM-Tree
------                          --------
Write path:                     Write path:
  put(k, v)                       put(k, v)
     |                               |
     v                               v
  find leaf (O(log N))          memtable (BTreeMap)
     |                          (no disk I/O)
     v
  write leaf page (1 I/O)
  + possibly split + write
    O(log N) dirty pages total

Read path:                      Read path:
  get(k)                          get(k)
     |                               |
  traverse root→leaf             memtable?
  O(log N) page reads            |  miss
  (1 per level)                  v
                               SSTable 1 (newest)
                                  |  miss
                                  v
                               SSTable 2 ...
                               O(levels) SSTables
```

For a tree of 1 billion keys with ORDER=512 (4096-byte pages with 8-byte keys), the tree is at most 4 levels deep. A get costs exactly 4 page reads. An LSM-Tree with 7 levels checks up to 7 SSTables on a read — and compaction adds write amplification on top.

**B+Tree is read-optimized**: O(log N) reads, predictable latency.
**LSM-Tree is write-optimized**: O(1) writes (memtable insert), but reads and compaction are expensive.

InnoDB and PostgreSQL choose B+Tree. RocksDB, Cassandra, and TiKV choose LSM-Tree. The choice depends on read/write ratio: if reads heavily outnumber writes, B+Tree wins. If writes are the bottleneck, LSM-Tree wins.

### The leaf linked list: why range scans are cheap

The most important structural difference between a B-Tree and a B+Tree is the leaf linked list. In a B-Tree (no plus), data lives in every node — internal and leaf. In a B+Tree, **all data lives in leaf nodes**, and leaf nodes are linked in key order.

```
Internal nodes (router-only, no data):
         [user:500]
        /            \
  [user:250]        [user:750]
  /       \          /      \

Leaf nodes (doubly-linked, hold all data):
[user:001..249] <-> [user:250..499] <-> [user:500..749] <-> [user:750..999]
```

To answer `range("user:200", "user:350")`:
1. Traverse from root to the leaf containing `user:200` — O(log N) traversal
2. Scan remaining keys in that leaf — O(keys per leaf)
3. Follow `next` pointer to adjacent leaf — O(1) per hop
4. Repeat until key exceeds `user:350`

Total cost: O(log N + result_size). Without the leaf linked list, step 3 would require a full tree traversal for each adjacent leaf — O(result_size times log N). For a range of 10,000 keys with tree height 4, that is 4 vs 40,000 page reads.

### Node splits

When a node reaches capacity (ORDER-1 keys), the next insert must split it:

```
Before split (ORDER=4, max 3 keys, leaf is full):
  Leaf: [a=1, b=2, c=3]

Insert d=4:
  Split at mid (index 2):
    Left:  [a=1, b=2]
    Right: [c=3, d=4]
    Promote: c (smallest key in right leaf)
  
  Parent internal node absorbs (c, right_ptr):
    [c] → (left_ptr, right_ptr)

If parent is also full, it splits too.
If root splits, a new root is created — the only way tree height increases.
```

This propagation is what makes inserts O(log N) in the worst case: a split can cascade all the way to the root. In practice, most inserts touch only 1–2 nodes.

The in-memory tree uses a Vec arena for nodes (index = node ID). This avoids Rust's reference cycle problems with doubly-linked structures and makes the page-ID abstraction clear for v1.

```rust
pub enum Node {
    Internal(InternalNode),
    Leaf(LeafNode),
}

pub struct InternalNode {
    pub keys: Vec<Key>,
    pub children: Vec<usize>,  // arena indices
}

pub struct LeafNode {
    pub keys: Vec<Key>,
    pub values: Vec<Value>,
    pub prev: Option<usize>,   // linked list
    pub next: Option<usize>,
}
```

**The insert algorithm** — `insert_recursive` descends the tree and returns `InsertResult { split: Option<(promoted_key, new_right_idx)> }`. If the child split, the parent incorporates the promoted key. If the root splits, a new root is created:

```rust
pub fn insert(&mut self, key: Key, value: Value) {
    let result = self.insert_recursive(0, key, value);  // 0 = root
    if let Some((mid_key, right_idx)) = result.split {
        // Root split: move current root to a new arena slot,
        // create new root pointing to both halves.
        let old_root_idx = self.allocate_node(self.nodes[0].clone());
        let new_root = Node::Internal(InternalNode {
            keys: vec![mid_key],
            children: vec![old_root_idx, right_idx],
        });
        self.nodes[0] = new_root;
    }
}
```

**Leaf split** — the tricky part is maintaining the linked list. When leaf L splits into L (left) and R (right):
- L.next = R
- R.prev = L
- R.next = old L.next
- old L.next.prev = R (if old L.next exists)

The promoted key is `R.keys[0]` — the smallest key in the right half. This is a B+Tree property: the promoted key stays in both the parent and the right leaf. In a B-Tree (without plus), the promoted key is removed from the node. The difference matters for range scans.

**Range scan using the leaf list**:

```rust
pub fn range(&self, start: &[u8], end: &[u8]) -> Vec<(Key, Value)> {
    let mut result = Vec::new();
    let mut leaf_idx = self.find_leaf(0, start);  // O(log N)

    loop {
        let leaf = self.nodes[leaf_idx].as_leaf();
        for (i, k) in leaf.keys.iter().enumerate() {
            if k.as_slice() < start { continue; }
            if k.as_slice() > end   { return result; }
            result.push((k.clone(), leaf.values[i].clone()));
        }
        match leaf.next {
            Some(next_idx) => leaf_idx = next_idx,  // O(1) hop
            None => break,
        }
    }
    result
}
```

This is O(result_size) after the initial O(log N) traversal. The loop never goes back up the tree.

Each node becomes a fixed 4096-byte page on disk. The root is always page 0. Page ID = byte offset in the file divided by PAGE_SIZE.

**Page format** (14-byte header + key slots + value/child slots):

```
Offset  Size  Field
──────  ────  ─────────────────────────────────────────────────
0       1     node_type (0x01=internal, 0x02=leaf)
1       2     key_count (u16 LE)
3       4     next_page (u32 LE) — leaf linked list
7       4     prev_page (u32 LE) — leaf linked list
11      3     padding

After header:
  Keys:     key_count × KEY_SLOT bytes
              [key_len(2)][key_bytes...(KEY_MAX-2 zero-padded)]
  
  Children (internal):
              (key_count+1) × 4 bytes (u32 LE page IDs)
  
  Values (leaf):
              key_count × VAL_SLOT bytes
              [val_len(2)][val_bytes...(VAL_SLOT-2 zero-padded)]
```

**Why fixed-size slots?** Variable-length serialization would require parsing the entire key section to find where children start. Fixed slots let us compute any child's offset as `HEADER + key_count * KEY_SLOT + child_index * 4` — O(1) random access within a page.

**LRU page cache** uses a `HashMap` for O(1) lookup and a `VecDeque` as the access-order queue:

```rust
pub struct PageCache {
    capacity: usize,
    map: HashMap<u32, Node>,
    order: VecDeque<u32>,  // front = most recently used
}

pub fn get(&mut self, page_id: u32) -> Option<&Node> {
    if self.map.contains_key(&page_id) {
        // Move to front (MRU)
        self.order.retain(|&id| id != page_id);
        self.order.push_front(page_id);
        self.map.get(&page_id)
    } else {
        None
    }
}
```

**Write amplification for one insert**: a leaf split writes the modified leaf + the new right leaf. If the parent internal node also splits, that's another 2 page writes. In the worst case (all ancestors split), it's O(log N) page writes — 4 pages for a billion-key tree. Each page is 4096 bytes, so one insert costs at most 4 × 4096 = 16 KB of disk I/O, regardless of key size.

**Why 4096-byte pages?** This matches the OS virtual memory page size on x86/ARM. The kernel never gives you a partial page on a crash — a 4096-byte write either completes or it doesn't (the "torn write" problem only applies within a page, not across pages). Our CRC check on open detects a corrupted page. A multi-page operation (node split) still needs the WAL (v2) to be atomic.

**The torn write problem** — without WAL, a node split involves at least 3 page writes:
1. Write right leaf (new page)
2. Write left leaf (updated next pointer)
3. Write parent internal node (new key + child pointer)

If a crash happens after step 1 but before step 3, the parent's pointer to the new right child points to a page that was written, but the parent's key array doesn't include the new key. The B+Tree is silently inconsistent. On reopen, some range of keys is unreachable.

**The WAL fix** — before writing any modified page, log the intent:

```
WAL record: [MAGIC(1)][CRC32(4)][LEN(4)][page_id(4)][page_content(4096)]
```

On recovery:
1. Open the WAL
2. Read records until CRC mismatch or EOF
3. For each valid record, write `page_content` to `page_id` in the data file
4. Open the tree (now consistent)

Because WAL records are written and fsynced before data pages, a crash at any point leaves a recoverable state:
- Crash before WAL write: neither WAL nor data changed — old state recovers correctly
- Crash after WAL write but before data write: WAL replay re-applies the data write
- Crash after both WAL and data write: WAL replay is idempotent (writes same bytes again)

**Free list** — page 1 is the free list header. When a leaf becomes empty after deletes, its page ID is added to the free list. On `allocate_page()`, we check the free list first:

```rust
fn allocate_page(&mut self) -> io::Result<u32> {
    let mut free_list = self.read_free_list()?;
    if let Some(page_id) = free_list.pop() {
        self.write_free_list(&free_list)?;
        return Ok(page_id);  // reuse freed page
    }
    // Extend the file
    let id = self.next_page_id;
    self.next_page_id += 1;
    Ok(id)
}
```

The free list update is itself a page write (page 1), so it must also be WAL-protected. This is why WAL + free list go together: without WAL, a crash during free list update corrupts the free list.

**Shadow paging as an alternative** — some databases (SQLite in WAL mode, LMDB) use copy-on-write instead of WAL. On a write, they allocate a new page, write the new content there, and update a pointer in a "root of truth" page. The old page is freed only after the pointer update is durable. This gives crash safety without a WAL, but requires maintaining two copies of every modified page chain — reads can hit either the old or new version depending on which root pointer you use. MVCC adds more versions. The read amplification cost is why most production databases prefer WAL.

## Stress tests

To verify correctness under load, the test suite includes:

**Leaf list integrity after many splits**: insert 50 keys to force multiple split cascades, then verify that a full range scan returns all 50 keys. If any `next` pointer is stale, the range scan will miss entries.

**Persistence and recovery**: insert keys, close the tree (drop), reopen, verify all keys are still accessible. Tests the page serialization round-trip.

**Overwrite correctness**: insert the same key multiple times with different values, verify only the latest value is returned. Checks that the binary search in leaf_insert finds and overwrites the existing slot.

**Range boundary conditions**: range with disjoint bounds (no results), range spanning multiple leaves, range of exactly one key.

### Part A — BTreeClient: the HTTP bridge

`BTreeClient.java` wraps the Rust B+Tree server's five REST endpoints using Spring Framework 6.1's `RestClient`. The range query is where B+Tree shines:

```java
public class BTreeClient {
    private final RestClient http;

    // POST /put — write a key-value pair
    public void put(String key, String value) {
        http.post().uri("/put")
            .contentType(MediaType.APPLICATION_JSON)
            .body(new PutRequest(key, value))
            .retrieve().toBodilessEntity();
    }

    // GET /get?key=... — returns Optional.empty() on 404
    public Optional<String> get(String key) {
        try {
            var resp = http.get().uri("/get?key={key}", key)
                    .retrieve().body(GetResponse.class);
            return resp != null ? Optional.of(resp.value()) : Optional.empty();
        } catch (RestClientResponseException e) {
            if (e.getStatusCode().value() == 404) return Optional.empty();
            throw e;
        }
    }

    // GET /range?start=&end= — B+Tree leaf-list walk at the server
    public List<KeyValue> range(String start, String end) {
        var result = http.get()
                .uri("/range?start={start}&end={end}", start, end)
                .retrieve().body(KeyValue[].class);
        return result != null ? List.of(result) : List.of();
    }

    // Java 17 records — immutable DTOs
    record PutRequest(String key, String value) {}
    public record KeyValue(String key, String value) {}
}
```

The `range()` method is O(result_size) at the Rust server (leaf-list walk), not O(result_size times log N). Results arrive pre-sorted — the Java caller does not need to sort.

### Part B — BTreeService: write-through cache (the buffer pool analogy)

The Spring service is the Java analog of InnoDB's buffer pool. InnoDB keeps hot B+Tree pages in memory — reads serve from the buffer pool without going to disk. `BTreeService` keeps hot keys in a Caffeine cache — reads serve from Caffeine without an HTTP round-trip to the Rust server.

```java
@Service
public class BTreeService implements HealthIndicator {
    private final BTreeClient client;
    private final Cache<String, String> kvCache;

    // Write-through: cache AND forward to B+Tree
    public void put(String key, String value) {
        client.put(key, value);     // B+Tree O(log N) page writes
        kvCache.put(key, value);    // Caffeine O(1)
    }

    // Cache-first: no HTTP for hot keys
    public Optional<String> get(String key) {
        var cached = kvCache.getIfPresent(key);
        if (cached != null) return Optional.of(cached);  // O(1), no network

        // Cache miss: B+Tree O(log N) page reads
        var result = client.get(key);
        result.ifPresent(v -> kvCache.put(key, v));  // populate on read-through
        return result;
    }

    // Range: bypass per-key cache (we don't know which keys are in cache)
    public List<BTreeClient.KeyValue> range(String start, String end) {
        return client.range(start, end);  // B+Tree leaf walk
    }

    // delete: evict + B+Tree delete
    public void delete(String key) {
        kvCache.invalidate(key);
        client.delete(key);
    }
}
```

**Why range queries bypass the cache**: the per-key cache can't answer "all keys between A and Z" without scanning the entire cache map — which is O(cache_size), not O(result_size). The B+Tree leaf walk is O(result_size + log N), so it's always faster than a full cache scan for ranges. If you need hot-range caching, you would need a separate range cache with invalidation on puts/deletes to the affected key range — a much more complex design.

## What surprised me

<WhatSurprisedMe insight="The reason databases use 4096-byte pages isn't just to match the OS page size — it's that the OS will never return a partial page on a crash (torn write only happens within a page). So a single page-write failure is detectable via CRC; a multi-page-write failure needs the WAL. I assumed WAL was about individual byte-level corruption. The actual threat is coarser: a node split touching 3 pages is the dangerous operation. If any of the 3 page writes are lost, the tree is inconsistent. The CRC per page catches physical corruption; the WAL catches logical inconsistency across pages. Two different problems, two different solutions." />
