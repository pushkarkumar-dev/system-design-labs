## Why this lab exists

Every database you've ever used has a Write-Ahead Log. MySQL's InnoDB redo log, Postgres WAL, RocksDB's WAL, etcd's WAL — all of them, before writing a single byte to their primary data structure, first write to the log. The log is the source of truth. Everything else is a cache of it.

You've read about this in Designing Data-Intensive Applications. You've answered "what happens when a database crashes mid-write?" in system design rounds. But until you've *fought* a CRC mismatch in your own recovery code, until you've measured that a single fsync costs 310μs regardless of payload size, until you've seen group commit turn 3,200 writes/sec into 112,000 writes/sec with a trivial code change — you don't *know* it.

The WAL is also the right first storage lab: small enough to fully implement in a weekend, hard enough to surface the real lessons, foundational enough that the LSM-tree, B+tree, and Raft labs all build on it.

## What we're building

A Write-Ahead Log library in Rust, implemented across four stages. Each stage adds one layer and benchmarks it.

**We build:**
- Append-only log with monotonically increasing offsets
- On-disk record format with magic bytes, CRC32 checksum, and length prefix
- Crash recovery that stops at the first corrupted record
- Group commit that batches N records into one fsync call
- An HTTP server exposing the WAL to other processes

**We explicitly skip (and why):**
- Segment rotation lifecycle management (covered in v3 skeleton only)
- Log compaction (that's the LSM-tree lab)
- Replication (that's the Raft lab)
- Concurrent writers (a single `Mutex<Wal>` is fine for the lessons here)

## Mental model

Think of the WAL as a tape. You can only append to the end. You can read from any position. You can never go back and change what you wrote.

When a database wants to write anything durable, it *first* appends to the tape (the WAL), *then* writes to the real data structure (the B-tree, the memtable, etc.). If it crashes between the two, recovery replays the tape to reconstruct what was lost.

The key invariant: **if `append()` returns, the record is durable**. That guarantee is what makes everything else in the database safe to be async, buffered, and optimized.

```
Write path:               Recovery path:
                          
  app write               open WAL file
     │                       │
     ▼                       ▼
  WAL append             read record loop
     │ (fsync)            ├─ good CRC → recover
     │                    └─ bad CRC  → stop here
     ▼                       │
  data structure          apply records
  update (async)          to rebuild state
```

## Stage-by-stage build

  title="In-memory append-only log"
  goals={[

    "Write tests: sequential offsets, replay skips, empty log",

  loc={55}
>

The smallest WAL that can exist. A `Vec<Vec<u8>>` with an offset counter. Nothing survives a restart. The goal is to make the algorithm visible without I/O noise.

```rust
pub struct Wal {
    records: Vec<Vec<u8>>,
}

impl Wal {
    pub fn append(&mut self, data: &[u8]) -> u64 {
        let offset = self.records.len() as u64;
        self.records.push(data.to_vec());
        offset
    }

    pub fn replay(&self, from_offset: u64) -> Vec<LogRecord> {
        self.records.iter().enumerate()
            .filter(|(i, _)| *i as u64 >= from_offset)
            .map(|(i, data)| LogRecord::new(i as u64, data))
            .collect()
    }
}
```

**The core lesson**: offset = position in an immutable sequence. Every durability mechanism in v1–v3 is layered on top of this invariant without changing it.

</LabStage>

  id="v1"
  title="File-backed WAL with fsync and crash recovery"
  goals={[
    "Design the on-disk record format (magic + CRC32 + len + data)",
    "Implement append() with BufWriter + flush + sync_all",
    "Implement recovery: stop at first bad magic or CRC mismatch",
    "Write a test that simulates a truncated write and verifies recovery stops correctly",
  ]}
  loc={140}
>

The on-disk format matters. Choose it poorly and you either can't detect corruption or waste space. Here's what we use:

```
┌──────────┬─────────────┬──────────────┬────────────────────┐
│ MAGIC(1) │  CRC32 (4)  │   LEN (4)    │  DATA (LEN bytes)  │
└──────────┴─────────────┴──────────────┴────────────────────┘
```

- **MAGIC = 0xAB** — detects reads starting at the wrong offset
- **CRC32** — detects corrupted or partially-written records
- **LEN** — tells the reader how many bytes to read for DATA
- **DATA** — the payload, verbatim

**Recovery stops at the first anomaly**, not at EOF. This is correct. A truncated write (power failure mid-record) leaves garbage at the end of the file. Stopping there means we recover all good records; the garbage is silently dropped. Postgres calls this the "crash boundary" and it's exactly where you'd expect to lose data on an unclean shutdown.

**The two-step flush**: `BufWriter::flush()` moves data from the userspace buffer to the kernel page cache. `sync_all()` (fsync) moves it from the kernel cache to the storage device. Both are required. Skipping `flush()` leaves data in your process memory. Skipping `sync_all()` leaves data in the OS cache, which survives a process crash but not a kernel panic or power failure.

This fsync cost is the entire story of v2.

</LabStage>

<LabStage
  id="v2"
  title="Group commit — one fsync covers N records"
  goals={[
    "Split append() into buffered write (no fsync) + commit() (fsync)",
    "Benchmark: compare v1 fsync-per-write vs v2 at batch sizes 1, 8, 16, 32, 64",
    "Understand the durability window tradeoff",
  ]}
  loc={90}
>

The fsync latency on a modern NVMe SSD is roughly **300–500μs**, regardless of how much data you're flushing. This is hardware physics — the drive has to guarantee writes are durable, which takes time.

v1 calls fsync once per record. At 300μs each, you're capped at ~3,300 writes/sec.

v2 separates writing from fsyncing. Writers call `append()` (which buffers), and a commit step calls `commit()` (which fsyncs once for all buffered records):

```rust
// v2 API — identical records, different durability contract
pub fn append(&mut self, data: &[u8]) -> io::Result<u64> {
    // Write to BufWriter — no fsync. Not durable yet.
    // ...
}

pub fn commit(&mut self) -> io::Result<u64> {
    self.writer.flush()?;    // kernel buffer
    self.writer.get_ref().sync_all()?; // storage device
    // One fsync covers every append() since the last commit()
    Ok(self.unflushed)
}
```

**The tradeoff**: records between `append()` and `commit()` are in the kernel page cache. They survive a process crash but not a power failure. The window of potential data loss equals your commit interval. This is why MySQL's `innodb_flush_log_at_trx_commit=2` (fsync on commit) vs `=1` (fsync on each write) is a real production decision with a real safety-vs-throughput tradeoff.

</LabStage>

    "Write a checkpoint file that records the last stable offset",
    "Verify recovery skips segments before the checkpoint",
  ]}
  loc={135}
>

**Segments** solve the "unbounded log file" problem. Instead of one growing file, the WAL keeps a set of numbered segment files (`00000001.log`, `00000002.log`, …). When the active segment reaches a size limit (say 64 MB), it's sealed and a new segment opens.

Old segments can be deleted once the data structure they protect (memtable, B-tree page) has been flushed to disk — this is the checkpoint. Recovery starts from the checkpoint, skipping earlier segments entirely.

**Memory-mapped replay** (`mmap`) lets the OS page in the file lazily as you read it. For sequential scans (recovery), this gives bandwidth close to raw disk throughput (~1.4 GB/s on NVMe) because the kernel can prefetch aggressively.

For v3, the schema extends to include a checkpoint file:

```
wal-dir/
├── 00000001.log  (sealed, after checkpoint)
├── 00000002.log  (sealed, after checkpoint)
├── 00000003.log  (active — appending here)
└── CHECKPOINT    ({"segment": 3, "offset": 1247})
```

Recovery: read CHECKPOINT → skip segments 1 and 2 → read segment 3 from offset 1247.

</LabStage>

## Stress tests

**What breaks at 10×?** At 10× the design load (~1M writes/min), v1's fsync queue backs up. The OS journal throttles new writes. Latency spikes to >1s and the server stops accepting connections. v2 with batch=32 handles this fine because throughput scales with batch size, not raw fsync count.

**At 100×?** The `BufWriter` fills up faster than the disk can drain. Backpressure is the real constraint. Production WALs add a bounded channel between writers and the sync thread; when the channel is full, writers block (bounded backpressure) or return errors (unbounded clients). We have neither — at 100×, our server runs out of file descriptors.

**Corruption injection**: stop the server mid-append with `kill -9`, restart. Recovery correctly discards the partial record (CRC mismatch stops the scan). All records before the crash boundary are recovered.

### Part A — WalClient: the HTTP bridge

The Rust WAL server exposes three routes. `WalClient.java` wraps them using Spring Framework 6.1's `RestClient` — the modern replacement for `RestTemplate`.

```java
public class WalClient {
    private final RestClient http;

    public WalClient(String baseUrl) {
        this.http = RestClient.builder()
                .baseUrl(baseUrl)
                .defaultHeader("Accept", MediaType.APPLICATION_JSON_VALUE)
                .build();
    }

    // POST /append — returns durable offset
    public long append(byte[] data) {
        var response = http.post()
                .uri("/append")
                .contentType(MediaType.APPLICATION_OCTET_STREAM)
                .body(data)
                .retrieve()
                .body(AppendResponse.class);
        if (response == null) throw new WalException("null response");
        return response.offset();
    }

    // GET /replay?from=N — returns all records from offset N
    public List<WalRecord> replay(long fromOffset) {
        return http.get()
                .uri("/replay?from={offset}", fromOffset)
                .retrieve()
                .body(new ParameterizedTypeReference<List<WalRecord>>() {});
    }

    // Java 16 records — immutable DTOs, no boilerplate
    record AppendResponse(long offset) {}

    public record WalRecord(long offset, String data) {
        public byte[] rawBytes() {
            return Base64.getDecoder().decode(data);
        }
        public String asString() {
            return new String(rawBytes(), StandardCharsets.UTF_8);
        }
    }
}
```

**Why `RestClient` over `RestTemplate`?** `RestTemplate` is in maintenance mode since Spring 5.0. `RestClient` is fluent, type-inferred, and throws `RestClientException` on non-2xx automatically. The learning: APIs that encode the happy path in the type system (`.retrieve().body(T.class)`) reduce error-handling boilerplate by ~40%.

### Part B — WalService: application-level wrapper

```java
@Service
public class WalService implements HealthIndicator {
    private final WalClient client;
    private final Cache<Long, byte[]> recentAppends; // Caffeine

    public WalService(WalClient client, WalProperties props) {
        this.client = client;
        this.recentAppends = Caffeine.newBuilder()
                .maximumSize(props.cache().maxEntries())
                .expireAfterWrite(props.cache().ttl())
                .recordStats()           // enables hit-rate metrics
                .build();
    }

    public long append(String entry) {
        var bytes = entry.getBytes(StandardCharsets.UTF_8);
        long offset = client.append(bytes);
        recentAppends.put(offset, bytes); // write-through: cache the append
        return offset;
    }

    public String get(long offset) {
        var cached = recentAppends.getIfPresent(offset);
        if (cached != null) return new String(cached, StandardCharsets.UTF_8);

        // Cache miss: fall back to WAL replay (reads from disk)
        return client.replay(offset).stream()
                .filter(r -> r.offset() == offset)
                .findFirst()
                .map(WalClient.WalRecord::asString)
                .orElseThrow(() -> new WalClient.WalException("not found: " + offset));
    }

    // Actuator health — surfaces at /actuator/health
    @Override
    public Health health() {
        try {
            var s = client.health();
            return Health.up()
                    .withDetail("nextOffset", s.nextOffset())
                    .withDetail("cacheHitRate",
                        String.format("%.1f%%", recentAppends.stats().hitRate() * 100))
                    .build();
        } catch (Exception e) {
            return Health.down().withException(e).build();
        }
    }
}
```

**The Caffeine cache** avoids WAL replay for read-after-write scenarios. Replay is O(n) from the start of the file — fine for recovery, expensive for hot-path reads. The cache hit-rate appears in `/actuator/health`, so you know whether to tune `wal.cache.max-entries`.

**`HealthIndicator`** is the right Spring abstraction here. It plugs into the actuator subsystem with zero extra config and surfaces alongside JVM health, disk health, and database health in a unified `/actuator/health` endpoint. This is the pattern real Spring services use — not a custom `/ping` endpoint.
