## Why this lab exists

Every notification system, every event-driven microservice architecture, every analytics pipeline has a pub/sub broker at its centre. AWS SNS, Google Cloud Pub/Sub, Azure Service Bus — they all solve the same core problem: one publisher, N subscribers, each getting every message independently.

Most engineers use pub/sub as a black box. They call `sns.Publish()`, assume the message reaches all subscribers, and move on. That works until something breaks: a slow subscriber backs up the publisher, a network partition causes message loss, a high-volume topic generates millions of deliveries that overwhelm the fan-out system. Understanding the internals is what separates debugging "why are my SNS deliveries slow?" from blindly increasing timeouts.

The fix is to build it yourself. Not all of it — the distributed coordination, the durability guarantees, the global scale — but the core fan-out engine: the thing that makes pub/sub work, and the thing that makes it fail at scale.

This lab builds:
- A synchronous fan-out broker where the publisher blocks until all subscribers receive
- An async delivery engine with exponential backoff retry and dead-letter queues
- Ordering keys that preserve per-key message sequence without global locks
- Push subscriptions where the broker POSTs to subscriber HTTP endpoints

After this lab, you will be able to explain why SNS fan-out is limited to a few thousand subscribers before it needs a hierarchical tree, what dead-letter queues are and when to use them, and why "ordering" in a pub/sub system is harder to guarantee than it sounds.

## What we're building

A pub/sub broker in Go, implemented across three stages. Each stage adds one architectural layer.

**We build:**
- A `Broker` with synchronous fan-out: `Publish(topic, body, attrs)` blocks until all subscribers' channels receive
- `MessageFilter`: a `func(Message) bool` predicate that lets each subscriber receive only matching messages
- Async fan-out with buffered channels per subscriber (capacity 1000)
- Exponential backoff retry (1s, 2s, 4s) with dead-letter topic on exhaustion
- `BackpressurePolicy`: `Drop` (non-blocking) or `Block` (publisher waits)
- `SubscriptionStats`: atomic counters for `Delivered`, `Dropped`, `Retried`, `DLQ`
- Ordering keys: per-key goroutine channels guarantee message order within a key
- `ACL`: topic-level allow/deny rules for publishers and subscribers by `clientID`
- Push subscriptions: broker POSTs messages to subscriber HTTP endpoints, retries on non-200
- An HTTP server so any language can publish and subscribe without a native client

**We explicitly skip:**
- Persistent message storage (messages live only in channel buffers)
- Distributed topics (single broker, no partitioning)
- Exactly-once delivery (at-least-once only)
- Message schema validation
- Fan-out trees for millions of subscribers

## Mental model

### The difference between a queue and a pub/sub broker

A **queue** has ownership semantics. One consumer gets each message. Message disappears after consumption. Use queues when you want work to be done exactly once — send emails, process orders, encode videos.

A **pub/sub broker** has copy semantics. Every subscriber gets every message. Publishers and subscribers are decoupled — the publisher does not know how many subscribers exist or who they are. Use pub/sub when you want every consumer to react independently to every event.

```
Queue model (SQS, RabbitMQ):
  Publisher → [msg A] [msg B] [msg C]
                  ↓           ↓
               Worker 1    Worker 2
              (A is gone)  (B is gone)

Pub/Sub model (SNS, Pub/Sub):
  Publisher → msg A
                 ↓   (copy)
               ├── Email service    (gets A)
               ├── Analytics system (gets A)
               └── Audit log        (gets A)
              (A delivered to all three independently)
```

The practical consequence: if you add a new subscriber to a pub/sub topic, every future message is delivered to it automatically. If you add a new worker to a queue, it competes with existing workers for messages (load balancing). These are completely different semantics, and choosing the wrong one is a common architecture mistake.

### Why fan-out is harder than it looks

The naive implementation of `Publish` is a `for` loop over subscribers:

```go
for _, sub := range topic.subscribers {
    sub.ch <- msg  // blocking send
}
```

This is synchronous fan-out. It works, but it has a critical flaw: the publisher's speed equals the speed of the **slowest subscriber**. If subscriber 3 out of 10 is slow (network congestion, processing backlog, GC pause), the publisher is blocked waiting for it. All other subscribers — which might be fast — also wait.

The v1 async fix replaces blocking sends with non-blocking sends into buffered channels. The publisher's goroutine puts the message into each subscriber's buffer and moves on. Each subscriber drains its own buffer at its own speed. This decouples publisher throughput from subscriber throughput entirely.

The cost: if a subscriber's buffer fills up (it is too slow to drain), you have a choice: drop messages (lose data) or block the publisher (back-pressure). Both are legitimate choices depending on your use case.

### What ordering means in a pub/sub system

"Ordered delivery" in pub/sub is almost always weaker than people expect. Google Cloud Pub/Sub and AWS SNS FIFO topics promise ordering **within an ordering key**, not globally. That is:

- Messages with key `"user-42"` arrive at each subscriber in the order they were published
- Messages with key `"user-99"` also arrive in publish order
- But a `"user-42"` message and a `"user-99"` message published close together may arrive in either order relative to each other

This weaker guarantee is implementable without a global lock: each ordering key gets its own serialised delivery goroutine. Different keys run in parallel. Same key is strictly sequential. This is exactly what our v2 implementation does.

### The Message type

```go
type Message struct {
    ID          string
    Topic       string
    Body        []byte
    Attributes  map[string]string
    PublishedAt time.Time
}
```

`Attributes` is the pub/sub equivalent of HTTP headers — key-value metadata that subscribers can filter on without parsing the body. AWS SNS calls these "message attributes". They enable a single "events" topic to serve subscribers that only care about `"type": "order"` or `"type": "payment"` — no need for separate topics per event type.

### The fan-out loop

```go
func (b *Broker) Publish(topicName string, body []byte, attrs map[string]string) (string, error) {
    t := b.getOrCreateTopic(topicName)
    msg := newMessage(topicName, body, attrs, "")

    t.mu.RLock()
    subs := make([]*Subscription, 0, len(t.subscribers))
    for _, s := range t.subscribers {
        subs = append(subs, s)
    }
    t.mu.RUnlock()

    for _, sub := range subs {
        if sub.Filter != nil && !sub.Filter(msg) {
            continue // filter rejected this message for this subscriber
        }
        sub.ch <- msg // synchronous: blocks until subscriber receives
    }
    return msg.ID, nil
}
```

The critical design choice: we copy the subscriber list under `RLock`, then release the lock before sending to channels. This prevents a deadlock where the subscriber's consumer goroutine tries to acquire the same lock that Publish is holding. The list copy is O(N subscribers) — acceptable since subscriber counts are typically small.

**The key test to write**: call `Publish` with 3 subscribers, each with a different `MessageFilter`. Assert only matching messages arrive. This is the test that validates that filters are evaluated per-subscriber, not per-message at publish time.

**The performance test to run mentally**: with 10 synchronous subscribers, what is the throughput? If each channel send takes 1 microsecond and you have 10 subscribers, each `Publish` call takes at least 10 microseconds — giving a ceiling of 100,000 msg/sec. With 1 subscriber it's 1 microsecond per publish — giving 1,000,000 msg/sec. Synchronous fan-out is O(N) in publish latency. This is what v1 fixes.

### The async delivery goroutine

The core change from v0: instead of `sub.ch <- msg` in the `Publish` loop, we do:

```go
func deliverAsync(sub *Subscription, msg Message) {
    if sub.backpressure == Drop {
        select {
        case sub.ch <- msg:
            // delivered to buffer
        default:
            atomic.AddInt64(&sub.stats.Dropped, 1)
            // channel full — message silently dropped
        }
    } else {
        // Block policy: park the publisher goroutine until space is available
        sub.ch <- msg
    }
}
```

The delivery goroutine runs independently per subscription:

```go
func (sub *Subscription) deliveryLoop() {
    defer sub.wg.Done()
    for {
        select {
        case <-sub.stopCh:
            return
        case msg := <-sub.ch:
            sub.deliverWithRetry(msg)
        }
    }
}
```

### Exponential backoff retry

Real-world delivery failures are transient: network blips, downstream service restarts, temporary rate limits. Exponential backoff gives failing services time to recover without hammering them:

```go
var retryDelays = []time.Duration{1*time.Second, 2*time.Second, 4*time.Second}

func (sub *Subscription) deliverWithRetry(msg Message) {
    for attempt := 0; attempt <= maxRetries; attempt++ {
        if attempt > 0 {
            atomic.AddInt64(&sub.stats.Retried, 1)
            time.Sleep(retryDelays[attempt-1])
        }
        if sub.callback == nil || sub.callback(msg) {
            atomic.AddInt64(&sub.stats.Delivered, 1)
            return
        }
    }
    // All retries exhausted: send to DLQ
    atomic.AddInt64(&sub.stats.DLQ, 1)
    sub.broker.Publish(sub.Topic+"-dlq", msg.Body, msg.Attributes)
}
```

The total retry window is 1+2+4 = 7 seconds. Google Cloud Pub/Sub uses exponential backoff starting at 10ms and going up to 600 seconds, with configurable maximum retries. AWS SNS has a fixed 4-phase retry schedule per delivery protocol (HTTP/SQS/Lambda/email).

### Dead-letter topics

The dead-letter topic (DLQ) is a separate topic that receives messages that could not be delivered after all retries. It serves two purposes:

1. **Observability**: you can inspect the DLQ to understand what failed and why
2. **Recovery**: after fixing the root cause, you can replay DLQ messages to the original topic

In our implementation, the DLQ topic is automatically named `"<original-topic>-dlq"`. A subscriber on the DLQ topic receives failed messages just like any other subscriber — they can alert, log, or re-enqueue them.

### Backpressure design

The `Drop` policy (default) keeps publishers fast but loses messages if subscribers are slow. The `Block` policy preserves every message but makes publishers slow when subscribers are slow — it propagates back-pressure to the publisher.

Which to choose? For analytics and logging (high-volume, loss-tolerant), use `Drop`. For financial transactions and audit events (low-volume, loss-critical), use `Block`. Real SNS is always at-least-once, which means it retries rather than drops — but it also enforces per-topic throughput limits to prevent one noisy publisher from starving others.

### Ordering keys: per-key channels

The naive approach to ordered delivery would be a global lock per subscription — only deliver one message at a time, in strict order. This serialises all messages across all keys, capping throughput at the rate of one delivery goroutine.

The correct approach is per-key parallelism: messages with different ordering keys are independent and can be delivered concurrently. Only messages with the *same* ordering key need serialisation.

```go
func deliverOrdered(sub *Subscription, msg Message) {
    if msg.OrderingKey == "" {
        deliverAsync(sub, msg) // no ordering needed
        return
    }

    sub.orderingMu.Lock()
    keyCh, ok := sub.orderingKeys[msg.OrderingKey]
    if !ok {
        // First message with this key: create a dedicated serialising channel.
        keyCh = make(chan Message, 1000)
        sub.orderingKeys[msg.OrderingKey] = keyCh
        go sub.keyLoop(keyCh) // one goroutine per ordering key
    }
    sub.orderingMu.Unlock()

    keyCh <- msg // deliver to per-key serialiser
}
```

The `keyLoop` goroutine reads from the per-key channel and forwards to `sub.ch` in strict order. The subscriber reads from a single `sub.ch` regardless of ordering keys, so the API is unchanged.

This is the same design Google Cloud Pub/Sub uses. Each ordering key maps to a "stream" within the subscription. Streams are independent; within a stream, messages are strictly ordered.

### Push subscriptions

A pull subscription requires the subscriber to poll for messages. A push subscription inverts control: the broker calls the subscriber's HTTP endpoint for each message.

```go
func (sub *Subscription) pushLoop() {
    client := &http.Client{Timeout: 10 * time.Second}
    for msg := range sub.ch {
        sub.pushWithRetry(client, msg)
    }
}

func postToWebhook(client *http.Client, url string, body []byte) bool {
    resp, err := client.Post(url, "application/json", bytes.NewReader(body))
    if err != nil { return false }
    defer resp.Body.Close()
    return resp.StatusCode == http.StatusOK
}
```

The broker sends:

```json
{
  "id": "a3f9...",
  "topic": "orders",
  "body": "base64-encoded-payload",
  "attributes": {"type": "order"},
  "publishedAt": "2026-05-08T12:00:00Z"
}
```

The subscriber responds with HTTP 200 to acknowledge, or any non-200 to trigger retry. This is identical to the AWS SNS HTTP/HTTPS delivery protocol, which uses the same acknowledgement model.

**Push vs. pull trade-offs**:
- Push has lower latency (broker drives delivery immediately) but requires the subscriber to be reachable over HTTP and able to handle the broker's request rate
- Pull gives the subscriber rate control (it only polls when ready) and works behind NAT, but adds latency equal to the poll interval

### ACL: topic-level access control

The `ACL` type enforces which `clientID` strings may publish to or subscribe from a topic:

```go
acl := NewACL().AllowPublisher("order-service").AllowSubscriber("analytics")
broker.CreateTopic("payments", acl)

// order-service can publish:
broker.PublishAs("order-service", "payments", body, attrs) // OK

// rogue-service cannot:
broker.PublishAs("rogue-service", "payments", body, attrs) // error
```

The allow/deny model: deny always wins over allow; an empty allow-list means "allow all". This is the same two-list model as AWS IAM's explicit deny semantics.

## Stress tests

**Slow subscriber with synchronous fan-out**: with v0's synchronous broker, subscribe 10 consumers where one consumer's goroutine does `time.Sleep(10 * time.Millisecond)` per message. Observe that overall publish throughput drops to 100 msg/sec — the slowest subscriber sets the pace. Switch to v1's async broker and observe throughput recover to the channel-limited rate. This single experiment demonstrates why synchronous fan-out is not production-ready.

**Buffer saturation under Drop policy**: with v1 async and Drop policy, subscribe a consumer that never reads from its channel. Publish messages at 100,000 msg/sec. After ~10ms the buffer of 1000 fills up. Observe the `Dropped` counter in `SubscriptionStats` climbing. The publisher is unaffected — it continues at full speed. This shows the Drop policy trades delivery reliability for publisher independence.

**Ordering key correctness under concurrent publishing**: with v2, create 4 goroutines each publishing 1000 messages to key `"stream-A"` and 4 goroutines publishing to key `"stream-B"`. Subscribe and assert all `"stream-A"` messages arrive in publish order (sequence numbers 0-999 from each goroutine are interleaved but each goroutine's sequence is intact within its own ordering key). This test exposes any race condition in the per-key channel creation code.

**Push endpoint failure cascade**: create 5 push subscriptions pointing at an HTTP test server that fails the first 2 requests and then succeeds. Publish 1 message. Observe the broker's retry goroutine making 3 HTTP calls per subscription (2 failures + 1 success), with 1s + 2s delay between them. With 5 subscriptions and 7 seconds of retry time, the broker's goroutine pool must handle all 5 concurrently without serialising them.

### The SNS analogy

Real Spring Cloud AWS uses `SnsTemplate` backed by the AWS SDK, which calls the SNS API over HTTPS with SigV4 authentication. Our broker speaks plain HTTP, so `PubSubClient` wraps `RestClient` instead. The conceptual contract is identical: `publish(topic, body, attributes)` delivers a message to all current subscribers.

The gap is not in the Java code — it's in what happens after `publish` returns. With real SNS, the message is durably stored and will be delivered even if the SNS service restarts. With our broker, the message lives in Go channel buffers and is lost on crash. The Java code looks identical either way; the difference is entirely in the broker's write path.

### Part A: PubSubClient — the HTTP bridge

`PubSubClient` wraps the broker HTTP API with three methods: `createTopic`, `publish`, and `createSubscription`. Under 60 lines, intentionally.

```java
public class PubSubClient {
    private final RestClient http;

    // Analogous to: snsTemplate.sendNotification(topic, message)
    public String publish(String topic, String body, Map attributes) {
        var payload = Map.of(
            "body", Base64.getEncoder().encodeToString(body.getBytes()),
            "attributes", attributes != null ? attributes : Map.of()
        );
        var resp = http.post()
                .uri("/topics/{topic}/publish", topic)
                .contentType(MediaType.APPLICATION_JSON)
                .body(payload)
                .retrieve()
                .body(Map.class);
        return (String) resp.get("messageId");
    }
}
```

The `body` is base64-encoded because JSON does not natively represent arbitrary bytes. SNS uses the same approach — message bodies are UTF-8 strings or base64-encoded binary.

### Part B: Fan-out demo

The demo creates one topic, subscribes three consumers (email, analytics, audit), and publishes 100 events. This is the canonical SNS fan-out pattern: one publish call drives N downstream systems.

```java
// Create three independent subscriptions on the same topic.
// Each subscription is a separate delivery queue — all get every message.
String emailSubId     = client.createSubscription("user-events");
String analyticsSubId = client.createSubscription("user-events");
String auditSubId     = client.createSubscription("user-events");

// One publish reaches all three.
client.publish("user-events", "user-42 purchased item-7", Map.of("action", "purchase"));
```

In real SNS, these subscriptions would be SQS queues, Lambda functions, or HTTP endpoints — not pull queues. But the semantics are identical: one `Publish` call, three independent deliveries.

### Part C: What SNSTemplate does under the hood

`SnsTemplate.sendNotification()` from Spring Cloud AWS:
1. Resolves the topic ARN from the topic name (a lookup in SNS)
2. Calls `AmazonSNS.publish(topicArn, message)` using the AWS SDK
3. AWS SDK signs the request with SigV4, POSTs to `https://sns.amazonaws.com`
4. SNS writes the message to its internal durable store, returns a `MessageId`
5. SNS asynchronously fans out to all subscriptions (SQS, Lambda, HTTP, email, SMS)

Our `PubSubClient.publish()` does steps 2-4 with HTTP instead of HTTPS+SigV4, and step 5 synchronously in Go goroutines instead of asynchronously at AWS scale. The result is the same: all subscribers get the message.
