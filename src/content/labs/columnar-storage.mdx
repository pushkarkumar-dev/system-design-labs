## Why this lab exists

Every analytics database you use — BigQuery, Snowflake, Redshift, DuckDB — stores its tables in a columnar format. When you run `SELECT SUM(price) FROM orders`, the database reads only the `price` column bytes. A row-oriented database like PostgreSQL reads every column in every row just to extract `price`. For a table with 8 columns and 100 million rows, that is an 8x difference in I/O before any work is done.

Apache Parquet is the de facto columnar file format for the data ecosystem: Spark, Hive, Arrow, Pandas, and DuckDB all read and write Parquet. Yet most engineers who use Parquet daily have never looked at how column projection, row group pruning, or dictionary encoding actually work at the byte level.

This lab changes that. You will build a Parquet-lite engine in three stages: from an in-memory column store that proves the cache efficiency claim, to row groups with dictionary encoding and predicate pushdown, to RLE and bit-packing with a binary file format that mirrors Parquet's footer-first design. By the end, every Parquet optimization you read about in a query optimizer blog post will be something you have implemented yourself.

## What we are building

A columnar storage engine where:

- The fundamental unit is a column: a `Vec<Value>` array, one element per row
- `Value` is a typed enum: `Int(i64)`, `Float(f64)`, `Str(String)`, `Bool(bool)`, `Null`
- Rows are decomposed into per-column appends on write; column arrays are scanned directly on read
- Row groups batch rows into fixed-size blocks (65,536 rows) with per-column min/max statistics
- Dictionary encoding compresses low-cardinality string columns to 1-byte codes
- RLE compresses sorted columns; bit-packing compresses small integers
- A binary file format (magic + row groups + footer offset table) provides persistence

**Explicit scope exclusions**: no Snappy/Zstd compression, no Bloom filters, no nested types (lists/structs), no schema evolution, no SIMD vectorization, no S3 byte-range reads, no concurrent readers.

## Mental model

### Row store vs column store: a cache line story

The performance difference between row and column stores is not about algorithms — it is about cache lines.

A modern CPU cache line is 64 bytes. When the CPU loads a memory address, it loads the entire 64-byte cache line containing that address. Cache efficiency means: what fraction of the loaded bytes do we actually use?

Consider a table with 8 columns of 8 bytes each. One row = 64 bytes = exactly one cache line.

```
Row store — one row per cache line:
  Cache line: [id:8][price:8][qty:8][status:8][region:8][ts:8][discount:8][active:8]
  SUM(price): load cache line, use 8 bytes, discard 56 bytes
  Cache efficiency: 8/64 = 12.5%

Column store — one column chunk per cache line:
  Cache line: [price:8][price:8][price:8][price:8][price:8][price:8][price:8][price:8]
  SUM(price): load cache line, use all 64 bytes (8 prices)
  Cache efficiency: 64/64 = 100%
```

This is not a theoretical difference. Modern CPUs process memory at 40-100 GB/s when they hit L3 cache, but drop to 10-20 GB/s on cache misses. A column store that scans a contiguous `Vec<i64>` runs at near-peak memory bandwidth; a row store that scans the same 1M integers is scattered across 8M cache line loads.

### Row groups: the unit of parallelism and pruning

A row group is a horizontal partition of the table — a fixed number of consecutive rows stored together. Parquet's default is 128K rows per group; we use 65,536.

```
Table: 1,000,000 rows
  Row group 0:  rows 0..65535      (Jan data)  min_ts=1700000000 max_ts=1700065535
  Row group 1:  rows 65536..131071 (Feb data)  min_ts=1700065536 max_ts=1700131071
  ...
  Row group 15: rows 983040..999999             min_ts=1701310464 max_ts=1701375999

Query: WHERE ts > 1701310000
  Row groups 0-14: max_ts < 1701310000 — skip entirely (0 bytes read)
  Row group 15: may contain matching rows — read and filter
```

The min/max statistics live in the file footer (and in our `ColumnStats` struct). Skipping a row group is free — it requires only reading 16 bytes of metadata, not any column data. For a query that touches 1% of rows, 99% of row groups are skipped.

### Dictionary encoding: turn strings into integers

A status column with values "active", "inactive", "pending", "suspended", "trial" has 5 distinct values but 1 million rows. Dictionary encoding stores each unique value once (the dictionary) and stores one 1-byte code per row.

```
Without dictionary (1M rows, avg 8 bytes/value):
  Raw storage: 1,000,000 * 8 = 8 MB

With dictionary (5 distinct values + 1 byte/row):
  Dictionary: 5 values * 8 bytes = 40 bytes
  Codes:      1,000,000 * 1 byte = 1 MB
  Total:      ~1 MB  (87.5% reduction)
```

On top of dictionary encoding, Parquet applies RLE (run-length encoding) to the code stream — if rows are sorted by status, consecutive codes are identical and compress further. Our implementation applies these encodings independently.

The key design decision in v0 is the data layout. A `RowStore` stores `Vec<HashMap<String, Value>>` — each element is a full row. A `ColumnStore` stores `HashMap<String, Vec<Value>>` — each entry is a full column. Flipping the nesting order is the entire columnar idea.

```rust
pub struct ColumnStore {
    pub columns: HashMap<String, ColumnData>,  // col_name -> Vec<Value>
    pub num_rows: usize,
}
```

**The append operation** decomposes a row into per-column pushes. Missing columns receive `Value::Null` — this ensures all columns stay the same length after every append, which is required for the row index to work:

```rust
pub fn append_row(&mut self, row: HashMap<String, Value>) {
    for (col_name, col_data) in &mut self.columns {
        let val = row.get(col_name).cloned().unwrap_or(Value::Null);
        col_data.push(val);
    }
    self.num_rows += 1;
}
```

**The scan operation** is O(1) — it returns a `&[Value]` slice directly from the `Vec`:

```rust
pub fn scan_column(&self, name: &str) -> Option<&[Value]> {
    self.columns.get(name).map(|c| c.as_slice())
}
```

No deserialization, no row reconstruction, no pointer chasing. The entire column is a contiguous array in memory. The CPU prefetcher can stream it at full memory bandwidth.

**The benchmark target** is `sum_int_column`. The column store implementation:

```rust
pub fn sum_int_column(&self, name: &str) -> Option<i64> {
    self.scan_column(name).map(|col| {
        col.iter().filter_map(|v| v.as_int()).sum()
    })
}
```

This iterates one `Vec<Value>` of length 1M. The row store implementation iterates `Vec<HashMap<String, Value>>` of length 1M, extracting one field from each map. The map lookup is O(1) but requires hashing and pointer following — exactly the cache inefficiency the column layout avoids.

**Predicate evaluation** in v0 iterates row-by-row within the column. This is not ideal — better implementations vectorize the predicate over the column array. But v0 demonstrates the correctness of column-level evaluation:

```rust
fn eval_predicate(&self, pred: &Predicate, row_idx: usize) -> bool {
    match pred {
        Predicate::Eq(col, expected) => {
            let actual = self.columns.get(col)
                .and_then(|c| c.get(row_idx));
            actual == Some(expected)
        }
        // ...
    }
}
```

The `And` combinator is recursive — it supports arbitrarily deep predicate trees with the same evaluation loop.

v1 introduces two independent features that interact: row groups give us the unit of pruning, and dictionary encoding compresses the column data within each group.

**Row group structure** — a `RowGroup` holds one column per key, plus computed stats:

```rust
pub struct RowGroup {
    pub columns: HashMap<String, ColumnData>,
    pub stats: HashMap<String, ColumnStats>,
    pub size: usize,
}

pub struct ColumnStats {
    pub min_int: Option<i64>,
    pub max_int: Option<i64>,
    pub null_count: usize,
    pub distinct_count: usize,
}
```

The stats are computed once when the row group is finalised (`finalise_stats`). After finalisation, no row-level data is needed to answer "can this group match predicate P?" — only the stats are consulted.

**Row group pruning** — the `can_satisfy` method:

```rust
pub fn can_satisfy(&self, pred: &Predicate) -> bool {
    match pred {
        Predicate::Eq(_, Value::Int(target)) => {
            match (self.min_int, self.max_int) {
                (Some(lo), Some(hi)) => *target >= lo && *target <= hi,
                _ => true,
            }
        }
        Predicate::Lt(_, Value::Int(threshold)) => {
            self.min_int.map_or(true, |min| min < *threshold)
        }
        _ => true, // conservative: don't prune unknown predicates
    }
}
```

If `can_satisfy` returns `false`, the entire row group is skipped — no column data is read. This is the key Parquet optimization: for a query touching 1% of rows, 99% of row groups are skipped in O(1) per group.

**The writer** buffers rows and flushes when the group fills:

```rust
pub fn write_row(&mut self, row: HashMap<String, Value>) {
    // append to current group...
    self.current_group.size += 1;
    if self.current_group.size >= ROW_GROUP_SIZE {
        self.flush_group();
    }
}
```

When `flush_group` is called, `finalise_stats` is computed over the completed group before it is moved to `finished_groups`. The new current group is empty.

**Dictionary encoding** — the encoding decision is per-column per-row-group. If a column has fewer than 256 distinct values and `distinct_count / total_count < 0.5`, we encode it:

```rust
pub fn encode_dict(col: &ColumnData) -> Option<DictColumn> {
    let mut dict: Vec<Value> = Vec::new();
    let mut index_map: HashMap<String, u8> = HashMap::new();
    let mut codes: Vec<u8> = Vec::with_capacity(col.len());

    for v in col {
        let key = value_key(v);
        let code = if let Some(&existing) = index_map.get(&key) {
            existing
        } else {
            if dict.len() >= 256 { return None; }
            let code = dict.len() as u8;
            dict.push(v.clone());
            index_map.insert(key, code);
            code
        };
        codes.push(code);
    }
    // only encode if worthwhile
    let ratio = dict.len() as f64 / col.len().max(1) as f64;
    if ratio >= 0.5 { return None; }
    Some(DictColumn { dict, codes })
}
```

A status column with 5 values and 1M rows: `dict` has 5 entries (~40 bytes total), `codes` has 1M entries (1 byte each = 1 MB). Raw storage of 1M strings would be 8-12 MB. Compression: ~99%.

**Decoding** is a simple code-to-value lookup:

```rust
pub fn decode_dict(dict_col: &DictColumn) -> ColumnData {
    dict_col.codes.iter().map(|&code| {
        dict_col.dict.get(code as usize).cloned().unwrap_or(Value::Null)
    }).collect()
}
```

This is also vectorisable — the code-to-value lookup is a table lookup, which CPUs handle efficiently via gather instructions.

v2 adds two more encoding schemes and a binary file format. The encodings are complementary: dictionary encoding helps low-cardinality columns, RLE helps sorted columns, and bit-packing helps integer columns with a small range.

**Run-length encoding** merges consecutive equal values into `(value, count)` pairs:

```rust
pub fn encode_rle(col: &[Value]) -> RleColumn {
    let mut runs: Vec<(Value, u32)> = Vec::new();
    for val in col {
        match runs.last_mut() {
            Some((last_val, count)) if last_val == val => { *count += 1; }
            _ => { runs.push((val.clone(), 1)); }
        }
    }
    RleColumn { runs, total_rows: col.len() }
}
```

A timestamp column sorted by hour: if there are 200 rows per hour and 5,000 hours, the encoded form has 5,000 run pairs instead of 1M values — a 200x reduction. At 16 bytes per run pair (8 for value + 4 for count + overhead), that is 80 KB vs 8 MB.

**Bit-packing** addresses integer columns where values fit in fewer than 64 bits. A column of small integers (0-255) wastes 56 of 64 bits per value in a `Vec<i64>`. Bit-packing stores each value in the minimum number of bits:

```rust
pub fn bitpack(values: &[i64], bits: u8) -> Vec<u8> {
    let total_bits = values.len() * bits as usize;
    let byte_count = (total_bits + 7) / 8;
    let mut output = vec![0u8; byte_count];
    let mut bit_pos = 0usize;
    for &v in values {
        for bit_idx in (0..bits).rev() {
            let bit = ((v >> bit_idx) & 1) as u8;
            let byte_idx = bit_pos / 8;
            let bit_within_byte = 7 - (bit_pos % 8);
            output[byte_idx] |= bit << bit_within_byte;
            bit_pos += 1;
        }
    }
    output
}
```

For 8-bit packing: 1M integers × 8 bits = 1M bytes. Raw: 1M × 8 bytes = 8 MB. That is 8x compression from bit-packing alone, before any other encoding.

**The file format** — Parquet's key design is footer-last: the file begins with raw data, and the footer (offset table) is at the end. To open a Parquet file, you seek to the last 8 bytes (the footer offset), then seek to the footer and read the offset table, then issue targeted byte-range reads for the columns you need.

```
File structure:
  MAGIC (8 bytes)
  ROW GROUP 0
    num_cols (u32 LE)
    row_count (u32 LE)
    for each column:
      name_len (u16 LE)
      name_bytes (UTF-8)
      encoding_type (u8)
      data_len (u32 LE)
      data_bytes
  ROW GROUP 1
    ...
  FOOTER
    num_groups (u32 LE)
    group_0_offset (u64 LE)
    group_1_offset (u64 LE)
    ...
    footer_offset (u64 LE)   <- last 8 bytes of file
```

The footer offset at the very end is the anchor: `read_file` seeks to `file_end - 8`, reads the footer offset, then seeks to the footer and reads all group offsets. This enables reading a specific row group without scanning the file sequentially.

```rust
pub fn read_file<R: Read + Seek>(reader: &mut R) -> io::Result<Vec<RowGroup>> {
    // Verify magic
    let mut magic = [0u8; 8];
    reader.read_exact(&mut magic)?;
    if &magic != MAGIC { return Err(...); }

    // Read footer offset from last 8 bytes
    reader.seek(SeekFrom::End(-8))?;
    let mut footer_offset_buf = [0u8; 8];
    reader.read_exact(&mut footer_offset_buf)?;
    let footer_offset = u64::from_le_bytes(footer_offset_buf);

    // Seek to footer and read group offsets
    reader.seek(SeekFrom::Start(footer_offset))?;
    // ...
}
```

This is identical in design to Apache Parquet's actual read path — the only difference is that Parquet stores many more fields per column chunk (encoding metadata, dictionary page, data pages, statistics, Bloom filter offset, etc.).

## Stress tests

To verify correctness under load, the test suite includes:

**Column-row equivalence**: build 1M rows in both `ColumnStore` and `RowStore`, verify that `sum_int_column` returns identical results. If the column decomposition loses any values, this fails.

**Row group split boundary**: write exactly `ROW_GROUP_SIZE + 1` rows to a `ParquetLiteWriter`, verify that `finish()` returns exactly 2 groups — the first with `ROW_GROUP_SIZE` rows and the second with 1. Tests the off-by-one in the flush trigger.

**Dictionary round-trip with all Value types**: encode and decode a column containing `Int`, `Float`, `Str`, `Bool`, and `Null` values. Verifies that the `value_key` function produces unique keys for all types.

**Predicate pushdown skips exactly the right groups**: build 3 row groups with non-overlapping integer ranges, query for a value in only the first group, verify that `count_skipped` returns 2.

**File format round-trip with mixed types**: write a file containing `Int`, `Float`, `Str`, `Bool`, and `Null` values, read it back, verify byte-for-byte equality of all column data.

### Part A — ColumnarClient: HTTP bridge to the Rust engine

`ColumnarClient.java` wraps the Rust columnar server's REST endpoints using Spring Framework 6.1's `RestClient`. The key operation is column-level aggregation — the Rust server returns only the aggregate, not the raw column data:

```java
public class ColumnarClient {
    private final RestClient http;

    public ColumnarClient(String baseUrl) {
        this.http = RestClient.builder()
                .baseUrl(baseUrl)
                .defaultHeader("Accept", "application/json")
                .build();
    }

    /** Sum an integer column. The Rust server does the scan; Java receives the aggregate. */
    public long sumColumn(String columnName) {
        var resp = http.get()
                .uri("/sum?col={col}", columnName)
                .retrieve()
                .body(SumResponse.class);
        return resp != null ? resp.sum() : 0L;
    }

    public record SumResponse(String column, long sum, long rows) {}
}
```

This mirrors how analytics systems like Presto or Trino work: the storage engine (Parquet file on HDFS/S3) returns aggregates or filtered column chunks to the query coordinator. The Java coordinator never sees raw row data.

### Part B — ParquetComparison: how parquet-mr reads column subsets

`ParquetComparison.java` documents the Apache Parquet (parquet-mr) read pattern — the production counterpart to our Parquet-lite engine. The key capability is column projection at the I/O layer:

```java
// Declare projection: only read "id" and "price" columns
MessageType projection = MessageTypeParser.parseMessageType(
    "message orders { required int64 id; required double price; }"
);

// Build reader with projection
ParquetReader<GenericRecord> reader = AvroParquetReader
    .<GenericRecord>builder(HadoopInputFile.fromPath(path, conf))
    .withConf(conf)
    .build();

// Only "id" and "price" bytes are read from disk.
// All other column chunks are skipped at the I/O layer.
GenericRecord record;
long sum = 0;
while ((record = reader.read()) != null) {
    sum += ((Number) record.get("price")).longValue();
}
reader.close();
```

The critical difference from our toy: parquet-mr skips non-projected columns at the I/O layer, not at the filter layer. The file format stores each column as a separate byte range per row group. The reader can seek directly to the `price` column range and skip all other column ranges without reading them. Our implementation reads all column data and filters in memory.

### Part C — What parquet-mr adds that our toy misses

The comparison table in `ParquetComparison.java` documents the gap:

## What surprised me

<WhatSurprisedMe insight="The column scan vs row scan gap surprised me. Summing an integer column across 1M rows in column store reads at 2.8 GB/s — just an array scan. The same operation in row store reads at 0.3 GB/s — a 9x difference. The row store does not read 9x more data in bytes; it reads the same number of integers. The difference is cache-line waste: a row has 8 columns (8 x 8 bytes = 64 bytes, one cache line), but we only need 1 column (8 bytes = 1/8 of the cache line). Every cache miss pays for 64 bytes but delivers only 8 useful bytes. Column store wastes 0 cache bandwidth." />

## Where to go deeper

- **Apache Parquet format spec**: the full binary format including encoding types, page headers, and statistics — [parquet.apache.org/docs/file-format/](https://parquet.apache.org/docs/file-format/)
- **Dremel paper (Google 2010)**: the original paper introducing nested columnar storage and the definition/repetition level encoding that Parquet uses for lists and structs
- **DuckDB's execution model**: how DuckDB achieves 100 GB/s scan speeds using vectorized (SIMD) column scans — [duckdb.org/why_duckdb](https://duckdb.org/why_duckdb)
- **Apache Arrow**: the in-memory columnar format that complements Parquet on disk — enables zero-copy data exchange between systems that all use Arrow buffers
- **Parquet encoding types**: PLAIN, RLE_DICTIONARY, DELTA_BINARY_PACKED, BYTE_STREAM_SPLIT — each optimised for different data distributions — [parquet.apache.org/docs/file-format/data-pages/encodings/](https://parquet.apache.org/docs/file-format/data-pages/encodings/)
