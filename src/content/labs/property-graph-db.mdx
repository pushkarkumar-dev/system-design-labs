## Why this lab exists

Every recommendation engine, fraud detection system, and social network runs on graphs. When LinkedIn answers "people you may know," it is traversing a graph where nodes are people and edges are connections. When a bank detects fraud, it is looking for suspicious transaction paths in a graph of accounts and transfers. The query "find all accounts within 3 hops of this flagged account" is trivial in a graph database and painful in a relational one — it requires 3 self-joins, each exponentially larger than the last.

This lab makes you build the data structure that powers those systems. You will go from a bare-bones in-memory adjacency list, to a mini Cypher query engine that parses and executes graph pattern matching, to a label-indexed, Dijkstra-capable, PageRank-equipped graph database. By the end, you will understand exactly why Neo4j's "index-free adjacency" is not marketing — it is a fundamental architectural choice that makes multi-hop traversal O(1) per hop regardless of graph size.

The reference for comparison throughout this lab is Neo4j — the dominant property graph database, written in Java, with a 15-year head start. Every design decision in our Rust toy is illuminated by asking: what does Neo4j do instead, and why?

## What we are building

A property graph database where:
- Nodes have a unique `NodeId`, one or more string labels, and a property map (`HashMap<String, Value>`)
- Edges are directed, have a string label, a `from` NodeId, a `to` NodeId, and properties
- `Value` is a tagged enum: `String`, `Int`, `Float`, `Bool`
- Queries are expressed in a subset of the Cypher query language
- v2 adds a label index, a property index, Dijkstra shortest path, and iterative PageRank

**Explicit scope exclusions**: no disk persistence, no ACID transactions, no MVCC, no concurrent access, no Cypher aggregations (COUNT, COLLECT), no schema enforcement, no relationship directions in index lookups.

## Mental model

### The property graph model

A property graph has two kinds of entities, both carrying arbitrary key-value properties:

```
Node: {id: 1, labels: ["Person"], properties: {name: "Alice", age: 30}}
Edge: {id: 1, from: 1, to: 2, label: "KNOWS", properties: {since: 2020}}
```

This is different from an RDF triple store (no literals as nodes, schema-free) and different from a relational database (no foreign-key joins — edges are first-class citizens).

The defining query is multi-hop traversal:

```
MATCH (alice:Person {name: "Alice"})-[:KNOWS*1..3]->(friend:Person)
RETURN friend.name
```

In SQL this is a recursive CTE or 3 self-joins. In our graph engine it is DFS with a depth limit. In Neo4j it follows physical relationship pointers — one pointer dereference per hop.

### Adjacency list: the backbone

Our graph is stored as three hash maps:

```
nodes:     HashMap<NodeId, Node>
edges:     HashMap<EdgeId, Edge>
adjacency: HashMap<NodeId, Vec<EdgeId>>
```

To find Alice's friends: look up `adjacency[alice_id]`, get a list of edge IDs, fetch each edge to find the `to` NodeId. Two HashMap lookups per hop. This is fast in practice (HashMap is O(1) amortized), but it is not the same as Neo4j's approach.

### What index-free adjacency actually means

Neo4j stores each node as a 15-byte record on disk:

```
Node record (15 bytes):
  in_use:        1 byte
  next_rel_id:   4 bytes  -- pointer to first relationship record
  next_prop_id:  4 bytes  -- pointer to first property record
  label_id:      2 bytes
  extra:         4 bytes
```

Each relationship record (34 bytes) stores pointers to the next relationship for BOTH the source and target node:

```
Relationship record (34 bytes):
  in_use:              1 byte
  first_node:          4 bytes
  second_node:         4 bytes
  relationship_type:   2 bytes
  first_node_prev_rel: 4 bytes  -- doubly-linked list for first_node
  first_node_next_rel: 4 bytes
  second_node_prev_rel:4 bytes  -- doubly-linked list for second_node
  second_node_next_rel:4 bytes
  next_prop_id:        4 bytes
  extra:               3 bytes
```

To find Alice's friends: read Alice's node record (1 disk read, 15 bytes), follow `next_rel_id` to her first relationship record (1 disk read, 34 bytes), follow `first_node_next_rel` for the next relationship. No index lookup required — you are following physical pointers on disk.

Our HashMap approach: `adjacency[alice_id]` is a hash lookup that may cause a cache miss. For a hot in-memory graph with 10,000 nodes, this is fast. For a cold on-disk graph with 1 billion nodes, following physical record pointers beats hash lookups decisively.

### The Cypher-lite query plan

Our mini Cypher parser produces a `QueryPlan` tree:

```
MATCH (n:Person {name: "Alice"})-[:KNOWS*1..3]->(m:Person) RETURN m

QueryPlan::PathMatch {
  from_label: Some("Person"),
  from_props: {"name": String("Alice")},
  rel_type: Some("KNOWS"),
  min_hops: 1, max_hops: 3,
  to_label: Some("Person"),
  where_pred: None,
  return_aliases: ["m"],
}
```

The executor converts this to: scan Person nodes, filter by `name = "Alice"`, DFS from each match up to 3 hops along KNOWS edges, filter targets by Person label, project the "m" alias.

The in-memory graph is three hash maps and two monotonic ID counters. The adjacency list maps each node to its outgoing edge IDs — not directly to neighbor NodeIds. This indirection lets edges carry properties (which you cannot store in a direct NodeId adjacency list).

```rust
pub struct Graph {
    pub nodes:     HashMap<NodeId, Node>,
    pub edges:     HashMap<EdgeId, Edge>,
    pub adjacency: HashMap<NodeId, Vec<EdgeId>>,
    next_node_id:  NodeId,
    next_edge_id:  EdgeId,
}
```

**Why `Vec<EdgeId>` instead of `Vec<NodeId>`?** If you store `Vec<NodeId>` directly, you lose the edge properties. When you ask "what is the weight of the KNOWS edge between Alice and Bob?", you need to look up the edge record. Storing `Vec<EdgeId>` in the adjacency list gives you O(1) access to both the neighbor and the edge properties.

**The `neighbors()` method** traverses the adjacency list:

```rust
pub fn neighbors(&self, node_id: NodeId) -> Vec<NodeId> {
    self.adjacency
        .get(&node_id)
        .map(|edge_ids| {
            edge_ids.iter()
                .filter_map(|eid| self.edges.get(eid).map(|e| e.to))
                .collect()
        })
        .unwrap_or_default()
}
```

Two HashMap lookups per hop: `adjacency[node_id]` to get edge IDs, then `edges[eid]` to get the target NodeId. Compare to Neo4j's one pointer dereference per hop.

**Multi-label nodes**: a node can have multiple labels. `add_node` accepts `Vec<String>` for labels. `find_nodes("Person")` returns nodes where `labels.contains("Person")` — a linear scan across all nodes. The label index (v2) makes this O(1).

**`Value` as a hashable enum**: `Value` needs to implement `Hash` and `Eq` so it can be used as a key in the property index (v2). `Float` hashes via `f64::to_bits()` — not IEEE-correct (NaN != NaN by default), but acceptable for a toy.

</LabStage>

The parser handles the 4 query patterns in a single pass. The key design decision is to represent the full path pattern as a single `QueryPlan::PathMatch` variant rather than composing `ScanNodes + Filter + Traverse`. This avoids the complexity of building a join operator.

**The tokenizer** handles the awkward graph syntax. The tricky tokens are `-[` (start of relationship pattern) and `]->` (end of relationship pattern). A naive character-by-character tokenizer will produce `-`, `[` and `]`, `-`, `>` separately. The tokenizer must look ahead to compose these:

```rust
'-' => {
    chars.next();
    if chars.peek() == Some(&'[') {
        chars.next();
        tokens.push("-[".to_string());
    } else {
        tokens.push("-".to_string());
    }
}
']' => {
    chars.next();
    if chars.peek() == Some(&'-') {
        chars.next();
        if chars.peek() == Some(&'>') {
            chars.next();
            tokens.push("]->".to_string());
        }
    }
}
```

The `..` range separator in `*1..3` is also ambiguous: it could be two `.` tokens (from property access like `n.name`). The tokenizer peeks ahead to distinguish `.` (property access) from `..` (range).

**The QueryPlan tree** is the intermediate representation between parsing and execution:

```rust
pub enum QueryPlan {
    ScanNodes(String),                          // scan all nodes with label
    Filter(Box<QueryPlan>, Predicate),          // filter sub-plan by predicate
    Traverse { source, rel_type, min_hops, max_hops },
    PathMatch { from_label, from_props, rel_type, min_hops, max_hops,
                to_label, where_pred, return_aliases, from_alias, to_alias },
    Project(Box<QueryPlan>, Vec<String>),       // keep only listed aliases
}
```

**DFS with a depth limit** for variable-length paths:

```rust
pub fn dfs_traverse(graph, start, rel_type, min_hops, max_hops) -> Vec<NodeId> {
    let mut results = Vec::new();
    let mut stack: Vec<(NodeId, usize)> = vec![(start, 0)];

    while let Some((current, depth)) = stack.pop() {
        if depth > max_hops { continue; }
        if depth >= min_hops && current != start {
            results.push(current);
        }
        if depth < max_hops {
            for edge in graph.outgoing_edges(current) {
                if rel_type.map_or(true, |t| edge.label == t) {
                    stack.push((edge.to, depth + 1));
                }
            }
        }
    }
    results
}
```

Note that the stack can revisit nodes at different depths. For a cyclic graph this causes infinite recursion — the production solution is a `visited` set per path (not per node, since the same node can appear on multiple paths). Our simple implementation uses a `(NodeId, depth)` visited pair to break cycles.

**`ResultRow`** is `HashMap<String, Node>`: maps alias names ("n", "m") to cloned node values. Cloning is expensive for large result sets — production query engines use row references, not copies. For a toy this is acceptable.

**The label index** is a `HashMap<String, HashSet<NodeId>>` maintained alongside the graph. Every `add_node` call updates the index in O(labels.len()) time:

```rust
pub fn insert(&mut self, node: &Node) {
    for label in &node.labels {
        self.inner.entry(label.clone()).or_default().insert(node.id);
    }
}
```

Lookup is O(1): `index.get("Person")` returns a `HashSet<NodeId>`. Before the index, `find_nodes("Person")` scanned all `n` nodes. After the index, it is a single HashMap lookup.

**The property index** handles `(label, property_name) -> value -> NodeIds`. This is how `MATCH (n:Person {name: "Alice"})` becomes O(1) instead of O(n):

```rust
pub fn lookup(&self, label: &str, prop: &str, value: &Value) -> HashSet<NodeId> {
    let key = (label.to_string(), prop.to_string());
    self.inner.get(&key).and_then(|m| m.get(value)).cloned().unwrap_or_default()
}
```

The property index is opt-in (you must call `create_index(label, prop)` explicitly). This mirrors Neo4j's `CREATE INDEX ON :Person(name)` command. Without the explicit declaration, the executor falls back to a linear scan with filter.

**Dijkstra with a binary heap** uses `std::collections::BinaryHeap`. Rust's `BinaryHeap` is a max-heap, so we reverse the ordering (min cost = max priority):

```rust
impl Ord for DijkstraState {
    fn cmp(&self, other: &Self) -> Ordering {
        other.cost.partial_cmp(&self.cost).unwrap_or(Ordering::Equal)
            .then_with(|| self.node.cmp(&other.node))
    }
}
```

Edge weights come from a named property: `shortest_path(from, to, Some("cost"))` reads `edge.properties["cost"]` as an `f64`. Missing weight property defaults to 1.0, making unweighted BFS and weighted Dijkstra share the same code path.

**PageRank power iteration** runs for a fixed number of iterations. The standard formula:

```
rank(v) = (1 - d) / N + d * sum(rank(u) / out_degree(u)) for all u -> v
```

Where `d` is the damping factor (0.85) and `N` is the total number of nodes. Dangling nodes (out-degree 0) distribute their rank equally to all nodes — otherwise rank leaks out of the graph and scores no longer sum to 1.0.

The sum-to-1 property (within floating-point tolerance) is a correctness invariant: if PageRank values do not sum to approximately 1.0 after convergence, there is a bug in the dangling-node handling.

## Stress tests

To verify correctness under load, the test suite includes:

**Label index consistency**: insert 1,000 Person nodes, verify the label index contains exactly 1,000 entries. Then verify the linear scan and the index agree on every NodeId.

**Cycle detection in DFS**: build a cyclic graph (A->B->C->A), run a variable-length path query, verify termination and no duplicate results in the output.

**Dijkstra optimality**: build a graph with two paths of different total weights, verify Dijkstra returns the lower-cost path (not just the fewer-hop path).

**PageRank sum invariant**: run PageRank on graphs of sizes 10, 100, 1000 with random topologies, verify the sum of all scores is always within 0.001 of 1.0.

**Property filter with index**: run the same query (`MATCH (n:Person {name: "Alice"}) RETURN n`) against graphs with and without a property index, verify both paths return identical results.

### Part A — GraphClient: raw Cypher via the Neo4j Java Driver

`GraphClient.java` runs the same four query patterns our Rust Cypher-lite engine supports, but against a real Neo4j instance using the official Java driver. The comparison is intentional: same query, two engines.

```java
@Component
public class GraphClient {
    private final Driver driver;

    // MATCH (n:Person)-[:KNOWS]->(m:Person) WHERE n.name = $name RETURN m
    // Rust equiv: executor.execute(PathMatch{rel_type: KNOWS}, graph)
    public List<Map<String, Object>> findFriends(String name) {
        try (Session session = driver.session()) {
            Result result = session.run(
                "MATCH (n:Person)-[:KNOWS]->(m:Person) WHERE n.name = $name RETURN m.name AS name",
                Values.parameters("name", name)
            );
            return result.list(Record::asMap);
        }
    }

    // MATCH (n:Person)-[:KNOWS*1..3]->(m:Person) WHERE n.name = $name RETURN m
    // Rust equiv: dfs_traverse(graph, alice_id, "KNOWS", 1, 3)
    public List<Map<String, Object>> findFriendsUpTo3Hops(String name) {
        try (Session session = driver.session()) {
            Result result = session.run(
                "MATCH (n:Person)-[:KNOWS*1..3]->(m:Person) WHERE n.name = $name RETURN DISTINCT m.name AS name",
                Values.parameters("name", name)
            );
            return result.list(Record::asMap);
        }
    }
}
```

**The architectural difference in one sentence**: when Neo4j executes `MATCH (n)-[:KNOWS]->(m)`, it follows a physical pointer chain stored in each relationship record. When our Rust executor runs the same query, it does `adjacency[node_id]` (HashMap lookup) + `edges[edge_id]` (another HashMap lookup) per hop. On a cold in-memory graph the difference is negligible. On a 50 billion-edge graph stored on NVMe, Neo4j's pointer chain beats hash map lookups by an order of magnitude.

### Part B — Spring Data Neo4j: the ORM path

Spring Data Neo4j (SDN) provides a higher-level API using `@Node` and `@Relationship` annotations — the Spring Data pattern familiar from Spring Data JPA:

```java
// Spring Data Neo4j entity — comparable to a JPA @Entity
@Node("Person")
public class PersonNode {
    @Id @GeneratedValue
    private Long id;

    private String name;
    private int age;

    @Relationship(type = "KNOWS", direction = OUTGOING)
    private List<PersonNode> friends;
}

// Repository — same pattern as JpaRepository
public interface PersonRepository extends Neo4jRepository<PersonNode, Long> {
    List<PersonNode> findByName(String name);

    @Query("MATCH (n:Person)-[:KNOWS*1..3]->(m:Person) WHERE n.name = $name RETURN m")
    List<PersonNode> findFriendsWithin3Hops(String name);
}
```

SDN translates `findByName("Alice")` into `MATCH (n:Person) WHERE n.name = "Alice" RETURN n` — the same query our Cypher-lite parser handles. The custom `@Query` for variable-length paths shows the limit of derived query names: some graph patterns cannot be expressed without explicit Cypher.
