## Why this lab exists

Every LLM API you call — OpenAI, Anthropic, Groq — likely runs speculative decoding internally. When you get tokens back at 80 tokens/sec instead of 25 tokens/sec, speculative decoding is probably why. It is the single most impactful serving optimization for autoregressive LLMs discovered in the past decade.

The trick sounds almost too good: use a small, fast model to *propose* K tokens speculatively, then use the large, slow model to *verify* all K tokens in one forward pass. The output distribution is provably identical to running the large model alone — you get the same quality at 2-4x the throughput. No model changes. No quality degradation. Just arithmetic.

Until you've implemented the rejection sampling criterion yourself, until you've derived the geometric series that gives you the expected speedup, until you've watched the acceptance rate determine everything — you don't truly understand *why* it works. This lab makes it concrete in three stages: from a 180-line draft-verify loop to tree attention that pushes speedup further.

## What we're building

A speculative decoding engine in Python, implemented across three stages. Each stage adds one concept and makes the speedup more measurable.

**We build:**
- Draft-then-verify core algorithm (v0): `speculative_step`, `AcceptanceRate`
- Batched decoding with `SpeedupStats` (v1): `BatchedSpeculativeDecoder`, geometric-series speedup formula
- Tree attention and self-speculative decoding (v2): `DraftTree`, `SelfSpeculator`, `AcceptanceMetrics`
- FastAPI server: `POST /generate`, `GET /stats`
- Spring AI Java integration: `SpeculativeClient`, transparency demonstration

**We explicitly skip (and why):**
- Real GPU model weights: we use mock models with configurable distributions so the lab runs on any laptop in under 5 minutes
- Custom CUDA kernels: tree attention masking requires GPU-specific implementation; the algorithm is shown on CPU
- Training the draft model: we simulate a high-acceptance draft; training a real draft is covered in the LLM inference and LoRA labs

## Mental model

The core intuition: the large target model's forward pass is expensive whether it processes 1 token or K+1 tokens. The attention computation is O(n^2) in sequence length, but for short speculative drafts the batch dimension (K+1 additional tokens) costs almost nothing on GPU. So you get K+1 output distributions for roughly the price of 1.

```
Standard decoding (1 target call = 1 token):

  Prompt → [Target FWD] → token_1
           [Target FWD] → token_2
           [Target FWD] → token_3
  3 target calls → 3 tokens → speedup = 1.0

Speculative decoding (1 target call = up to K+1 tokens):

  Prompt → [Draft FWD x5] → [t1, t2, t3, t4, t5] (fast, cheap)
         → [Target FWD x1] → verify positions 0..4 simultaneously
         → Accept t1, t2, t3 (rejected at t4) → resample t4'
         → bonus token at position 4

  1 target call → 4 tokens → speedup = 4.0 for this step
```

The acceptance criterion is rejection sampling: accept draft token `t` at position `i` if `uniform(0,1) < min(1, p_target(t) / p_draft(t))`. When `p_target` and `p_draft` agree (both assign high probability to `t`), the token is accepted with probability near 1. When they disagree, the token is rejected and we resample from the corrected distribution `max(0, p_target - p_draft) / Z`.

This corrected distribution is the key: sampling from it produces the correct marginal distribution. The math guarantees that the output of speculative decoding has exactly the same distribution as the output of the target model alone — provably.

The entire algorithm is one function: `speculative_step`. Understanding it means understanding all of speculative decoding — everything else is optimization.

**Phase 1 — Draft:**

```python
draft_tokens = []
draft_probs = []
current_ctx = list(context_ids)

for _ in range(K):
    probs = draft_model.get_probs(current_ctx)          # fast
    token = torch.multinomial(probs, num_samples=1)
    draft_tokens.append(token)
    draft_probs.append(probs[token].item())
    current_ctx.append(token)
```

This is just standard autoregressive sampling from the small draft model, K times. Each call is cheap — the draft model might be 10-100x smaller than the target.

**Phase 2 — Verify (the key insight):**

In a real transformer, one forward pass over `context + K draft tokens` produces K+1 output distributions simultaneously (one for each position). On GPU with sequence lengths under 100, adding K+1 tokens to a batch costs nearly nothing extra. Our mock simulates this with K+1 sequential calls — same semantics, same correctness, just slower.

```python
verify_ctx = list(context_ids)
target_probs_list = []
for i in range(K):
    tp = target_model.get_probs(verify_ctx)    # would be batched in real system
    target_probs_list.append(tp)
    verify_ctx.append(draft_tokens[i])
target_bonus_probs = target_model.get_probs(verify_ctx)
```

**Phase 3 — Accept/reject:**

```python
for i in range(K):
    token = draft_tokens[i]
    ratio = p_target[i][token] / p_draft[i]
    if uniform() < min(1.0, ratio):
        accepted.append(token)
    else:
        # Resample from the corrected distribution
        corrected = clamp(p_target[i] - p_draft_full[i], min=0) / Z
        resampled = multinomial(corrected)
        accepted.append(resampled)
        break   # discard remaining draft tokens
```

The corrected distribution `max(0, p_target - p_draft) / Z` is what preserves the output quality. Sampling from it on rejection ensures that the marginal distribution over the accepted token is exactly `p_target`, even though the draft proposed something else.

**Phase 4 — Bonus:**

Always add one token sampled from `target_bonus_probs` at the position after all accepted tokens. This guarantees progress of at least 1 token per target call, even if all K drafts are rejected.

**The acceptance rate invariant:**

The key test: when `draft_model` and `target_model` return identical distributions (e.g., both uniform), `min(1, p_target/p_draft) = min(1, 1) = 1` — every draft token is accepted with probability 1. This means `n_accepted == K` always, and `total_tokens == K + 1` (K accepted + 1 bonus). Your test verifies this over 50 steps: any deviation means the sampling or probability computation has a bug.

v1 extends the core algorithm to handle a batch of B prompts simultaneously and adds the statistics that make speedup concrete and measurable.

**Why batching matters:**

In GPU inference, the fixed overhead of a target model forward pass (loading weights from HBM, dispatch overhead) is the same whether you run 1 sequence or B sequences in parallel. If each sequence contributes K+1 tokens per forward pass, B sequences contribute B*(K+1) tokens per forward pass. Standard decoding of B sequences: B target calls per step. Batched speculative: 1 target call for all B sequences — ideally.

Our implementation processes each sequence in `decode_batch` sequentially per step (Python has no GPU parallelism here), but we count statistics correctly — one "target call" per batch step, not per sequence.

**The speedup formula:**

The expected number of tokens accepted from K draft proposals at acceptance rate `alpha`:

```
E[accepted] = sum(alpha^i for i in range(K))
            = (1 - alpha^K) / (1 - alpha)   [geometric series]
```

At `alpha=0.82`, `K=5`:
```
1 + 0.82 + 0.6724 + 0.5514 + 0.4521 = 3.496
```

Plus 1 bonus token = 4.496 tokens per target call on average. Standard decoding gives 1 token per target call. Speedup = 4.496... but in practice the integer floor of accepted tokens and the resampling overhead bring it to approximately 3.2x in simulations.

`SpeedupStats.speedup_vs_standard = tokens_generated / target_calls`

This is the real-world measurement, not the theoretical formula. Run `decode_batch` over many prompts and compare `speedup_vs_standard` against `expected_speedup(acceptance_rate, K)` — they should be close.

**`target_calls` counts steps, not tokens:**

This is a common confusion. If you generate 30 tokens with speedup=3.0, `target_calls` should be approximately 10, not 30. Each `speculative_step` call is exactly 1 target call. The `tokens_generated / target_calls` ratio is what shows the speedup.

**Breakeven analysis:**

Speculative decoding only helps if the draft model is fast enough. If the draft model costs `d` fraction of the target's time and we run K drafts:

```
Net speedup = tokens_per_target_call / (1 + K * d)
```

At `d=0.10` (draft 10x faster) and `K=5` with `alpha=0.82`:
```
Net speedup = 3.49 / (1 + 5*0.10) = 3.49 / 1.5 = 2.33x
```

Still profitable. But at `d=0.30` (draft only 3x faster):
```
Net speedup = 3.49 / (1 + 5*0.30) = 3.49 / 2.5 = 1.40x
```

Barely worth it. This is why draft model selection matters enormously in production.

v2 pushes speculative decoding in two orthogonal directions: better draft coverage (tree attention) and no separate draft model (self-speculative).

**Tree attention intuition:**

Linear drafting proposes one path: `[t1, t2, t3, t4, t5]`. If `t2` is rejected, you discard `t3, t4, t5`. But what if you had proposed 3 alternatives at position 2? At least one of them might be accepted, recovering the tokens that would be wasted.

Tree drafting explores a branching tree of candidates:

```
Depth 0:  t0a, t0b             (W=2 candidates)
Depth 1:  t1aa, t1ab, t1ba, t1bb  (4 nodes)
Depth 2:  8 leaf paths

Target verifies all 8 paths in one forward pass using tree attention masking:
  - t0a only attends to context
  - t1aa attends to context + t0a
  - t1ab attends to context + t0a   (same parent as t1aa)
  - t1ba attends to context + t0b   (different parent)
  ...
```

The tree attention mask ensures each node only attends to its own ancestors, not to sibling branches. This is where the custom CUDA kernel requirement comes in — standard causal attention assumes a linear sequence, not a tree.

Our `verify_tree` checks all paths sequentially but picks the longest accepted path — the same result a GPU tree attention would produce.

**Self-speculative decoding:**

The simplest possible draft: stop the forward pass at an intermediate layer and use that layer's output as the draft distribution. The full forward pass is the target.

```python
# One forward pass, two exit points:
x = embed(tokens) + pos_embed(positions)
x = layer_0(x)
early_logits = early_unembed(x[:, -1, :])   # draft: after 1 layer

x = layer_1(x)
full_logits = full_unembed(x[:, -1, :])     # target: after 2 layers
```

The accept/reject criterion is identical. The efficiency gain: you don't need to maintain a separate draft model, download extra weights, or warm up a separate GPU. The draft comes "for free" from the early layers of the model you're already running.

In practice, self-speculative requires the model to be trained with an auxiliary early-exit head. Medusa (2024) extends this further with multiple decoding heads at different positions, each proposing a token at a different offset from the current position.

**AcceptanceMetrics gives you percentile information:**

`mean_accepted` is the average. But `p95_accepted` tells you what happens on easy inputs (in-domain, high-certainty tokens), while `p50_accepted` tells you the median. If `p95_accepted = 4.8` and `mean_accepted = 2.1`, you have high variance — the acceptance rate fluctuates significantly across different input types. This is the signal to implement adaptive K.

`target_calls_per_1k_tokens = 1000 / (mean_accepted + 1)` is the headline infrastructure metric: how many target model calls do you need per 1000 output tokens? At `mean_accepted=3.2`, you need `1000 / 4.2 = 238` calls instead of 1000 — a 4.2x reduction in inference infrastructure cost.

## Stress tests

**Acceptance rate vs K**: run 1000 speculative steps with K=1, 3, 5, 8, 10 and measure `speedup_vs_standard` at each K. You should see speedup increase with K (more draft tokens amortize the target call), then plateau as `alpha^K` becomes negligible. At `alpha=0.82`, the optimal K is around 7-9 for draft models 10x faster than target.

**Acceptance rate distribution**: log `n_accepted` for each step over 1000 steps. Plot the distribution — it should be roughly geometric with parameter `alpha`. Most steps accept all K tokens (probability `alpha^K = 0.82^5 = 0.37` at alpha=0.82). Some steps accept 0 draft tokens (probability `1-alpha = 0.18` for the first rejection). The distribution tells you the tail behavior.

**Tree vs linear comparison**: run 500 steps with linear draft (width=1, depth=5) and tree draft (width=2, depth=3) at the same target model. Compare `mean_accepted_per_step`. The tree should outperform linear at the same total draft token budget because it explores more paths. The difference narrows as acceptance rate increases.

**Self-speculative layer depth**: vary the early exit layer in `SelfSpeculator` from layer 0 to the penultimate layer. Measure acceptance rate at each exit point. You should see acceptance rate increase as the exit layer gets closer to the final layer (early exit is worse draft; final layer early exit approaches target quality). Plot this curve — it shows how many layers of the model are "wasted" on refinement that doesn't change the top token.

**Batch size scaling**: run `decode_batch` with batch_size=1, 2, 4, 8 and fixed total tokens. In our CPU simulation, time scales linearly with batch_size (no GPU parallelism). But `target_calls` stays the same because we count one step as one target call regardless of batch size. This demonstrates how GPU batching would eliminate the batch_size scaling: the same number of target calls processes more prompts.

### Part A — SpeculativeClient: direct HTTP access

`SpeculativeClient.java` wraps our FastAPI server. It demonstrates the key property of speculative decoding from a caller's perspective: **the API is identical to standard decoding**. You send a prompt, you receive tokens. The speedup is invisible to the caller.

```java
public class SpeculativeClient {
    public GenerateResponse generate(List<Integer> promptTokens) {
        var body = new GenerateRequest(promptTokens, props.maxTokens(), props.k());
        return http.post()
            .uri("/generate")
            .body(body)
            .retrieve()
            .body(GenerateResponse.class);
    }

    public StatsResponse stats() {
        return http.get().uri("/stats").retrieve().body(StatsResponse.class);
    }
}
```

The response includes `speedupVsStandard` and `acceptanceRate` — metrics that are invisible to the caller during generation but available for monitoring. This is the correct production pattern: expose metrics via `/stats`, not in the generation response itself.

### Part B — Speedup math in Java

The geometric series formula translates directly to Java:

```java
double alpha = 0.82;  // acceptance rate
int K = 5;            // speculation width
double geoSum = 0.0;
for (int i = 0; i < K; i++) {
    geoSum += Math.pow(alpha, i);
}
// geoSum = 1 + 0.82 + 0.6724 + 0.5514 + 0.4521 = 3.496
// Plus 1 bonus token: 4.496 tokens per target call
// Speedup vs standard (1 token per call): ~3.2x after practical corrections
```

This is a useful back-of-envelope calculation for sizing inference infrastructure. If your target model costs 50ms per call and you achieve 3.2x speedup, effective latency per token drops from 50ms to 15ms — the difference between real-time and batch-feeling generation.

- **System design question**: At what draft-to-target cost ratio does speculative decoding with K=5 break even? (Answer: if the draft costs more than 1/(K+1) = 16.7% of the target's time, the overhead exceeds the gain — you need an even smaller draft model or lower K.)
- **System design question**: How does speculative decoding interact with beam search? (Speculative decoding is designed for greedy/temperature sampling, not beam search. Beam search maintains B candidate sequences simultaneously — speculative decoding can be applied per-beam, but the combinatorics make tree attention exponentially more expensive.)
