## Why this lab exists

Every language runtime you use daily runs on a virtual machine. CPython compiles your Python to bytecode before interpreting it. The JVM runs Java, Kotlin, Scala, and Clojure on the same bytecode format. V8 compiles JavaScript to its own internal bytecode (Ignition) before JIT-compiling hot paths to native code. Even Rust's MIR (Mid-level Intermediate Representation) is a simplified instruction format that passes through several VM-like transformation stages before becoming machine code.

Yet "how does a VM work" is one of those questions that seems well-documented but actually isn't — most explanations stop at "it's a loop over instructions" without showing the call stack mechanics, how closures work, or why tail call optimization is fundamentally a different model of function invocation.

This lab builds a bytecode VM from scratch in three stages. By the end, you'll be able to read JVM bytecode output from `javap` and understand what the JVM is actually doing for every Java construct — loops, method calls, lambdas, and the conspicuous absence of tail call optimization.

## What we're building

A stack-based bytecode VM in Rust, built across three stages.

**We build:**
- A stack machine with 17 instructions (PUSH, POP, ADD, SUB, MUL, DIV, NEG, EQ, LT, JUMP, JUMP_IF_FALSE, LOAD, STORE, CALL, RETURN, HALT, and TAIL_CALL)
- A call stack with frames (`ip` + `stack_base`), enabling recursive function calls
- Closures with upvalues: functions that capture variables from their enclosing scope
- Tail call optimization: `TAIL_CALL` reuses the current frame instead of pushing a new one
- A JVM bytecode comparison showing how each concept maps to real JVM instructions

**We explicitly skip (and why):**
- A parser or compiler (we hand-write bytecode directly — the interpreter is the lesson)
- Register-based layout (covered in WhatTheToyMisses — this is a conscious stack-only design)
- Garbage collection (values are Rust-owned with `Rc`; see WhatTheToyMisses)
- JIT compilation (out of scope — an entire separate lab)

## Mental model

A stack VM is a calculator with memory. Every instruction either pushes a value onto a stack or pops values from it, computes something, and pushes the result.

Here is the execution trace for `2 + 3 * 4 = 14`:

```
Instruction    Stack (bottom → top)
───────────────────────────────────────────────────────
PUSH 2         [2]
PUSH 3         [2, 3]
PUSH 4         [2, 3, 4]
MUL            [2, 12]      ← pops 3, 4; pushes 3*4=12
ADD            [14]         ← pops 2, 12; pushes 2+12=14
HALT           result: 14
```

No registers. No named variables. Just a pile of values where every instruction knows exactly how many things to consume and how many to produce.

The call stack is layered on top of this. When a function is called, we push a **frame** onto a separate call stack. A frame records two things:

- `ip` — where to resume in the *caller's* code when we return
- `stack_base` — the index in the value stack where this frame's local variables start

`LOAD(idx)` pushes `stack[stack_base + idx]` and `STORE(idx)` writes to it. This is why local variable access doesn't require knowing absolute positions — just offsets from the frame's base.

**Comparison with JVM:** The JVM is a stack VM with the same mental model. `iload_0` pushes local variable 0 (the first argument or `this`). `iadd` pops two ints and pushes their sum. `ireturn` pops the top int and returns it to the caller. `invokevirtual` pushes a new frame. Our `LOAD/STORE/CALL/RETURN` map to these JVM instructions one-to-one.

## Prerequisites

None. A basic grasp of how function calls work (call stack, local variables) helps but isn't required — we build that model from scratch here.

The first thing to understand is the dispatch loop. Every VM is a loop with a `match` (or switch) on the current instruction:

```rust
loop {
    let instr = self.code[self.ip].clone();
    self.ip += 1;   // advance before execution (like a CPU program counter)

    match instr {
        Instruction::Push(v) => self.stack.push(v),
        Instruction::Add => {
            let b = self.stack.pop()?;  // right operand
            let a = self.stack.pop()?;  // left operand
            self.stack.push(Value::Int(a + b));
        }
        Instruction::Halt => return Ok(self.stack.last().cloned()?),
        // ...
    }
}
```

The `ip` advances *before* executing the instruction, just like a CPU's program counter. Jumps are relative: `JUMP(offset)` sets `ip = ip + offset - 1` (the `-1` corrects for the pre-advance).

**The frame invariant:** When `CALL` is executed, the current state (`code`, `ip`, `stack_base`) is pushed onto `call_stack`. The callee runs. When `RETURN` fires, the frame is popped, the value stack is truncated back to `frame.stack_base`, and the return value is pushed. The caller resumes exactly where it left off.

```rust
Instruction::Return => {
    let ret_val = self.pop()?;
    if let Some(frame) = self.call_stack.pop() {
        self.stack.truncate(frame.stack_base);
        self.code = frame.code;
        self.ip = frame.ip;
        self.stack.push(ret_val);
    }
}
```

**Verify your v0 with this trace.** The loop `sum = 0; for i in 1..=10: sum += i` as VM bytecode:

```
slot 0 = count (starts at 10)
slot 1 = acc   (starts at 0)

ip=0:  PUSH 10    → [10]
ip=1:  PUSH 0     → [10, 0]
ip=2:  LOAD 0     → [10, 0, 10]
ip=3:  PUSH 0     → [10, 0, 10, 0]
ip=4:  EQ         → [10, 0, false]   (10 == 0?)
ip=5:  JumpIfFalse(2) → false, so skip to body
...
ip=16: LOAD 1     → result: 55
ip=17: HALT
```

**JVM correlation:** Run `javap -c` on any compiled Java class and you'll see the same pattern. `iload_1` is `LOAD(1)`. `iadd` is `ADD`. `if_icmpge` is `LT` + `JumpIfFalse`. `goto` is `JUMP`. The only difference is that JVM bytecode is more granular (separate instructions for int, long, float, double) and uses a compact binary format instead of a Rust enum.

A closure is a function that has been paired with the environment it was created in. The "environment" is a list of **upvalues** — variable bindings captured from the enclosing scope.

Consider this pseudocode:

```
function make_adder(n):
    return function(x):  ← this inner function is a closure
        return x + n     ← 'n' is captured from the outer scope
```

When `make_adder(10)` is called, the inner function is created with `n=10` captured as an upvalue. When the returned closure is later called with `x=5`, it reads `n` from its upvalue slot and returns `15`.

In bytecode:

```
// Top-level program
PUSH 10               → stack: [10]   (this is n)
CLOSURE(code, 1)      → pop 1 upvalue (10), push <closure>
PUSH 5                → the argument x
CALL_CLOSURE(1)       → call the closure with 1 argument

// Inside the closure body:
GET_UPVALUE(0)        → push n (10)
LOAD(0)               → push x (5)
ADD                   → 15
RETURN
```

The `CLOSURE` instruction takes the top `N` values from the value stack and stores them as the closure's upvalue array. `GET_UPVALUE(idx)` pushes `upvalues[idx]`. `SET_UPVALUE(idx)` writes to `upvalues[idx]` — this is how a closure can maintain mutable state (like a counter).

**Frame extension for closures:** When `CALL_CLOSURE` fires, we save not just `code` and `ip` but also `upvalues` in the call frame. When `RETURN` unwinds the frame, the caller's upvalue context is restored.

```rust
Instruction::CallClosure(n_args) => {
    let closure = /* pop from stack */;
    // Save current context
    let frame = Frame {
        code: self.code.clone(),
        ip: self.ip,
        stack_base: self.stack_base,
        upvalues: self.upvalues.clone(), // ← save caller's upvalues
    };
    self.call_stack.push(frame);
    // Switch to callee
    self.code = closure.code;
    self.upvalues = closure.upvalues;   // ← install callee's upvalues
    self.ip = 0;
}
```

**JVM correlation:** Java lambdas use `invokedynamic` + `LambdaMetafactory`. On first call, the JVM generates a class implementing the functional interface. The lambda body becomes a synthetic method. Captured effectively-final locals become fields on the generated object. This is structurally identical to our `Closure{code, upvalues}`: the synthetic method is `code`, the captured fields are `upvalues`.

Run `javap -p -c` on a class with a lambda that captures a local and look for `invokedynamic` followed by `invokevirtual lambda$methodName$0`. The `$0` is the generated synthetic method.

**The "effectively final" rule:** Java enforces that lambda captures must be effectively final (not reassigned after capture). This is because Java's capture model copies the value — the lambda holds its own copy, and mutations to the original local would not be seen by the lambda. Our VM's `SET_UPVALUE` mutates only the closure's own copy. Real upvalue sharing by reference (Lua's UpValue cell model) requires heap allocation.

**What is a tail call?** A call is in tail position if its result is the function's final result — nothing happens to the return value after the call returns. In `return f(n-1, acc)`, the call to `f` is in tail position because `return` does nothing to the value.

**Why does tail position matter?** With a normal call, the current frame must stay alive until the callee returns (in case the return value needs to be used). In tail position, the current frame's only job is to pass the callee's return value straight to its own caller — it's redundant. TCO eliminates it.

**Before TCO (naive CALL):**

```
fib(5)   → fib(4)   → fib(3)   → fib(2)   → fib(1)
frame 1    frame 2    frame 3    frame 4    frame 5

Stack depth = 5. At n=100000, stack depth = 100000. StackOverflow.
```

**After TCO (TAIL_CALL):**

```
fib(5)   → (frame reused)   → (frame reused)   → ... → fib(0)
frame 1        frame 1              frame 1           frame 1

Stack depth = 1, always. n=100000 is fine.
```

**The TAIL_CALL implementation:**

```rust
Instruction::TailCall(n_args) => {
    let closure = /* pop from stack */;

    // Collect the new arguments
    let args_start = self.stack.len() - n_args;
    let new_args: Vec<ExtValue> = self.stack[args_start..].to_vec();

    // REUSE the current frame: truncate stack back to stack_base,
    // then push the new arguments in the same slots
    self.stack.truncate(self.stack_base);
    for arg in new_args {
        self.stack.push(arg);
    }

    // Switch to the new callee's code
    self.code = closure.code;
    self.upvalues = closure.upvalues;
    // Reset ip — "restart" the function with new args
    self.ip = 0;

    // CRUCIALLY: call_stack.len() does NOT increase.
    // This is the entire optimization.
}
```

The key line is `self.stack.truncate(self.stack_base)` followed by pushing the new arguments. The old arguments are gone, the new ones are in their place, and `ip = 0` means we start the function over. From the caller's perspective, nothing changed — it still sees one frame below it. From the callee's perspective, it starts fresh with new arguments.

**Measuring the difference:**

```
fib(30) naive:
  result: 832040
  calls:  2,692,537   ← exponential growth (each call branches into two)
  time:   ~22ms

fib(30) tail-recursive:
  result: 832040
  calls:  31           ← exactly n+1 steps
  time:   ~0.04ms
```

**Why the JVM deliberately skips TCO:** The JVM specification guarantees that stack traces are observable via `Thread.getStackTrace()`, the `java.lang.StackTraceElement` API, and virtually every profiler, APM, and debugger in the Java ecosystem. If the JVM reused frames for tail calls, a stack trace inside `fib(100000)` would show a single frame instead of 100000 — silently changing the observable behavior of correct programs.

This is not an oversight. It is a deliberate trade-off: correctness of observability over memory efficiency of deep recursion. The practical consequence is that idiomatic functional-style recursion (common in Scala, Clojure, and Kotlin) must be handled at the language/compiler level, not the JVM level.

**Scala's `@tailrec`:** Scala's compiler detects tail-recursive methods and rewrites them to a `while` loop before generating JVM bytecode. The generated bytecode uses `goto` (JVM's `GOTO` instruction) instead of `invokevirtual`. The transformation is:

```scala
// Scala source
@tailrec
def fib(n: Int, a: Long = 0, b: Long = 1): Long =
  if n == 0 then a else fib(n - 1, b, a + b)

// What the compiler actually emits (simplified):
// while (n != 0) { val tmp = b; b = a+b; a = tmp; n -= 1 }; return a
```

This is exactly what our TAIL_CALL does — reuse the frame, reset to the start, update the arguments. The compiler does it; the JVM never sees a recursive call.

## Stress tests

**Stack overflow without TCO:** Call `fib_naive(50000)` with the recursive Rust implementation. You'll hit a stack overflow at around 8,000 frames deep (depending on your OS thread stack size, typically 8MB on Linux, 8MB on macOS). The recursion tree tries to branch twice at each node — at depth 50000, you'd need roughly 2^50000 frames to finish if you could. You'll overflow well before that.

**TCO handles depth 100,000:** `fib_tco(100_000)` completes in microseconds with a single frame in the call stack. The integer result wraps (i64 overflow is expected for large Fibonacci numbers), but the computation completes without any stack allocation.

**Upvalue mutation:** A closure that increments `upvalue[0]` on each call: start with `0`, call 5 times, get `1, 2, 3, 4, 5`. Verifies that `SET_UPVALUE` modifies the closure's own copy of the captured variable across calls.

**String concatenation:** `ADD` on two strings concatenates. This exercises the same code path as integer ADD but with a different type branch — verifies the type dispatch in the instruction handler. A closure capturing a string prefix and concatenating it with each argument demonstrates string upvalues.

### Part A — Reading JVM bytecode

Every compiled Java class file contains bytecode in a compact binary format. `javap -c ClassName` disassembles it. Compile `BytecodeComparison.java` and run:

```bash
mvn compile
javap -c -p target/classes/dev/pushkar/vm/BytecodeComparison.class
```

#### Method 1: `factorial(int n)` — iterative

The JVM bytecode for the iterative factorial loop:

```
public int factorial(int);
  Code:
     0: iconst_1          // push int constant 1 → acc = 1
     1: istore_2          // local[2] = pop()     (store acc)
     2: iload_1           // push local[1]        (n)
     3: ifle          21  // if n <= 0, jump to ireturn
     6: iload_2           // push acc
     7: iload_1           // push n
     8: imul              // pop 2 ints, push acc*n
     9: istore_2          // acc = acc*n
    10: iinc    1, -1     // local[1] -= 1        (n--)
                          // iinc is a JVM optimization: modifies a local
                          // variable in-place without a stack round-trip.
                          // Our VM has no equivalent — we use LOAD/SUB/STORE.
    13: iload_1           // push n
    14: ifgt           6  // if n > 0, jump to loop body
    17: iload_2           // push acc (result)
    18: ireturn           // return int at top of stack to caller
```

Compare this directly to our v0 VM loop. `iload_1` = `LOAD(1)`. `imul` = `MUL`. `istore_2` = `STORE(2)`. `ifgt` = `LT` + `JumpIfFalse`. The only alien instruction is `iinc` — a JVM-specific shortcut for incrementing or decrementing a local variable without stack operations.

#### Method 2: `factorialRecursive(int n)` — recursive

```
public int factorialRecursive(int);
  Code:
     0: iload_1           // push n
     1: ifgt           6  // if n > 0, go to recursive case
     4: iconst_1          // base case: push 1
     5: ireturn
     6: aload_0           // push 'this' (the receiver — JVM instance methods
                          // receive 'this' as implicit local[0])
     7: iload_1           // push n
     8: iconst_1          // push 1
     9: isub              // n - 1
    10: invokevirtual #7  // Call factorialRecursive(n-1)
                          // ← THIS IS CALL in our VM.
                          // Creates a new JVM stack frame.
                          // At depth 100000, you have 100000 frames.
    13: iload_1           // push n
    14: imul              // result * n
    15: ireturn
```

`invokevirtual` is the JVM's `CALL_CLOSURE`. It looks up the method in the constant pool (index `#7`), creates a new stack frame, copies the arguments into the new frame's local variable slots, and starts executing at the method's entry point.

#### Method 3: `addWithLambda(int x, int y)` — closures via invokedynamic

```
public int addWithLambda(int, int);
  Code:
     0: iload_1           // push x
     1: iload_2           // push y
     2: invokedynamic #13, 0   // Bootstrap: LambdaMetafactory
                               // Generates a class implementing IntBinaryOperator.
                               // The lambda body goes into a synthetic method:
                               //   private static int lambda$addWithLambda$0(int a, int b)
                               //   { return a + b; }
                               // The "captured" values (x, y) here are passed as
                               // arguments to the bootstrap, not captured as fields
                               // (because the lambda body uses its own params, not
                               // outer-scope variables).
     7: astore_3          // store the IntBinaryOperator object
     8: aload_3           // push it
     9: iload_1
    10: iload_2
    11: invokeinterface #17, 3  // IntBinaryOperator.applyAsInt(x, y)
    16: ireturn
```

`invokedynamic` is Java 7+'s mechanism for deferred method dispatch. On first call, the bootstrap method (`LambdaMetafactory.metafactory`) generates a class at runtime that:
1. Implements `IntBinaryOperator`
2. Has `applyAsInt` delegate to the synthetic `lambda$addWithLambda$0` method
3. Captures any effectively-final outer-scope locals as constructor parameters (fields)

This is structurally our `Closure{code, upvalues}`. The synthetic method is `code`. The captured constructor parameters are `upvalues`.

#### Method 4: `computeFib(int n, long a, long b)` — tail-recursive, no JVM TCO

```
public long computeFib(int, long, long);
  Code:
     0: iload_1           // push n
     1: ifne           6  // if n != 0, jump to recursive case
     4: lload_2           // push 'a' (long — uses lload not iload)
     5: lreturn           // return a
     6: aload_0           // push 'this'
     7: iload_1           // push n
     8: iconst_1
     9: isub              // n - 1
    10: lload          4  // push 'b' (long arg at local[4])
    12: lload_2           // push 'a'
    13: lload          4  // push 'b' again
    15: ladd              // a + b (new 'b' argument)
    16: invokevirtual #7  // ← new frame every time, even in tail position
                          // The JVM sees a call in tail position and creates
                          // a new frame anyway. Stack grows by 1 per call.
                          // At n=10000, stack depth = 10000.
    19: lreturn
```

Even though the `invokevirtual` at line 16 is immediately followed by `lreturn` (the return value of `computeFib(n-1, b, a+b)` is the return value of this function, unchanged), the JVM still creates a new frame. It must: the JVM spec guarantees that `Thread.getStackTrace()` returns accurate frame information. If TCO silently reused this frame, the stack trace would show 1 frame instead of N, breaking debuggers and profilers.

### Part B — The Scala `@tailrec` solution

Since the JVM won't do TCO, Scala does it at the compiler level. `@tailrec` transforms the source code before generating bytecode:

```scala
// Scala source (tail-recursive)
@scala.annotation.tailrec
def fib(n: Int, a: Long = 0L, b: Long = 1L): Long =
  if n == 0 then a else fib(n - 1, b, a + b)
```

The Scala compiler detects the tail call and generates:

```
// Simplified: what the compiler emits
0: iload_1         // n
1: ifeq   14       // if n == 0, return a
4: iinc  1, -1     // n -= 1
7: ...             // compute new a, b
10: goto  0        // ← GOTO instead of invokevirtual
                   // No new frame. Same frame, new args, ip reset.
```

`goto` is our `TAIL_CALL`. The compiler performed the optimization; the JVM bytecode has no recursive call at all. This is why `@tailrec`-annotated methods can handle arbitrarily deep "recursion" — at the bytecode level, they're while loops.

## What surprised me

<WhatSurprisedMe insight="The most surprising thing about implementing a VM is how much of the interpreter's work is bookkeeping, not computation. Out of 100 instructions executed in a typical program, maybe 20 are actual math operations (ADD, MUL, SUB). The rest are PUSH, POP, LOAD, STORE — moving values around to set up for the real operations and clean up after them. In a profiler, the hottest paths are the frame-push/pop code and the stack bounds checks, not the arithmetic. This is why register VMs are faster: they eliminate redundant moves by naming intermediate values directly (like CPU registers), so the ratio of math to bookkeeping flips from 20:80 to something closer to 60:40." />
