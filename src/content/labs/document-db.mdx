## Why this lab exists

Every backend engineer reaches the moment where a relational schema becomes the enemy. You need to add a new field to a table with 200 million rows. The ALTER TABLE locks the table for 40 minutes. The deployment window is Saturday at 3 AM. You wonder, briefly, if there is a better way.

There is — and document databases are one answer to it. MongoDB, CouchDB, and Firestore all let you insert documents with arbitrary shapes and evolve the schema by simply writing new fields on new documents. The old documents stay as they are. No migration. No downtime.

But "no schema" sounds like magic. It isn't. Someone still has to decide how to store `{name: "Alice", age: 30}` on disk. Someone still has to answer "give me all users where role equals admin" in under a millisecond. This lab tears open the black box: how does schemaless storage work, what does BSON actually look like on disk, and what does a secondary index really cost?

You'll build:
- A schemaless in-memory document store that can hold millions of documents with different shapes
- A BSON-inspired binary encoder that writes type information alongside values
- A secondary index system that makes field lookups 11x faster at the cost of ~15% write overhead

After this lab, you will understand the exact tradeoff MongoDB makes when it tells you "indexes improve read performance but slow down writes." You will have felt it in benchmarks.

## What we're building

A document database library in Rust across three stages. Each stage adds one layer and benchmarks it.

**We build:**
- Schemaless document store: insert any JSON, retrieve by ID, query by field equality
- BSON-like binary format: type-tagged fields, recoverable after restart
- Secondary indexes: field value maps to a list of document IDs
- An HTTP server that exposes all operations over REST
- A Java/Spring client that mirrors the Spring Data MongoTemplate API

**We explicitly skip (and why):**
- A query language with operators like `$gt`, `$in`, `$regex` — that's a whole compiler
- MVCC and concurrent writer isolation — that requires snapshot management
- Sharding — single-node is enough to learn the core lessons
- Change streams / oplog — that's a separate persistence concern

## Mental model

### Why schemaless?

SQL tables have a fixed schema. Every row has exactly the same columns. Adding a column requires ALTER TABLE — a metadata change that rewrites or locks the table. For a 10-row table, irrelevant. For a table with 200 million rows, ALTER TABLE can hold an exclusive lock for minutes. Even with online DDL tools, the operation generates significant I/O and replication lag.

Document databases sidestep this entirely. A document is a self-describing JSON object. Every document carries its own field names. Two documents in the same collection can have completely different shapes:

```
Collection: "users"

Doc 1: {"name": "Alice", "email": "alice@example.com", "age": 30}
Doc 2: {"name": "Bob", "email": "bob@example.com", "age": 25, "premium": true}
Doc 3: {"name": "Carol", "email": "carol@example.com"}  ← no "age" field
```

Adding a new field — say, `lastLogin` — to future documents requires zero schema migration. New documents get it; old ones don't. The application handles the absence gracefully. This is the schema evolution story that drove MongoDB's adoption.

### Collections as namespaces

A collection is a namespace — nothing more. It is not a table with a schema. It is not even a file with a fixed format. It is a name that groups related documents so you can query them together. Under the hood, `insert("users", doc)` is equivalent to `collections["users"][new_uuid()] = doc`. Collections are created implicitly when the first document is inserted.

### The read path

Every query starts with a question: do we have an index for this filter? If yes, look up the document IDs from the index and fetch each one by ID (O(log N) for the index lookup, then O(1) per ID). If no, scan every document in the collection and check the filter (O(n)). Secondary indexes exist solely to convert O(n) into O(log N). That is their entire purpose.

```
find({email: "alice@example.com"})
         │
         ▼ index exists on "email"?
    YES ─── BTreeMap lookup ──► [doc-id-42]
         │                           │
         │                           ▼
    NO ──── full collection scan ──► filter each document
```

The v0 store is two nested HashMaps and a UUID generator. That is the entire implementation.

```rust
pub struct DocumentStore {
    collections: HashMap<String, HashMap<DocId, Document>>,
}

impl DocumentStore {
    pub fn insert(&mut self, collection: &str, mut doc: Document) -> DocId {
        let id = new_doc_id();  // uuid::Uuid::new_v4().to_string()
        if let Some(obj) = doc.as_object_mut() {
            obj.insert("_id".to_string(), Value::String(id.clone()));
        }
        self.collections
            .entry(collection.to_string())
            .or_insert_with(HashMap::new)
            .insert(id.clone(), doc);
        id
    }

    pub fn find(&self, collection: &str, filter: &Filter) -> Vec<Document> {
        match self.collections.get(collection) {
            None => vec![],
            Some(col) => col.values()
                .filter(|doc| matches_filter(doc, filter))
                .cloned()
                .collect(),
        }
    }
}
```

The filter check is dead simple:

```rust
pub fn matches_filter(doc: &Document, filter: &Filter) -> bool {
    if filter.is_empty() { return true; }
    let obj = match doc.as_object() { Some(o) => o, None => return false };
    for (key, expected) in filter {
        match obj.get(key) {
            Some(actual) if actual == expected => {}
            _ => return false,
        }
    }
    true
}
```

An empty filter returns all documents. This is the MongoDB `find({})` behaviour.

**The key insight**: document databases store `serde_json::Value`, not typed rows. The database never needs to know that `age` is an integer or `name` is a string. The application knows; the database just stores and retrieves. This is why there is no migration — the database has no schema to migrate.

**What `_id` embedding gives you**: the document returned by `find` includes its own ID in a `_id` field. This mirrors MongoDB's `_id` convention and lets callers identify documents from query results without a separate lookup.

v0 loses all data on restart. v1 fixes that by writing documents to disk in a binary format. Why binary and not JSON? JSON is text. You can read it without a schema. But JSON does not tell you the type of each field: is `30` an integer or a float? Is `"true"` a string or a boolean? Binary formats embed the type alongside the value. That is the core idea of BSON.

### The format

```
Document record:
┌──────────────┬──────────────────┬───────────────┬─────────────────────┐
│ doc_len: u32 │ id_len: u8       │ id: bytes     │ field_count: u16 …  │
└──────────────┴──────────────────┴───────────────┴─────────────────────┘

Each field:
┌─────────────┬───────────────┬──────────────┬──────────────────────────┐
│ key_len: u8 │ key: bytes    │ type_tag: u8 │ value_data               │
└─────────────┴───────────────┴──────────────┴──────────────────────────┘

Type tags:
  0x01 = String  (len: u32 LE, data: utf8 bytes)
  0x02 = Int64   (8 bytes LE)
  0x03 = Float64 (8 bytes LE)
  0x04 = Bool    (1 byte: 0x00=false, 0x01=true)
  0x05 = Null    (0 bytes)
  0x06 = Array   (count: u32 LE, then count values without keys)
  0x07 = Object  (field_count: u16 LE, then fields)
```

### Byte-by-byte walkthrough

Let's encode `{name: "Alice", age: 30}`:

```
Field: name
  key_len:   0x04           (4 bytes for "name")
  key:       6E 61 6D 65    ("name" in UTF-8)
  type_tag:  0x01           (String)
  str_len:   05 00 00 00    (5 bytes for "Alice", little-endian)
  str_data:  41 6C 69 63 65 ("Alice" in UTF-8)

Field: age
  key_len:   0x03           (3 bytes for "age")
  key:       61 67 65       ("age" in UTF-8)
  type_tag:  0x02           (Int64)
  value:     1E 00 00 00 00 00 00 00  (30 as i64 little-endian)
```

The decoder reads the type tag first, then knows exactly how many bytes to consume. A `String` reads 4 bytes for length, then that many bytes for the string. An `Int64` reads exactly 8 bytes. A `Null` reads zero bytes. This is what "self-describing" means: the encoding carries enough information to deserialize without a schema.

**Forward compatibility**: if a newer version adds type tag `0x08 = DateTime`, an older reader that doesn't recognize `0x08` can still skip it — but only if the format includes enough information to determine the byte size of unknown types. Our format does not fully support this, but real BSON handles it by including length prefixes for complex types.

### Offset index

Reading a document by ID should not require scanning the entire file. We maintain an in-memory offset index:

```rust
offsets: HashMap<String, HashMap<DocId, u64>>
//       collection    doc_id  byte offset
```

When we open the store, we scan every collection file and record where each document starts. Reading by ID then requires a single `seek()` to the offset and a single read of `doc_len` bytes.

The trade-off: the offset index lives in memory. For a store with 10 million documents, that is roughly 10 million UUID strings (~36 bytes each) plus their offsets (~8 bytes) = ~440 MB just for the index. Real databases (MongoDB, WiredTiger) use a persistent B-tree for the primary index. Our approach is fine for the lab but would not scale to hundreds of millions of documents.

v1 find is O(n): scan every document, check the filter. For 10k documents with a simple equality filter, that is 10k field comparisons. For 10 million documents, it is 10 million comparisons. Secondary indexes solve this.

### The index structure

```rust
indexes: HashMap<String, HashMap<String, BTreeMap<String, Vec<DocId>>>>
//       collection  field_name  field_value     matching doc IDs
```

A secondary index on `email` looks like:

```
indexes["users"]["email"] = BTreeMap {
    "alice@example.com" -> ["doc-id-1"],
    "bob@example.com"   -> ["doc-id-2", "doc-id-7"],
    "carol@example.com" -> ["doc-id-3"],
    ...
}
```

Finding users where `email == "alice@example.com"` is now:
1. Look up `indexes["users"]["email"]["alice@example.com"]` — O(log N) BTreeMap lookup
2. Fetch `doc-id-1` from disk by its offset — O(1) seek + read

Instead of O(n) for n documents, it's O(log N) for the index lookup plus O(k) for the k matching documents. For a unique field like email, k is almost always 1.

### Write amplification

Every `insert` that updates an indexed field must update the BTreeMap too:

```rust
pub fn insert(&mut self, collection: &str, doc: Document) -> io::Result<DocId> {
    // 1. Write to disk (always)
    let encoded = encode_document(&id, &doc);
    // ... append to file ...

    // 2. Update offset index (always)
    self.offsets[collection].insert(id.clone(), offset);

    // 3. Update every secondary index (only for indexed fields)
    if let Some(col_indexes) = self.indexes.get_mut(collection) {
        for (field, field_index) in col_indexes.iter_mut() {
            if let Some(val) = doc.as_object().and_then(|o| o.get(field)) {
                field_index.entry(index_key(val)).or_default().push(id.clone());
            }
        }
    }
    Ok(id)
}
```

With 1 secondary index, each insert does 1 BTreeMap insert in addition to the disk write. Our benchmarks show ~15% write overhead for one index. With 5 indexes, expect ~40% overhead. With 10 indexes, write throughput roughly halves. MongoDB's documentation warns about this explicitly: "each index adds overhead to write operations."

### Index selectivity

Not all indexes are equally useful. The key metric is **cardinality** — how many distinct values a field takes:

```
Field: active (boolean)    Cardinality: 2
  → index splits collection into ~50/50 halves
  → find({active: true}) still fetches ~half the collection
  → essentially useless

Field: role (5 values)     Cardinality: 5
  → index splits collection into ~20% slices
  → find({role: "admin"}) fetches ~20% of collection
  → marginally useful

Field: email (one per user) Cardinality: N
  → index lookup returns exactly 1 document
  → find({email: "alice@example.com"}) is O(1)
  → perfect index
```

The formula: if a collection has N documents and a field has K distinct values, a BTreeMap lookup returns on average N/K documents. The index is only worthwhile when N/K is small — ideally 1.

MongoDB's query planner tracks index statistics (nReturned, keysExamined) to decide whether to use an index or do a full scan. For a low-cardinality field, the full scan can be faster because it avoids the BTreeMap lookup and per-doc seek overhead.

## Stress tests

**What breaks at 10× load?**

At 10× the write rate, the in-memory BTreeMap for each secondary index starts competing for allocator lock time. Rust's default allocator handles this fine up to a point, but at high concurrency (many tokio tasks all inserting simultaneously), you start to see lock contention on the `Mutex<DocumentStore>`. The fix in production: use a sharded lock (one mutex per collection) or a lock-free concurrent hash map.

**At 100× load?**

The offset index (`HashMap<DocId, u64>`) grows proportionally with document count. At 10 million documents, it occupies roughly 440 MB (36 bytes per UUID string + 8 bytes per offset, plus HashMap overhead). The store starts competing with the OS page cache for memory. Real databases use a persistent B-tree for the primary index precisely to keep memory usage bounded.

**Corruption resilience test:**

```bash
# Insert 100 documents
for i in $(seq 1 100); do
  curl -s -X POST http://localhost:8080/collections/items/docs \
    -H 'Content-Type: application/json' \
    -d "{\"n\": $i}" > /dev/null
done

# Kill the server mid-write
kill -9 $(pgrep docdb-server)

# Corrupt the last few bytes of the collection file
truncate -s -10 /tmp/docdb-data/items.db

# Restart and verify
cargo run --bin docdb-server -- --dir /tmp/docdb-data --port 8080
curl -s -X POST http://localhost:8080/collections/items/find \
  -d '{"filter": {}}' -H 'Content-Type: application/json' | jq length
```

You will get back 99 or 98 documents — the partially written record at the end is discarded by the decoder stopping at the first parse error. This is the same "stop at first corrupt record" behavior as WAL crash recovery.

### Part A — DocumentDbClient: the HTTP bridge

`DocumentDbClient.java` wraps the four REST endpoints using Spring Framework 6.1's `RestClient`. The class is deliberately kept under 60 lines — retry logic and circuit-breaking belong in the `DocumentDbTemplate` or a higher layer, not here.

```java
public class DocumentDbClient {
    private final RestClient http;

    public DocumentDbClient(String baseUrl) {
        this.http = RestClient.builder()
                .baseUrl(baseUrl)
                .defaultHeader("Accept", MediaType.APPLICATION_JSON_VALUE)
                .build();
    }

    public String insert(String collection, Map<String, Object> doc) {
        var response = http.post()
                .uri("/collections/{col}/docs", collection)
                .contentType(MediaType.APPLICATION_JSON)
                .body(doc)
                .retrieve()
                .body(InsertResponse.class);
        if (response == null) throw new DocumentDbException("null response");
        return response.id();
    }

    public List<Map<String, Object>> find(String collection, Map<String, Object> filter) {
        var body = Map.of("filter", filter);
        var result = http.post()
                .uri("/collections/{col}/find", collection)
                .contentType(MediaType.APPLICATION_JSON)
                .body(body)
                .retrieve()
                .body(new ParameterizedTypeReference<List<Map<String, Object>>>() {});
        return result != null ? result : List.of();
    }

    public void createIndex(String collection, String field) {
        http.post()
                .uri("/collections/{col}/indexes", collection)
                .contentType(MediaType.APPLICATION_JSON)
                .body(Map.of("field", field))
                .retrieve()
                .toBodilessEntity();
    }

    record InsertResponse(String id) {}
}
```

### Part B — DocumentDbTemplate: the MongoTemplate analogue

`DocumentDbTemplate` adds POJO serialization on top of `DocumentDbClient` using Jackson's `ObjectMapper`. This is the same design as Spring Data MongoDB's `MongoTemplate`:

```java
@Service
public class DocumentDbTemplate {
    private final DocumentDbClient client;
    private final ObjectMapper objectMapper;

    // Generic insert: POJO → Map → HTTP
    public <T> String insert(String collection, T document) {
        Map<String, Object> map = objectMapper.convertValue(document, Map.class);
        return client.insert(collection, map);
    }

    // Generic find: HTTP → Map → POJO
    public <T> List<T> find(String collection,
                             Map<String, Object> filter,
                             Class<T> type) {
        return client.find(collection, filter).stream()
                .map(map -> objectMapper.convertValue(map, type))
                .toList();
    }
}
```

The `convertValue` call from `Map` to `T` uses Jackson's object mapper pipeline — it respects `@JsonProperty`, `@JsonIgnore`, custom serializers, and all other Jackson annotations. This is exactly how Spring Data MongoDB converts BSON documents to POJOs.

**Why this mirrors MongoTemplate**: MongoDB's `MongoTemplate` has `insert(String collectionName, Object objectToSave)` and `find(Query query, Class<T> entityClass, String collectionName)`. Our `DocumentDbTemplate` has the same shape. The difference is that `MongoTemplate` uses a MongoDB `MongoClient` internally; ours uses `RestClient` to talk to the Rust server. This deliberate mirroring means that when you eventually wire a real MongoDB deployment, the calling code in your services is unchanged — only the Spring bean configuration changes.

## Where to go deeper

- **BSON specification** — the full type system, including Date, Decimal128, Regex, and the deprecated Undefined: [bsonspec.org](https://bsonspec.org/)
- **MongoDB WiredTiger integration** — how MongoDB maps BSON documents to WiredTiger's key-value storage engine, including the document-level locking and snapshot isolation: [source.wiredtiger.com](https://source.wiredtiger.com/)
- **MongoDB index internals** — compound indexes, covered queries (index-only scans), and the query planner's selectivity estimation: [mongodb.com/docs/manual/indexes/](https://www.mongodb.com/docs/manual/indexes/)
- **CouchDB's B-tree storage** — CouchDB uses an append-only B-tree (no WAL) where every write creates a new B-tree root, enabling MVCC without locking: [guide.couchdb.org/draft/btree.html](https://guide.couchdb.org/draft/btree.html)
- **Firestore's design** — Google Cloud Firestore is a document database built on Bigtable (wide-column store); the translation layer between JSON documents and Bigtable rows is instructive: [cloud.google.com/firestore/docs/data-model](https://cloud.google.com/firestore/docs/data-model)

## Related labs and questions

- **Previous in storage**: WAL (lab 5) — our v1 disk store needs a WAL to be crash-safe; right now a partial write at shutdown loses the last document
- **Previous in storage**: LSM-Tree KV Store (lab 1) — MongoDB's WiredTiger uses a B-tree, not an LSM tree; CockroachDB's Pebble uses an LSM tree; the tradeoff is write amplification vs. read amplification
- **System design companion**: When would you choose a document database over a relational one? Answer: when your data is naturally hierarchical (orders with line items), when your schema evolves frequently, or when your access pattern is always "fetch the whole entity" rather than arbitrary joins. When you need multi-collection ACID transactions, a relational database is still the better choice.
- **Next in distributed**: Raft From Scratch — how would you make this document database fault-tolerant? Each Raft log entry would be an insert or createIndex operation; the replicated state machine would replay them on each node.
